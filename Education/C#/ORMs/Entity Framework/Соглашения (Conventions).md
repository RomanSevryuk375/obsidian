**Соглашения (Conventions)** в [[Entity Framework]] Core — это набор правил «по умолчанию». Благодаря им EF Core может «угадать», как должна выглядеть база данных, просто посмотрев на твои C#.

Философия называется **"Convention over Configuration"** (Соглашение важнее конфигурации). Это значит: если ты следуешь стандартным правилам именования, тебе не нужно писать лишний код для настройки.

Вот самые важные соглашения, которые работают «из коробки».

---

### 1. Соглашение об именовании таблиц

EF Core смотрит на название свойства `DbSet<T>` в твоем `DbContext`.

**Код C#:**
```csharp
public class AppDbContext : DbContext
{
    public DbSet<Product> Products { get; set; } // назовет таблицу "Products"
}
```

*   **Результат в [[БД]]:** Таблица будет называться `Products`.

---

### 2. Первичный ключ ([[Primary Key]])

**Правило:** Свойство должно называться `Id` или `[ИмяКласса]Id` (регистр не важен).

```csharp
public class User
{
    public int Id { get; set; } // Станет Primary Key (PK)
    // ИЛИ
    // public int UserId { get; set; } // Тоже станет PK
}
```

*   **Результат в БД:** Создается колонка `Id` с ограничением `PRIMARY KEY`.
*   **Автоинкремент:** Если тип данных числовой (`int`, `long`) или `Guid`, EF Core по умолчанию настраивает автогенерацию значений (Identity в [[SQL Server]]).

---

### 3. Внешние ключи ([[Foreign Keys]]) и Связи

**Правило:** EF Core ищет свойство с именем по шаблону:
1.  `[ИмяНавигационногоСвойства]Id` (например, `BlogId`)
2.  `[ИмяКласса]Id` (например, `BlogId`)
3.  `[ИмяНавигационногоСвойства][ИмяPKЦелевогоКласса]` (например, `BlogId` или `BlogTargetId`)

```csharp
public class Post
{
    public int Id { get; set; }

    public Blog Blog { get; set; }  // Навигационное свойство
    
    public int BlogId { get; set; } // Внешний ключ (Foreign Key)
}
```

**Если свойство найдено:** EF использует его как внешний ключ.
**Если свойство НЕ найдено:** EF создаст **"Теневое свойство" ([[Shadow Property]])**. В классе C# его не будет, но в таблице базы данных появится скрытая колонка `BlogId`.

*   **Результат в БД:** В таблице `Post` создается колонка `BlogId` с `FOREIGN KEY`, ссылающимся на `Blog`. Также автоматически создается **Индекс** для этой колонки (для скорости JOIN'ов).

### 4. Типы данных и Nullable

| Тип C#     | Тип SQL (SQL Server) | Примечание                               |
| :--------- | :------------------- | :--------------------------------------- |
| `int`      | `int`                |                                          |
| `string`   | `nvarchar(max)`      | По умолчанию строка максимальной длины   |
| `decimal`  | `decimal(18,2)`      | Часто нужно настраивать точность вручную |
| `bool`     | `bit`                | 0 или 1                                  |
| `DateTime` | `datetime2`          |                                          |
| `byte[]`   | `varbinary(max)`     |                                          |

**Nullable (Можно ли хранить NULL):**

*   **Значимые типы (`int`, `bool`, `double`):**
    *   `int Age` -> `NOT NULL` (Обязательное поле)
    *   `int? Age` -> `NULL` (Может быть пустым)

*   **Ссылочные типы (`string`):**
    *   Если в проекте включен `<Nullable>enable</Nullable>` (стандарт в .NET 6+):
        *   `string Name` -> `NOT NULL`
        *   `string? Name` -> `NULL`
    *   Если Nullable отключен: `string` по умолчанию допускает `NULL`.

---

### 5. Примеры того, что НЕ попадает в соглашения

Не все свойства автоматически попадают в базу.

*   `static` свойства — игнорируются.
*   `readonly` поля — игнорируются.
*   Свойства без `get` или `set` (или приватные сеттеры) — могут требовать дополнительной настройки, но обычно игнорируются или вызывают ошибки, если EF не может к ним доступиться.

---

### Что делать, если соглашения не подходят?

Иногда тебе нужно нарушить правила. Например, назвать таблицу `tbl_Users`, а не `Users`, или ограничить строку 50 символами.

Для этого есть два пути:
1.  **[[Data Annotations]]:** Атрибуты прямо над свойствами.
2.  **[[Fluent API]]:** Настройка в методе `OnModelCreating` (самый мощный способ).

**Итог:** Соглашения экономят время. Ты просто пишешь классы, соблюдая простые правила именования (`Id`, `EntityId`), а EF Core делает всю грязную работу по созданию схемы БД.
