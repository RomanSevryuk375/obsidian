# Микросервисная архитектура (MSA)

### Суть (The "What")
**Микросервисная архитектура** — это архитектурный стиль, при котором приложение строится как набор слабосвязанных, автономных сервисов. Каждый сервис отвечает за узкую бизнес-задачу (Bounded Context), имеет собственную базу данных и развертывается независимо.

MSA решает задачу **масштабируемости организации и системы**: она позволяет сотням разработчиков работать над одним продуктом, не блокируя друг друга, и масштабировать только те компоненты, которые реально нагружены (например, сервис корзины в черную пятницу).

---

### Как это работает (The "How" / Nutshell style)

#### 1. Физическая и логическая изоляция
В отличие от монолита, где компоненты общаются через вызовы методов в рамках одного домена приложения (`AppDomain`), микросервисы разделены границей процесса и сети.
*   **CLR Isolation:** Каждый сервис — это отдельный экземпляр рантайма .NET. Падение или утечка памяти в одном сервисе физически не может "уронить" соседний.
*   **Data Sovereignty:** Принцип "Database per Service" исключает использование JOIN между таблицами разных сервисов. Данные согласуются через события (Eventual Consistency).

#### 2. Межсервисные коммуникации
*   **Синхронные ([[gRPC]]/[[HTTP]]):** Прямой вызов. В .NET gRPC предпочтительнее для внутренних связей, так как использует бинарный протокол Protobuf и HTTP/2, что снижает нагрузку на CPU и сетевой трафик.
*   **Асинхронные (Message Broker):** Использование RabbitMQ, Kafka или Azure Service Bus. Сервисы общаются через "интеграционные события", что гарантирует отказоустойчивость: если получатель временно недоступен, сообщение подождет в очереди.

#### 3. Инфраструктурная обвязка
Для работы MSA в .NET требуются дополнительные компоненты:
*   **API Gateway:** Единая точка входа (YARP, Ocelot).
*   **Service Discovery:** Механизм поиска динамических IP-адресов сервисов.
*   **Sidecar Pattern:** Вынос сквозной логики (логирование, метрики) в отдельные контейнеры.

### Структура проекта (Standard Microservice Solution)

Типичный современный репозиторий микросервиса на .NET 8/9 следует принципам **Clean Architecture** (Чистая архитектура):

```text
MyMicroservice.sln
│
├── src/
│   ├── Services/
│   │   ├── Ordering.API/                # Входная точка 
│   │   │   ├── Controllers/
│   │   │   ├── Extensions/              # Конфигурация серви
│   │   │   ├── Program.cs               # Настройка конвейера (Middleware)
│   │   │   └── appsettings.json
│   │   │
│   │   ├── Ordering.Application/        # Бизнес-логика 
│   │   │   ├── Behaviours/              # Валидация, Логирование 
│   │   │   ├── Commands/                # Логика изменения данных
│   │   │   ├── Queries/                 # Логика чтения данных
│   │   │   └── IntegrationEvents/       # Обработчики событий из др сервисов
│   │   │
│   │   ├── Ordering.Domain/             # Ядро 
│   │   │   ├── Exceptions/              # Специфичные ошибки домена
│   │   │   ├── Models/
│   │   │   └── RepositoryInterfaces/    # Контракты для доступа к данным
│   │   │
│   │   └── Ordering.Infrastructure/     # Детали реализации 
│   │       ├── Data/                    # Entity Framework Core конфигурации
│   │       ├── Logging/
│   │       └── MessageBus/               # Реализация клиента RabbitMQ/Kafka
│   │
├── tests/
│   ├── Ordering.UnitTests/
│   ├── Ordering.IntegrationTests/
│   └── Ordering.FunctionalTests/
│
├── docker-compose.yml                   # Оркестрация локальной инфраструктуры
└── k8s/                                 # Манифесты для развертывания в Kubernetes
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Распределенный монолит
> Самая опасная ошибка — создание сервисов, которые не могут работать друг без друга. Если при падении Сервиса А Сервис Б сразу выдает 500 ошибку, у вас не микросервисы, а "монолит на минималках" с накладными расходами сети. Используйте **Retry** и **Circuit Breaker**.

> [!WARNING] Ошибка: Общий код (Shared Libraries)
> Не выносите бизнес-логику в общие NuGet-пакеты. Это создает жесткую связь между сервисами. Делить можно только инфраструктурные вещи (логирование, обертки над шиной данных). Копирование DTO между сервисами часто дешевле, чем управление зависимостями общей библиотеки.

**Best Practices:**
1.  **Idempotency:** Любой обработчик сообщений должен быть идемпотентным (повторная обработка того же сообщения не должна менять состояние системы).
2.  **Health Checks:** Используйте `Microsoft.Extensions.Diagnostics.HealthChecks`. Оркестратор должен знать, жив ли процесс сервиса и готова ли его БД к работе.
3.  **Observability:** Используйте **OpenTelemetry** для сквозного проброса `CorrelationId`. Без этого отладка запроса, прошедшего через 5 сервисов, превращается в ад.
4.  **Database isolation:** Никогда не позволяйте Сервису А подключаться к базе данных Сервиса Б напрямую. Только через API.

> [!TIP] Нюанс реализации: Outbox Pattern
> Для обеспечения надежности передачи событий из .NET сервиса используйте **Transactional Outbox Pattern**. Сначала событие сохраняется в локальную БД сервиса в той же транзакции, что и бизнес-данные, а отдельный фоновый воркер (BackgroundService) перекладывает его в брокер сообщений. Это гарантирует доставку "At-least-once".
