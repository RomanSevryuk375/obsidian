### Суть (The "What")
**Посредник (Mediator)** — это поведенческий шаблон проектирования, который обеспечивает слабую связанность между объектами, заставляя их взаимодействовать друг с другом не напрямую, а через специальный объект-посредник.

Шаблон решает задачу **хаотичной зависимости между компонентами** (проблема "спагетти-связей"). Вместо того чтобы объекты хранили ссылки друг на друга и общались по принципу "каждый с каждым" (сложность связей $O(N^2)$), они общаются только с Посредником (сложность связей $O(N)$), который перенаправляет запросы нужным адресатам. Это заменяет топологию связей типа «паутина» на топологию «звезда».

### Как это работает (The "How" / Nutshell style)

#### 1. Участники паттерна
*   **Коллеги (Colleagues):** Классы, логику которых необходимо координировать. Они изолированы друг от друга, не знают о существовании других коллег и хранят ссылку только на интерфейс Посредника.
*   **Посредник (Mediator):** Центральный узел, который знает обо всех коллегах и координирует их действия. Когда один из коллег сообщает о событии, Посредник принимает решение, кому переслать это уведомление или какое действие выполнить в ответ.

#### 2. Механизм передачи сообщений
*   Коллега отправляет сообщение или сигнал Посреднику: `_mediator.Notify(this, "Event")`.
*   Посредник анализирует отправителя и тип события, а затем вызывает соответствующие методы у других зарегистрированных коллег.
*   На низком уровне это избавляет от необходимости внедрять десятки интерфейсов в конструкторы классов, заменяя их одной зависимостью от Посредника.

> [!INFO] Нюанс: Отличие от Брокера сообщений (Message Broker)
> Паттерн Посредник обычно работает внутрипроцессно (In-Memory) на уровне объектов приложения (например, координация элементов интерфейса или вызов обработчиков команд). Брокер сообщений ([[RabbitMQ]], Kafka) — это инфраструктурный компонент для межпроцессного взаимодействия по сети.

### Структура папок и пример реализации ([[C#]] 12)

Реализация авиадиспетчерской службы (Посредник), координирующей посадку самолетов (Коллеги) без их прямого взаимодействия:

```text
MediatorPattern/
├── Abstract/
│   ├── IControlTower.cs          # Интерфейс Посредника
│   └── Aircraft.cs               # Базовый класс Коллеги
├── Concrete/
│   ├── AirportControlTower.cs    # Реализация Посредника
│   └── Plane.cs                  # Реализация конкретного Коллеги
└── Program.cs
```

#### 1. Интерфейсы и базовые классы (Abstract)
```csharp
namespace MediatorPattern.Abstract;

// Интерфейс Посредника
public interface IControlTower
{
    void RegisterAircraft(Aircraft aircraft);
    bool RequestLanding(Aircraft aircraft);
}

// Абстрактный Коллега с первичным конструктором C# 12
public abstract class Aircraft(string callSign, IControlTower controlTower)
{
    public string CallSign { get; } = callSign;
    protected readonly IControlTower ControlTower = controlTower;

    public abstract void Land();
}
```

#### 2. Реализация компонентов (Concrete)
```csharp
using System;
using System.Collections.Generic;
using MediatorPattern.Abstract;

namespace MediatorPattern.Concrete;

// Конкретный Посредник
public class AirportControlTower : IControlTower
{
    private readonly List<Aircraft> _aircrafts = [];
    private bool _isRunwayFree = true; // Состояние, которым управляет Посредник

    public void RegisterAircraft(Aircraft aircraft) => _aircrafts.Add(aircraft);

    public bool RequestLanding(Aircraft aircraft)
    {
        if (_isRunwayFree)
        {
            _isRunwayFree = false; // Полоса занята
            return true;
        }
        return false;
    }

    public void ReleaseRunway() => _isRunwayFree = true;
}

// Конкретный Коллега
public class Plane(string callSign, IControlTower controlTower) : Aircraft(callSign, controlTower)
{
    public override void Land()
    {
        Console.WriteLine($"[Plane {CallSign}]: Запрашиваю посадку...");
        
        if (ControlTower.RequestLanding(this))
        {
            Console.WriteLine($"[Plane {CallSign}]: Посадка успешна.");
        }
        else
        {
            Console.WriteLine($"[Plane {CallSign}]: Полоса занята, ухожу на второй круг.");
        }
    }
}
```

#### 3. Использование в Program.cs
```csharp
using MediatorPattern.Abstract;
using MediatorPattern.Concrete;

var tower = new AirportControlTower();

Aircraft boeing = new Plane("Boeing-747", tower);
Aircraft airbus = new Plane("Airbus-A320", tower);

tower.RegisterAircraft(boeing);
tower.RegisterAircraft(airbus);

boeing.Land(); // Займет полосу
airbus.Land(); // Будет вынужден уйти на второй круг

tower.ReleaseRunway(); // Освобождаем через диспетчера
airbus.Land(); // Теперь посадка успешна
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Божественный Посредник (God Object)
> Самая частая ошибка при реализации паттерна — перенос всей бизнес-логики координируемых классов внутрь самого Посредника. Посредник превращается в гигантский, монолитный класс, который знает абсолютно обо всем, нарушая принцип единственной ответственности ([[Single Responsibility Principle|SRP]]). Посредник должен заниматься исключительно *маршрутизацией и координацией*, но не логикой работы самих коллег.

> [!WARNING] Ошибка: Скрытые зависимости (Hidden Dependencies)
> Использование Посредника делает зависимости между классами неявными. Глядя на конструктор класса, принимающий только `IMediator`, вы не можете сразу сказать, от каких именно других компонентов зависит его работа. Это усложняет чтение кода и статический анализ архитектуры.

**Best Practices:**
1.  **Иммутабельность сообщений:** Передавайте между коллегами и посредником неизменяемые объекты данных (`readonly record` или `struct`), чтобы исключить случайное изменение данных одним из коллег в процессе транзита.
2.  **Слабая связь:** Всегда работайте через интерфейсы Посредника (`IMediator` / `IControlTower`). Это позволяет легко тестировать коллег изолированно, подменяя посредника на мок-объект.
3.  **Использование готовых библиотек:** В реальных .NET проектах для реализации этого паттерна на уровне архитектуры приложения ([[CQRS]]) де-факто стандартом является библиотека `MediatR`. Писать кастомные медиаторы стоит только для специфических локальных задач (например, координация сложных элементов UI).
4.  **Разделение интерфейсов:** Если у вас очень много разных типов сообщений, разделите один большой интерфейс посредника на несколько специализированных контрактов.

> [!TIP] Нюанс: События вместо Посредника
> Если координация требуется только в одном направлении (например, объект А должен просто узнать, что в объекте Б что-то произошло), использование событий C# (`event`) или реактивных расширений (Rx.NET) часто оказывается более лаконичным решением, чем полноценный паттерн Посредник.
