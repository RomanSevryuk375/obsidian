**Проблема N+1** — это самая распространенная и коварная проблема производительности при работе с [[ORM]] (включая [[Entity Framework|EF Core]]). Она возникает, когда код делает **один** запрос к базе данных для получения списка объектов, а затем делает еще **N** дополнительных запросов для получения связанных данных для каждого объекта из этого списка.

В итоге вместо 1 эффективного запроса база данных получает **1001 запрос** (если в списке 1000 элементов).

---
### 1. Анатомия проблемы (На примере Счетов)

Допустим, у нас есть таблица счетов (`Bills`), и мы хотим вывести ID счета и имя его статуса (`BillStatus`).

#### Плохой код (Как возникает N+1)

```csharp
// 1. ПЕРВЫЙ ЗАПРОС (1)
// Получаем список счетов. Допустим, их вернулось 100 штук.
var bills = context.Bills.ToList(); 

foreach (var bill in bills)
{
    // 2. ЦИКЛ ЗАПРОСОВ (N)
    // Если включен Lazy Loading, то при обращении к bill.Status 
    // EF Core молча идет в базу данных за именем статуса.
    // Это произойдет 100 раз.
    Console.WriteLine($"Счет {bill.Id}: {bill.Status.Name}");
}
```

**Что происходит в SQL:**

1.  `SELECT * FROM bills` (1 раз)
2.  `SELECT * FROM bill_statuses WHERE Id = 10` (для 1-го счета)
3.  `SELECT * FROM bill_statuses WHERE Id = 12` (для 2-го счета)
4.  ...
5.  `SELECT * FROM bill_statuses WHERE Id = 55` (для 100-го счета)

**Почему это плохо:**
Даже если каждый маленький запрос выполняется 1 мс, накладные расходы на сетевое общение (Network Latency) убьют скорость. Если пинг до базы 5 мс, то 100 запросов займут **500 мс** (полсекунды!) просто на ожидание сети.

---

### 2. Как избежать (Решения)

Есть два основных способа решения: **[[Eager Loading]]** (Жадная загрузка) и **[[Projections]]** (Проекции).

#### ✅ Решение 1: Eager Loading (.Include)
Мы говорим [[EF Core]]: «Когда будешь грузить счета, сразу подтяни их статусы через [[JOIN]]».

```csharp
var bills = context.Bills
    .Include(b => b.Status) // <--- РЕШЕНИЕ
    .ToList();

foreach (var bill in bills)
{
    // Данные уже в памяти, запросов к БД нет
    Console.WriteLine($"Счет {bill.Id}: {bill.Status.Name}");
}
```

**Что происходит в [[SQL]]:**
Один эффективный запрос:
```sql
SELECT b.*, s.* 
FROM bills AS b
LEFT JOIN bill_statuses AS s ON b.StatusId = s.Id
```

#### ✅ Решение 2: Проекции (.Select) — Самый быстрый способ
Если тебе не нужны полные объекты сущностей, а только конкретные поля (например, для отправки [[DTO]] на фронтенд), используй `.Select`. [[EF Core]] сам поймет, что нужно сделать JOIN.

```csharp
var billDtos = context.Bills
    .Select(b => new 
    {
        Id = b.Id,
        StatusName = b.Status.Name // EF Core видит обращение к Status и делает JOIN сам
    })
    .ToList();
```
**Плюс:** В отличие от `Include`, это вытягивает из базы не все колонки (`SELECT *`), а только две нужные (`Id` и `Name`). Это экономит трафик.

### 3. Сложный случай: Коллекции (Списки внутри списков)

Допустим, нужно вывести счета и список платежей по каждому.

**❌ Плохо (N+1):**
```csharp
var bills = context.Bills.ToList();
foreach(var bill in bills)
{
    // Запрос за платежами для каждого счета
    var count = bill.Payments.Count(); 
}
```

**✅ Хорошо (Include):**
```csharp
var bills = context.Bills
    .Include(b => b.Payments)
    .ToList();
```

**⚠️ Нюанс с производительностью:**
Если у каждого счета 100 платежей, то один огромный [[JOIN]] может вернуть слишком много дублирующихся данных (проблема Cartesian Explosion).
В этом случае (EF Core 5+) используй **[[AsSplitQuery()]]**:

```csharp
var bills = context.Bills
    .Include(b => b.Payments)
    .AsSplitQuery() // <--- Разбить на 2 оптимизированных запроса
    .ToList();
```
Это сделает 2 запроса (один за счетами, второй за всеми платежами сразу) вместо N+1. Это всё равно намного быстрее, чем цикл.

### 4. Как обнаружить N+1?

Часто мы не замечаем эту проблему на локальной машине, где база данных летает, а данных мало (2-3 записи). Проблемы начинаются на продакшене.

#### Способ А: Логирование
Включи логирование [[SQL]]-запросов в консоль (в `appsettings.json` или `Program.cs`). Если видишь в консоли поток одинаковых запросов типа `SELECT ... WHERE Id = ?`, бегущих подряд — это оно.

#### Способ Б: Профайлеры
Используй SQL Server Profiler (для MSSQL) или специальные инструменты мониторинга (Application Insights, MiniProfiler), которые подсвечивают "Duplicate Queries".

### Резюме

1.  **N+1** — это когда ты в цикле `foreach` обращаешься к навигационному свойству (`bill.Status`), которое не было загружено заранее.
2.  **Лекарство №1:** Использовать `.Include(b => b.Status)`.
3.  **Лекарство №2:** Использовать `.Select(b => new { ... })` (проекции в [[DTO]]).
4.  **Внимательность:** Если видишь цикл `foreach` по сущностям [[БД]] — у тебя в голове должна загораться красная лампочка: "А всё ли я загрузил до входа в этот цикл?".
