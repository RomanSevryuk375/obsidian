### Суть (The "What")
**Inversion of Control (IoC)** — это фундаментальный архитектурный принцип, при котором контроль над потоком выполнения программы передается от вашего кода внешнему каркасу (фреймворку). В традиционном программировании ваш код сам решает, когда создавать объекты, как управлять их жизненным циклом и какие библиотеки вызывать. При IoC фреймворк берет управление на себя и вызывает ваш код в нужные моменты.

Этот принцип широко известен как **«Голливудский принцип»**: *«Не звоните нам, мы сами вам позвоним»* («Don't call us, we'll call you»). Он решает задачу жесткой связанности компонентов и превращает ваше приложение из монолитного сценария в набор расширяемых модулей (плагинов) для фреймворка.

### Как это работает (The "How" / Nutshell style)

#### 1. Смена потока управления
*   **Традиционный подход:** Метод `Main` запускает программу, сам создает экземпляр класса `Database`, делает запрос, форматирует результат и выводит на экран. Весь поток выполнения жестко закодирован.
*   **Подход IoC:** Фреймворк (например, ASP.NET Core) запускает хост, слушает сетевой порт, настраивает конвейер обработки, а затем сам создает экземпляр вашего контроллера и вызывает нужный метод действия, когда на сервер приходит [[HTTP]]-запрос. Ваш код пассивен и ждет вызова.

#### 2. Способы реализации IoC
Инверсия управления — это широкое архитектурное понятие, которое реализуется через различные паттерны:
*   **Паттерн "Шаблонный метод" (Template Method):** Базовый класс фреймворка определяет алгоритм, но оставляет абстрактные "заглушки". Вы переопределяете их в наследнике, а фреймворк вызывает их в нужном порядке.
*   **Событийно-ориентированная модель (Event-driven):** Вы регистрируете обработчик событий (делегат), а операционная система или среда выполнения сама вызывает его при возникновении события (например, клик мышкой или получение данных из сокета).
*   **Внедрение зависимостей ([[Dependency Injection]]):** Вы регистрируете свои классы в контейнере, а он сам решает, когда их создавать, как внедрять друг в друга и когда уничтожать.

#### 3. Различия между IoC, [[Dependency Inversion Principle|DIP]] и [[Dependency Injection|DI]]
Эти три термина часто путают на собеседованиях. Их иерархия выглядит следующим образом:
1.  **IoC (Inversion of Control):** Общая архитектурная философия (высокоуровневый принцип смены потока управления).
2.  **DIP (Dependency Inversion Principle):** Принцип проектирования классов (буква "D" в [[SOLID]]), утверждающий, что модули не должны зависеть от конкретных реализаций, а должны зависеть от абстракций.
3.  **DI (Dependency Injection):** Конкретный паттерн проектирования, который реализует DIP и управляется IoC-контейнером.

### Структура папок и сравнение подходов

Для демонстрации разницы между прямым управлением (Legacy) и инвертированным (IoC):

```text
MyProject.Demo/
├── LegacyStyle/                 # Прямое управление
│   ├── ConsoleWriter.cs
│   └── ReportManager.cs
└── IoCStyle/                    # Инверсия управления
    ├── IOutputWriter.cs         # Контракт
    ├── ConsoleOutputWriter.cs   # Реализация контракта
    └── ReportManagerIoC.cs      # Класс, не управляющий созданием зависимостей
```

#### 1. Классический стиль (Высокая связанность)
Класс сам контролирует создание зависимости. Мы не можем заменить консоль на файл без изменения этого класса.

```csharp
namespace MyProject.Demo.LegacyStyle;

public class ReportManager
{
    private readonly ConsoleWriter _writer;

    public ReportManager()
    {
        // Жесткая связь: класс САМ решает, как создавать зависимость
        _writer = new ConsoleWriter(); 
    }

    public void GenerateReport()
    {
        _writer.Write("Report generated.");
    }
}

public class ConsoleWriter { public void Write(string m) => Console.WriteLine(m); }
```

#### 2. Стиль IoC / DIP (Слабая связанность)
Класс делегирует создание зависимости внешнему исполнителю (например, DI-контейнеру).

```csharp
namespace MyProject.Demo.IoCStyle;

public interface IOutputWriter { void Write(string message); }

public class ConsoleOutputWriter : IOutputWriter 
{ 
    public void Write(string message) => Console.WriteLine(message); 
}

// Использование первичного конструктора C# 12
// Класс не знает, кто и как создаст IOutputWriter. Управление инвертировано.
public class ReportManagerIoC(IOutputWriter writer)
{
    private readonly IOutputWriter _writer = writer;

    public void GenerateReport()
    {
        _writer.Write("Report generated via IoC.");
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Борьба с фреймворком
> Попытка вручную управлять процессами, которые уже инвертированы фреймворком. Например, самостоятельное создание экземпляров контроллеров, ручное управление потоками в ASP.NET через `new Thread()` вместо использования пула или TPL, или ручное чтение сырых сокетов там, где фреймворк предоставляет готовые конвейеры.

> [!WARNING] Ошибка: Чрезмерная связанность через статические методы
> Использование глобальных статических вызовов (вроде `DateTime.Now` или обращений к глобальным синглтонам) неявно возвращает поток управления внутрь вашего класса и мешает инверсии управления. Оборачивайте такие вызовы в абстракции (например, `IDateTimeProvider`).

**Best Practices:**
1.  **Позвольте фреймворку делать его работу:** Не пытайтесь продублировать функции маршрутизации, авторизации или сериализации, которые уже встроены в конвейеры .NET.
2.  **Зависимость от абстракций:** Проектируйте связи между вашими классами по принципу DIP — это обеспечит легкую интеграцию ваших модулей в IoC-контейнер.
3.  **Событийно-ориентированный дизайн:** Используйте делегаты, события и реактивное программирование внутри домена приложения, чтобы инвертировать зависимости между вашими внутренними модулями.

> [!TIP] Архитектурный вывод
> Применение IoC превращает ваше приложение из монолитного исполняемого файла в набор подключаемых модулей (плагинов). Фреймворк становится главной платформой, а ваш код — лишь набором инструкций, которые платформа вызывает по мере необходимости.
