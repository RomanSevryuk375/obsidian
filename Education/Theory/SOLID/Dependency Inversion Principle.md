### Суть (The "What")
**Dependency Inversion Principle (DIP)** — это пятый и заключительный принцип [[SOLID]], который гласит:
1. Модули верхнего уровня не должны зависеть от модулей нижнего уровня. Оба должны зависеть от абстракций.
2. Абстракции не должны зависеть от деталей. Детали должны зависеть от абстракций.

DIP решает задачу **жесткой связности (tight coupling)**. Без этого принципа изменение в низкоуровневом компоненте (например, замена SQL на MongoDB) вынуждает переписывать высокоуровневую бизнес-логику. DIP "разворачивает" стрелку зависимости: теперь не логика подстраивается под инструменты, а инструменты реализуют [[Интерфейсы|интерфейсы]], необходимые логике.

### Как это работает (The "How" / Nutshell style)

#### 1. Инверсия владения (Inversion of Control)
В традиционном подходе класс `Service` сам создает экземпляр `Database`. В DIP класс `Service` лишь объявляет: "Мне нужен кто-то, кто умеет выполнять `Save()`". 
*   **[[Метаданные]]:** На уровне сборки это означает, что высокоуровневый проект больше не имеет ссылки (Reference) на проект с реализацией БД. Вместо этого оба проекта ссылаются на общую сборку с интерфейсами.

#### 2. Механика вызова через [[VMT]]
Когда высокоуровневый модуль вызывает метод интерфейса:
*   Процессор не знает заранее адрес исполняемого кода.
*   [[CLR]] обращается к **[[MethodTable]]** текущего объекта в куче.
*   Через **Interface Dispatch Table** находится адрес конкретной реализации (например, `SqlSave`).
*   Происходит переход к коду. Эта индирекция — цена, которую мы платим за гибкость, но благодаря [[JIT]]-оптимизациям она минимальна.

#### 3. Внедрение зависимостей (Dependency Injection)
DIP — это концепция, а DI — это основной инструмент её реализации. В .NET управление зависимостями обычно берет на себя встроенный **IoC-контейнер** (`IServiceProvider`), который сам создает объекты и "пробрасывает" их в [[Конструкторы|конструкторы]].

> [!INFO] Нюанс: Жизненный цикл (Lifetimes)
> При использовании DIP через DI-контейнер критически важно управлять временем жизни объектов: `Transient` (каждый раз новый), `Scoped` (один на запрос) или `Singleton` (один на всё время). Ошибка в выборе lifetime может привести к утечкам памяти или повреждению данных в многопоточной среде.

### Пример кода ([[MOC#/MOC|C#]] 12)

```csharp
using System;

// 1. Абстракция (Интерфейс) — принадлежит уровню логики
public interface IMessageSender
{
    void Send(string message);
}

// 2. Низкоуровневая деталь 1
public class EmailSender : IMessageSender
{
    public void Send(string message) => Console.WriteLine($"Email sent: {message}");
}

// 3. Низкоуровневая деталь 2
public class SmsSender : IMessageSender
{
    public void Send(string message) => Console.WriteLine($"SMS sent: {message}");
}

// 4. Модуль верхнего уровня (Бизнес-логика)
// Использует Primary Constructor (C# 12) для внедрения зависимости
public class NotificationService(IMessageSender sender)
{
    private readonly IMessageSender _sender = sender;

    public void Notify(string text)
    {
        // Логика не знает, КТО именно отправит сообщение
        _sender.Send(text);
    }
}

public class Program
{
    public static void Main()
    {
        // В реальном приложении это делает DI-контейнер
        IMessageSender emailProvider = new EmailSender();
        var service = new NotificationService(emailProvider);
        
        service.Notify("Hello SOLID!");
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Service Locator
> Использование контейнера внутри класса (например, `container.Resolve<IService>()`) — это антипаттерн Service Locator. Он скрывает зависимости класса, делает его нетестируемым и нарушает DIP, так как класс начинает зависеть от самого контейнера. Зависимости должны передаваться **только** через конструктор.

> [!WARNING] Ошибка: Зависимость от реализации
> Если метод вашего интерфейса возвращает `List<T>` (конкретный класс), а не `IEnumerable<T>` или `IReadOnlyList<T>`, вы частично нарушаете DIP, так как абстракция начинает зависеть от деталей реализации коллекции.

**Best Practices:**
1.  **Constructor Injection:** Это основной и самый прозрачный способ реализации DIP. Используйте первичные конструкторы C# 12 для лаконичности.
2.  **Проектируйте от [[Интерфейсы|интерфейсов]]:** Сначала опишите, как компоненты будут общаться (контракт), и только потом пишите реализацию.
3.  **Composition Root:** Настраивайте все зависимости в одной точке входа в приложение (в ASP.NET Core это `Program.cs`).
4.  **Абстрагируйте внешние системы:** Файловая система, системное время (DateTime.Now), сетевые вызовы — всё это должно быть скрыто за интерфейсами, чтобы код можно было тестировать в изоляции.

> [!TIP] Нюанс: Пакетная организация
> Согласно "Nutshell", истинная инверсия зависимостей достигается тогда, когда интерфейсы лежат в той же сборке, что и высокоуровневая логика, а не в сборке с реализациями. Это гарантирует, что низкоуровневые плагины зависят от ядра, а не наоборот.
