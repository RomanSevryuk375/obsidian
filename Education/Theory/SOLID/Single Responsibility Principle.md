### Суть (The "What")
**Single Responsibility Principle (SRP)** — это первый принцип [[SOLID]], который гласит: «У класса должна быть только одна причина для изменения». Это означает, что каждый класс или модуль должен отвечать за одну конкретную часть функциональности системы.

Принцип решает проблему **сильной связанности (tight coupling)** и **низкой сплоченности (low cohesion)**. Без SRP возникают «божественные объекты» (God Objects) — огромные классы, которые делают слишком много. Изменение в одной части такого класса может непредсказуемо сломать другую его часть.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Высокая сплоченность (High Cohesion)
С точки зрения архитектуры, SRP стремится к максимизации сплоченности. Это означает, что все методы и поля внутри класса должны быть логически тесно связаны друг с другом для решения общей задачи. Если половина методов класса использует одну группу полей, а вторая половина — другую, это явный признак нарушения SRP и сигнал к разделению.

#### 2. Причины для изменения (Reasons for Change)
Ответственность в контексте SRP эквивалентна «причине для изменения». 
*   Если класс сохраняет данные в БД и генерирует отчет в PDF, у него две причины для изменения: смена провайдера базы данных и смена библиотеки для генерации PDF.
*   Разделение ответственности упрощает **Unit-тестирование**, так как каждый класс становится изолированным и предсказуемым.

#### 3. Влияние на компиляцию и [[JIT]]
На низком уровне использование множества мелких [[Классы и объекты|классов]] вместо одного огромного имеет свои нюансы:
*   **Типы в метаданных:** Каждый новый класс создает запись в таблице типов ([[MethodTable]]) в Loader Heap.
*   **Инлайнинг (Inlining):** Небольшие методы в узкоспециализированных классах легче поддаются оптимизации JIT-компилятором (инлайнингу), что повышает производительность.
*   **Локальность кэша:** Работа с компактными объектами в куче может быть более эффективной для кэша процессора (L1/L2), чем работа с перегруженными объектами.

> [!INFO] Нюанс: Границы ответственности
> Важно не впадать в крайность. SRP не означает, что в классе должен быть только один метод. Ответственность — это логическая область (например, «Управление данными пользователя»), а не атомарное действие.

### Пример кода ([[MOC#/MOC|C#]] 12)

#### Плохо: Нарушение SRP
Класс управляет и данными, и логикой отправки, и сохранением.
```csharp
public class UserRegistration
{
    public void Register(string email, string password)
    {
        // Логика регистрации
        Console.WriteLine($"User {email} registered.");

        // Прямое сохранение в БД (Нарушение)
        File.WriteAllText("db.txt", $"User: {email}");

        // Отправка email (Нарушение)
        Console.WriteLine($"Sending email to {email}...");
    }
}
```

#### Хорошо: Соблюдение SRP
Разделение на специализированные компоненты через первичные конструкторы C# 12.
```csharp
// 1. Ответственность за хранение данных
public class UserRepository
{
    public void Save(string data) => File.WriteAllText("db.txt", data);
}

// 2. Ответственность за коммуникации
public class EmailService
{
    public void SendWelcomeEmail(string email) => Console.WriteLine($"Email to {email}");
}

// 3. Координатор регистрации (Оркестратор)
public class UserRegistrationService(UserRepository repository, EmailService emailService)
{
    public void Register(string email)
    {
        // Только бизнес-логика
        repository.Save($"User: {email}");
        emailService.SendWelcomeEmail(email);
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Shotgun Surgery (Стрельба дробью)
> Обратная сторона нарушения SRP. Если для внедрения одного небольшого изменения вам приходится править 10 разных мелких классов, значит, вы «переделили» систему, и ответственность размазана слишком тонко.

> [!WARNING] Ошибка: Layering Trap (Ловушка слоев)
> Не путайте SRP с архитектурными слоями. Класс в слое Data Access должен отвечать только за доступ к данным, но не должен пытаться отвечать за все данные приложения сразу.

**Best Practices:**
1.  **Названия классов:** Если вы не можете дать классу короткое и четкое имя без использования слова «And» (например, `UserSaverAndEmailer`), скорее всего, вы нарушаете SRP.
2.  **Размер класса:** Стремитесь к тому, чтобы класс помещался на один-два экрана монитора. Если в классе более 200 строк, проверьте его на наличие скрытых ответственностей.
3.  **Использование интерфейсов:** Описывайте ответственности через интерфейсы (`IEmailService`, `IUserRepository`). Это позволит легко заменять реализации, не меняя логику оркестраторов.
4.  **TDD (Test Driven Development):** Написание тестов перед кодом заставляет вас делать классы маленькими и независимыми, что автоматически ведет к соблюдению SRP.

> [!TIP] Нюанс: Частичные классы (Partial Classes)
> Использование `partial` не помогает соблюсти SRP. Оно лишь разделяет физический файл на части, но логически класс остается единым целым со всеми его ответственностями. Не используйте `partial` для сокрытия архитектурных дефектов.
