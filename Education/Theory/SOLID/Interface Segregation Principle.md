### Суть (The "What")
**Interface Segregation Principle (ISP)** — это четвертый принцип [[SOLID]], который гласит: «Клиенты не должны зависеть от методов, которые они не используют». 

Принцип решает проблему **«толстых» интерфейсов (Fat Interfaces)**. Когда [[Интерфейсы|интерфейс]] перегружен [[Методы|методами]], подходящими для разных сценариев, классы-реализаторы вынуждены предоставлять пустые реализации или выбрасывать исключения для методов, которые им не нужны. Это приводит к избыточной связности (coupling) и хрупкости архитектуры: изменение в одной части интерфейса заставляет перекомпилировать и тестировать всех его потребителей.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Дробление контрактов
С точки зрения архитектуры .NET, интерфейс — это минимальный контракт. ISP требует, чтобы эти контракты были гранулярными. Класс может реализовывать **неограниченное количество интерфейсов**, что в [[MOC#/MOC|C#]] является единственным способом реализации «множественного наследования» поведения.

#### 2. Интерфейсные таблицы (ITables) и VSD
На низком уровне каждый интерфейс, который реализует класс, создает запись в его **Interface Map** (внутри [[MethodTable]]). 
*   **Virtual Stub Dispatch (VSD):** [[CLR]] оптимизирует вызовы интерфейсов. Если интерфейс маленький, поиск нужного метода в таблицах происходит быстрее за счет кэширования заглушек.
*   Чем больше методов в интерфейсе, тем «раздутее» таблицы метаданных и тем сложнее [[JIT]]-компилятору оптимизировать вызовы.

#### 3. Явная реализация интерфейса (Explicit Implementation)
C# предоставляет механизм явной реализации (`void IScanner.Scan()`), который идеально дополняет ISP. Он позволяет классу реализовать интерфейс, но «спрятать» его методы из публичного API самого класса. Методы будут доступны только при приведении объекта к типу интерфейса. Это сохраняет чистоту класса, даже если он реализует десятки мелких контрактов.

> [!INFO] Нюанс: Наследование интерфейсов
> ISP не запрещает наследование интерфейсов. Вы можете создать `IPrinter` и `IScanner`, а затем объединить их в `IMultiFunctionDevice : IPrinter, IScanner`. Это позволяет клиентам выбирать нужный уровень абстракции.

### Пример кода (C# 12)

#### Плохо: Нарушение ISP
Класс `SimplePrinter` вынужден реализовывать метод `Fax`, который он не поддерживает.

```csharp
public interface IMultiFunctionDevice
{
    void Print();
    void Scan();
    void Fax();
}

public class SimplePrinter : IMultiFunctionDevice
{
    public void Print() => Console.WriteLine("Printing...");
    public void Scan() => Console.WriteLine("Scanning...");
    // Нарушение ISP и LSP (Liskov Substitution)
    public void Fax() => throw new NotSupportedException(); 
}
```

#### Хорошо: Соблюдение ISP
Разделение на мелкие контракты и использование явной реализации.

```csharp
public interface IPrinter { void Print(); }
public interface IScanner { void Scan(); }
public interface IFax { void Fax(); }

// Реализует только то, что может
public class PhotoScanner : IScanner
{
    public void Scan() => Console.WriteLine("Scanning photo...");
}

// Реализует всё, используя явную реализацию для чистоты API
public class AllInOneMachine : IPrinter, IScanner, IFax
{
    public void Print() => Console.WriteLine("Laser Printing...");
    public void Scan() => Console.WriteLine("High-res Scanning...");
    
    // Явная реализация: метод Fax не будет виден через переменную типа AllInOneMachine
    void IFax.Fax() => Console.WriteLine("Sending Fax...");
}

public class Program
{
    public static void Main()
    {
        var machine = new AllInOneMachine();
        machine.Print(); // Доступно
        // machine.Fax(); // Ошибка компиляции - метод скрыт
        
        if (machine is IFax faxDevice) 
            faxDevice.Fax(); // Доступно через интерфейс
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Интерфейс для одного метода
> Не стоит превращать каждый метод в отдельный интерфейс. ISP призывает к логическому разделению ролей. Если `Read()` и `Write()` всегда используются вместе в 99% случаев, их разделение лишь усложнит код (Over-engineering).

> [!WARNING] Ошибка: Default Interface Methods (DIM)
> C# 8+ позволяет добавлять реализацию методов в интерфейс по умолчанию. Это может показаться решением проблемы ISP (можно просто не переопределять ненужное), но это ловушка. DIM часто скрывают плохой дизайн, где интерфейс делает слишком много.

**Best Practices:**
1.  **Принцип «Клиент диктует интерфейс»:** При проектировании интерфейса думайте о том, как его будет использовать клиент, а не о том, как его удобно реализовать.
2.  **Композиция интерфейсов:** Если вы создаете новый интерфейс, проверьте, нельзя ли его составить из уже существующих мелких интерфейсов.
3.  **Адаптеры:** Если вы работаете с «толстым» сторонним интерфейсом, создайте свой маленький интерфейс-адаптер, чтобы ваш код зависел только от необходимых методов.
4.  **Hiding with Explicit:** Используйте явную реализацию интерфейсов, чтобы избежать конфликтов имен и не засорять подсказки IntelliSense для пользователей вашего класса.

> [!TIP] Нюанс: Размер сборки
> Большое количество интерфейсов увеличивает размер метаданных сборки. Однако в современных .NET приложениях это нивелируется при использовании **Trimming** (самоотсечения лишнего кода) при публикации приложения, так как неиспользуемые интерфейсные связи легко вычисляются компилятором.
