### Суть (The "What")
**Open/Closed Principle (OCP)** — это второй принцип [[SOLID]], который гласит: «Программные сущности ([[Классы и объекты|классы]], модули, функции) должны быть **открыты для расширения**, но **закрыты для модификации**».

Принцип решает проблему **регрессии**: каждое изменение в уже протестированном и работающем коде несет риск появления новых багов. OCP позволяет добавлять новую функциональность в систему, не затрагивая существующий исходный код, а создавая новые классы на основе абстракций.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Механика абстракции
Реализация OCP опирается на **динамический полиморфизм**. Вместо того чтобы писать код, зависящий от конкретных типов данных, разработчик создает зависимости от интерфейсов или абстрактных классов.
*   **Closed for Modification:** Базовый класс или интерфейс определяет неизменный контракт. Код, использующий этот контракт (клиентский код), остается стабильным.
*   **Open for Extension:** Поведение системы расширяется путем создания новых реализаций интерфейса или наследников абстрактного класса.

#### 2. [[CLR]] и стабильность сборок
С точки зрения .NET, OCP критически важен для разработки библиотек (NuGet-пакетов).
*   Если библиотека требует изменения своего исходного кода для добавления новой функции, это заставляет всех пользователей библиотеки перекомпилировать свои проекты и рисковать совместимостью.
*   Правильно спроектированная по OCP библиотека предоставляет точки расширения ([[Интерфейсы|интерфейсы]]/[[Виртуальные методы и свойства|виртуальные методы]]), позволяя пользователям внедрять свою логику без доступа к исходникам библиотеки.

#### 3. Паттерны-помощники
OCP часто реализуется через следующие паттерны:
*   **[[Strategy]] (Стратегия):** Инкапсуляция алгоритма в отдельный класс.
*   **[[Decorator]] (Декоратор):** Добавление обязанностей объекту динамически.
*   **Template Method:** Определение скелета алгоритма в базовом классе.

> [!INFO] Нюанс: VMT и Dispatch
> На низком уровне соблюдение OCP увеличивает использование Таблиц виртуальных методов (VMT). Каждый раз, когда мы расширяем систему новым классом, CLR добавляет новую MethodTable. Вызов метода через интерфейс стоит несколько дополнительных тактов процессора, но это цена за модульность и надежность системы.

### Пример кода ([[MOC#/MOC|C#]] 12)

#### Плохо: Нарушение OCP
Добавление нового типа отчета требует изменения метода `Generate`, что нарушает принцип "закрытости для модификации".

```csharp
public class ReportGenerator
{
    public void Generate(string type)
    {
        if (type == "PDF") { /* логика PDF */ }
        else if (type == "Excel") { /* логика Excel */ }
        // Чтобы добавить HTML, придется менять этот класс
    }
}
```

#### Хорошо: Соблюдение OCP
Использование интерфейса и первичных конструкторов C# 12.

```csharp
// 1. Абстракция закрыта для модификации
public interface IReportFormatter
{
    void Format();
}

// 2. Расширение системы новыми классами
public class PdfFormatter : IReportFormatter
{
    public void Format() => Console.WriteLine("Formatting PDF...");
}

public class ExcelFormatter : IReportFormatter
{
    public void Format() => Console.WriteLine("Formatting Excel...");
}

// 3. Клиентский код открыт для любого количества новых форматов
public class ReportService(IEnumerable<IReportFormatter> formatters)
{
    public void GenerateAll()
    {
        foreach (var formatter in formatters)
        {
            formatter.Format();
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Предсказание будущего (Speculative Generality)
> Не создавайте абстракции "на будущее" для кода, который никогда не будет расширяться. Это усложняет архитектуру без реальной пользы. Применяйте OCP только тогда, когда вы точно знаете, что вариативность логики неизбежна.

> [!WARNING] Ошибка: Хрупкий базовый класс (Fragile Base Class)
> При расширении через наследование будьте осторожны: изменение реализации `virtual` метода в базовом классе может сломать логику всех наследников. Предпочитайте расширение через интерфейсы (композицию).

**Best Practices:**
1.  **Dependency Injection:** Используйте DI-контейнеры для регистрации новых реализаций. Это позволяет менять поведение программы простым изменением конфигурации (регистрации типов).
2.  **Pattern Matching:** В C# 12 можно использовать `switch` на типах, но это часто является признаком нарушения OCP. Если вам приходится постоянно обновлять `switch` при добавлении нового класса — перенесите логику в сам класс через `override`.
3.  **Default Interface Methods (DIM):** Используйте реализации в интерфейсах по умолчанию (C# 8+) аккуратно. Они помогают расширять интерфейсы без поломки старых реализаций, но могут запутать иерархию вызовов.
4.  **Plugin Architecture:** Проектируйте критические узлы системы так, чтобы они могли подгружать реализации из внешних сборок (DLL) — это высшая форма соблюдения OCP.

> [!TIP] Нюанс: Абстракция vs Интерфейс
> Nutshell рекомендует использовать интерфейсы для OCP, когда поведение (способность) может быть реализовано совершенно разными типами. Абстрактные классы лучше подходят, когда расширения имеют общую внутреннюю структуру и состояние.
