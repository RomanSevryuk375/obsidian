### Суть (The "What")
**Liskov Substitution Principle (LSP)** — это третий принцип [[SOLID]], сформулированный Барбарой Лисков. Он гласит: «Объекты в программе должны быть заменяемыми на экземпляры их подтипов без изменения правильности работы программы».

LSP решает задачу **корректности иерархии наследования**. Он гарантирует, что производный класс не просто копирует сигнатуры методов базового класса, но и полностью соблюдает его **поведенческий контракт**. Если код ожидает `Bird`, он должен корректно работать и с `Duck`, и с `Penguin`, не натыкаясь на неожиданные исключения или изменение логики.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Контрактное программирование
[[Абстрактные классы и члены классов|Базовый класс]] устанавливает правила игры (инварианты). Наследник обязан их соблюдать:
*   **Предусловия не могут быть усилены:** Метод наследника не должен требовать от входных данных больше, чем базовый метод.
*   **Постусловия не могут быть ослаблены:** Если базовый метод гарантирует возврат положительного числа, наследник не может вернуть отрицательное.
*   **Инварианты должны сохраняться:** Внутреннее состояние объекта, которое считалось валидным в базовом классе, должно оставаться валидным и в производном.

#### 2. Проблема "Квадрат-Прямоугольник"
Классический пример нарушения LSP. С точки зрения математики квадрат — это прямоугольник. Но в программировании, если у прямоугольника можно независимо менять ширину и высоту, а квадрат при изменении ширины вынужден менять и высоту, он нарушает ожидания пользователя, работающего с ним как с прямоугольником.

#### 3. Низкоуровневая безопасность ([[CLR]])
В .NET приведение типа к базовому (**[[Upcasting]]**) всегда безопасно на уровне памяти, так как поля базового класса в структуре объекта в куче всегда располагаются первыми. Однако LSP отвечает за **логическую безопасность**: CLR не знает о ваших бизнес-правилах, поэтому ответственность за соблюдение LSP лежит целиком на разработчике.

> [!INFO] Нюанс: Ковариантность и Контравариантность
> LSP тесно связан с поддержкой вариантности в [[MOC#/MOC|C#]]. Возвращаемые типы могут быть ковариантными (наследник может вернуть более специфичный тип), а аргументы — контравариантными. Это позволяет типам оставаться "взаимозаменяемыми" на уровне сигнатур.

### Пример кода (C# 12)

#### Плохо: Нарушение LSP
Использование наследования там, где нарушается логика поведения базового класса.

```csharp
public class Bird
{
    public virtual void Fly() => Console.WriteLine("Flying...");
}

public class Penguin : Bird
{
    public override void Fly() 
    {
        // Нарушение LSP: ожидание поведения "полет" прерывается исключением
        throw new NotSupportedException("Penguins can't fly!");
    }
}
```

#### Хорошо: Соблюдение LSP
Разделение интерфейсов и использование композиции.

```csharp
// Базовый контракт для всех птиц
public abstract class Bird(string name)
{
    public string Name { get; } = name;
}

// Выделение способности к полету в отдельный интерфейс
public interface IFlyable
{
    void Fly();
}

public class Duck(string name) : Bird(name), IFlyable
{
    public void Fly() => Console.WriteLine($"{Name} is flying.");
}

public class Penguin(string name) : Bird(name)
{
    // Пингвин — птица, но он не реализует IFlyable. Контракт не нарушен.
    public void Swim() => Console.WriteLine($"{Name} is swimming.");
}

public class NatureService
{
    public void MakeBirdsFly(IEnumerable<IFlyable> flyingBirds)
    {
        foreach (var bird in flyingBirds)
        {
            bird.Fly(); // Здесь никогда не будет NotSupportedException
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: NotImplementedException
> Если вы переопределяете метод базового класса и выбрасываете `NotImplementedException` или `NotSupportedException`, вы почти наверняка нарушаете LSP. Это означает, что ваша иерархия наследования построена неверно ("is-a" не выполняется).

> [!WARNING] Ошибка: Проверка типов (is/as) в циклах
> Если вам приходится проверять фактический тип объекта в цикле по базовому классу (`if (item is Penguin) ...`), чтобы избежать ошибки или выполнить спец-логику — это признак нарушения LSP. Поведение должно быть полиморфным.

**Best Practices:**
1.  **Приоритет интерфейсов:** Используйте интерфейсы для описания способностей (`IFlyable`, `IDisposable`). Это позволяет объектам быть заменяемыми без привязки к жесткой иерархии классов.
2.  **Неизменяемость (Immutability):** Проблема Квадрата-Прямоугольника исчезает, если объекты неизменяемы (`readonly record`). Если свойства нельзя менять, то и инварианты нельзя нарушить.
3.  **Композиция вместо наследования:** Если класс "почти как тот", но с другими правилами поведения, лучше включить экземпляр "того" класса внутрь нового, чем наследоваться от него.
4.  **Unit-тесты базового класса:** Прогоняйте тесты базового класса на объектах производных классов. Если тесты падают — LSP нарушен.

> [!TIP] Нюанс: Тест "Утки"
> Если это выглядит как утка и крякает как утка, но нуждается в батарейках — у вас, вероятно, неправильная абстракция. LSP требует, чтобы "батарейки" (детали реализации) не мешали объекту быть "уткой" в любом контексте.
