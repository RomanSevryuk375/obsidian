### Суть (The "What")
**Объект-значение (Value Object)** — это паттерн тактического проектирования [[DDD]], представляющий объект, который описывает некую характеристику предметной области, но **не имеет уникального идентификатора (Identity)**. Два таких объекта считаются абсолютно равными, если совпадают значения всех их свойств.

Value Objects решают проблему **«одержимости примитивами» (Primitive Obsession)**. Вместо использования разрозненных базовых типов (например, `string` для адреса или `decimal` для денег), данные и правила их валидации объединяются в единый, семантически значимый тип. Это повышает типобезопасность и защищает бизнес-логику от некорректных состояний.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Структурное равенство (Value Equality)
Поскольку у Value Object нет ID (в отличие от Entity), логика оператора `==` и метода `Equals` должна проверять совпадение каждого поля объекта. 
*   **Исторический подход:** Разработчикам приходилось писать абстрактный базовый класс `ValueObject`, использующий [[Рефлексия|рефлексию]] для обхода полей, или вручную переопределять `Equals` и `GetHashCode()`.
*   **Современный подход:** Использование типов `record` ([[MOC|C#]] 9) или `readonly record struct` (C# 10+). Компилятор автоматически генерирует высокооптимизированный [[IL]]-код для покомпонентного сравнения всех свойств, исключая накладные расходы на рефлексию.

#### 2. Абсолютная неизменяемость (Immutability)
Value Object создается один раз и навсегда. 
*   У него не должно быть публичных или приватных сеттеров (`set`).
*   Любая бизнес-операция, которая логически "изменяет" объект, на низком уровне должна **возвращать новый экземпляр** Value Object с обновленными данными. Это делает такие объекты абсолютно потокобезопасными (Thread-Safe) и защищает агрегаты от скрытых побочных эффектов.

#### 3. Память: [[Stack]] vs [[Куча|Heap]]
Выбор между `record class` и `readonly record struct` для реализации Value Object влияет на [[GC]] (Garbage Collector):
*   `readonly record struct` размещается в **стеке** (Stack). Это идеальный выбор для небольших объектов (Деньги, Координаты, Возраст), так как они не создают нагрузки на кучу.
*   `record class` размещается в **куче** (Heap). Используется для тяжелых Value Objects (например, сложный Адрес из 10 полей), чтобы избежать накладных расходов на копирование памяти при передаче в методы.

> [!INFO] Нюанс: Интеграция с ORM (EF Core)
> Value Objects не имеют собственных таблиц в реляционных базах данных. В Entity Framework Core они маппятся в виде колонок таблицы той сущности (Entity), которой принадлежат. До EF Core 8 это делалось через `.OwnsOne()`, а начиная с EF Core 8 появилась нативная поддержка **Complex Types** (`[ComplexType]`), которая идеально ложится на семантику Value Objects.

### Пример кода ([[MOC|C#]] 12)

Реализация паттерна "Деньги" (Money) с использованием современных возможностей C#.

```csharp
using System;

// Используем readonly record struct для размещения в стеке и бесплатного Value Equality
public readonly record struct Money
{
    public decimal Amount { get; }
    public string Currency { get; }

    // Конструктор с инкапсулированной логикой валидации
    public Money(decimal amount, string currency)
    {
        if (amount < 0)
            throw new ArgumentOutOfRangeException(nameof(amount), "Сумма не может быть отрицательной");
            
        ArgumentException.ThrowIfNullOrWhiteSpace(currency);
        if (currency.Length != 3)
            throw new ArgumentException("Код валюты должен состоять из 3 символов ISO", nameof(currency));

        Amount = amount;
        Currency = currency.ToUpper();
    }

    // Бизнес-поведение: сложение возвращает НОВЫЙ объект
    public static Money operator +(Money left, Money right)
    {
        if (left.Currency != right.Currency)
            throw new InvalidOperationException("Нельзя складывать разные валюты");

        return new Money(left.Amount + right.Amount, left.Currency);
    }

    // Фабричные методы для удобства
    public static Money Usd(decimal amount) => new(amount, "USD");
    public static Money Eur(decimal amount) => new(amount, "EUR");
}

public class Program
{
    public static void Main()
    {
        Money price1 = Money.Usd(100.50m);
        Money price2 = new(100.50m, "usd"); // В конструкторе валюта приведется к "USD"

        // 1. Проверка структурного равенства (True, так как Amount и Currency совпадают)
        Console.WriteLine(price1 == price2); 

        // 2. Использование перегруженного оператора (создаст новый объект в стеке)
        Money total = price1 + Money.Usd(50m);
        Console.WriteLine($"{total.Amount} {total.Currency}"); // 150.50 USD

        // 3. Защита инварианта: Money.Usd(-10) выбросит исключение
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Одержимость примитивами (Primitive Obsession)
> Передача в метод сигнатуры `void Pay(decimal amount, string currency)` — это риск ошибки. Разработчик может перепутать порядок аргументов, передать отрицательное число или пустую валюту. Если заменить это на `void Pay(Money amount)`, компилятор и система типов защитят вас от подавляющего большинства ошибок.

> [!WARNING] Ошибка: Изменяемые свойства
> Если вы сделаете свойство Value Object изменяемым (например, `public decimal Amount { get; set; }`), вы нарушите архитектуру. Если этот Value Object используется как ключ в `Dictionary` или элемент в `HashSet`, изменение свойства изменит его хеш-код, и объект навсегда "потеряется" в коллекции.

**Best Practices:**
1.  **Always Valid:** Value Object никогда не должен существовать в невалидном состоянии. Валидация всех параметров должна происходить исключительно в его конструкторе.
2.  **Замена базового класса:** В современном .NET (C# 9+) откажитесь от написания кастомных базовых классов `ValueObject` с тяжелой рефлексией. `record` и `readonly record struct` делают это быстрее, безопаснее и на уровне IL-кода.
3.  **Богатая логика:** Value Object — это не просто структура данных. В него нужно выносить логику форматирования, вычислений и сравнений, которая относится к этим данным (например, метод `IsExpired()` внутри `ExpirationDate`).
4.  **Сравнение с Null:** При использовании `record class` всегда помните, что это ссылочный тип, и он может быть `null`. При использовании `readonly record struct` вы защищены от `null` (хотя объект может иметь значение `default`, что требует обработки в конструкторе без параметров, если он используется).

> [!TIP] Нюанс: Использование 'with'
> Если Value Object реализован как `record`, используйте выражение `with` для создания копий с небольшими изменениями. Это не нарушает неизменяемость: `var discountedPrice = price with { Amount = 90m };`. Однако убедитесь, что валидация не будет пропущена (в C# 10+ для рекордов можно определить конструктор копирования, чтобы защитить оператор `with`).
