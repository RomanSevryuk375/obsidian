### Суть (The "What")
**Bounded Context (Ограниченный контекст)** — это ключевой паттерн стратегического проектирования в [[DDD (Domain-Driven Design)|DDD]]. Это строгая логическая (а часто и физическая) граница, внутри которой применяется единая, непротиворечивая модель предметной области и строго соблюдается **Единый язык (Ubiquitous Language)**.

Он решает проблему **семантической двусмысленности и жесткого зацепления**. В масштабах предприятия один и тот же термин (например, "Клиент" или "Продукт") означает совершенно разные вещи для отдела продаж, бухгалтерии и службы доставки. Bounded Context позволяет создать для каждого отдела свою независимую модель "Продукта", вместо того чтобы строить один гигантский класс `Product` с сотней полей для всех случаев жизни.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Границы Единого языка
Внутри контекста каждое понятие имеет только одно строгое определение. 
*   В контексте **Продаж (Sales)** "Продукт" — это цена, скидки и маркетинговое описание.
*   В контексте **Склада (Inventory)** "Продукт" — это габариты, вес и номер стеллажа.
Попытка объединить их в один класс приведет к тому, что изменения в логике склада будут ломать код отдела продаж.

#### 2. Проекция на архитектуру .NET
В .NET Bounded Context чаще всего реализуется как:
*   Отдельный **микросервис** (в микросервисной архитектуре).
*   Изолированный **модуль** с собственными проектами `Domain`, `Application`, `Infrastructure` (в модульном монолите).

#### 3. Database per Context (Изоляция данных)
Фундаментальное правило: один Bounded Context = одна база данных (или изолированная схема).
Вы не имеете права делать SQL `JOIN` между таблицами разных контекстов. Если контексту "Доставка" нужно узнать имя клиента из контекста "Продажи", он должен получить эти данные через API или подписаться на событие, сохранив нужную часть данных в *свою* базу.

#### 4. [[Context Mapping]] (Связь между контекстами)
Контексты неизбежно должны общаться. Для этого используются паттерны интеграции:
*   **Anti-Corruption Layer (Антикоррупционный слой):** Фасад, который транслирует (маппит) термины чужого контекста в термины вашего контекста, защищая вашу модель от "загрязнения" чужими абстракциями.
*   **Integration Events (Интеграционные события):** Асинхронный обмен сообщениями (через [[RabbitMQ]]/Kafka). Контекст "Продажи" публикует `OrderPlaced`, а контекст "Склад" ловит его и реагирует.

> [!INFO] Нюанс: Shared Kernel (Общее ядро)
> Иногда два контекста тесно связаны и делят общую часть домена (например, класс `Money` или базовые перечисления). Этот общий код выносится в Shared Kernel (отдельный NuGet-пакет). Но использовать это нужно крайне осторожно: изменение в Shared Kernel требует синхронного обновления всех зависящих контекстов, что убивает автономию.

### Пример кода ([[MOC|C#]] 12)

Этот пример показывает, как одна и та же физическая сущность ("Книга") моделируется совершенно по-разному в двух разных ограниченных контекстах (Продажи и Доставка).

```csharp
using System;

// ==========================================================
// BOUNDED CONTEXT A: Sales (Отдел продаж)
// Фокус на деньгах, скидках и каталоге
// ==========================================================
namespace BoundedContexts.Sales.Domain;

// Товар в контексте продаж (Primary Constructor C# 12)
public class CatalogProduct(Guid id, string name, decimal price)
{
    public Guid Id { get; } = id;
    public string Name { get; } = name;
    public decimal Price { get; private set; } = price;

    public void ApplyDiscount(decimal percentage)
    {
        Price -= Price * (percentage / 100);
    }
}


// ==========================================================
// BOUNDED CONTEXT B: Shipping (Служба доставки)
// Фокус на логистике, вес и габариты. Цена здесь не нужна!
// ==========================================================
namespace BoundedContexts.Shipping.Domain;

// Тот же самый товар, но смоделированный для логистики
public class ShippableItem(Guid id, double weightKg, string dimensions)
{
    public Guid Id { get; } = id; // Связь с продуктом из Sales идет только по ID
    public double WeightKg { get; } = weightKg;
    public string Dimensions { get; } = dimensions;

    public decimal CalculateShippingCost(decimal baseRatePerKg)
    {
        // Логика зависит от веса, а не от цены товара
        return (decimal)WeightKg * baseRatePerKg;
    }
}


// ==========================================================
// АНТИКОРРУПЦИОННЫЙ СЛОЙ (ACL) внутри контекста Shipping
// ==========================================================
namespace BoundedContexts.Shipping.Infrastructure.Acl;

public record ExternalProductDto(Guid Id, string Name, double Weight, string SizeBox);

// Служба, которая переводит "чужой" DTO в "свою" доменную модель
public class ShippingAntiCorruptionLayer
{
    public ShippableItem TranslateFromExternal(ExternalProductDto externalData)
    {
        // Контекст логистики берет только то, что ему нужно, игнорируя остальное
        return new ShippableItem(
            id: externalData.Id,
            weightKg: externalData.Weight,
            dimensions: externalData.SizeBox
        );
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Единая модель предприятия (Enterprise Data Model)
> Попытка создать один гигантский класс `User` или `Product`, который содержит все возможные поля для всех отделов компании. Это делает сущность раздутой (God Object), а код — немасштабируемым. Каждое изменение потребует согласования со всеми командами.

> [!WARNING] Ошибка: Общая база данных (Shared Database)
> Если контекст "Склад" обращается к таблице `Products` контекста "Продажи" напрямую через SQL-запрос, границы контекста прорваны. При изменении схемы БД в "Продажах" упадет код "Склада". Контексты должны общаться исключительно через API или события.

**Best Practices:**
1.  **Дублирование данных — это нормально:** Если "Доставке" нужно имя клиента, чтобы напечатать этикетку, скопируйте это имя в базу данных "Доставки" в момент создания заказа. Итоговая согласованность (Eventual Consistency) лучше жесткой связанности.
2.  **Защищайте свои границы:** Используйте Anti-Corruption Layer при интеграции со старыми (Legacy) системами или внешними API. Не пускайте чужие модели ([[DTO]]) в свой Domain.
3.  **Определяйте границы по бизнес-функциям:** Bounded Context должен соответствовать организационной структуре компании или конкретному бизнес-процессу (Биллинг, Инвентаризация, Поддержка), а не техническим слоям (Фронтенд, Бэкенд).
4.  **Разные команды:** В идеале каждый Bounded Context разрабатывается, тестируется и развертывается одной выделенной командой разработчиков, что обеспечивает максимальную автономию.

> [!TIP] Нюанс: Именование
> Чтобы подчеркнуть границы, используйте разные пространства имен и даже разные имена классов для одной и той же физической вещи. В Billing это может быть `Payer`, в Shipping — `Recipient`, а в IAM — `User`.
