### Суть (The "What")
**Context Mapping (Карта контекстов)** — это инструмент стратегического проектирования [[DDD]], который описывает топологию взаимосвязей и способы интеграции между различными ограниченными контекстами ([[Bounded Context]]). 

Он решает задачу **управляемой интеграции**. Поскольку каждый контекст имеет свою собственную модель (Единый язык), прямая передача объектов из одного контекста в другой невозможна. Context Mapping определяет архитектурные и "политические" (межкомандные) правила того, как модели транслируются и обмениваются данными без потери своей автономности.

---

### Как это работает (The "How" / Nutshell style)

В архитектуре .NET границы контекстов часто совпадают с границами микросервисов или модулей. Пересечение этой границы — это пересечение процессов (сетевой вызов, сериализация), поэтому Context Mapping опирается на следующие паттерны:

#### 1. Anti-Corruption Layer (ACL - Антикоррупционный слой)
*   **Концепция:** Ваш контекст (клиент) создает изолирующий слой-фасад. Он перехватывает запросы к чужому контексту и переводит "чужую" модель в "свою".
*   **Реализация .NET:** Реализуется через паттерны `Adapter` и `Facade`. На низком уровне это означает жесткое разделение: [[HTTP]]/[[gRPC]] клиент получает `ExternalDto`, а затем слой ACL маппит его в `InternalDomainEntity`. Внешний [[DTO]] никогда не пересекает границу слоя Domain.

#### 2. Open Host Service (OHS) и Published Language (PL)
*   **Концепция:** Контекст (поставщик) предоставляет стабильный, версионированный API для всех желающих, используя общепринятый язык обмена данными.
*   **Реализация .NET:** OHS — это ваш Web API. PL (Опубликованный язык) — это контракты Protobuf (`.proto` файлы) в gRPC или спецификация OpenAPI (Swagger) в REST. Они не привязаны к внутренней структуре C# классов (Entity Framework моделей).

#### 3. Shared Kernel (Общее ядро)
*   **Концепция:** Два контекста разделяют общий кусок доменной модели.
*   **Реализация .NET:** Общая сборка (`.dll`), распространяемая через внутренний NuGet-сервер. В ней хранятся базовые интерфейсы (например, `IDomainEvent`), абстракции или примитивные объекты-значения (например, структура `Money`).
*   **Нюанс:** Приводит к жесткому зацеплению. Изменение NuGet-пакета требует синхронной перекомпиляции обоих микросервисов.

#### 4. Conformist (Конформист)
*   **Концепция:** Ваш контекст слепо принимает модель чужого контекста, так как у вас нет ресурсов на написание ACL, а поставщик отказывается менять свой API.
*   **Реализация .NET:** В вашем коде бизнес-логики начинают использоваться DTO из стороннего SDK (например, объекты из библиотеки Stripe или AWS).

### Пример кода ([[MOC|C#]] 12)

Реализация паттерна **Anti-Corruption Layer (ACL)**. Контекст "Биллинг" запрашивает данные у контекста "Идентификация" (IAM). Чтобы не загрязнять свой домен чужими структурами, Биллинг использует ACL.

```csharp
using System;
using System.Threading.Tasks;

// ==========================================================
// 1. ВНЕШНЯЯ МОДЕЛЬ (Чужой Bounded Context - IAM)
// ==========================================================
// Это DTO, который мы получаем по сети (например, десериализованный JSON).
// Нам не нужны поля 'LastLoginIp' и 'IsEmailVerified' в биллинге.
public record IamUserDto(Guid UserId, string FullName, string LastLoginIp, bool IsEmailVerified);


// ==========================================================
// 2. ВНУТРЕННЯЯ МОДЕЛЬ (Наш Bounded Context - Billing)
// ==========================================================
// Это сущность, которая понятна нашему бизнес-домену.
public class Payer(Guid accountId, string displayName)
{
    public Guid AccountId { get; } = accountId;
    public string DisplayName { get; } = displayName;
    public decimal Balance { get; private set; } = 0;

    public void Charge(decimal amount) => Balance -= amount;
}


// ==========================================================
// 3. ANTI-CORRUPTION LAYER (Инфраструктурный слой Billing)
// ==========================================================
// Интерфейс лежит в слое Application (Инверсия зависимостей)
public interface IPayerProvider
{
    Task<Payer> GetPayerAsync(Guid id);
}

// Реализация лежит в слое Infrastructure
public class IamAntiCorruptionLayer(IIamHttpClient externalClient) : IPayerProvider
{
    private readonly IIamHttpClient _externalClient = externalClient;

    public async Task<Payer> GetPayerAsync(Guid id)
    {
        // 1. Получаем чужую модель (ExternalDto)
        IamUserDto externalUser = await _externalClient.FetchUserAsync(id);

        // 2. Транслируем (маппим) чужую модель в нашу доменную сущность
        // Защищаем домен биллинга от изменений в IAM
        return new Payer(
            accountId: externalUser.UserId, 
            displayName: externalUser.FullName
        );
    }
}

// Заглушка HTTP клиента для компиляции
public interface IIamHttpClient { Task<IamUserDto> FetchUserAsync(Guid id); }
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Протекающая модель (Leaky Domain)
> Самая частая архитектурная ошибка — использование автогенерированных классов из Swagger/gRPC клиентов напрямую в ядре (Domain/Application) вашего приложения. При любом изменении API поставщиком (чужим контекстом) ваше ядро перестанет компилироваться. Модели поставщика должны останавливаться в инфраструктурном слое (ACL).

> [!WARNING] Ошибка: Раздутый Shared Kernel
> Помещение бизнес-логики, сущностей (Entities) или сервисов в общий NuGet-пакет превращает микросервисную архитектуру в **распределенный монолит**. Общее ядро должно содержать только максимально абстрактные типы (интерфейсы, базовые исключения, примитивы).

**Best Practices:**
1.  **Dumb Pipes, Smart Endpoints:** В Context Mapping интеграция через брокер сообщений (RabbitMQ) не должна содержать логики маршрутизации на самом брокере (Dumb Pipes). Вся логика трансформации и понимания контрактов (Smart Endpoints) должна лежать в ACL-слое конкретного микросервиса.
2.  **Versioning (OHS):** Если вы выступаете в роли поставщика (Open Host Service), всегда версионируйте свои API (`/api/v1/...` или пакеты gRPC `package mycompany.billing.v1;`). Это защитит ваших клиентов от поломок.
3.  **Устойчивость (Resilience):** ACL-слой должен быть защищен паттернами `Retry` и `Circuit Breaker` (через библиотеку Polly), так как он осуществляет переходы между границами контекстов (сетевые вызовы), которые по своей природе ненадежны.
4.  **Разные модели для CRUD и Событий:** DTO, возвращаемый по REST API, и тело интеграционного события (Integration Event) в Kafka для одной и той же сущности — это разные контракты. Относитесь к ним как к разным частям вашего Опубликованного языка (PL).
