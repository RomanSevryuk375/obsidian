### Суть (The "What")
**Оповещения и Публикация (Notifications & Publishing)** — это встроенный в MediatR механизм реализации паттерна «Издатель-Подписчик» (Publish-Subscribe) для обмена сообщениями типа «один ко многим» внутри одного процесса (в памяти приложения). 

Этот паттерн решает задачу **слабой связанности при реакции на события** (побочные эффекты). Когда в системе происходит важное бизнес-событие (например, «Заказ оформлен»), вместо того чтобы основной метод создания заказа напрямую зависел от сервиса отправки SMS, сервиса обновления склада и сервиса аналитики, он просто публикует событие. Любое количество независимых обработчиков могут подписаться на это событие и отреагировать на него, не зная друг о друге и об инициаторе события.

### Как это работает (The "How" / Nutshell style)

#### 1. Многоадресная рассылка
В отличие от запросов (`IRequest`), которые направляются строго одному обработчику, оповещения обрабатываются по другой схеме:
*   **Оповещение (`INotification`):** Маркерный интерфейс для [[DTO]]-события.
*   **Обработчики (`INotificationHandler<T>`):** Любое количество классов может реализовать этот интерфейс для одного и того же типа оповещения.
*   При вызове `_mediator.Publish(notification)` MediatR запрашивает у [[Dependency Injection|DI]]-контейнера **все** зарегистрированные реализации `INotificationHandler` для данного события и передает сообщение каждой из них.

#### 2. Стратегии публикации (Notification Publisher)
По умолчанию MediatR выполняет обработчики событий **последовательно** в рамках одного потока выполнения (один `await` за другим). Это поведение можно изменить при настройке MediatR с помощью свойства `NotificationPublisher`:
*   **ForeachAwaitPublisher (по умолчанию):** Вызывает обработчики один за другим. Если один из обработчиков упадет с ошибкой, выполнение последующих прервется.
*   **TaskWhenAllPublisher:** Запускает все обработчики параллельно с помощью `Task.WhenAll`. Это ускоряет работу, но требует осторожности с потокобезопасностью (например, нельзя использовать один и тот же scoped-контекст БД параллельно в разных обработчиках).

#### 3. Транзакционность и Исключения
Поскольку публикация происходит в памяти процесса, по умолчанию обработчики выполняются в рамках той же транзакции базы данных (если транзакция была открыта в вызывающем коде). Исключение, выброшенное в любом обработчике, по умолчанию прервет основной процесс и откатит транзакцию.

### Структура папок и пример реализации ([[C#]] 12)

Рекомендуемая структура папок, разделяющая само событие и его независимых подписчиков:

```text
Ordering.Application/
└── Orders/
    └── Events/
        ├── OrderPlacedNotification.cs       # Оповещение (Record)
        └── Handlers/
            ├── SendEmailConfirmation.cs     # Подписчик 1 (Primary Constructor)
            └── UpdateInventory.cs           # Подписчик 2 (Primary Constructor)
```

#### 1. Оповещение (OrderPlacedNotification.cs)
Используем `record` для гарантии иммутабельности события:

```csharp
using MediatR;
using System;

namespace Ordering.Application.Orders.Events;

// Оповещение не несет логики, только данные о свершившемся факте
public record OrderPlacedNotification(Guid OrderId, Guid CustomerId, decimal TotalAmount) : INotification;
```

#### 2. Подписчик 1: Отправка Email (SendEmailConfirmation.cs)
```csharp
using System;
using System.Threading;
using System.Threading.Tasks;
using MediatR;

namespace Ordering.Application.Orders.Events.Handlers;

public class SendEmailConfirmation(IEmailSender emailSender) : INotificationHandler<OrderPlacedNotification>
{
    private readonly IEmailSender _emailSender = emailSender;

    public async Task Handle(OrderPlacedNotification notification, CancellationToken cancellationToken)
    {
        // Логика отправки письма клиенту
        await _emailSender.SendAsync(notification.CustomerId, "Ваш заказ успешно оформлен!", cancellationToken);
    }
}

// Заглушка для компиляции
public interface IEmailSender { Task SendAsync(Guid id, string m, CancellationToken ct); }
```

#### 3. Подписчик 2: Обновление Склада (UpdateInventory.cs)
```csharp
using System;
using System.Threading;
using System.Threading.Tasks;
using MediatR;

namespace Ordering.Application.Orders.Events.Handlers;

public class UpdateInventory(IInventoryService inventoryService) : INotificationHandler<OrderPlacedNotification>
{
    private readonly IInventoryService _inventoryService = inventoryService;

    public async Task Handle(OrderPlacedNotification notification, CancellationToken cancellationToken)
    {
        // Логика резервирования товаров на складе
        await _inventoryService.ReserveForOrderAsync(notification.OrderId, cancellationToken);
    }
}

// Заглушка для компиляции
public interface IInventoryService { Task ReserveForOrderAsync(Guid id, CancellationToken ct); }
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Тяжелая синхронная логика в обработчиках
> Не выполняйте длительные, тяжелые или нестабильные операции (например, сложные интеграционные запросы к сторонним сервисам) напрямую в обработчиках оповещений, если они выполняются последовательно. Это заблокирует основной поток бизнес-логики (оформление заказа будет висеть, пока отправляется email). 
> **Решение:** Обработчик должен лишь зафиксировать задачу (например, записать сообщение в Outbox-таблицу или брокер сообщений RabbitMQ) и мгновенно вернуть управление.

> [!WARNING] Ошибка: Необработанные исключения (Exception Swallowing)
> Если один из подписчиков падает с ошибкой, по умолчанию падает весь вызывающий процесс. Если побочный эффект не является критическим для завершения основной транзакции (например, сбой аналитики не должен мешать созданию заказа), оборачивайте код обработчика в `try-catch` или используйте специализированные издатели с перехватом ошибок.

**Best Practices:**
1.  **Иммутабельность:** Объявляйте `INotification` как `readonly record`. Обработчики не должны иметь возможности изменить данные события в процессе рассылки.
2.  **Асинхронность по умолчанию:** Пишите обработчики асинхронно, пробрасывая [[CancellationToken]] во все вложенные вызовы.
3.  **Изоляция доменных событий:** Используйте `Publish` для обработки доменных событий ([[Domain Events]]). Это позволяет держать сущности чистыми: сущность просто накапливает события в процессе работы, а инфраструктурный уровень (например, [[Entity Framework|EF Core]] при `SaveChanges`) публикует их через MediatR.
4.  **Кастомный Publisher для надежности:** Если вам нужно параллельное выполнение, но с гарантией работы всех обработчиков при падении одного из них, напишите свою реализацию `INotificationPublisher`, которая перехватывает исключения каждого обработчика и агрегирует их в конце.

> [!TIP] Настройка параллельного выполнения
> Чтобы включить параллельное выполнение обработчиков во всем приложении, настройте MediatR в файле конфигурации:
> ```csharp
> builder.Services.AddMediatR(cfg => {
>     cfg.RegisterServicesFromAssembly(typeof(Program).Assembly);
>     cfg.NotificationPublisher = new TaskWhenAllPublisher();
> });
> ```
