# Запросы и Обработчики (Request-Response) в MediatR

### Суть (The "What")
**Запросы и Обработчики (Request-Response)** — это основной механизм взаимодействия в MediatR, реализующий паттерн внутрипроцессного обмена сообщениями типа «один к одному». Каждому отправленному объекту запроса соответствует ровно один обработчик, который возвращает результат.

Этот паттерн решает задачу **изоляции сценариев использования (Use Cases)**. Вместо того чтобы класс контроллера или API-эндпоинта зависел от десятков различных сервисов, он зависит только от одной абстракции — `IMediator`. Контроллер отправляет медиатору простой объект запроса, а медиатор сам находит и запускает нужный обработчик, инкапсулирующий всю бизнес-логику конкретного сценария.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Разделение данных и логики
Паттерн жестко разделяет данные запроса и логику его обработки на два класса:
*   **Запрос (`IRequest<TResponse>`):** Простой объект переноса данных ([[DTO]]). Он декларирует свое намерение и тип возвращаемого значения `TResponse`. Не содержит никакой логики.
*   **Обработчик (`IRequestHandler<TRequest, TResponse>`):** Класс, содержащий метод `Handle`. Он принимает объект запроса, выполняет бизнес-логику и возвращает результат.

#### 2. Динамическое разрешение зависимостей ([[Dependency Injection|DI]])
При вызове метода `_mediator.Send(request, cancellationToken)` происходят следующие шаги:
1.  MediatR анализирует тип переданного объекта запроса.
2.  Через встроенный DI-контейнер .NET (`IServiceProvider`) запрашивается соответствующая реализация `IRequestHandler<TRequest, TResponse>`.
3.  Рантайм создает экземпляр обработчика, автоматически внедряя в его конструктор все необходимые зависимости (например, контекст БД, репозитории, внешние клиенты).
4.  Вызывается метод `Handle`, и результат возвращается отправителю.

#### 3. Жизненный цикл ([[Service Lifetimes]])
По умолчанию при регистрации MediatR через `AddMediatR` все обработчики регистрируются в контейнере как **Transient** (создаются заново при каждом запросе).
*   Это гарантирует отсутствие "грязного" или устаревшего состояния (state pollution) внутри обработчика между разными вызовами.
*   Это обеспечивает потокобезопасность: каждый поток работает со своим изолированным экземпляром обработчика.

### Структура папок и пример реализации ([[C#]] 12)

Рекомендуемая структура папок по принципу **Vertical Slices** (вертикальные срезы), где все файлы одной бизнес-фичи лежат в одной директории:

```text
Ordering.Application/
└── Orders/
    └── Queries/
        └── GetOrderDetails/
            ├── GetOrderDetailsQuery.cs          # Запрос (Record)
            ├── GetOrderDetailsQueryHandler.cs   # Обработчик (Primary Constructor)
            └── OrderDetailsResponse.cs          # DTO ответа
```

#### 1. Запрос и Модель ответа (GetOrderDetailsQuery.cs & OrderDetailsResponse.cs)
Используем `record` для обеспечения иммутабельности данных:

```csharp
using MediatR;
using System;

namespace Ordering.Application.Orders.Queries.GetOrderDetails;

// Модель ответа (DTO)
public record OrderDetailsResponse(Guid Id, string CustomerName, decimal TotalAmount);

// Запрос, возвращающий OrderDetailsResponse. Иммутабелен по умолчанию.
public record GetOrderDetailsQuery(Guid OrderId) : IRequest<OrderDetailsResponse?>;
```

#### 2. Обработчик запроса (GetOrderDetailsQueryHandler.cs)
Реализация логики с использованием первичного конструктора C# 12 для внедрения зависимостей:

```csharp
using System;
using System.Threading;
using System.Threading.Tasks;
using MediatR;

namespace Ordering.Application.Orders.Queries.GetOrderDetails;

// Обработчик принимает запрос и возвращает DTO
public class GetOrderDetailsQueryHandler(IOrderRepository repository) 
    : IRequestHandler<GetOrderDetailsQuery, OrderDetailsResponse?>
{
    private readonly IOrderRepository _repository = repository;

    public async Task<OrderDetailsResponse?> Handle(GetOrderDetailsQuery request, CancellationToken cancellationToken)
    {
        // Обязательно пробрасываем cancellationToken в асинхронные вызовы
        var order = await _repository.GetByIdAsync(request.OrderId, cancellationToken);
        
        if (order == null) return null;

        return new OrderDetailsResponse(order.Id, order.CustomerName, order.TotalAmount);
    }
}

// Заглушка репозитория для компиляции
public interface IOrderRepository 
{ 
    Task<dynamic?> GetByIdAsync(Guid id, CancellationToken ct); 
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Анемичные обработчики (Transit Handlers)
> Избегайте создания обработчиков, которые просто перенаправляют вызов во внешний сервис: `return _orderService.GetDetails(request.Id);`. Это приводит к дублированию слоев (Over-engineering). Обработчик *должен* быть местом выполнения сценария использования. Избавьтесь от промежуточных сервисов-прослоек и пишите логику сценария прямо в обработчике.

> [!WARNING] Ошибка: Игнорирование CancellationToken
> Всегда пробрасывайте [[CancellationToken]] во все асинхронные операции внутри обработчика (вызовы к БД, внешним API). Если клиент прервет [[HTTP]]-запрос (например, закроет вкладку), отмена распространится по цепочке, и сервер не будет тратить ресурсы на выполнение ненужной работы.

**Best Practices:**
1.  **Иммутабельность:** Объявляйте все запросы как `readonly record`. Данные запроса не должны изменяться во время прохождения конвейера MediatR.
2.  **Vertical Slices (Группировка):** Размещайте запрос, его обработчик и DTO ответа в одной папке (или даже в одном файле, если они небольшие). Они неразрывно связаны, и разделение их по разным проектам только усложняет навигацию.
3.  **Не возвращайте доменные сущности:** Обработчик запроса (Query) должен возвращать только плоские DTO (как `OrderDetailsResponse`), оптимизированные под требования вызывающей стороны, а не внутренние доменные модели базы данных.
4.  **Валидация вне обработчика:** Не пишите код проверки параметров (например, `if (id == Guid.Empty)`) внутри метода `Handle`. Используйте [[Pipeline Behaviors|IPipelineBehavior]] совместно с [[FluentValidation]] для автоматической предварительной проверки всех запросов.

> [!TIP] Нюанс: MediatR.Contracts
> Начиная с версии MediatR 12.0, интерфейсы контрактов (такие как `IRequest`) вынесены в отдельный легковесный NuGet-пакет `MediatR.Contracts`. Это позволяет подключать его в чистый слой доменных моделей (`Domain`), не затягивая туда всю тяжелую инфраструктуру самой библиотеки MediatR.
