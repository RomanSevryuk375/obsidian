**Explicit Loading (Явная загрузка)** — это техника, когда ты загружаешь связанные данные **позже**, уже после того, как основной объект был получен из базы.

Если [[Eager Loading]] (`Include`)— это "Дайте мне бургер сразу с картошкой и колой" (одним заказом),
то **Explicit Loading** — это "Дайте мне бургер. ... (спустя время) ... А теперь донесите, пожалуйста, колу".

Это делается через API отслеживания изменений: `context.Entry(...)`.

---

### 1. Зачем это нужно?

Казалось бы, зачем делать два запроса к [[БД]] вместо одного?

1.  **Условная загрузка:** Ты не знаешь заранее, понадобятся ли тебе связанные данные. Например, ты показываешь список счетов (`Bill`). И только если пользователь нажмет кнопку "Показать детали", ты подгружаешь список платежей (`Payments`).
2.  **Фильтрация перед загрузкой:** Самая мощная фишка. В отличие от [[Lazy Loading]], Explicit Loading позволяет отфильтровать или отсортировать дочерние данные **до** того, как они попадут в память.
3.  **Память:** Если у объекта 1000 связанных записей, а тебе нужны только 5 последних, Explicit Loading спасет твою оперативную память.

---

### 2. Синтаксис

Есть два метода, в зависимости от типа связи:
*   `Collection(...).Load()` — для связей "один-ко-многим" (списки, `List`, `ICollection`).
*   `Reference(...).Load()` — для связей "один-к-одному" или "многие-к-одному" (одиночные объекты).

#### Пример на твоем `BillEntity`

Представим, что мы уже загрузили счет, но забыли (или не захотели) сразу загрузить платежи и заказ.

```csharp
// 1. Загружаем только сам счет (1 запрос к БД)
var bill = await context.Bills.FindAsync(10);

// ... какая-то логика ...
if (userClickedDetails)
{
    // 2. Явно загружаем коллекцию платежей (2-й запрос к БД)
    await context.Entry(bill)
        .Collection(b => b.Payments)
        .LoadAsync();
        
    // 3. Явно загружаем связанный заказ (3-й запрос к БД)
    await context.Entry(bill)
        .Reference(b => b.Order)
        .LoadAsync();
}
```

### 3. "Киллер-фича": Метод [[Query()]]

Это главное отличие от [[Lazy Loading]]. Перед вызовом `.Load()`, ты можешь вызвать `.Query()`. Это возвращает обычный `IQueryable`, к которому можно применить LINQ (`Where`, `OrderBy`, `Count`).

**Сценарий:** У счета может быть 1000 попыток оплаты. Нам не нужны они все, нам нужны только успешные или только количество попыток.

```csharp
var bill = await context.Bills.FindAsync(10);

// Пример А: Загрузить только успешные платежи
// Данные материализуются и попадут в bill.Payments
await context.Entry(bill)
    .Collection(b => b.Payments)
    .Query() // Превращаем в запрос
    .Where(p => p.IsSuccessful)
    .OrderByDescending(p => p.PaymentDate)
    .LoadAsync(); 

// Пример Б: Просто узнать количество платежей, НЕ загружая их в память
// Очень эффективно: SQL вернет одно число (COUNT), а не массив данных
int paymentCount = await context.Entry(bill)
    .Collection(b => b.Payments)
    .Query()
    .CountAsync();
```

### 4. Проверка загрузки (IsLoaded)

Иногда полезно знать, загружены ли уже данные, чтобы [[Entity Framework]] не делал лишний запрос.

```csharp
var entry = context.Entry(bill).Collection(b => b.Payments);

if (!entry.IsLoaded)
{
    await entry.LoadAsync();
}
```

### 5. Explicit Loading vs [[Eager Loading]] vs [[Lazy Loading]]

| Тип          | Метод             | Кол-во запросов | Когда использовать?                                                                     |
| :----------- | :---------------- | :-------------- | :-------------------------------------------------------------------------------------- |
| **Eager**    | `.Include()`      | 1 (обычно)      | Когда данные нужны **сразу и всегда** (например, вывести счет и его статус на экран).   |
| **Explicit** | `.Entry().Load()` | [[N plus 1]]         | Когда данные нужны **редко** или нужна **фильтрация** (только активные заказы клиента). |
| **Lazy**     | `virtual` св-ва   | [[N plus 1]](неявно) | В энтерпрайзе лучше **избегать**. Слишком легко случайно убить базу циклом foreach.     |
