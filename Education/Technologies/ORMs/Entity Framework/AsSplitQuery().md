**AsSplitQuery()** — это механизм оптимизации загрузки связанных данных, появившийся в [[Entity Framework|EF Core]] 5.0. Он решает одну из главных проблем производительности [[ORM]] при использовании `Include` — **Взрыв Декартова произведения ([[Cartesian Explosion]])**.

Давай разберем, что это за проблема и как `AsSplitQuery` её решает.

---

### 1. Проблема: Один огромный запрос (Single Query)

По умолчанию (если не указать иное), EF Core пытается загрузить все данные **одним** [[SQL]]-запросом, используя `JOIN`.

Представим твою систему счетов (`Bill`). Допустим, у счета есть:
*   Коллекция `Payments` (Платежи) — 10 штук.
*   Коллекция `HistoryLogs` (История изменений) — 20 записей.

Ты пишешь код:
```csharp
var bills = await context.Bills
    .Include(b => b.Payments)
    .Include(b => b.HistoryLogs)
    .FirstOrDefaultAsync(b => b.Id == 1);
```

**Что происходит в SQL (Single Query):**
```sql
SELECT b.*, p.*, h.*
FROM Bills AS b
LEFT JOIN PaymentNotes AS p ON b.Id = p.BillId
LEFT JOIN HistoryLogs AS h ON b.Id = h.BillId
WHERE b.Id = 1
```

**Результат (Декартово произведение):**
База данных вернет таблицу, где количество строк = `1 (Счет) * 10 (Платежей) * 20 (Логов) = 200 строк`.

**Почему это плохо:**
В каждой из этих 200 строк информация о самом счете (`Id`, `Amount`, `CreatedAt`, `StatusId`...) будет **дублироваться**.
Если у тебя в таблице `Bills` есть тяжелые текстовые поля (например, `Description` на 1 Кб), то ты скачаешь этот килобайт 200 раз (200 Кб мусора), хотя он нужен всего один раз.

Это тратит трафик сети и оперативную память сервера.

---

### 2. Решение: Разделение запросов (Split Query)

Когда ты добавляешь `.AsSplitQuery()`, ты говоришь [[Entity Framework|EF Core]]: *"Не пытайся сделать всё одним махом. Лучше сделай несколько маленьких и чистых запросов"*.

```csharp
var bill = await context.Bills
    .Include(b => b.Payments)
    .Include(b => b.HistoryLogs)
    .AsSplitQuery() // <--- ВКЛЮЧАЕМ РЕЖИМ
    .FirstOrDefaultAsync(b => b.Id == 1);
```

**Что происходит в [[SQL]] (будет 3 запроса):**

1.  **Запрос 1:** Грузим сам счет.
    ```sql
    SELECT * FROM Bills WHERE Id = 1
    ```
    *(Вернулась 1 строка)*

2.  **Запрос 2:** Грузим платежи для этого счета.
    ```sql
    SELECT * FROM PaymentNotes WHERE BillId = 1
    ```
    *(Вернулось 10 строк)*

3.  **Запрос 3:** Грузим логи для этого счета.
    ```sql
    SELECT * FROM HistoryLogs WHERE BillId = 1
    ```
    *(Вернулось 20 строк)*

**Итог:** Мы скачали `1 + 10 + 20 = 31` строку данных без дубликатов. [[Entity Framework|EF Core]] в памяти сам соберет это в один объект `BillEntity`.

---

### 3. Когда использовать AsSplitQuery?

Это не "серебряная пуля", у подхода есть свои плюсы и минусы.

| Характеристика          | Single Query (По умолчанию)          | Split Query (AsSplitQuery)                     |
| :---------------------- | :----------------------------------- | :--------------------------------------------- |
| **Количество запросов** | 1 (всегда)                           | [[N+1]] (где N — количество Include коллекций) |
| **Объем данных**        | Большой (много дубликатов)           | Минимальный (без дубликатов)                   |
| **Нагрузка на [[БД]]**  | Сложные [[JOIN]], грузят CPU базы    | Простые SELECT, легкие для базы                |
| **Целостность данных**  | ✅ Гарантирована (Snapshot Isolation) | ⚠️ Возможна рассинхронизация (см. ниже)        |

#### Рекомендация:
Используй `AsSplitQuery`, когда:
1.  Ты загружаешь **несколько коллекций** одновременно (`Include(a).Include(b)`).
2.  Связанные коллекции **большие** (сотни элементов).
3.  Основная таблица содержит **"тяжелые" колонки** (большие строки, JSON, XML).

Не используй, когда:
1.  Ты грузишь одну маленькую коллекцию (1-2 элемента). Один [[JOIN]] будет быстрее из-за отсутствия лишних сетевых вызовов (Latency).
2.  Ты используешь `Skip/Take` (пагинацию) со сложной сортировкой — иногда Split Query может вести себя неэффективно.

### 4. Риск: Согласованность данных ([[Data Consistency]])

Это важный профессиональный нюанс.

При **[[Single Query]]** база данных гарантирует целостность моментального снимка. Ты получаешь данные, актуальные на момент выполнения запроса.

При **Split Query** между первым запросом (получение Счета) и вторым запросом (получение Платежей) может пройти время (допустим, 10 мс).
В эти 10 мс кто-то может добавить новый платеж.
В итоге ты можешь получить ситуацию:
*   В Счете поле `TotalPaidAmount` еще старое.
*   А в списке `Payments` новый платеж уже есть.

**Решение:** Если для твоей задачи это критично, нужно оборачивать операцию в Транзакцию:
```csharp
using var transaction = context.Database.BeginTransaction();
// ... запрос с AsSplitQuery ...
transaction.Commit();
```
*Но в 99% случаев чтения данных (Read-Only) этим пренебрегают.*

### 5. Глобальная настройка

Если ты решил, что в твоем проекте почти всегда лучше использовать Split Query (например, у тебя везде огромные графы данных), ты можешь включить это глобально в `Program.cs`, чтобы не писать `AsSplitQuery()` везде.

```csharp
services.AddDbContext<AppDbContext>(options =>
    options.UseNpgsql(connectionString, o => 
        // Включаем Split Query по умолчанию для всего проекта
        o.UseQuerySplittingBehavior(QuerySplittingBehavior.SplitQuery)));
```

Если вдруг в конкретном месте тебе понадобится старое поведение, ты напишешь `.AsSingleQuery()`.

### Пример для твоего кода

В твоем случае со счетами и платежами это очень актуально:

```csharp
public async Task<BillEntity> GetBillWithDetails(long id)
{
    return await _context.Bills
        .Include(b => b.Status)     // Одиночная связь (JOIN ок)
        .Include(b => b.Order)      // Одиночная связь (JOIN ок)
        .Include(b => b.Payments)   // <--- КОЛЛЕКЦИЯ! Потенциальный дубляж
        .AsSplitQuery()             // <--- Оптимизируем загрузку платежей
        .FirstOrDefaultAsync(b => b.Id == id);
}
```

Здесь `Status` и `Order` скорее всего будут загружены вместе с `Bills` (так как это Single Reference), а `Payments` — отдельным запросом. [[Entity Framework|EF Core]] достаточно умен, чтобы комбинировать подходы.
