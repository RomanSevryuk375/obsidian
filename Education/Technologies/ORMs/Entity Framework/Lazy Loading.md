**Lazy Loading (Ленивая загрузка)** — это механизм, при котором связанные данные загружаются из [[БД]] автоматически **в тот самый момент, когда ты впервые обращаешься к свойству**.

Это звучит как магия и кажется очень удобным («не надо писать `Include`, всё само подтянется»), но в современной Web-разработке (особенно в высоконагруженных [[API]]) это считается **антипаттерном**.

---

### 1. Как это работает 

Чтобы Lazy Loading заработал в [[Entity Framework]] Core, нужно выполнить три условия:

1.  Подключить пакет `Microsoft.EntityFrameworkCore.Proxies`.
2.  Включить настройку в `OnConfiguring`: `.UseLazyLoadingProxies()`.
3.  Сделать все навигационные свойства **`virtual`**.

**Твой класс:**
```csharp
public class BillEntity
{
    public int Id { get; set; }
    
    public virtual BillStatusEntity Status { get; set; } // Ключевое слово virtual
}
```

**Что делает EF Core (Proxies):**
В runtime (во время работы программы) EF Core создает **класс-наследник** (Proxy), который выглядит примерно так (псевдокод):

```csharp
// Сгенерированный EF Core класс в памяти
public class BillEntityProxy : BillEntity
{
    private readonly DbContext _context;
    
    public override BillStatusEntity Status
    {
        get
        {
            // Перехватываем обращение к свойству
            if (base.Status == null)
            {
                // Если данных нет — делаем синхронный запрос в БД!
                // SELECT * FROM bill_statuses WHERE Id = ...
                this.LoadStatusFromDb(); 
            }
            return base.Status;
        }
        set => base.Status = value;
    }
}
```

Ты этого кода не видишь, но именно так работает «магия». Ты пишешь `var s = bill.Status.Name`, а под капотом летит SQL-запрос.

### 2. Главная проблема: [[N plus 1]]

Это самая частая причина тормозов в приложениях с Lazy Loading.

Представим, ты хочешь вывести список из 100 счетов и название их статуса.

```csharp
// 1 запрос: SELECT * FROM Bills
var bills = context.Bills.ToList(); 

foreach (var bill in bills)
{
    // В момент обращения к bill.Status летит запрос в БД!
    // И так для каждого из 100 счетов.
    Console.WriteLine(bill.Status.Name);
}
```

**Итог:**
*   Без Lazy Loading (с `Include`): **1 запрос** ([[JOIN]]).
*   С Lazy Loading: **101 запрос** (1 для списка + 100 маленьких запросов для каждого статуса).

База данных захлебнется от количества мелких обращений (Network Roundtrips).

### 3. Проблема Сериализации ([[JsonSerializer]])

Это классическая ошибка новичка в [[Web API]].

Ты пишешь контроллер:
```csharp
[HttpGet]
public List<BillEntity> Get()
{
    // Загружаем только счета
    return context.Bills.ToList(); 
}
```

А теперь представь, что делает JSON-сериализатор (System.Text.Json или Newtonsoft), чтобы отправить ответ клиенту. Он проходит по **всем** публичным свойствам объекта, чтобы превратить их в текст.

1.  Сериализатор трогает `bill.Status`.
2.  Lazy Loading просыпается: «Ага, меня позвали!» -> Делает запрос в БД.
3.  Сериализатор идет внутрь `Status`, видит там коллекцию `Bills`.
4.  Lazy Loading просыпается -> Делает запрос, чтобы загрузить все счета с этим статусом.
5.  В этих счетах он снова видит `Status`...

**Результат:**
1.  Либо приложение выкачает **всю базу данных** в память.
2.  Либо упадет с ошибкой `Cycle Detected` (циклическая ссылка).
3.  Либо просто будет отвечать 30 секунд.

---

### 4. Синхронность ([[Блокировка потоков]])

Lazy Loading в текущей реализации EF Core — **синхронный**.
Когда срабатывает геттер свойства (`get`), он не может сделать `await`.

В [[ASP.NET]] Core мы боремся за [[Aсинхронность]] (`async/await`), чтобы один поток мог обслуживать тысячи запросов.
Lazy Loading замораживает поток на время запроса к [[БД]] (блокирующий I/O). Если база ответит через 500мс, поток будет стоять и ждать, не обслуживая других пользователей.

---

### 5. Когда Lazy Loading полезен?

Неужели он настолько плох? Нет, у него есть ниша:

1.  **Desktop-приложения (WPF, WinForms):** Там нет проблемы 1000 одновременных пользователей. Пользователь открыл карточку клиента — данные подгрузились. Удобно.
2.  **Сложная бизнес-логика ([[Domain Model]]):** Иногда внутри метода модели (`bill.CalculatePenalty()`) нужны данные, и ты не хочешь думать, подгрузили их снаружи через `Include` или нет.
3.  **Прототипирование:** Когда нужно быстро накидать админку и плевать на производительность.

---

### Итог: Почему стоит избегать в [[Web API]]?

В твоем проекте `CRMSystem` (судя по структуре, это бэкенд/[[API]]) **лучше не использовать Lazy Loading**.

1.  **Непредсказуемость:** Глядя на код `var x = bill.Status.Name`, ты не знаешь, произойдет запрос к [[БД]] или нет. Это скрывает проблемы производительности.
2.  **[[N plus 1]]:** Очень легко случайно положить базу в цикле.
3.  **Контроль:** В [[API]] ты должен точно знать, какие данные ты отдаешь клиенту. [[Eager Loading]] (`Include`) или [[Explicit Loading]] (`Load`) дают тебе полный контроль.
