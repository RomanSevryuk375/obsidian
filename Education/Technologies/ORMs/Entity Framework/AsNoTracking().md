**AsNoTracking()** — это один из самых простых и эффективных способов ускорить работу [[Entity Framework|EF Core]]. Это флаг, который говорит контексту: **«Не следи за этими объектами, я просто хочу их прочитать»**.

В высоконагруженных системах (HighLoad) использование этого метода для GET-запросов является **обязательным стандартом**.

Давай разберем, что происходит «под капотом» и почему это так важно.

---

### 1. Как работает стандартный запрос ([[Tracking]])

По умолчанию, когда ты делаешь запрос:
```csharp
var bill = await context.Bills.FirstOrDefaultAsync(b => b.Id == 10);
```

Происходит тяжелая работа:
1.  **Материализация:** EF создает объект `Bill` из данных [[SQL]].
2.  **Снапшот (Snapshot):** EF создает **копию** этого объекта и сохраняет её в `ChangeTracker` (в памяти контекста).
3.  **Отслеживание:** Теперь EF сравнивает (при каждом `SaveChanges`) текущий объект со снапшотом, чтобы понять, изменил ли ты что-то.

Это нужно для того, чтобы работал код:
```csharp
bill.Amount = 500;
await context.SaveChangesAsync(); // EF знает, что Amount изменился
```

### 2. Как работает AsNoTracking()

Когда ты пишешь:
```csharp
var bill = await context.Bills
    .AsNoTracking() // <--- Выключаем слежку
    .FirstOrDefaultAsync(b => b.Id == 10);
```

Происходит только:
1.  **Материализация:** [[Entity Framework|EF]] создает объект и отдает его тебе.
2.  **Забывание:** Контекст тут же «забывает» про этот объект. В `ChangeTracker` пусто.

**Результат:**
*   **Скорость:** Работает быстрее (не тратится время на создание снапшотов).
*   **Память:** Потребляет меньше RAM (не хранятся копии объектов).

---

### 3. Когда использовать?

#### Сценарий 1: Read-Only [[API]] (GET запросы)
90% запросов в [[Web API]] — это чтение. Списки товаров, просмотр профиля, отчеты.
В методах контроллера, которые просто возвращают данные ([[DTO]]) и ничего не меняют, **всегда** используй `AsNoTracking`.

```csharp
[HttpGet("{id}")]
public async Task<ActionResult<BillDto>> GetBill(long id)
{
    var bill = await _context.Bills
        .AsNoTracking() // Мы не будем делать Update, слежка не нужна
        .Include(b => b.Status)
        .FirstOrDefaultAsync(b => b.Id == id);

    return _mapper.Map<BillDto>(bill);
}
```

#### Сценарий 2: Генерация отчетов
Если ты выгружаешь 10,000 записей для Excel-отчета, `ChangeTracker` без `AsNoTracking` просто «взорвет» память сервера, пытаясь отследить 10,000 объектов.

---

### 4. Когда НЕ использовать?

####  Сценарий: Планируется изменение (Update/Delete)
Если ты собираешься менять объект в **этом же** запросе.

```csharp
// ПЛОХОЙ ПРИМЕР
var bill = await _context.Bills.AsNoTracking().FirstAsync(b => b.Id == 1);
bill.Amount = 999;
await _context.SaveChangesAsync(); // НИЧЕГО НЕ ПРОИЗОЙДЕТ!
```
Так как слежка была выключена, `SaveChanges` подумает, что ничего не изменилось. [[SQL]] запрос `UPDATE` не будет отправлен.

### 5. Нюанс: [[Identity Resolution]] (Разрешение идентичности)

В режиме отслеживания (`Tracking`) EF Core гарантирует, что если в результате SQL-запроса одна и та же строка (например, `StatusId=1`) встретилась 5 раз, то в памяти C# это будет **один и тот же экземпляр объекта**.

В режиме `AsNoTracking` [[Entity Framework|EF Core]] не проверяет, загружал ли он этот объект раньше.
Если у тебя 5 счетов со статусом "Paid", EF создаст **5 разных объектов** Status (хотя ID у них одинаковый).

**Проблема:** Это может увеличить потребление памяти, если дублей очень много.
**Решение (EF Core 5+):**
```csharp
.AsNoTrackingWithIdentityResolution()
```
Это компромисс. Он не следит за изменениями (быстро), но проверяет дубликаты ID, чтобы не создавать лишние объекты в памяти (экономично). Используй это при загрузке больших графов данных с повторяющимися связями.

### 6. Глобальная настройка (QueryTrackingBehavior)

Если у тебя [[API]], где почти всё — это чтение (например, новостной портал), ты можешь отключить трекинг **по умолчанию** для всего [[DbContext]].

В `Program.cs` или при регистрации контекста:
```csharp
services.AddDbContext<AppDbContext>(options =>
    options
        .UseNpgsql(...)
        // По умолчанию все запросы будут NoTracking
        .UseQueryTrackingBehavior(QueryTrackingBehavior.NoTracking));
```

Тогда, если тебе вдруг нужно будет что-то изменить, ты включишь трекинг явно:
```csharp
var bill = context.Bills
    .AsTracking() // Включаем обратно для конкретного запроса
    .First();
bill.Amount = 500;
context.SaveChanges();
```

### 7. Секрет про `.Select` (Проекции)

Если ты используешь проекцию в [[DTO]] или анонимный тип:

```csharp
var dtos = context.Bills
    .Select(b => new BillDto { Id = b.Id, Sum = b.Amount }) // Проекция
    .ToList();
```

[[Entity Framework|EF Core]]** автоматически** применяет логику `AsNoTracking`. Он понимает, что ты возвращаешь не сущность `BillEntity`, а какой-то `BillDto`, который он всё равно не умеет отслеживать.
Поэтому писать `.AsNoTracking().Select(...)` можно, но технически это избыточно (хотя и не вредно).

### Резюме

1.  **GET-запросы:** Всегда `AsNoTracking()`.
2.  **POST/PUT/DELETE:** Используй обычный трекинг (без `AsNoTracking`).
3.  **Большие отчеты:** Обязательно `AsNoTracking()`.
4.  **Сложные связи:** Если много дублирующихся вложенных объектов — `AsNoTrackingWithIdentityResolution()`.
