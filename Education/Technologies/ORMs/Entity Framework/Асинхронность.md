**Асинхронность** в [[Entity Framework|EF Core]] — это фундамент масштабируемости современных веб-приложений.

Если коротко: **Синхронный код блокирует поток**, пока база данных думает. **Асинхронный код освобождает поток**, позволяя серверу обрабатывать другие запросы в это время.

В [[ASP.NET]] Core использование асинхронных методов (`...Async`) является **обязательным стандартом**.

---

### 1. Зачем это нужно? (Управление потоками)

Представь, что твой [[Web API]] работает на веб-сервере ([[Kestrel]]). У сервера есть ограниченный запас рабочих рук — **Пул потоков ([[Thread Pool]])**. Допустим, их 100 штук.

#### Сценарий А: Синхронный (Плохо)
Ты пишешь: `var users = context.Users.ToList();`

1.  Приходит [[HTTP-запрос]]. Сервер выделяет **Поток №1**.
2.  Поток отправляет [[SQL]]-запрос в базу данных.
3.  [[БД]] выполняет сложный поиск (допустим, 2 секунды).
4.  Всё это время **Поток №1 просто стоит и ждет**. Он заблокирован. Он не может делать ничего полезного.
5.  Если придет 101-й запрос, а все 100 потоков "ждут" базу, сервер перестанет отвечать (Thread Starvation).

#### Сценарий Б: Асинхронный (Хорошо)
Ты пишешь: `var users = await context.Users.ToListAsync();`

1.  Приходит HTTP-запрос. Сервер выделяет **Поток №1**.
2.  Поток отправляет SQL-запрос.
3.  Встречая `await`, **Поток №1 возвращается в пул**. Он свободен! Он может пойти обрабатывать запрос от другого пользователя.
4.  Через 2 секунды база отвечает.
5.  Система берет **любой свободный поток** (например, Поток №5) и продолжает выполнение твоего кода с того же места.

**Итог:** Асинхронность не делает выполнение *одного* запроса быстрее (иногда даже чуть медленнее из-за накладных расходов). Но она позволяет серверу обрабатывать **тысячи запросов одновременно**, используя всего пару сотен потоков.

### 2. Основные методы

Почти у каждого синхронного метода [[LINQ]] в [[Entity Framework|EF Core]] есть асинхронный брат-близнец. Они находятся в пространстве имен `Microsoft.EntityFrameworkCore`.

| Синхронный (Не использовать) | Асинхронный (Использовать) | Что делает |
| :--- | :--- | :--- |
| `ToList()` | `ToListAsync()` | Выполняет запрос и возвращает список. |
| `ToArray()` | `ToArrayAsync()` | Возвращает массив. |
| `First()` / `FirstOrDefault()` | `FirstAsync()` / `FirstOrDefaultAsync()` | Ищет первый элемент. |
| `Single()` / `SingleOrDefault()` | `SingleAsync()` / `SingleOrDefaultAsync()` | Ищет единственный элемент. |
| `Any()` | `AnyAsync()` | Проверяет наличие (`EXISTS`). |
| `All()` | `AllAsync()` | Проверяет все элементы. |
| `Count()` | `CountAsync()` | Возвращает количество. |
| `SaveChanges()` | `SaveChangesAsync()` | Сохраняет изменения (COMMIT). |

**Обрати внимание:** Методы `Where`, `OrderBy`, `Select`, `Include` — **не имеют** асинхронных версий. Почему? Потому что они не обращаются к базе. Они просто строят запрос в памяти. Обращение происходит только при "материализации" (ToList, Count и т.д.).

### 4. [[Cancellation Token]] (Отмена запросов) — Pro уровень

Что если пользователь запросил тяжелый отчет, база начала его считать, а пользователь передумал и закрыл вкладку браузера?

Без **CancellationToken** база продолжит работать впустую, сжигая ресурсы процессора.
[[Entity Framework|EF Core]] поддерживает отмену операций.

```csharp
[HttpGet("heavy-report")]
// ASP.NET Core сам передаст токен отмены в этот параметр, если клиент разорвет соединение
public async Task<IActionResult> GetReport(CancellationToken cancellationToken)
{
    var data = await _context.Bills
        .Where(b => b.Amount > 1000)
        // Передаем токен в метод EF Core
        .ToListAsync(cancellationToken); 

    return Ok(data);
}
```
Если пользователь закроет браузер, `ToListAsync` выбросит исключение `OperationCanceledException`, и [[SQL]]-запрос на сервере базы данных будет отменен (команда `KILL` или аналог).

### 5. IAsyncEnumerable (Стриминг данных)

В [[Entity Framework|EF Core]] есть возможность читать данные потоком, по мере их поступления из базы, не загружая весь список в память сразу.

Обычный `ToListAsync()` ждет, пока база отдаст **все** строки, потом создает огромный `List` в памяти, и только потом отдает его дальше.

С `AsAsyncEnumerable()` можно обрабатывать строки по одной:

```csharp
// Это супер-эффективно по памяти для огромных выборок
public async Task ProcessLargeTable()
{
    var billsQuery = _context.Bills.AsAsyncEnumerable();

    await foreach (var bill in billsQuery)
    {
        // Этот код выполняется сразу, как только из БД пришла очередная строка.
        // Нам не нужно ждать загрузки всей таблицы.
        ProcessBill(bill);
    }
}
```

### 6. Частые ошибки

1.  **`async void`:** Никогда не используй `async void` (кроме обработчиков событий в WPF/WinForms). Всегда используй `async Task`. Если в `async void` методе произойдет ошибка, она убьет весь процесс приложения.
2.  **Смешивание Sync и Async:**
    Не делай так: `context.Bills.ToListAsync().Result`. Это называется "Sync-over-Async". Это может привести к мертвым блокировкам (Deadlocks) и убивает всю пользу асинхронности. Используй `await`.
3.  **Асинхронность в [[LINQ]] выражениях:**
    ```csharp
    // ОШИБКА!
    var result = await context.Bills
        .Where(async b => await CheckExternalApi(b.Id)) // Нельзя использовать async внутри Where
        .ToListAsync();
    ```
    [[SQL]] не умеет ждать внешний [[API]]. Такие вещи нужно делать либо через `Client Evaluation` (загрузить всё, потом фильтровать), либо менять логику.

### Резюме

*   Используй асинхронные методы (`...Async`) **везде**, где есть обращение к базе данных (`I/O bound operations`).
*   Используй `await`, чтобы не блокировать поток.
*   Передавай `CancellationToken`, чтобы не делать лишнюю работу, если клиент ушел.
*   Это стандарт де-факто для разработки бэкенда на .NET.
