### Суть (The "What")
**Асинхронность** — это модель программирования, позволяющая выполнять ресурсоемкие операции (ввод-вывод, сетевые запросы, обращение к БД) без блокировки вызывающего потока. 

Асинхронность решает две главные задачи:
1. **Масштабируемость (Scalability):** Позволяет серверу (ASP.NET) обрабатывать тысячи запросов, используя минимальное количество потоков, так как потоки не простаивают в ожидании ответа от внешних систем.
2. **Отзывчивость (Responsiveness):** Предотвращает "зависание" графического интерфейса (UI), освобождая основной поток для взаимодействия с пользователем.

### Как это работает (The "How" / Nutshell style)

#### 1. Конечный автомат ([[State Machine]])
Когда компилятор видит ключевое слово `async`, он преобразует метод в **структуру-конечный автомат**. 
*   Метод выполняется синхронно до первого ключевого слова `await`. 
*   Если задача, которую мы ждем, еще не завершена, метод возвращает управление (объект [[Task]]) вызывающему коду, а состояние метода (локальные переменные) сохраняется в куче (или стеке, если используется [[ValueTask]]).
*   Когда операция завершается, выполнение возобновляется с сохраненной точки.

#### 2. Магия "без потоков" (I/O Bound)
Для операций ввода-вывода (чтение файла, сетевой запрос) **новый поток не создается**. 
*   [[CLR]] передает запрос операционной системе. 
*   ОС использует механизмы прерываний и порты завершения ввода-вывода (**IOCP — I/O Completion Ports**). 
*   Пока данные читаются, поток из пула .NET свободен и может выполнять другие задачи. 
*   Когда данные готовы, ОС сигнализирует рантайму, и поток из пула "подхватывает" продолжение метода.

#### 3. Контекст синхронизации (SynchronizationContext)
В десктопных приложениях (WPF, WinForms) по умолчанию после `await` выполнение возвращается в исходный (UI) поток. В ASP.NET Core контекста синхронизации нет — продолжение всегда выполняется в любом свободном потоке из пула.

#### 4. ValueTask
Введен для оптимизации. Если метод часто возвращает результат синхронно (например, данные уже есть в кэше), `ValueTask` позволяет избежать аллокации объекта в куче, что снижает нагрузку на Garbage Collector.

> [!INFO] Нюанс:ConfigureAwait(false)
> В библиотечном коде всегда используйте `Task.ConfigureAwait(false)`. Это говорит рантайму, что нам не нужно возвращаться в исходный контекст (например, UI-поток), что повышает производительность и предотвращает Deadlocks в старых фреймворках (.NET Framework).

### Пример кода ([[MOC|C#]] 12)

```csharp
using System;
using System.Net.Http;
using System.Threading;
using System.Threading.Tasks;

public class AsyncService
{
    private readonly HttpClient _client = new();

    // ValueTask эффективен, если результат часто берется из кэша
    public async ValueTask<string> GetDataAsync(string url, CancellationToken ct = default)
    {
        try
        {
            // Метод выполняется синхронно до этой строки
            // Поток освобождается на время выполнения GetStringAsync
            string result = await _client.GetStringAsync(url, ct).ConfigureAwait(false);
            
            return result;
        }
        catch (OperationCanceledException)
        {
            return "Operation was cancelled";
        }
    }

    public async Task ProcessMultipleTasks()
    {
        Task<string> task1 = _client.GetStringAsync("https://api.a.com");
        Task<string> task2 = _client.GetStringAsync("https://api.b.com");

        // Запускаем задачи параллельно и ждем их завершения
        string[] results = await Task.WhenAll(task1, task2);
        
        foreach (var r in results) Console.WriteLine(r.Length);
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Async Void
> Никогда не используйте `async void`, за исключением обработчиков событий (UI). Исключения в `async void` методах не могут быть пойманы вызывающим кодом и приводят к немедленному падению всего процесса (Crash). Используйте `async Task`.

> [!WARNING] Ошибка: Блокирующий вызов (Sync over Async)
> Использование `.Result` или `.Wait()` для асинхронных задач блокирует поток. В средах с контекстом синхронизации (WPF, старый ASP.NET) это гарантированно приводит к **Deadlock** (взаимной блокировке).

**Best Practices:**
1. **Async all the way:** Асинхронность должна пронизывать приложение от верхнего уровня (Controller/UI) до нижнего (DB/Network).
2. **[[CancellationToken]]:** Всегда передавайте токены отмены в асинхронные методы. Это позволяет пользователю или системе прервать затянутую операцию.
3. **Task.WhenAll:** Если задачи не зависят друг от друга, запускайте их одновременно через `WhenAll`, а не последовательно через `await task1; await task2;`.
4. **ValueTask для горячих путей:** Используйте `ValueTask`, если метод вызывается очень часто и в большинстве случаев завершается мгновенно (синхронно).

> [!TIP] Нюанс: IAsyncEnumerable (C# 8+)
> Для получения потоковых данных (например, чтение строк из БД по мере их поступления) используйте `await foreach` и тип `IAsyncEnumerable<T>`. Это позволяет обрабатывать данные "порциями", не загружая всю коллекцию в память целиком.
