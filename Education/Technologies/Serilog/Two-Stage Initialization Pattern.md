### Суть (The "What")
**Two-Stage Initialization** — это архитектурный паттерн настройки логирования в .NET приложениях. Он разделяет процесс запуска логгера на два шага: создание временного "начального" логгера (Bootstrap Logger) и его последующую замену полноценным логгером, интегрированным в контейнер зависимостей (DI).

Этот паттерн решает проблему **"слепого пятна" при запуске**: если ошибка произойдет во время считывания конфигурации `appsettings.json` или сборки DI-контейнера, обычный логгер еще не будет готов, и причина падения приложения не будет зафиксирована.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Стадия начальной загрузки (Bootstrap)
На первом этапе создается статический экземпляр `Log.Logger`. Он настраивается минимально (обычно только вывод в консоль), так как на этой стадии еще недоступны механизмы конфигурации .NET и внешние сервисы.
*   **Низкий уровень:** Метод `.CreateBootstrapLogger()` создает специальный тип логгера — `ReloadableLogger`. Это прокси-объект, который позволяет безопасно подменить внутреннюю реализацию логгера "на лету" без потери текущих потоков логирования.

#### 2. Стадия полной инициализации
После того как `Host` построен и конфигурация загружена, вызывается интеграция с контейнером (через `UseSerilog` или `AddSerilog`). 
*   **Механика:** [[Serilog]] считывает финальные настройки, подключает сложные [[Sinks]] (файлы, базы данных, [[Seq]]) и **перехватывает** управление у Bootstrap-логгера.
*   Все сообщения, которые стояли в очереди или генерировались в момент переключения, корректно обрабатываются обновленным конвейером.

#### 3. Взаимодействие с [[CLR]] и потоками
`Log.Logger` является статическим глобальным ресурсом. Прямая перезапись статического поля новым объектом во время работы приложения может привести к состоянию гонки (race condition). Паттерн `ReloadableLogger` использует внутреннюю синхронизацию, гарантируя, что старые вызовы допишут данные, а новые — перейдут на новую конфигурацию.

> [!INFO] Нюанс: Очистка ресурсов
> Приложение на этапе Bootstrap владеет дескрипторами (например, консоли). При переходе на второй этап старые ресурсы не закрываются принудительно, а плавно передаются новому инстансу, что исключает ошибки доступа к выводу.

### Пример кода ([[MOC|C#]] 12)

Современная реализация в `Program.cs` с использованием `Primary Constructors` и блоков `try-catch-finally`.

```csharp
using Serilog;
using Serilog.Events;

// ШАГ 1: Bootstrap Logger
// Настраивается максимально рано, чтобы поймать ошибки конфигурации
Log.Logger = new LoggerConfiguration()
    .MinimumLevel.Override("Microsoft", LogEventLevel.Information)
    .Enrich.FromLogContext()
    .WriteTo.Console()
    .CreateBootstrapLogger();

try
{
    Log.Information("Приложение запускается (Стадия Bootstrap)...");

    var builder = WebApplication.CreateBuilder(args);

    // ШАГ 2: Полная инициализация
    // Теперь Serilog получает доступ к builder.Configuration и DI
    builder.Services.AddSerilog((services, lc) => lc
        .ReadFrom.Configuration(builder.Configuration)
        .ReadFrom.Services(services)
        .Enrich.FromLogContext()
        .WriteTo.Console()
        .WriteTo.File("logs/full-log.txt", rollingInterval: RollingInterval.Day));

    var app = builder.Build();
    
    app.UseSerilogRequestLogging(); // Оптимизация логов HTTP
    app.MapGet("/", () => "Hello World!");

    app.Run();
}
catch (Exception ex)
{
    // Фиксация фатальных ошибок, произошедших на любом этапе запуска
    Log.Fatal(ex, "Критический сбой при запуске приложения");
}
finally
{
    // Обязательный сброс буферов и закрытие логгера
    Log.CloseAndFlush();
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Прямое использование .CreateLogger() для Bootstrap
> Если вызвать `.CreateLogger()` вместо `.CreateBootstrapLogger()` на первом этапе, логгер станет неизменяемым (Frozen). При попытке интеграции с `HostBuilder` система не сможет подменить конфигурацию, и вы либо получите ошибку, либо ваше приложение продолжит использовать примитивный начальный логгер, игнорируя настройки из `appsettings.json`.

> [!WARNING] Ошибка: Пропуск блока finally
> Если не вызвать `Log.CloseAndFlush()`, асинхронные приемники (Sinks), такие как File или Seq, могут не успеть записать последние сообщения из памяти на диск перед тем, как процесс ОС будет завершен. Это приведет к потере информации о самой важной — последней — ошибке.

**Best Practices:**
1.  **Минимализм на 1 этапе:** Не подключайте сложные Sinks (например, SQL Server) в Bootstrap Logger. Если база данных будет недоступна, Bootstrap Logger сам упадет с ошибкой, и вы не узнаете причину.
2.  **ReadFrom.Services:** На втором этапе обязательно используйте `.ReadFrom.Services(services)`. Это позволяет Serilog извлекать зарегистрированные в DI `Enrichers` и другие зависимости.
3.  **Иерархия уровней:** В Bootstrap Logger устанавливайте уровень `Information` или выше, чтобы не засорять вывод до того, как будут применены фильтры из файла конфигурации.
4.  **LogContext:** Всегда включайте `.Enrich.FromLogContext()` на обоих этапах для поддержки сквозной трассировки (`CorrelationId`).

> [!TIP] Нюанс из Nutshell
> Использование паттерна двухэтапной инициализации критически важно в контейнеризированных средах (Docker/Kubernetes), так как ошибки маппинга томов конфигурации или секретов происходят до старта основного кода приложения. Только Bootstrap Logger способен сообщить оркестратору, почему контейнер ушел в `CrashLoopBackOff`.
