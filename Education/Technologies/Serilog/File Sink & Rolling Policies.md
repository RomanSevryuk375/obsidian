### Суть (The "What")
**File Sink** — это расширение [[Serilog]], предназначенное для записи структурированных логов в текстовые файлы на диске. 
**Rolling Policies (Политики ротации)** — это механизмы автоматического управления жизненным циклом файлов логов. Они определяют, когда текущий файл должен быть закрыт и создан новый (например, при достижении определенного размера или наступлении нового дня), а также когда старые файлы должны быть удалены.

Это решает проблему **неконтролируемого роста логов**: без ротации один файл может забить всё дисковое пространство сервера, а поиск в текстовом документе размером в несколько гигабайт становится технически невозможным.

### Как это работает (The "How" / Nutshell style)

#### 1. Механика записи и блокировки (Locking)
По умолчанию Serilog удерживает файл открытым для записи, используя монопольный доступ.
*   **Синхронизация:** Внутри Sink использует `lock` для обеспечения потокобезопасности, чтобы сообщения от разных потоков не перемешивались в одной строке.
*   **Shared Access:** Можно настроить `FileShare.Read`, чтобы внешние программы (например, текстовые редакторы) могли читать файл, пока приложение в него пишет.
*   **Низкий уровень (OS):** Запись в файл — это дорогая системная операция. Для оптимизации Serilog использует внутренний буфер (по умолчанию 32 КБ). Данные сбрасываются на диск либо при заполнении буфера, либо принудительно через интервал `flushToDiskInterval`.

#### 2. Триггеры ротации (Rolling Triggers)
Ротация может происходить по двум критериям:
*   **Time-based (Интервальная):** Параметр `rollingInterval`. Файлы создаются ежечасно, ежедневно, ежемесячно. К имени файла добавляется временной штамп (например, `log20240520.txt`).
*   **Size-based (По размеру):** Параметры `fileSizeLimitBytes` и `rollOnFileSizeLimit`. Как только файл достигает предела (по умолчанию 1 ГБ), создается новый с порядковым номером в имени.

#### 3. Политика удержания (Retention)
Параметр `retainedFileCountLimit` определяет, сколько файлов логов хранить. 
*   При создании нового файла Serilog сканирует директорию и удаляет самые старые файлы, выходящие за лимит. 
*   **Нюанс:** Удаление происходит по дате последнего изменения файла, определяемой файловой системой (OS).

> [!INFO] Нюанс: Асинхронная запись (Async Wrapper)
> Для высоконагруженных систем запись в файл должна быть обернута в `WriteTo.Async()`. Это переносит тяжелые I/O операции в фоновый поток и использует `Channel<T>` (или `BlockingCollection`) в памяти как буфер, предотвращая задержки в основном бизнес-потоке.

### Пример кода ([[MOC|C#]] 12)

Конфигурация File Sink с продвинутыми политиками ротации и асинхронностью.

```csharp
using Serilog;

Log.Logger = new LoggerConfiguration()
    .MinimumLevel.Debug()
    // Обертка Async предотвращает блокировку вызывающего потока дисковыми операциями
    .WriteTo.Async(a => a.File(
        path: "logs/app-log.txt",
        // Ротация каждый день
        rollingInterval: RollingInterval.Day, 
        // Создавать новый файл при достижении 10 МБ
        fileSizeLimitBytes: 10 * 1024 * 1024,
        rollOnFileSizeLimit: true,
        // Хранить только последние 31 файл
        retainedFileCountLimit: 31,
        // Оптимизация: сбрасывать буфер на диск каждые 1 сек
        buffered: true,
        flushToDiskInterval: TimeSpan.FromSeconds(1),
        // Шаблон сообщения (Semantic Logging)
        outputTemplate: "{Timestamp:yyyy-MM-dd HH:mm:ss.fff zzz} [{Level:u3}] {Message:lj}{NewLine}{Exception}"
    ))
    .CreateLogger();

try 
{
    Log.Information("Приложение запущено. Система логирования настроена.");
    // Логика...
}
finally 
{
    // Критически важно: без этого асинхронный буфер не успеет записаться при закрытии
    Log.CloseAndFlush();
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Забытый CloseAndFlush
> При использовании асинхронного логирования или `buffered: true` данные сначала попадают в RAM. Если приложение упадет или завершится без вызова `Log.CloseAndFlush()`, последние (и самые важные) строки логов будут безвозвратно потеряны.

> [!WARNING] Ошибка: Слишком мелкая ротация
> Установка `RollingInterval.Minute` или очень маленького `fileSizeLimitBytes` порождает тысячи мелких файлов. Это перегружает таблицу дескрипторов файловой системы (MFT в NTFS) и замедляет работу антивирусов и систем резервного копирования.

**Best Practices:**
1.  **Buffered Writing:** Включайте `buffered: true`, если вы пишете много логов. Это значительно снижает нагрузку на CPU за счет уменьшения количества системных вызовов `WriteFile`.
2.  **Retention Policy:** Всегда явно задавайте `retainedFileCountLimit`. По умолчанию он равен 31. Убедитесь, что это значение коррелирует с доступным местом на диске.
3.  **Hooks:** Используйте `FileLifecycleHooks` (например, из пакета `Serilog.Sinks.File.Gzip`), если вам нужно автоматически сжимать архивные логи для экономии места.
4.  **JSON для машин:** Если логи будут читаться парсерами (например, Filebeat), пишите их в формате JSON (`JsonFormatter`), а не через `outputTemplate`.

> [!TIP] Нюанс из Nutshell
> Файловая система — это разделяемый ресурс. Если несколько инстансов вашего приложения пытаются писать в один и тот же файл, используйте `FileShare.ReadWrite` или пакет `Serilog.Sinks.File.Header` для корректной координации, но лучше — разводите логи разных инстансов по разным папкам.
