### Суть (The "What")
**Message Templates** — это стандарт описания логов, который разделяет сообщение на неизменяемую текстовую часть (шаблон) и именованные аргументы (данные). 
**Semantic Logging (Семантическое логирование)** — это подход, при котором лог воспринимается не как строка текста, а как структурированное событие, имеющее тип и набор свойств.

Этот подход решает проблему "черного ящика" текстовых логов. Вместо того чтобы парсить гигабайты текста регулярными выражениями, вы получаете базу данных событий, где можно мгновенно найти все записи по условию `CartId == 500` или `ElapsedMs > 1000`.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Механика шаблонизации
В отличие от `string.Format` или интерполяции строк (`$""`), Message Templates используют именованные плейсхолдеры: `Log.Information("Processed {Count} items", count);`.
*   **Парсинг:** Библиотека логирования ([[Serilog]] или Microsoft.Extensions.Logging) анализирует строку шаблона.
*   **Захват свойств:** Каждому значению в скобках присваивается имя свойства из переданных аргументов.
*   **Хеширование:** Шаблон сообщения (Message Template) становится идентификатором типа события. Это позволяет группировать миллионы логов "Processed {Count} items" как один тип события, даже если значения `Count` всегда разные.

#### 2. Деструктуризация (The `@` Operator)
На низком уровне логгеры обычно вызывают `ToString()` у переданных объектов. Чтобы заставить логгер разобрать объект на части (сериализовать в JSON-структуру), используется оператор деструктуризации `@`.
*   `{User}` — вызовет `User.ToString()`.
*   `{@User}` — разложит объект на свойства `{ "Name": "Ivan", "Id": 1 }`.

#### 3. Отложенный рендеринг (Deferred Rendering)
Строка сообщения фактически не формируется (не склеивается) в момент вызова метода `Log`.
*   Данные сохраняются в структурированном виде (например, в `LogEvent`).
*   Если уровень лога (например, `Debug`) отключен, никакие операции со строками вообще не производятся, что дает прирост производительности.
*   Рендеринг в текстовый вид происходит только в самом конце — например, при выводе в консоль.

#### 4. Глобальный контекст ([[Enrichment & Context|Enrichment]])
Семантическое логирование позволяет "обогащать" события данными, которые не передаются в конкретный вызов. Используя `AsyncLocal<T>`, контекст (например, `CorrelationId` или `UserId`) неявно прикрепляется ко всем логам в рамках одного потока управления или асинхронной цепочки.

### Пример кода ([[MOC|C#]] 12)

```csharp
using Microsoft.Extensions.Logging;
using System.Diagnostics;

public record Order(int Id, decimal Total, string Currency);

public class OrderProcessor(ILogger<OrderProcessor> logger)
{
    private readonly ILogger<OrderProcessor> _logger = logger;

    public void Process(Order order)
    {
        var sw = Stopwatch.StartNew();

        // 1. ПЛОХО: Интерполяция строк. 
        // Данные превращаются в текст ДО логгера. Поиск по OrderId невозможен.
        _logger.LogInformation($"Order {order.Id} processed in {sw.ElapsedMilliseconds}ms");

        // 2. ХОРОШО: Message Template. 
        // Свойства OrderId и ElapsedMs сохраняются отдельно.
        _logger.LogInformation("Order {OrderId} processed in {ElapsedMs}ms", order.Id, sw.ElapsedMilliseconds);

        try
        {
            // Имитация ошибки
            if (order.Total < 0) throw new ArgumentException("Invalid amount");
        }
        catch (Exception ex)
        {
            // 3. Семантическая деструктуризация. 
            // Оператор '@' (в Serilog) сохранит весь объект заказа для анализа.
            _logger.LogError(ex, "Failed to process order {@OrderDetails}", order);
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Переменные в именах свойств
> Никогда не делайте так: `Log.Info("User {User" + id + "} logged in")`. Это приведет к тому, что каждое сообщение будет иметь уникальный шаблон. Внутренний кэш парсера шаблонов переполнится, что вызовет утечку памяти (Memory Leak) и огромную нагрузку на GC. Шаблон всегда должен быть константной строкой.

> [!WARNING] Ошибка: Путаница с типами данных в хранилище
> Если в одном месте вы залогируете `{TransactionId}` как `int`, а в другом — как `string`, системы вроде Elasticsearch могут выдать ошибку маппинга (Mapping Collision). Договоритесь о типах общих свойств заранее.

**Best Practices:**
1.  **PascalCase для свойств:** Используйте `{OrderId}` вместо `{orderId}`. Это стандарт для JSON-полей во многих приемниках (Seq, Elastic).
2.  **Семантическая уникальность:** Дайте свойствам уникальные имена. Вместо общего `{Id}` используйте `{CustomerId}` или `{RequestId}`. Это упростит фильтрацию в глобальном поиске.
3.  **LoggerMessage.Define:** Для высокопроизводительных участков кода (Hot Paths) используйте Source Generators или `LoggerMessage.Define`, чтобы избежать парсинга шаблона при каждом вызове.
4.  **Exceptions:** Всегда передавайте объект исключения первым аргументом. Не вставляйте `ex.Message` в строку шаблона — это разрушает структуру данных.

> [!TIP] Нюанс из Nutshell
> При использовании семантического логирования объем данных логов может вырасти в 5-10 раз из-за хранения метаданных и JSON-структур. Настраивайте политики ротации файлов и очистки индексов в хранилищах заранее.
