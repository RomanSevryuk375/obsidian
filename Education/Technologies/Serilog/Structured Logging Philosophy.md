### Суть (The "What")
**Структурированное логирование** — это подход к ведению журналов, при котором запись рассматривается не как плоский текст (строка), а как набор данных в формате «ключ-значение» с привязкой к шаблону сообщения. 

Традиционное логирование заставляет программиста писать сообщения для людей (`log.Info("User 123 logged in")`). Структурированное логирование заставляет думать о машинной обработке (`log.Info("User {UserId} logged in", 123)`). Это решает проблему **нечитаемости логов для систем анализа**: вместо написания сложных регулярных выражений (Regex) для парсинга текстовых файлов, вы получаете чистый JSON, который можно мгновенно фильтровать по конкретным полям в системах вроде [[Elasticsearch]] или [[Seq]].

### Как это работает (The "How" / Nutshell style)

#### 1. Шаблоны сообщений ([[Message Templates & Semantic Logging|Message Templates]])
В основе лежит стандарт **MessageTemplate.NET**. Когда вы вызываете метод лога, происходит не интерполяция строки, а разделение на:
*   **Message Template:** неизменяемая строка с плейсхолдерами.
*   **Properties:** именованный список аргументов.

На низком уровне это позволяет системам агрегации группировать миллионы записей по хешу шаблона, несмотря на то, что значения аргументов у них разные.

#### 2. Деструктуризация ([[Destructuring & Serialization|Destructuring]])
В отличие от классического `ToString()`, структурированные логгеры (например, [[Serilog]]) умеют выполнять **глубокий обход объекта** (Object Graph) с помощью рефлексии. 
*   Оператор `@` (деструктуризация) заставляет логгер превратить объект в иерархическую структуру JSON.
*   Это позволяет сохранить состояние системы в момент ошибки не в виде "обрывка строки", а в виде полноценного снимка данных.

#### 3. Обогащение контекстом ([[Enrichment & Context|Enrichment]])
Благодаря использованию `AsyncLocal<T>`, структурированное логирование позволяет добавлять свойства к логам "сбоку", не передавая их в каждый вызов. 
*   **CorrelationId:** позволяет связать десятки логов из разных микросервисов в одну цепочку одного пользовательского запроса.
*   **Environment:** автоматическое добавление версии приложения, имени сервера или ID потока.

#### 4. Производительность и ленивые вычисления
Современные библиотеки логирования сначала проверяют уровень важности (`LogLevel`). Если уровень `Debug` отключен, то парсинг шаблона и деструктуризация объектов **не производятся**, что сводит накладные расходы к проверке одного `bool` флага.

### Пример кода ([[MOC|C#]] 12)

```csharp
using Microsoft.Extensions.Logging;
using System.Diagnostics;

// Использование Primary Constructor (C# 12)
public class OrderProcessor(ILogger<OrderProcessor> logger)
{
    private readonly ILogger<OrderProcessor> _logger = logger;

    public void Process(Order order)
    {
        // 1. Плохо: Интерполяция строк (Убивает структуру)
        // В логах будет: "Processing order 42" - это просто текст.
        _logger.LogInformation($"Processing order {order.Id}");

        // 2. Хорошо: Структурированный подход
        // В логах будет: { "Template": "Processing order {OrderId}", "OrderId": 42 }
        _logger.LogInformation("Processing order {OrderId}", order.Id);

        try
        {
            // Имитация логики
            if (order.Amount > 1000) 
                throw new InvalidOperationException("Too expensive");
        }
        catch (Exception ex)
        {
            // 3. Деструктуризация объекта (Serilog style @)
            // Мы сохраняем ВЕСЬ объект заказа в JSON для анализа ошибки
            _logger.LogError(ex, "Error processing order {@OrderDetails}", order);
        }
    }
}

public record Order(int Id, decimal Amount);
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Интерполяция строк (The $ Trap)
> Никогда не используйте `$` внутри методов логирования: `_logger.LogInfo($"User {id}...")`. Это превращает данные в строку до того, как логгер их увидит. Вы теряете возможность поиска по полю `id` в базе логов, а также "засоряете" кэш шаблонов логгера уникальными строками, что может привести к утечке памяти.

> [!WARNING] Ошибка: Переменные имена свойств
> Имена внутри скобок `{Property}` должны быть константными и осмысленными. Избегайте использования индексов `{0}`. Используйте семантические имена, например `{TransactionId}` или `{CustomerId}`, чтобы во всех сервисах компании одно и то же понятие называлось одинаково.

**Best Practices:**
1.  **High-performance Logging:** В критических секциях (миллионы вызовов) используйте `LoggerMessage.Define` или Source Generators (`[LoggerMessage]`). Это исключает аллокации на парсинг шаблона в рантайме.
2.  **Naming Convention:** Используйте PascalCase для имен свойств в шаблонах (например, `{OrderId}`), чтобы они соответствовали стандартам JSON-полей.
3.  **Log Context:** Используйте `ILogger.BeginScope` для добавления свойств (например, `RequestID`) ко всем последующим логам в рамках текущего метода или запроса.
4.  **Exceptions:** Всегда передавайте объект исключения первым аргументом в `LogError`. Не вставляйте `ex.Message` в строку шаблона — логгеры сами извлекут StackTrace и детали.

> [!TIP] Нюанс: Уровни логов
> `Information` — это бизнес-события. `Debug` — детали для разработчика. Никогда не логгируйте персональные данные (пароли, номера карт) даже на уровне `Trace`, так как логи часто хранятся в открытом виде в системах агрегации.
