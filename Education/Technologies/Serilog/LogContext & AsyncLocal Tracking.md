### Суть (The "What")
**LogContext** — это механизм динамического обогащения логов метаданными, которые привязываются к текущему потоку выполнения или асинхронной цепочке. 
**AsyncLocal\<T>** — это низкоуровневый примитив .NET, позволяющий хранить данные в локальном контексте потока, которые "перетекают" вместе с выполнением кода через точки `await`.

Эта связка решает задачу **сквозной трассировки (Correlation)**. Она позволяет добавить информацию (например, `TransactionId` или `UserId`) в одной точке (например, в Middleware) и автоматически прикреплять её ко всем логам, созданным глубоко в бизнес-логике, без необходимости явной передачи этих данных через параметры методов.

### Как это работает (The "How" / Nutshell style)

#### 1. Механика AsyncLocal\<T>
В основе `LogContext` лежит системный класс `System.Threading.AsyncLocal<T>`. 
*   **ExecutionContext:** Когда среда выполнения ([[CLR]]) переключает управление между потоками (например, при завершении задачи и возврате в `TaskScheduler`), она копирует структуру `ExecutionContext`.
*   **Copy-on-Write:** Данные в `AsyncLocal` ведут себя по принципу "копирования при записи". Если дочерний поток или задача изменяет значение, это не влияет на родительский контекст, но дочерний контекст видит все значения родителя.

#### 2. Стек свойств LogContext
`LogContext` реализует структуру данных типа **Stack** (LIFO) внутри `AsyncLocal`. 
*   Метод `LogContext.PushProperty("Key", "Value")` возвращает объект `IDisposable`.
*   При вызове `PushProperty` новое свойство добавляется на вершину стека контекста.
*   При вызове `.Dispose()` (обычно через блок `using`) свойство удаляется, восстанавливая предыдущее состояние контекста.

#### 3. Жизненный цикл в CLR
При вызове `_logger.LogInformation(...)` [[Serilog]] обращается к текущему значению в `AsyncLocal`. Он извлекает все накопленные в стеке свойства и добавляет их в `LogEvent`. Поскольку `AsyncLocal` привязан к логической цепочке вызовов, это работает корректно даже при параллельном выполнении тысяч независимых запросов в одном процессе.

> [!INFO] Нюанс: Производительность
> Каждый `PushProperty` создает аллокацию объекта `IDisposable` и обновляет структуру данных в `AsyncLocal`. В высоконагруженных циклах (миллионы операций в секунду) чрезмерное использование `LogContext` может привести к заметному давлению на 0-е поколение сборщика мусора ([[GC]] Gen 0).

### Пример кода ([[MOC|C#]] 12)

Пример имитации обработки заказа, где контекст пробрасывается через асинхронные вызовы.

```csharp
using Serilog;
using Serilog.Context;

// Настройка Serilog (Enrich.FromLogContext обязателен!)
Log.Logger = new LoggerConfiguration()
    .Enrich.FromLogContext()
    .WriteTo.Console(outputTemplate: "{Timestamp:HH:mm} [{CorrelationId}] {Message}{NewLine}")
    .CreateLogger();

public class OrderProcessor(ILogger logger) // Primary constructor
{
    public async Task ProcessAsync(Guid orderId)
    {
        // 1. Помещаем CorrelationId в контекст асинхронной цепочки
        using (LogContext.PushProperty("CorrelationId", orderId))
        {
            logger.LogInformation("Начало обработки заказа");

            await StepOneAsync();
            await StepTwoAsync();

            logger.LogInformation("Заказ успешно обработан");
        }
    }

    private async Task StepOneAsync()
    {
        await Task.Delay(100); // Имитация I/O
        // Контекст CorrelationId сохраняется здесь автоматически
        logger.LogInformation("Шаг 1 выполнен");
    }

    private async Task StepTwoAsync()
    {
        // Можно вложить новый контекст
        using (LogContext.PushProperty("SubStep", "Validation"))
        {
            await Task.Delay(50);
            logger.LogInformation("Шаг 2 выполнен");
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Забытый Dispose
> Если не обернуть `PushProperty` в блок `using`, свойство останется в контексте потока даже после выхода из метода. Если поток вернется в пул (Thread Pool) и будет переиспользован для другой задачи, в её логи могут попасть "чужие" данные, что полностью дезинформирует при анализе инцидентов.

> [!WARNING] Ошибка: Доступ к LogContext из другого потока
> `LogContext` (через `AsyncLocal`) не передает данные в потоки, созданные вручную через `new Thread().Start()` или через нативные вызовы, если не была выполнена явная инициализация контекста. Используйте `Task.Run()`, который корректно захватывает и передает `ExecutionContext`.

**Best Practices:**
1.  **Enrich.FromLogContext():** Всегда проверяйте, что этот метод вызван при настройке `LoggerConfiguration`, иначе данные из `LogContext.PushProperty` будут игнорироваться.
2.  **Использование в [[Middleware]]:** В ASP.NET Core приложениях это идеальное место для установки `TraceId` или `UserId` для всего HTTP-запроса.
3.  **Имена свойств:** Используйте константные строки для ключей свойств (например, `nameof(orderId)`), чтобы избежать опечаток и облегчить поиск в системах агрегации ([[Seq]], [[Elasticsearch]]).
4.  **Короткие области видимости:** Держите блоки `using (LogContext.PushProperty(...))` как можно более компактными, чтобы минимизировать риск "загрязнения" логов лишними данными.

> [!TIP] Нюанс для Senior-разработчиков
> Если вам нужно передать контекст в фоновую задачу (например, `Fire-and-forget`), помните, что `LogContext` будет очищен сразу после выхода из родительского метода (когда сработает `Dispose`). Если фоновая задача должна сохранить контекст, значения нужно захватить явно и пересоздать `PushProperty` уже внутри фонового потока.
