Команда переносит изменения из рабочей директории (Working Directory) в **Index** (Staging Area), подготавливая "снэпшот" для следующего [[git commit]].

---

## Шпаргалка

```bash
# 1. Добавить конкретный файл (или несколько)
git add README.md src/main.js

# 2. Добавить ВСЕ изменения в текущей папке и подпапках
# Включает новые (untracked), измененные и удаленные файлы
git add .

# 3. Добавить только уже отслеживаемые файлы
# Обновляет изменения в известных файлах, но игнорирует новые (Untracked)
git add -u

# 4. Интерактивное добавление (Patch Mode) — PRO уровень
# Позволяет добавить в индекс не весь файл, а отдельные куски кода (hunks).
# Полезно для разделения логических изменений внутри одного файла.
git add -p
```

## Важные нюансы
*   **Двухэтапность:** Git не коммитит изменения сразу. `git add` позволяет собрать точный состав коммита (например, разбить работу на атомарные части), даже если вы изменили 10 файлов.
*   **Снэпшот на момент добавления:** Если вы сделали `git add file.txt`, а затем снова изменили этот файл, то в коммит попадет *версия на момент выполнения команды add*. Вторую правку нужно добавить снова.
*   **Игнорирование:** Если файл попадает под правила [[.gitignore]], `git add .` его пропустит (если не использовать флаг `-f`).

## Связанные темы
*   [[git status]] — проверка того, что попало в Staging Area (зеленое) и что нет (красное).
*   [[git commit]] — фиксация того, что было добавлено.
*   [[git restore]] — удаление файла из Staging Area (операция unstage).
*   [[git diff]] — просмотр изменений перед добавлением (`--staged` покажет разницу между индексом и последним коммитом).
