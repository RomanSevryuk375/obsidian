Стандарт оформления сообщений коммитов (формат `<type>[scope]: <description>`), позволяющий автоматизировать семантическое версионирование (SemVer) и генерацию Changelog, делая историю понятной для людей и машин.

---
## Структура
`тип(область): описание`

### Основные типы
1.  **feat:** (Feature) Новый функционал. Триггерит `MINOR` релиз.
2.  **fix:** Исправление бага. Триггерит `PATCH` релиз.
3.  **BREAKING CHANGE:** (или `feat!:`) Ломающие изменения. Триггерит `MAJOR` релиз.

### Вспомогательные типы (обычно не триггерят релиз)
*   **docs:** Документация.
*   **style:** Форматирование, пробелы, точки с запятой (без изменения логики).
*   **refactor:** Правка кода без фиксов или новых фич.
*   **perf:** Улучшение производительности.
*   **test:** Добавление или правка тестов.
*   **chore:** Обслуживание сборки, конфигов, зависимостей.

## Пример использования

```bash
# 1. Добавление новой фичи (Minor update: 1.1.0 -> 1.2.0)
git commit -m "feat(auth): добавить вход через Google OAuth"

# 2. Исправление ошибки (Patch update: 1.2.0 -> 1.2.1)
git commit -m "fix(cart): исправить расчет суммы при пустой корзине"

# 3. Ломающее изменение (Major update: 1.2.1 -> 2.0.0)
# Обратите внимание на восклицательный знак "!" перед двоеточием
git commit -m "feat!: удалить поддержку Node.js 14"

# 4. Технические задачи (No release / Hidden)
git commit -m "chore(deps): обновить React до версии 18"
git commit -m "docs: добавить схему архитектуры в README"
```

## Инструменты
Для обеспечения соблюдения стандарта часто используют **Husky** + **Commitlint** (в JS экосистеме) или **pre-commit** хуки.

## Связанные темы
*   [[git commit]] — команда, к которой применяется этот стандарт.
*   [[git log]] — чтение структурированной истории.
*   [[git tag]] — создание релизов на основе истории коммитов.
*   [[Git Hooks]] — автоматическая проверка формата сообщения перед коммитом.
