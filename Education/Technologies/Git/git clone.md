Команда скачивает полную копию удаленного репозитория (код + история изменений) на локальную машину и автоматически создает ссылку `origin` на источник.

---

## Протоколы
Выбор URL зависит от метода доступа (SSH ключи и аутентификация):
*   **[[HTTPS]]:** `https://github.com/user/repo.git` (просто, но требует ввод токена при push).
*   **[[SSH]]:** `git@github.com:user/repo.git` (стандарт индустрии, безопасный, без паролей).

## Шпаргалка

```bash
# 1. Базовое клонирование
# Создает директорию с именем репозитория в текущей папке
git clone git@github.com:company/backend-api.git

# 2. Клонирование в папку с другим именем
git clone git@github.com:company/backend-api.git my-api

# 3. Клонирование конкретной ветки
# Полезно, чтобы сразу работать в нужной ветке, не переключаясь
git clone -b develop git@github.com:company/backend-api.git

# 4. Shallow Clone (DevOps Optimization)
# Скачивает только последний коммит (history depth = 1).
# Используется в CI/CD пайплайнах для ускорения загрузки.
git clone --depth 1 git@github.com:company/backend-api.git

# 5. Рекурсивное клонирование
# Автоматически инициализирует и обновляет подмодули
git clone --recursive git@github.com:company/complex-project.git
```

## Что происходит "под капотом"
Команда `clone` — это обертка над серией команд. Она последовательно выполняет:
1.  `git init` (создает новый репозиторий).
2.  `git remote add origin <url>` (настраивает связь с сервером).
3.  [[git fetch]] (скачивает все объекты и ссылки).
4.  [[git checkout]] (переключает HEAD на основную ветку).

## Связанные темы
*   [[git init]] — создание репозитория с нуля (если нечего клонировать).
*   [[git remote]] — управление удаленными источниками.
*   [[git submodule]] — работа с вложенными репозиториями.
*   SSH ключи и аутентификация — настройка доступа для клонирования по [[SSH]].
