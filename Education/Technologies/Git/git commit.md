Команда берет подготовленные файлы из [[git add|Staging Area]] и сохраняет их как неизменяемый снимок (snapshot) в истории репозитория, присваивая уникальный хеш (SHA-1) и обновляя указатель [[HEAD и указатели|HEAD]].

## Шпаргалка

```bash
# 1. Стандартный коммит с сообщением
# Фиксирует только то, что было добавлено через git add
git commit -m "feat: добавить валидацию email"

# 2. Коммит "все включено" (Skip Staging)
# Автоматически делает git add для отслеживаемых файлов.
# ВНИМАНИЕ: Не добавляет новые (untracked) файлы!
git commit -am "fix: исправить опечатку в конфиге"

# 3. Пустой коммит
# Полезно для триггера CI/CD пайплайнов без изменения кода
git commit --allow-empty -m "ci: пересоздать docker образ"

# 4. Исправление последнего коммита (--amend)
# Используется, если забыли добавить файл или ошиблись в сообщении.
# ВАЖНО: Это переписывает историю (меняет хеш SHA-1)!
git add forgotten_file.js
git commit --amend --no-edit  # Добавить файл, не меняя сообщение
git commit --amend -m "feat: новое правильное сообщение"
```

> [!WARNING] Опасная зона
> Никогда не используйте `--amend` для коммитов, которые вы уже отправили в удаленный репозиторий (через `git push`). Это вызовет рассинхронизацию истории у коллег.

## Анатомия хорошего коммита
1.  **Атомарность:** Один логический шаг — один коммит.
2.  **Заголовок:** До 50 символов, повелительное наклонение ("Add login", не "Added login").
3.  **Тело:** Ответ на вопрос "почему" и "зачем" (контекст изменений).
