### Суть (The "What")
**Dependency Injection (DI)** — это паттерн проектирования, который является практической реализацией принципа инверсии управления ([[Inversion of Control]], IoC). Вместо того чтобы класс сам создавал свои зависимости (например, через `new DatabaseContext()`), они поставляются ("внедряются") ему извне внешним управляющим механизмом.

DI решает задачу **слабой связанности (loose coupling)**. Благодаря отделению создания объектов от их использования, код становится тестируемым (зависимости легко подменить на моки), гибким и готовым к изменениям реализации без изменения бизнес-логики.

### Как это работает (The "How" / Nutshell style)

#### 1. Встроенный [[Inversion of Control|IoC]]-контейнер .NET
Платформа .NET имеет встроенный легковесный контейнер внедрения зависимостей, представленный интерфейсами `IServiceCollection` (для регистрации) и `IServiceProvider` (для разрешения/получения зависимостей).

#### 2. Жизненные циклы служб ([[Service Lifetimes]])
Контейнер берет на себя управление жизненным циклом создаваемых объектов. В .NET доступны три варианта:
*   **Transient (Временный):** Создается новый экземпляр при каждом запросе из контейнера. Подходит для легковесных служб без состояния (`stateless`).
*   **Scoped (В рамках области):** Создается один экземпляр на каждую "область" (Scope). В веб-приложениях областью является один HTTP-запрос. Идеально для работы с контекстом БД (`DbContext`) или транзакциями.
*   **Singleton (Одиночка):** Создается один раз при первом обращении (или при старте) и живет все время работы приложения. Подходит для кэшей, конфигураций и потокобезопасных утилит.

#### 3. Внедрение через конструктор (Constructor Injection)
Основной и самый безопасный способ внедрения. Контейнер анализирует конструктор класса, находит все требуемые интерфейсы, автоматически создает их экземпляры (если они зарегистрированы) и передает их в конструктор.

> [!DANGER] Плененная зависимость (Captive Dependency)
> Это критическая ошибка проектирования. Она возникает, когда долгоживущая служба (например, Singleton) принимает в конструктор короткоживущую службу (например, Scoped). В этом случае Scoped-объект будет удерживаться в памяти вечно, что может привести к утечкам памяти или ошибкам параллельного доступа (например, у `DbContext`). В режиме `Development` .NET автоматически проверяет это и выбрасывает исключение при старте (`ValidateScopes = true`).

### Структура папок и пример реализации

Рекомендуемая организация сервисного слоя в проекте:

```text
MyProject.API/
├── Services/
│   ├── IOrderService.cs         # Интерфейс (Контракт)
│   └── OrderService.cs          # Реализация
└── Program.cs                   # Точка входа и регистрация зависимостей
```

#### 1. Объявление сервиса с первичным конструктором ([[C#]] 12)
Используем лаконичный синтаксис C# 12 для автоматического внедрения зависимостей без ручного объявления полей:

```csharp
using System;
using System.Threading.Tasks;

namespace MyProject.API.Services;

// Интерфейс
public interface IOrderService
{
    Task CreateOrderAsync(Guid productId);
}

// Реализация с первичным конструктором.
// Контейнер автоматически внедрит Scoped-репозиторий и Singleton-логгер.
public class OrderService(IOrderRepository repository, ILogger<OrderService> logger) : IOrderService
{
    public async Task CreateOrderAsync(Guid productId)
    {
        logger.LogInformation("Processing order for product: {ProductId}", productId);
        await repository.SaveOrderAsync(productId);
    }
}

// Заглушки для компиляции
public interface IOrderRepository { Task SaveOrderAsync(Guid id); }
public interface ILogger<T> { void LogInformation(string m, object p); }
```

#### 2. Регистрация в Program.cs
```csharp
// Program.cs
var builder = WebApplication.CreateBuilder(args);

// Регистрация сервисов с правильным временем жизни
builder.Services.AddSingleton<ILogger<OrderService>, ConsoleLogger<OrderService>>(); // Одиночка
builder.Services.AddScoped<IOrderRepository, SqlOrderRepository>();                 // Scoped (БД)
builder.Services.AddTransient<IOrderService, OrderService>();                       // Временный

var app = builder.Build();
app.Run();
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Service Locator
> Избегайте передачи `IServiceProvider` напрямую в конструктор класса для извлечения зависимостей "на лету" (например, `provider.GetService<IOrderService>()`). Это превращает DI в паттерн Service Locator. Он скрывает реальные зависимости класса, делает код непрозрачным и усложняет написание Unit-тестов. Всегда запрашивайте зависимости явно в конструкторе.

> [!WARNING] Ошибка: Циклические зависимости (Circular Dependency)
> Ситуация, когда Классу А для создания нужен Класс Б, а Классу Б нужен Класс А. Контейнер не сможет разрешить эту цепочку и выбросит исключение при попытке создания. Решение: вынести общую логику в третий класс или использовать интерфейсы более высокого уровня абстракции.

**Best Practices:**
1.  **Зависимость от интерфейсов:** Регистрируйте и запрашивайте интерфейсы (`IOrderService`), а не конкретные классы (`OrderService`). Это основа принципа инверсии зависимостей ([[Dependency Inversion Principle|DIP]]).
2.  **Разделение конфигурации:** Если зависимостей много, выносите их регистрацию в методы расширения (например, `builder.Services.AddApplicationServices()`), чтобы не раздувать `Program.cs`.
3.  **Не делайте утилиты Scoped:** Если класс не хранит состояние и не работает с БД (например, хелпер для шифрования строк), регистрируйте его как `Singleton` или `Transient`, чтобы не плодить лишние объекты при каждом [[HTTP]]-запросе.
4.  **Dispose в DI:** Контейнер автоматически вызывает `Dispose()` для всех созданных им объектов, реализующих `IDisposable`, в конце их жизненного цикла (например, в конце Scope). Не вызывайте `Dispose()` для внедренных зависимостей вручную.

> [!TIP] Нюанс: [FromServices] в ASP.NET Core
> Если зависимость нужна только в одном конкретном методе контроллера или Minimal API, и ее нецелесообразно внедрять через конструктор (например, тяжелый сервис, используемый в редком эндпоинте), вы можете внедрить его непосредственно в параметры метода с помощью атрибута `[FromServices]`.
