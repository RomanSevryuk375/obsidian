# Message Brokers: RabbitMQ & Apache Kafka

> [!abstract] Определение
> **Брокер сообщений** — это промежуточное ПО (middleware), которое позволяет микросервисам обмениваться данными асинхронно, обеспечивая слабую связность (loose coupling) и отказоустойчивость системы.

## 1. Основные концепции

### Очередь (Queue) vs Поток (Stream)
- **Очередь (Queue)**: Работает по принципу "отправил и забыл". Как только потребитель подтверждает получение (ACK), сообщение удаляется из брокера. 
    - *Аналогия:* Почтовый ящик.
- **Поток (Stream)**: Это неизменяемый лог событий (Append-only log). Сообщения не удаляются после прочтения, они хранятся заданное время (retention period). Потребитель сам управляет своим указателем (offset) в потоке.
    - *Аналогия:* Видеокассета, которую можно перематывать и пересматривать.

## 2. Сравнение RabbitMQ vs Kafka

### Архитектурная разница
| Характеристика | RabbitMQ | Apache Kafka |
| :--- | :--- | :--- |
| **Философия** | **Smart Broker / Dumb Consumer**. Брокер следит за состоянием очередей и доставкой. | **Dumb Broker / Smart Consumer**. Брокер просто хранит байты, потребитель сам следит за тем, что он прочитал. |
| **Маршрутизация** | Богатая (Exchanges: Direct, Topic, Fanout, Headers). | Простая (через ключи партиционирования в топиках). |
| **Масштабирование** | Вертикальное + Кластеризация (сложнее масштабировать на терабайты). | Горизонтальное (легко масштабируется добавлением брокеров/партиций). |

### Когда выбрать RabbitMQ?
- Нужна **сложная маршрутизация** сообщений.
- Требуются **приоритеты** сообщений.
- Гарантированная доставка и подтверждение на уровне каждого сообщения.
- Небольшие или средние нагрузки, где важна простота разработки.

### Когда выбрать Kafka?
- **Огромные объемы данных** (миллионы сообщений в секунду).
- Нужно **хранение истории событий** (Event Sourcing).
- **Аналитика в реальном времени** и Stream Processing.
- Возможность "переповтора" событий за прошлый период (Replay).

## 3. Интеграция в .NET

### Роль [[MassTransit]]
Использовать нативные SDK (RabbitMQ.Client или Confluent.Kafka) в микросервисах — это "велосипедостроение". 
**MassTransit** — это абстракция (Bus), которая:
1. Позволяет переключаться между брокерами ([[RabbitMQ]]/[[Kafka]]/Azure SB) сменой одной строки конфига.
2. Реализует паттерны "из коробки" (Retry, Circuit Breaker, Outbox).
3. Автоматически создает очереди и обменники.

### Пример кода (MassTransit + RabbitMQ)

```csharp
// 1. Определение сообщения (Contract)
public record OrderCreated(Guid OrderId, decimal Amount);

// 2. Consumer (Логика обработки)
public class OrderCreatedConsumer : IConsumer<OrderCreated>
{
    public async Task Consume(ConsumeContext<OrderCreated> context)
    {
        var message = context.Message;
        // Логика обработки...
        await Task.CompletedTask;
    }
}

// 3. Отправка сообщения из контроллера/сервиса
public class OrderService(IPublishEndpoint publishEndpoint)
{
    public async Task CreateOrder()
    {
        await publishEndpoint.Publish(new OrderCreated(Guid.NewGuid(), 100.0m));
    }
}
```

## 4. Ключевые паттерны

### Retry Logic (Повторные попытки)
В распределенных системах ошибки неизбежны (сеть моргнула, БД недоступна). 
- **Immediate Retry**: Попробовать сразу же.
- **Exponential Backoff**: Увеличивать интервал между попытками (1с, 5с, 30с...).
- *В MassTransit конфигурируется одной строкой:* `endpoint.UseMessageRetry(r => r.Exponential(5, 1s, 30s, 5s));`

### Dead Letter Queue (DLQ)
Если сообщение не удалось обработать после всех попыток (например, из-за ошибки в коде или битых данных), оно перемещается в **DLQ**.
- **Зачем?** Чтобы не блокировать очередь "ядовитыми сообщениями" (Poison Messages).
- **Что делать?** Архитектор/DevOps анализирует DLQ, исправляет баг и переотправляет (Redeliver) сообщения в основную очередь.
