### Суть (The "What")
**Захватывание переменных (Capturing)** — это механизм, при котором [[Лямбда-выражения|лямбда-выражение]] или локальная функция обращается к переменным, объявленным во внешней области видимости (за пределами самой функции). Комбинация функции и захваченных ею переменных называется **замыканием (closure)**.

Этот механизм решает задачу передачи контекста в [[Делегаты|делегаты]] и функции обратного вызова, позволяя писать гибкий код, который "помнит" данные из окружения, где он был создан.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Генерация скрытого класса (DisplayClass)
Когда лямбда захватывает переменную, компилятор [[MOC#/MOC|C#]] выполняет "преобразование замыкания":
*   Создается скрытый класс (часто с именем `<>c__DisplayClass`), который служит контейнером для захваченных переменных.
*   Локальные переменные, которые должны "жить" дольше своего родительского метода, превращаются в **поля** этого скрытого класса.
*   Лямбда-выражение становится **методом** экземпляра этого класса.

#### 2. Захват по ссылке (Reference Capture)
Важный нюанс: лямбда захватывает не *значение* переменной на момент создания, а **ссылку (доступ)** к самой переменной. Если значение переменной изменится во внешнем методе *после* создания лямбды, лямбда при своем выполнении увидит это новое значение. Это называется поздним связыванием (late binding).

#### 3. Влияние на память
Переменные, которые в обычном методе находились бы в стеке (и удалились бы по завершении метода), из-за захвата переносятся в **управляемую кучу (Heap)**. Это продлевает их время жизни до тех пор, пока делегат с замыканием не станет недоступным для сборщика мусора ([[GC]]).

> [!INFO] Нюанс: Статические лямбды (C# 9+)
> Если лямбда помечена как `static`, она не может захватывать переменные извне. Это заставляет разработчика передавать данные явно через аргументы, что предотвращает создание скрытых классов и лишние аллокации в куче.

### Пример кода (C# 12)

```csharp
using System;

public class ClosureDemo
{
    public void Execute()
    {
        int factor = 10; // Локальная переменная
        
        // Лямбда захватывает 'factor'
        Action print = () => Console.WriteLine($"Factor: {factor}");
        
        // Изменяем переменную после определения лямбды
        factor = 20;
        
        // Вывод будет 20, так как захвачена переменная, а не значение
        print(); 

        // Использование статической лямбды для предотвращения захвата
        // Этот код не скомпилируется, если мы попытаемся захватить 'factor'
        Action printSafe = static () => Console.WriteLine("Static lambda cannot capture");
        printSafe();
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Утечки памяти (Memory Leaks)
> Поскольку замыкание удерживает ссылку на скрытый класс, а скрытый класс удерживает ссылку на захваченную переменную, лямбда может "держать в живых" тяжелые объекты гораздо дольше, чем нужно. Если в лямбде используется ссылка на `this`, будет удерживаться весь экземпляр класса.

> [!WARNING] Ошибка: Захват переменных цикла
> До C# 5.0 захват переменной `i` в цикле `for` приводил к тому, что все лямбды видели одно и то же последнее значение `i`. В современном C# это поведение исправлено для `foreach` (каждая итерация создает новую переменную), но в `for` (классическом) переменная все еще одна на весь цикл.
> **Решение:** Объявляйте временную переменную внутри цикла: `var temp = i;`.

**Best Practices:**
1.  **Static Lambdas:** Всегда используйте `static` для лямбд, если они не требуют захвата контекста. Это гарантирует отсутствие скрытых аллокаций.
2.  **Минимизируйте захват:** Если нужно передать в лямбду только значение переменной, а не ссылку на неё, передавайте копию через аргумент или локальную константу, чтобы "отвязать" лямбду от изменения внешнего состояния.
3.  **Анализируйте область:** Если замыкание используется в методах, вызываемых миллионы раз (например, обработка изображений или сетевых пакетов), ручная передача аргументов всегда будет быстрее, чем создание скрытого `DisplayClass` и доступ к его полям.
4.  **Scope control:** Будьте особенно внимательны при захвате `this`. Это приводит к тому, что объект не может быть очищен GC, пока делегат жив.

> [!TIP] Нюанс: Local Functions vs Lambdas
> Локальные функции (методы внутри методов) в некоторых случаях более эффективны, так как позволяют компилятору точнее управлять захватом переменных (иногда используя `ref` для передачи контекста), что уменьшает количество аллокаций в куче по сравнению с делегатами.
