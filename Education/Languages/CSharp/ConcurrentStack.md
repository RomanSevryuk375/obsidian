### Суть (The "What")
`ConcurrentStack<T>` — это высокопроизводительная, потокобезопасная коллекция, реализующая структуру данных **LIFO (Last-In, First-Out — «последним пришел, первым ушел»)**. Она позволяет множеству потоков параллельно и безопасно добавлять элементы на вершину стека и извлекать их оттуда.

Она решает задачу **многопоточного LIFO-управления задачами или ресурсами**. Это лучший выбор для алгоритмов параллельного обхода в глубину (DFS), систем отмены транзакций в многопоточной среде, а также для пулов ресурсов, где предпочтительно переиспользовать объект, который использовался последним (для улучшения локальности кэша процессора).

---

### Как это работает (The "How" / Nutshell style)

#### 1. Алгоритм стека Трайбера (Lock-Free Treiber Stack)
В отличие от [[ConcurrentQueue]]`<T>`, которая построена на сегментированных массивах, `ConcurrentStack<T>` внутри реализован как **односвязный список узлов** (singly-linked list):
*   Каждый элемент оборачивается во внутренний класс `Node`, содержащий значение и ссылку на следующий узел `Next`.
*   Коллекция хранит единственный указатель на вершину стека — `m_head`.
*   **Механика CAS (Compare-And-Swap):** Операции добавления (`Push`) и извлечения (`TryPop`) полностью свободны от блокировок ядра (lock-free). Они используют атомарный метод `Interlocked.CompareExchange`, чтобы безопасно обновить указатель `m_head`. Если во время обновления другой поток успел изменить вершину, текущий поток просто повторяет попытку в сверхбыстром цикле ([[Spin-lock|spinning]]).

#### 2. Оптимизация пакетных операций (PushRange / TryPopRange)
Это важнейшая архитектурная особенность `ConcurrentStack<T>`, выделяющая его на фоне других конкурентных коллекций:
*   При добавлении элементов по одному под высокой нагрузкой потоки могут постоянно конфликтовать за обновление указателя `m_head` (высокая конкуренция/contention).
*   **Пакетные операции:** Методы `PushRange(T[] items)` и `TryPopRange(T[] items)` позволяют добавить или извлечь целый массив элементов **за одну атомарную операцию CAS**. На уровне кода создается локальная цепочка узлов, которая затем привязывается к `m_head` одним движением указателя. Это радикально снижает конкуренцию потоков.

#### 3. Стоимость Count
*   **Сложность:** Как и в случае со связными списками, вычисление свойства `Count` имеет сложность **$O(n)$**. Рантайму приходится проходить по всей цепочке узлов от вершины до основания, чтобы посчитать их количество.

### Пример кода ([[C#]]12)

Демонстрация базовой работы и эффективного использования пакетных операций.

```csharp
using System;
using System.Collections.Concurrent;
using System.Threading.Tasks;

public class ConcurrentStackDemo
{
    private static readonly ConcurrentStack<string> _actionHistory = new();

    public static async Task Main()
    {
        // 1. Пакетное добавление элементов (высокая производительность)
        string[] initialBatch = ["Action_1", "Action_2", "Action_3"];
        _actionHistory.PushRange(initialBatch); // Добавлено атомарно за один CAS-цикл

        // 2. Параллельный запуск задач
        Task pushTask = Task.Run(() =>
        {
            for (int i = 4; i <= 6; i++)
            {
                _actionHistory.Push($"Action_{i}");
                Thread.Sleep(50);
            }
        });

        Task popTask = Task.Run(() =>
        {
            int poppedCount = 0;
            while (poppedCount < 6)
            {
                // Извлечение всегда условное
                if (_actionHistory.TryPop(out string? action))
                {
                    Console.WriteLine($"Отменено (Pop): {action}");
                    poppedCount++;
                }
                else
                {
                    Thread.Sleep(10); // Ждем, если стек временно пуст
                }
            }
        });

        await Task.WhenAll(pushTask, popTask);

        // 3. Пример пакетного извлечения
        _actionHistory.PushRange(["New_1", "New_2", "New_3"]);
        string[] buffer = new string[2];
        
        // Пытаемся забрать сразу 2 элемента
        int popped = _actionHistory.TryPopRange(buffer);
        Console.WriteLine($"Успешно извлечено пакетом: {popped} элементов."); // 2
        Console.WriteLine($"Элементы в буфере: {buffer[0]}, {buffer[1]}"); // New_3, New_2 (LIFO порядок)
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Проверка IsEmpty перед TryPop
> Код `if (!_stack.IsEmpty) { _stack.TryPop(out var x); }` содержит Race Condition. Между проверкой `IsEmpty` и вызовом `TryPop` другой поток может извлечь элемент, и ваш вызов вернет `false` или приведет к ошибке логики. Всегда вызывайте `TryPop` напрямую и проверяйте его возвращаемое значение.

> [!WARNING] Ошибка: Чрезмерный вызов Count
> Избегайте использования `Count > 0` в циклах ожидания. Это приводит к постоянному обходу всей цепочки узлов стека в памяти, что нагружает шину процессора. Всегда используйте свойство `IsEmpty`, работающее за $O(1)$.

**Best Practices:**
1.  **Используйте диапазоны (Ranges):** Если у вас есть возможность обрабатывать элементы пакетами, всегда отдавайте приоритет методам `PushRange` и `TryPopRange`. Это снижает нагрузку на CAS-блокировки в сотни раз при высокой параллельной нагрузке.
2.  **Защита от утечек памяти:** Помните, что `ConcurrentStack` удерживает ссылки на объекты в куче через свои узлы. Если стек используется для временного хранения тяжелых объектов, не забывайте очищать его через `Clear()`, когда он больше не нужен, чтобы [[GC]] мог утилизировать объекты.
3.  **Не используйте для очередей:** Очередь (FIFO) на стеке (LIFO) через реверс — это крайне неэффективный алгоритм. Для очередей используйте `ConcurrentQueue<T>`.

> [!TIP] Нюанс: Структура узлов
> Каждый узел `Node` в `ConcurrentStack` — это отдельный объект в куче. В отличие от `ConcurrentQueue`, которая выделяет память сегментами по 32 элемента, стек выделяет память под каждый элемент отдельно. Это может создавать дополнительную нагрузку на GC (Garbage Collector) при экстремально высокой частоте операций `Push/Pop`. В таких случаях для пулов объектов часто предпочтительнее использовать `ConcurrentBag`.
