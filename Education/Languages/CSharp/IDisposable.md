### Суть (The "What")
`IDisposable` — это [[Интерфейсы|интерфейс]], предоставляющий механизм для **детерминированного освобождения [[unmanaged code|неуправляемых ресурсов]]**. В отличие от управляемой памяти, которой заведует Garbage Collector ([[GC]]) в неопределенные моменты времени, ресурсы ОС (дескрипторы файлов, сокеты, соединения с БД) требуют немедленного закрытия после использования.

Интерфейс решает проблему ограниченности системных ресурсов: если полагаться только на GC, приложение может исчерпать лимит открытых файлов или подключений к базе данных гораздо раньше, чем сработает автоматическая очистка памяти.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Детерминизм против Недетерминизма
GC работает по собственному расписанию. Метод `Dispose()` позволяет программисту явно сказать: "Я закончил работу с этим ресурсом, освободи его прямо сейчас".

#### 2. Взаимодействие со сборщиком мусора
Если класс реализует `IDisposable` и имеет **финализатор** (деструктор), то при вызове `Dispose()` обычно вызывается метод `GC.SuppressFinalize(this)`. 
*   **Механика:** Это сообщает GC, что объект уже очищен вручную и его не нужно помещать в очередь финализации (`F-Reachable queue`). Это экономит ресурсы процессора и позволяет объекту удалиться из памяти за один цикл сборки, а не за два.

#### 3. Стандартный паттерн Dispose (Disposable Pattern)
Для классов, которые могут иметь наследников, Албахари рекомендует использовать защищенный виртуальный метод `Dispose(bool disposing)`:
*   `disposing == true`: Вызов из кода. Можно безопасно освобождать и управляемые (другие `IDisposable` объекты), и неуправляемые ресурсы.
*   `disposing == false`: Вызов из финализатора. Можно освобождать **только** неуправляемые ресурсы. Обращаться к управляемым объектам нельзя, так как они уже могут быть уничтожены GC.

#### 4. Оператор using
Синтаксис `using (var x = ...)` или `using var x = ...` является синтаксическим сахаром для блока `try/finally`. Компилятор гарантирует вызов `x.Dispose()` в блоке `finally`, даже если внутри `try` возникло исключение.

#### 5. IAsyncDisposable (C# 8.0+)
Для ресурсов, требующих асинхронного закрытия (например, запись буфера в сетевой поток перед закрытием), используется интерфейс `IAsyncDisposable` с методом `ValueTask DisposeAsync()`.

### Пример кода ([[MOC|C#]] 12)

```csharp
using System;
using System.IO;

public class ResourceWrapper : IDisposable
{
    private FileStream? _fileStream;
    private bool _isDisposed;

    public ResourceWrapper(string path)
    {
        _fileStream = new FileStream(path, FileMode.OpenOrCreate);
    }

    // Публичный метод интерфейса
    public void Dispose()
    {
        Dispose(true);
        // Сообщаем GC, что финализатор вызывать не нужно
        GC.SuppressFinalize(this);
    }

    protected virtual void Dispose(bool disposing)
    {
        if (_isDisposed) return;

        if (disposing)
        {
            // 1. Освобождаем управляемые ресурсы
            _fileStream?.Dispose();
            _fileStream = null;
        }

        // 2. Освобождаем неуправляемые ресурсы (если есть IntPtr, дескрипторы)
        // В данном примере FileStream сам управляет дескриптором

        _isDisposed = true;
    }

    // Финализатор (нужен только если класс напрямую владеет IntPtr)
    ~ResourceWrapper() => Dispose(false);

    public void DoWork()
    {
        // Проверка состояния перед использованием
        ObjectDisposedException.ThrowIf(_isDisposed, this);
        Console.WriteLine("Working with file...");
    }
}

// Использование
public class Program
{
    public static void Main()
    {
        // Современный using declaration (C# 8+)
        using var wrapper = new ResourceWrapper("test.dat");
        wrapper.DoWork();
    } // Здесь автоматически вызовется Dispose()
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Доступ к объектам в финализаторе
> В методе `Dispose(false)` (вызов из финализатора) никогда не обращайтесь к другим управляемым объектам-полям вашего класса. Порядок сборки мусора не гарантирован, и эти объекты могут быть уже невалидны.

> [!WARNING] Ошибка: Игнорирование исключений в Dispose
> Метод `Dispose` не должен выбрасывать исключения. Если очистка может завершиться сбоем, используйте `try-catch` внутри `Dispose`. Брошенное исключение в блоке `finally` (куда компилятор прячет Dispose) может скрыть оригинальное исключение, ставшее причиной очистки.

**Best Practices:**
1.  **Идемпотентность:** Метод `Dispose` должен быть написан так, чтобы его можно было вызывать многократно без ошибок (второй и последующие вызовы просто ничего не делают).
2.  **Thread Safety:** Если ваш объект может использоваться в разных потоках, убедитесь, что вызов `Dispose` потокобезопасен (используйте `lock` или `Interlocked` для флага `_isDisposed`).
3.  **SafeHandle:** Вместо хранения сырых указателей `IntPtr` и написания финализаторов, используйте `System.Runtime.InteropServices.SafeHandle`. Он берет на себя логику взаимодействия с GC.
4.  **ThrowIfDisposed:** В публичных методах класса всегда проверяйте состояние объекта и выбрасывайте `ObjectDisposedException`, если `Dispose` уже был вызван.

> [!TIP] Нюанс: Container Ownership
> Если ваш класс получает `IDisposable` объект через конструктор (Dependency Injection), обычно он **не должен** вызывать для него `Dispose`. Правило хорошего тона: ресурсы освобождает тот, кто их создал (владелец).
