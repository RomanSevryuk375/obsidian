### Суть (The "What")
Блок `finally` — это завершающая часть конструкции [[Обработка исключений|try-catch-finally]], которая гарантированно выполняется после выхода из блоков `try` и `catch`, независимо от того, возникло ли исключение или был ли выполнен переход (например, `return`).

Основная задача `finally` — обеспечение **детерминированной очистки ресурсов** (закрытие файлов, сетевых соединений, освобождение дескрипторов). Это критически важно, так как сборщик мусора ([[GC]]) управляет только памятью, но не внешними ресурсами.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Гарантии выполнения
[[CLR]] (Common Language Runtime) гарантирует выполнение блока `finally` практически в любой ситуации. Даже если внутри блока `try` или `catch` вызывается `return`, `break` или `continue`, управление сначала передается в `finally`, и только после его завершения происходит фактический выход из метода.

#### 2. Случаи невыполнения
Существует лишь несколько сценариев, когда `finally` не будет выполнен:
*   Вызов `Environment.FailFast()`.
*   Критическое падение процесса (Stack Overflow, Access Violation).
*   Принудительное завершение процесса через Task Manager или системный сигнал (Kill).
*   Отключение электропитания.

#### 3. Наложение исключений (Shadowing)
Если исключение возникло в блоке `try`, а затем еще одно исключение возникло в блоке `finally`, то первое исключение будет **потеряно**. Стек вызовов будет содержать информацию только о последнем исключении из `finally`.

#### 4. Взаимосвязь с оператором using
Оператор `using` является синтаксическим сахаром для блока `try-finally`. Код:
```csharp
using var resource = new MyResource();
resource.DoWork();
```
Разворачивается компилятором в вызов `resource.Dispose()` именно внутри блока `finally`, что гарантирует освобождение ресурса даже при сбое.

> [!IMPORTANT] Нюанс: Запрет на передачу управления
> В C# запрещено использовать операторы перехода (`return`, `goto`, `break`, `continue`) внутри блока `finally`, если их цель находится вне этого блока. Блок должен быть выполнен до конца без прерывания "со стороны".

### Пример кода ([[MOC#/MOC|C#]] 12)

```csharp
using System;
using System.IO;

public class ResourceHandler(string path)
{
    private readonly string _path = path;

    public void WriteLog(string message)
    {
        StreamWriter? writer = null;
        try
        {
            writer = new StreamWriter(_path, append: true);
            writer.WriteLine(message);
            // Даже если здесь вызвать return, блок finally выполнится
        }
        catch (IOException ex)
        {
            Console.WriteLine($"Ошибка ввода-вывода: {ex.Message}");
        }
        finally
        {
            // Ручная очистка ресурса, если не используется оператор using
            if (writer != null)
            {
                writer.Close();
                Console.WriteLine("Ресурс успешно закрыт в блоке finally.");
            }
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Исключения внутри finally
> Никогда не допускайте возникновения исключений в блоке `finally`. Если код в `finally` может дать сбой, оберните его в дополнительный `try-catch`. Выброс исключения из `finally` затрет оригинальную ошибку из `try`, что сделает отладку невозможной.

> [!WARNING] Ошибка: Изменение возвращаемого значения
> Хотя `return` запрещен в `finally`, технически можно изменить состояние ссылочного типа, который возвращается из `try`. Однако это считается плохой практикой, так как делает поведение метода непредсказуемым.

**Best Practices:**
1.  **Предпочитайте using:** Вместо ручного написания `try-finally` для типов, реализующих `IDisposable`, всегда используйте оператор `using` (или `using var` в C# 8+).
2.  **Только очистка:** В блоке `finally` должна находиться только логика освобождения ресурсов. Не пишите там бизнес-логику или сложные вычисления.
3.  **Проверка на null:** При ручной очистке в `finally` всегда проверяйте объект ресурса на `null`, так как исключение могло возникнуть еще до инициализации ресурса в блоке `try`.
4.  **Атомарность:** Помните, что `finally` не делает операции атомарными. Если данные были частично записаны в файл до исключения, `finally` закроет файл, но не откатит изменения.

> [!TIP] Нюанс: Состояние стека
> Блок `finally` выполняется во время второй фазы обработки исключения (Stack Unwinding). В этот момент система ищет и выполняет все `finally` блоки по пути от места возникновения ошибки до найденного `catch`.
