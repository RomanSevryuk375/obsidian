# VMT (Virtual Method Table)

### Суть (The "What")
**VMT (Virtual Method Table)** — это структура данных, используемая средой выполнения ([[CLR]]) для реализации [[Динамический полиморфизм|динамического полиморфизма]]. Она представляет собой массив указателей на адреса исполняемого кода виртуальных методов типа.

VMT решает задачу **позднего связывания (late binding)**: определения конкретной реализации метода не на этапе компиляции, а во время выполнения, основываясь на фактическом типе объекта в куче, а не на типе ссылки.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Структура объекта в памяти
Каждый объект ссылочного типа в управляемой [[Куча|куче]] имеет стандартный заголовок, состоящий из двух полей (на x64 архитектуре):
*   **Sync Block Index (8 байт):** Используется для блокировок и хранения хеш-кода.
*   **Type Handle / Method Table Pointer (8 байт):** Указатель на структуру `MethodTable` для данного типа.

#### 2. [[MethodTable]] и VMT
`MethodTable` — это блок данных в **Loader Heap**, описывающий тип. VMT является частью этой таблицы.
*   **Слоты (Slots):** Каждый виртуальный метод занимает определенный индекс (слот) в таблице.
*   **Наследование:** Когда создается `MethodTable` для производного класса, CLR копирует таблицу базового класса. 
*   **Переопределение ([[Динамический полиморфизм|Override]]):** Если производный класс использует `override`, адрес метода в соответствующем слоте заменяется адресом новой реализации. Если не использует — в слоте остается адрес метода базового класса.

#### 3. Механизм вызова (Virtual Dispatch)
При вызове виртуального метода происходит следующее:
1. CLR обращается к объекту по ссылке.
2. Извлекает указатель на `MethodTable` из заголовка объекта.
3. Переходит по фиксированному смещению в VMT (индекс метода известен на этапе компиляции).
4. Извлекает адрес кода из слота и осуществляет переход (`call` по адресу).

> [!INFO] Нюанс: Прямой вызов (Direct Call)
> Если метод не помечен как `virtual`, компилятор генерирует инструкцию `call`, которая обращается напрямую к адресу метода в памяти. Для виртуальных методов генерируется инструкция `callvirt`, которая инициирует поиск через VMT.

#### 4. JIT-оптимизация: Devirtualization
Современный [[JIT]]-компилятор может выполнять "девиртуализацию". Если он может доказать (например, через статический анализ), что тип объекта в данной точке кода не изменится (например, класс помечен как `sealed`), он заменяет `callvirt` на обычный `call`, полностью исключая накладные расходы VMT.

### Пример кода ([[MOC#/MOC|C#]] 12)

```csharp
using System;

public class Base
{
    public void NonVirtual() => Console.WriteLine("Base Direct");
    public virtual void Virtual() => Console.WriteLine("Base VMT Slot 1");
}

public class Derived : Base
{
    // Скрывает метод (не занимает новый слот в VMT для Base)
    public new void NonVirtual() => Console.WriteLine("Derived Direct");
    
    // Перезаписывает адрес в VMT Slot 1
    public override void Virtual() => Console.WriteLine("Derived VMT Slot 1");
}

public class Program
{
    public static void Main()
    {
        Base obj = new Derived();

        // 1. Прямой вызов: компилятор смотрит на тип ссылки (Base).
        // Результат: "Base Direct"
        obj.NonVirtual();

        // 2. Виртуальный вызов: CLR идет в MethodTable объекта Derived, 
        // находит Slot 1 и берет оттуда адрес.
        // Результат: "Derived VMT Slot 1"
        obj.Virtual();
    }
}
```

### Ошибки и Best Practices

> [!WARNING] Нюанс: callvirt и NullCheck
> Инструкция `callvirt` выполняет проверку на `null` перед вызовом. Интересный факт: компилятор C# использует `callvirt` даже для невиртуальных методов экземпляра, чтобы обеспечить генерацию `NullReferenceException` при попытке вызвать метод у `null`-объекта.

> [!DANGER] Anti-pattern: Глубокие иерархии и ITable
> Вызовы через [[Интерфейсы|интерфейсы]] работают сложнее, чем через VMT классов. Они используют механизм **Virtual Stub Dispatch (VSD)** и дополнительные таблицы (Interface Dispatch Map). Глубокие иерархии с множественными интерфейсами могут незначительно снижать производительность из-за промахов в кэше поиска заглушек (stubs).

**Best Practices:**
1.  **Используйте sealed:** Пометка классов как `sealed` — это не только архитектурное решение, но и подсказка JIT-компилятору для девиртуализации вызовов.
2.  **Избегайте virtual в горячих циклах:** Если метод вызывается миллионы раз в секунду, лишняя индирекция (переход по указателю) может стать "узким местом".
3.  **Понимайте разницу между new и override:** `new` не меняет слот в VMT базового типа, он создает новый метод, не связанный с полиморфизмом базового класса.
4.  **Проверка производительности:** На современных CPU стоимость VMT вызова минимальна благодаря предсказателю переходов, но она возрастает, если в одной точке вызова чередуются объекты разных типов (полиморфный вызов становится "грязным" для кэша процессора).

> [!TIP] Нюанс: Размер MethodTable
> `MethodTable` содержит не только VMT, но и данные о размере экземпляра, флаги типа, указатель на родительскую таблицу и информацию о реализации интерфейсов. Это "мозг" типа в среде выполнения.
