### Суть (The "What")
`LinkedList<T>` — это последовательная обобщенная коллекция, представляющая собой двусвязный список. В отличие от [[List]]`<T>`, который хранит данные в непрерывном массиве, `LinkedList<T>` состоит из независимых узлов, каждый из которых ссылается на своего соседа слева и справа.

Он решает задачу **сверхбыстрой вставки и удаления элементов в любом месте списка за константное время $O(1)$** (при условии, что у вас уже есть ссылка на узел). При этом не требуется сдвигать элементы в памяти или перевыделять массив.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Анатомия узла (LinkedListNode\<T\>)
Базовым элементом коллекции является класс `LinkedListNode<T>`. Каждый такой узел содержит:
*   `Value` — непосредственно хранящиеся данные типа `T`.
*   `List` — ссылка на родительский список `LinkedList<T>`, которому принадлежит узел.
*   `Next` — ссылка на следующий узел (или `null`).
*   `Previous` — ссылка на предыдущий узел (или `null`).

#### 2. Распределение памяти и локальность данных
Поскольку каждый узел — это отдельный объект типа `LinkedListNode<T>` в [[Куча|куче]]:
*   **Накладные расходы (Overhead):** Каждый элемент списка оборачивается в отдельный объект. На x64-системах это дает минимум 24-32 байта накладных расходов памяти (заголовок объекта + ссылки) на каждый элемент, помимо размера самого значения `T`.
*   **Промахи кэша (Cache Misses):** Узлы выделяются в куче хаотично. Они не лежат в памяти последовательно (в отличие от массива или `List<T>`). Это лишает процессор преимуществ кэширования данных (CPU Cache Locality), что делает последовательный обход двусвязного списка физически более медленным, чем обход массива.

#### 3. Алгоритмическая сложность
*   **Вставка/Удаление в начало или конец:** **$O(1)$** (список хранит прямые ссылки на первый и последний узлы — свойства `First` и `Last`).
*   **Вставка/Удаление в середину (после конкретного узла):** **$O(1)$** (просто перенаправляются ссылки `Next` и `Previous` соседних узлов).
*   **Поиск по значению или индексация:** **$O(n)$**. Коллекция не поддерживает прямой доступ по индексу (нет индексатора `list[i]`). Чтобы найти элемент, рантайму приходится последовательно перебирать узлы с начала или конца.

### Пример кода ([[C#]] 12)

```csharp
using System;
using System.Collections.Generic;

public class LinkedListDemo
{
    public static void Main()
    {
        // 1. Создание двусвязного списка
        LinkedList<string> logChain = new();

        // 2. Быстрое добавление в начало и конец - O(1)
        LinkedListNode<string> nodeFirst = logChain.AddFirst("Log: Application Started");
        LinkedListNode<string> nodeLast = logChain.AddLast("Log: Application Stopped");

        // 3. Вставка относительно конкретного узла - O(1)
        LinkedListNode<string> nodeMiddle = logChain.AddAfter(nodeFirst, "Log: User Logged In");
        logChain.AddBefore(nodeLast, "Log: User Logged Out");

        // 4. Последовательный обход с начала в конец - O(n)
        Console.WriteLine("Прямой обход:");
        LinkedListNode<string>? current = logChain.First;
        while (current != null)
        {
            Console.WriteLine(current.Value);
            current = current.Next; // Переход по ссылке
        }

        // 5. Обратный обход с конца в начало - O(n)
        Console.WriteLine("\nОбратный обход:");
        current = logChain.Last;
        while (current != null)
        {
            Console.WriteLine(current.Value);
            current = current.Previous; // Переход назад
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Чрезмерная нагрузка на память и GC
> Использование `LinkedList<int>` для хранения миллионов чисел — грубая архитектурная ошибка. Каждое число `int` (4 байта) будет обернуто в объект узла (не менее 24 байт накладных расходов). Это создаст миллионы мелких объектов в куче, которые парализуют работу сборщика мусора ([[GC]]). Для чисел всегда используйте `List<T>`.

> [!WARNING] Ошибка: Поиск перед удалением
> Если вы хотите удалить элемент по значению через `list.Remove("value")`, то сложность операции составит $O(n)$, так как списку сначала придется линейно найти этот узел. Преимущество $O(1)$ работает только тогда, когда у вас уже есть готовая ссылка на `LinkedListNode<T>`.

**Best Practices:**
1.  **Редкие операции поиска, частые вставки:** Используйте `LinkedList<T>` только тогда, когда вам нужно постоянно вставлять и удалять элементы в разных частях списка, и вы делаете это, имея на руках прямые ссылки на узлы.
2.  **Замена для очередей с обходом:** `LinkedList<T>` удобен, если вам нужен функционал очереди, но с возможностью обхода и изменения элементов в середине списка (чего не позволяет стандартный [[Queue]]`<T>`).
3.  **Повторное использование узлов:** Вы можете удалять узел из одного списка и добавлять его в другой без выделения новой памяти: `list.Remove(node); otherList.AddFirst(node);`. Это позволяет переносить данные без аллокаций.

> [!TIP] Практика в реальном железе
> Из-за архитектуры современных процессоров и кэширования памяти, в реальных тестах обычный `List<T>` часто работает быстрее, чем `LinkedList<T>`, даже при удалении и вставке элементов в середину, если размер списка не превышает нескольких тысяч элементов. Накладные расходы на промахи кэша процессора в двусвязных списках превышают затраты на копирование плоского массива.
