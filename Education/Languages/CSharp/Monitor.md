### Суть (The "What")
**Monitor** — это низкоуровневый механизм синхронизации (эксклюзивная блокировка), на котором базируется оператор [[lock]]. Он позволяет ограничить доступ к блоку кода таким образом, чтобы в каждый момент времени только один поток мог выполнять этот код для конкретного объекта.

Помимо обеспечения взаимного исключения (mutual exclusion), `Monitor` предоставляет функционал для **сигнализации** между потоками через методы `Wait`, `Pulse` и `PulseAll`, что позволяет строить сложные сценарии взаимодействия потоков.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Связь с объектом и Sync Block
Каждый объект в управляемой куче .NET имеет заголовок, содержащий **Sync Block Index**. 
*   По умолчанию этот индекс пуст. Когда вызывается `Monitor.Enter(obj)`, [[CLR]] выделяет структуру данных (Sync Block) из внутреннего пула и записывает её индекс в заголовок объекта.
*   Эта структура содержит идентификатор потока-владельца и счетчик рекурсии (один и тот же поток может входить в блокировку многократно).

#### 2. Оператор lock и try-finally
Компилятор C# разворачивает оператор `lock(obj)` в конструкцию с использованием `Monitor`. Начиная с версии 4.0, это делается максимально безопасно:

```csharp
bool lockTaken = false;
try {
    Monitor.Enter(obj, ref lockTaken);
    // Критическая секция
}
finally {
    if (lockTaken) Monitor.Exit(obj);
}
```
Флаг `lockTaken` необходим, чтобы гарантировать, что `Exit` вызовется только в том случае, если `Enter` действительно захватил блокировку (предотвращает ошибки при исключениях внутри `Enter`).

#### 3. Сигнализация: Wait и Pulse
Это более продвинутый уровень синхронизации:
*   **Wait(obj):** Поток временно освобождает блокировку объекта и переходит в очередь ожидания. Он "засыпает" до тех пор, пока другой поток не пошлет сигнал.
*   **Pulse(obj):** Уведомляет один поток в очереди ожидания, что он может проснуться и попытаться снова захватить блокировку после выхода текущего потока.
*   **PulseAll(obj):** Пробуждает все ожидающие потоки.

> [!INFO] Нюанс: Гибридная блокировка
> `Monitor` не сразу усыпляет поток. Сначала он выполняет короткое "вращение" ([[Spin-lock|spin-lock]]) в пользовательском режиме, надеясь, что блокировка освободится быстро. Если этого не происходит, поток переходит в режим ожидания ядра ОС, что экономит CPU, но требует контекстного переключения.

### Пример кода ([[MOC|C#]] 12)

Реализация классической задачи "Производитель-Потребитель" через `Wait` и `Pulse`.

```csharp
using System;
using System.Collections.Generic;
using System.Threading;

public class ThreadSafeQueue
{
    private readonly object _locker = new();
    private readonly Queue<int> _queue = new();

    public void Produce(int value)
    {
        lock (_locker)
        {
            _queue.Enqueue(value);
            Console.WriteLine($"Produced: {value}");
            // Оповещаем ждущего потребителя
            Monitor.Pulse(_locker);
        }
    }

    public int Consume()
    {
        lock (_locker)
        {
            // Пока очередь пуста, отпускаем замок и ждем сигнала
            while (_queue.Count == 0)
            {
                Monitor.Wait(_locker);
            }
            return _queue.Dequeue();
        }
    }
}

public class Program
{
    public static void Main()
    {
        var buffer = new ThreadSafeQueue();
        
        Task.Run(() => {
            for(int i = 1; i <= 5; i++) {
                Thread.Sleep(500);
                buffer.Produce(i);
            }
        });

        for(int i = 1; i <= 5; i++) {
            Console.WriteLine($"Consumed: {buffer.Consume()}");
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Использование Wait вне блокировки
> Методы `Wait`, `Pulse` и `PulseAll` можно вызывать только в том случае, если текущий поток уже владеет блокировкой объекта (находится внутри `lock` или `Monitor.Enter`). В противном случае возникнет `SynchronizationLockException`.

> [!WARNING] Ошибка: Бесконечное ожидание
> Всегда используйте цикл `while` вокруг `Monitor.Wait`. Это защищает от "ложных пробуждений" (spurious wakeups) и гарантирует, что условие действительно соблюдено после того, как поток снова захватил замок.

**Best Practices:**
1.  **Приватный объект для блокировки:** Всегда используйте `private readonly object _locker = new();`. Блокировка на `this` или `typeof(T)` опасна взаимными блокировками со сторонним кодом.
2.  **Минимизация времени удержания:** Выполняйте только критически важные операции внутри `Monitor`. Длительные вычисления или I/O внутри блокировки убивают производительность.
3.  **Monitor.TryEnter:** Если вы не хотите блокировать поток навсегда, используйте `TryEnter` с таймаутом. Это позволяет реализовать логику отмены или альтернативного поведения при занятом ресурсе.
4.  **PulseAll вместо Pulse:** Если вы не уверены точно, какой именно поток должен проснуться (например, несколько разных условий ожидания на одном объекте), используйте `PulseAll`, чтобы избежать "зависания" нужного потока.

> [!TIP] Нюанс: Потоковая безопасность статических членов
> При работе со статическими полями часто блокируют `typeof(MyClass)`. Albahari рекомендует создавать для этих целей отдельный приватный статический объект блокировки, чтобы избежать раскрытия объекта синхронизации внешним пользователям класса.
