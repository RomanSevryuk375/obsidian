### Суть (The "What")
**yield return** — это контекстное ключевое слово, используемое для создания итератора. Оно позволяет методу возвращать элементы последовательности по одному, по мере необходимости, вместо того чтобы генерировать и хранить всю коллекцию в памяти целиком.

Итераторы решают задачу **отложенного выполнения (deferred execution)** и эффективного управления памятью при работе с потенциально бесконечными или очень большими наборами данных.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Магия компилятора: [[State Machine|Конечный автомат]]
Когда компилятор видит `yield return`, он полностью переписывает метод:
*   Создается скрытый приватный класс (или структура), реализующий интерфейсы [[IEnumerable]]`<T>` и `IEnumerator<T>`.
*   Локальные переменные метода "поднимаются" (lifting) в поля этого класса, чтобы сохранять свое состояние между вызовами.
*   Код метода превращается в метод `MoveNext()` этого класса, представляющий собой большой блок `switch`, который управляет переходами между состояниями (от одного `yield` до другого).

#### 2. Жизненный цикл итератора
1.  **Вызов метода:** Код внутри метода **не выполняется**. Компилятор просто создает и возвращает экземпляр сгенерированного класса-состояния.
2.  **Первый MoveNext() (обычно через foreach):** Выполнение начинается с начала метода и идет до первого `yield return`.
3.  **Приостановка:** Метод "замораживается", текущее значение возвращается в `Current`, а управление передается вызывающему коду.
4.  **Возобновление:** При следующем вызове `MoveNext()` выполнение продолжается ровно с той точки, где оно остановилось, сохраняя значения всех локальных переменных.

#### 3. yield break
Инструкция `yield break` немедленно завершает итерацию. Она сообщает конечному автомату перейти в финальное состояние, после чего `MoveNext()` будет всегда возвращать `false`.

#### 4. Ограничения (Compiler constraints)
*   Нельзя использовать `yield return` внутри блоков `try-catch` (но можно внутри `try-finally`).
*   Метод не может содержать параметры с модификаторами `ref` или `out`.
*   Метод не может быть анонимным (лямбда-выражением).

> [!INFO] Нюанс: IAsyncEnumerable`<T>`
> Начиная с C# 8.0, `yield return` можно использовать в методах, возвращающих `IAsyncEnumerable`. Это позволяет создавать асинхронные потоки данных, которые можно перебирать через `await foreach`.

### Пример кода ([[MOC|C#]] 12)

```csharp
using System;
using System.Collections.Generic;

public class IteratorDemo
{
    public static void Main()
    {
        // Вызов метода не приводит к выводу текста на консоль (отложенное выполнение)
        var sequence = GetNumbers(1, 5);

        Console.WriteLine("Начало итерации:");
        foreach (var num in sequence)
        {
            Console.WriteLine($"Получено число: {num}");
        }
    }

    public static IEnumerable<int> GetNumbers(int start, int end)
    {
        for (int i = start; i `<`= end; i++)
        {
            Console.WriteLine($"--- Генерирую число {i} ---");
            
            // Состояние метода сохраняется здесь
            yield return i; 

            if (i == 3)
            {
                Console.WriteLine("Досрочный выход.");
                yield break;
            }
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Множественное перечисление (Multiple Enumeration)
> Если итератор выполняет тяжелые вычисления или запросы к БД, каждый раз, когда вы вызываете `foreach` (или методы LINQ вроде `Count()`, `Any()`), код итератора выполняется заново. Если данные нужны несколько раз, сохраните их в список через `.ToList()`.

> [!WARNING] Ошибка: Валидация аргументов
> Поскольку выполнение итератора отложено, любые ошибки валидации аргументов (например, `ArgumentNullException`) возникнут не в момент вызова метода, а позже — когда кто-то начнет перебор данных. 
> **Решение:** Используйте обычный метод для проверки аргументов, который вызывает приватную локальную функцию с `yield return`.

**Best Practices:**
1.  **Потоковая обработка:** Используйте `yield return` для обработки строк из больших файлов или данных из БД порциями. Это удерживает потребление оперативной памяти на низком уровне.
2.  **Finally блоки:** Помните, что блок `finally` внутри итератора выполнится либо при завершении итерации, либо если вызывающий код преждевременно прекратит перебор (например, через `break` внутри `foreach`). Это гарантирует закрытие ресурсов.
3.  **Не возвращайте null:** Метод, возвращающий `IEnumerable`, никогда не должен возвращать `null`. Если данных нет, он просто не должен выполнять ни одного `yield return` (или выполнить `yield break`).
4.  **Конечные автоматы:** Помните, что каждый итератор — это аллокация объекта в куче. В высокопроизводительных циклах (Hot Paths) иногда лучше использовать обычные массивы или `Span<T>`.

> [!TIP] Нюанс: Изменение коллекции
> Если итератор обходит коллекцию, которую вы модифицируете в процессе перебора, вы получите `InvalidOperationException`. Итераторы .NET следят за версией коллекции.
