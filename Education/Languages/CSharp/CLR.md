# CLR (Common Language Runtime)

### Суть (The "What")
**CLR** — это виртуальная машина, исполнительная среда платформы .NET, в которой выполняется код. Все программы, написанные на [[MOC#/MOC|C#]], компилируются не в машинный код, а в промежуточный язык **[[IL]] (Intermediate Language)**. CLR берет на себя задачу трансформации этого кода в инструкции конкретного процессора непосредственно во время выполнения.

CLR решает задачу **абстракции от аппаратного обеспечения и операционной системы**, а также обеспечивает автоматическое управление памятью, безопасность типов и обработку исключений. Код, работающий под управлением CLR, называется **управляемым ([[managed code]])**.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Двухэтапная компиляция
Процесс превращения исходного кода в работающее приложение проходит через два фильтра:
1.  **Компилятор языка (csc.exe):** Превращает C# в сборку (Assembly — `.dll` или `.exe`), содержащую IL-код и метаданные.
2.  **[[JIT]]-компилятор (Just-In-Time):** Внутри CLR превращает IL в нативный машинный код прямо перед выполнением конкретного метода.

#### 2. Компоненты среды
*   **[[VES]] (Virtual Execution System):** Загружает сборки, управляет потоками и распределяет память.
*   **[[Метаданные]]:** CLR использует метаданные (описания типов, методов, атрибутов) для обеспечения межъязыкового взаимодействия и реализации [[Рефлексия|рефлексии]].
*   **Garbage Collector ([[GC]]):** Автоматически отслеживает ссылки на объекты в [[Куча|куче]] и освобождает память, когда объекты становятся недоступными.

#### 3. Уровни выполнения (Tiered Compilation)
Современная CLR (начиная с .NET Core 3.0+) использует многоуровневую компиляцию:
*   **Tier 0:** Быстрая компиляция без оптимизаций для максимально быстрого старта приложения.
*   **Tier 1:** Если метод вызывается часто, CLR перекомпилирует его с применением глубоких оптимизаций (инлайнинг, векторизация).

> [!INFO] Нюанс: Managed vs Unmanaged
> Управляемый код выполняется в "песочнице" CLR, которая гарантирует безопасность. Однако CLR позволяет вызывать неуправляемый код (например, системные Win32 API или C++ библиотеки) через механизм P/Invoke, что требует осторожности при работе с памятью.

#### 4. Type Safety и верификация
Перед выполнением IL-кода CLR проводит **верификацию**. Это процесс проверки того, что код не пытается получить доступ к памяти по недопустимым адресам и не нарушает правила типизации (например, не вызывает метод класса `B` через ссылку на класс `A`).

### Пример кода (C# 12)

Хотя CLR работает скрыто, мы можем взаимодействовать с её механизмами через пространство имен `System.Runtime`.

```csharp
using System;
using System.Runtime.CompilerServices;
using System.Runtime.InteropServices;

public class ClrInternals
{
    public static void Main()
    {
        // 1. Принудительная JIT-компиляция метода через CLR Helpers
        RuntimeHelpers.PrepareMethod(typeof(ClrInternals).GetMethod(nameof(TargetMethod))!.MethodHandle);
        
        // 2. Получение информации о версии рантайма
        string version = RuntimeInformation.FrameworkDescription;
        Console.WriteLine($"Running on: {version}");

        TargetMethod();
    }

    // Подсказка JIT-компилятору: не встраивать этот метод (NoInlining)
    [MethodImpl(MethodImplOptions.NoInlining)]
    public static void TargetMethod()
    {
        Console.WriteLine("This method was JIT-compiled specifically for this CPU.");
    }
}
```

### Ошибки и Best Practices

> [!WARNING] Ошибка: Чрезмерная генерация кода в Runtime
> Использование `Reflection.Emit` или динамических типов (`dynamic`) заставляет CLR генерировать новый код и метаданные прямо во время выполнения. Это увеличивает потребление памяти (Working Set) и может замедлить работу из-за постоянной работы JIT-компилятора.

> [!DANGER] Anti-pattern: Ручное управление временем жизни GC
> Вызов `GC.Collect()` вручную почти всегда является ошибкой. Это заставляет CLR приостанавливать все потоки (Stop-the-world) и пересчитывать граф объектов, что нарушает механизмы самооптимизации рантайма. Доверяйте сборщику мусора.

**Best Practices:**
1.  **Native [[AOT]] (Ahead-Of-Time):** Для микросервисов, где важна скорость холодного старта, рассмотрите компиляцию Native AOT (доступна в .NET 7/8+). Она исключает JIT из процесса, компилируя код в машинные инструкции заранее.
2.  **Generic-типы:** Используйте обобщения. CLR генерирует специализированный машинный код для каждого значимого типа (например, `List<int>`, `List<double>`), что исключает упаковку ([[boxing]]) и повышает производительность.
3.  **AppDomain (Legacy):** В современном .NET (.NET Core/5/6/7/8) больше нет нескольких доменов приложений в одном процессе. Для изоляции кода теперь используются `AssemblyLoadContext`.

> [!TIP] Нюанс: Преимущество JIT над C++
> Теоретически, JIT-компилятор может генерировать более эффективный код, чем статический C++ компилятор, так как JIT знает точную модель процессора (например, поддержку наборов инструкций AVX-512), на котором запущена программа прямо сейчас.
