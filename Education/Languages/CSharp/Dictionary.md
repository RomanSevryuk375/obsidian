### Суть (The "What")
**Dictionary<TKey, TValue>** — это универсальная коллекция, представляющая собой хеш-таблицу для хранения пар "ключ-значение". Она обеспечивает максимально быстрый доступ к значениям по уникальному ключу.

Словарь решает задачу **эффективного поиска**: в отличие от списка (`List<T>`), где поиск элемента требует перебора ($O(n)$), словарь позволяет получить значение за константное время — **$O(1)$** в среднем случае.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Внутреннее устройство ([[Hash Table]])
Внутри словаря находятся два основных массива: `buckets` (индексы) и `entries` (данные).
*   **Хеширование:** Когда вы добавляете ключ, вызывается метод `GetHashCode()`. Полученное число подвергается операции взятия остатка от деления на размер массива `buckets`, что определяет "корзину".
*   **Buckets и Entries:** Массив `buckets` хранит индекс первого элемента в массиве `entries`. Массив `entries` содержит сами ключи, значения и "указатель" (индекс) на следующий элемент в случае коллизии.

#### 2. Разрешение коллизий (Chaining)
Коллизия возникает, когда два разных ключа выдают один и тот же индекс корзины.
*   [[MOC|C#]] использует метод **цепочек (chaining)**: элементы с одинаковым индексом корзины выстраиваются в односвязный список внутри массива `entries`.
*   При поиске рантайм находит нужную корзину, а затем перебирает список, сравнивая ключи с помощью метода `Equals()`.

#### 3. Сравнение ключей (IEqualityComparer)
Словарь критически зависит от двух методов:
1.  `GetHashCode()`: для быстрого нахождения корзины.
2.  `Equals()`: для финального подтверждения идентичности ключа (особенно при коллизиях).
Вы можете передать кастомный `IEqualityComparer<TKey>` в конструктор, чтобы изменить логику сравнения (например, сделать строковые ключи нечувствительными к регистру).

#### 4. Ресайзинг и емкость
Когда количество элементов приближается к размеру массивов, словарь выполняет **Rehash**:
*   Создаются новые массивы большего размера (обычно размер увеличивается до следующего простого числа).
*   Все существующие элементы перераспределяются по новым корзинам. Это дорогая операция $O(n)$.

> [!IMPORTANT] Нюанс: Порядок элементов
> В .NET Dictionary **не гарантирует** сохранение порядка добавления элементов. Хотя в текущих реализациях порядок часто сохраняется до первого удаления или ресайза, полагаться на это в логике приложения категорически нельзя. Если нужен порядок — используйте `OrderedDictionary`.

### Пример кода (C# 12)

```csharp
using System;
using System.Collections.Generic;

public class DictionaryDemo
{
    public static void Main()
    {
        // 1. Инициализация через Collection Expressions (C# 12)
        // Также задаем начальную емкость для оптимизации
        Dictionary<int, string> users = new(capacity: 10)
        {
            [1] = "Artyom",
            [2] = "Maria",
            [3] = "Ivan"
        };

        // 2. Безопасное извлечение (TryGetValue)
        if (users.TryGetValue(2, out string? name))
        {
            Console.WriteLine($"Found user: {name}");
        }

        // 3. Добавление или обновление (индексатор)
        users[4] = "Elena"; // Добавит новый
        users[1] = "Artyom (Admin)"; // Обновит существующий

        // 4. Использование KeyValuePair при итерации
        foreach (KeyValuePair<int, string> kvp in users)
        {
            Console.WriteLine($"ID: {kvp.Key}, Name: {kvp.Value}");
        }

        // 5. Словарь с кастомным компаратором
        var config = new Dictionary<string, string>(StringComparer.OrdinalIgnoreCase)
        {
            ["Timeout"] = "30"
        };
        Console.WriteLine(config["timeout"]); // Вернет "30" благодаря компаратору
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Обращение по несуществующему ключу
> Вызов `var val = dict[nonExistentKey];` выбросит `KeyNotFoundException`. Всегда используйте `dict.TryGetValue()` или `dict.ContainsKey()`, если не уверены на 100% в наличии ключа.

> [!WARNING] Ошибка: Изменение ключа
> Если ключ является ссылочным типом и его внутреннее состояние изменится так, что поменяется `GetHashCode()`, объект в словаре станет "невидимым". Словарь будет искать его в старой корзине, а объект логически должен быть в новой. **Ключи должны быть неизменяемыми.**

**Best Practices:**
1.  **Initial Capacity:** Если вы заранее знаете примерный размер данных (например, 1000 элементов), всегда указывайте `capacity` в конструкторе. Это предотвратит множественные дорогостоящие операции ресайза.
2.  **Выбор ключа:** Используйте типы с быстрыми и надежными методами `GetHashCode`. `int`, `Guid`, `string` — хорошие ключи. Большие структуры в качестве ключей — плохая практика.
3.  **ConcurrentDictionary:** Если доступ к словарю осуществляется из нескольких потоков, используйте `System.Collections.Concurrent.ConcurrentDictionary` вместо обычного `lock(dict)`.
4.  **Избегайте боксинга:** Для значимых типов в качестве ключей всегда используйте стандартный `Dictionary<TKey, TValue>`. Старый `Hashtable` (из `System.Collections`) будет приводить к упаковке (boxing) каждого `int`-ключа в `object`, что уничтожит производительность.

> [!TIP] Нюанс: Метод Add vs Индексатор
> Метод `dict.Add(key, value)` выбросит исключение, если ключ уже существует. Индексатор `dict[key] = value` просто перезапишет значение. Используйте `Add`, если логика программы подразумевает, что дубликатов быть не может.
