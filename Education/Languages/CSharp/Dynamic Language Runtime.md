### Суть (The "What")
**DLR** — это среда выполнения, работающая поверх [[CLR]], которая добавляет в .NET поддержку динамической типизации. Главным инструментом взаимодействия с ней в [[MOC|C#]] является ключевое слово `dynamic`.

DLR решает задачу **позднего связывания (late binding)**. Она позволяет писать код, в котором проверка типов, поиск методов и свойств происходят не во время компиляции, а во время выполнения программы. Это критично при работе с COM-объектами, динамическими языками (IronPython), рефлексией или структурами данных с неопределенной заранее схемой (например, JSON или результаты парсинга XML).

### Как это работает (The "How" / Nutshell style)

#### 1. Тип dynamic vs object
На уровне [[IL]]-кода тип `dynamic` представлен как `System.Object`. Однако компилятор помечает переменные `dynamic` специальным атрибутом `DynamicAttribute`.
*   **object:** Компилятор требует явного приведения типов для доступа к членам.
*   **dynamic:** Компилятор "пропускает" проверки и генерирует инструкции для DLR, чтобы та нашла нужный член в рантайме.

#### 2. Механизм Call Sites и кэширование
Каждый динамический вызов превращается в **Call Site** (место вызова). Это сложная структура, которая:
1.  **Связывание (Binding):** При первом вызове анализирует объект и находит подходящий метод/свойство.
2.  **Кэширование (Caching):** Запоминает результат связывания. Если при следующем вызове типы аргументов совпадают, DLR выполняет вызов мгновенно, минуя стадию поиска. Это делает `dynamic` значительно быстрее "чистой" рефлексии.

#### 3. IDynamicMetaObjectProvider
Типы могут сами определять свое динамическое поведение, реализуя этот интерфейс. 
*   **ExpandoObject:** Позволяет добавлять и удалять свойства "на лету" (как в JavaScript).
*   **DynamicObject:** Базовый класс, упрощающий создание типов с кастомной логикой разрешения имен (например, для маппинга XML-тегов на свойства).

> [!INFO] Нюанс: Статическое связывание внутри динамики
> Если внутри динамического вызова используются аргументы статических типов, DLR учитывает их иерархию при поиске перегрузок методов, имитируя поведение обычного компилятора C# в реальном времени.

### Пример кода (C# 12)

```csharp
using System;
using System.Dynamic;
using System.Collections.Generic;

public class DynamicDemo
{
    public static void Main()
    {
        // 1. ExpandoObject: динамическое добавление свойств
        dynamic person = new ExpandoObject();
        person.FirstName = "Artyom";
        person.Age = 30;
        
        // Можно добавить даже делегат (метод)
        person.SayHello = new Action(() => Console.WriteLine($"Hello, {person.FirstName}"));
        
        person.SayHello();

        // 2. Взаимодействие с коллекциями (ExpandoObject реализует IDictionary)
        var personDict = (IDictionary<string, object>)person;
        personDict["LastName"] = "Dev";
        
        Console.WriteLine($"{person.FirstName} {person.LastName}");

        // 3. Кастомный динамический объект
        dynamic config = new SimpleConfig();
        config.DbPath = "server=localhost;db=main;"; // Вызовет TrySetMember
        Console.WriteLine(config.DbPath);           // Вызовет TryGetMember
    }
}

public class SimpleConfig : DynamicObject
{
    private readonly Dictionary<string, object> _values = [];

    public override bool TrySetMember(SetMemberBinder binder, object? value)
    {
        _values[binder.Name] = value ?? "";
        return true;
    }

    public override bool TryGetMember(GetMemberBinder binder, out object? result)
    {
        return _values.TryGetValue(binder.Name, out result);
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: dynamic в горячих циклах (Hot Paths)
> Несмотря на кэширование, динамический вызов все равно медленнее статического. В критических для производительности участках (обработка миллионов транзакций) используйте строгую типизацию.

> [!WARNING] Ошибка: RuntimeBinderException
> Любая опечатка в имени метода или передача неверного типа аргумента не будет подсвечена IDE и приведет к падению программы в рантайме. Используйте `dynamic` только там, где статика бессильна.

**Best Practices:**
1.  **Локализация:** Ограничивайте область видимости `dynamic`. Не позволяйте динамическим типам "протекать" в публичные интерфейсы (Public API) ваших библиотек.
2.  **Замена рефлексии:** Если вам нужно часто вызывать методы через рефлексию, `dynamic` — более быстрый и читаемый вариант благодаря внутреннему кэшированию DLR.
3.  **JSON и XML:** Используйте `dynamic` для быстрой прототипной работы со слабоструктурированными данными, но переходите к типизированным DTO (через `System.Text.Json`), когда структура стабилизируется.
4.  **Unit-тесты:** Код с использованием `dynamic` требует 100% покрытия тестами, так как компилятор не может гарантировать корректность вызовов.

> [!TIP] Нюанс: dynamic и методы расширения
> Методы расширения нельзя вызывать динамически, так как связывание происходит в рантайме, а методы расширения разрешаются компилятором на основе пространств имен (`using`). Для их вызова используйте синтаксис статического метода: `MyExtensions.Method(dynamicObj)`.
