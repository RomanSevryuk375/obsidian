### Суть (The "What")
`List<T>` — это последовательная обобщенная коллекция элементов, доступ к которым осуществляется по индексу. Он представляет собой динамически расширяемую обертку над обычным одномерным массивом `T[]`.

`List<T>` решает задачу **преодоления фиксированного размера массивов**. Он позволяет добавлять и удалять элементы "на лету", сохраняя при этом главное преимущество массива — сверхбыстрый произвольный доступ по индексу за константное время **$O(1)$**.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Внутреннее устройство и поля
Внутри класса `List<T>` объявлены два ключевых поля:
*   `T[] _items` — плоский массив в управляемой куче, где физически лежат данные.
*   `int _size` — количество фактически добавленных элементов (свойство `Count`).

#### 2. Разница между Count и Capacity
*   **Count (Размер):** Количество реально добавленных элементов.
*   **Capacity (Емкость):** Длина внутреннего массива `_items.Length`. Емкость всегда больше или равна размеру.

#### 3. Стратегия автоматического расширения
Когда при вызове `Add()` размер `Count` достигает предела емкости `Capacity`:
1.  Выделяется новый массив в куче, размер которого обычно **удваивается** (если текущая емкость 0, по умолчанию устанавливается размер 4).
2.  Элементы из старого массива копируются в новый с помощью высокопроизводительного системного метода `Array.Copy`.
3.  Старый массив теряет ссылки и со временем утилизируется сборщиком мусора ([[GC]]).
4.  Сложность операции `Add()` является **амортизированной $O(1)$**. Это значит, что большинство добавлений происходят мгновенно, но редкие вызовы (требующие расширения) занимают время $O(n)$.

#### 4. Вставка и удаление в середине
Методы `Insert(index, item)` и `RemoveAt(index)` имеют алгоритмическую сложность **$O(n)$**. При добавлении или удалении элемента в середине списка, [[CLR]] вынуждена физически сдвигать в памяти все последующие элементы с помощью копирования массива.

#### 5. [[JIT]]-компиляция обобщенного списка
*   Для **значимых типов** (например, `List<int>`) JIT-компилятор создает уникальный машинный код. Элементы лежат в массиве плотно, без упаковки ([[Boxing Unboxing|boxing]]) и накладных расходов.
*   Для **ссылочных типов** (например, `List<string>`) JIT-компилятор генерирует один общий код, так как все ссылки имеют одинаковый размер (указатель).

### Пример кода ([[C#]] 12)

```csharp
using System;
using System.Collections.Generic;

public class ListDemo
{
    public static void Main()
    {
        // 1. Инициализация через Collection Expressions (C# 12)
        // Если емкость не указана, она вычисляется автоматически
        List<string> fruits = ["Apple", "Banana", "Orange"];

        // 2. Инициализация с заранее заданной емкостью (Best Practice)
        List<int> numbers = new(capacity: 100);
        Console.WriteLine($"Count: {numbers.Count}, Capacity: {numbers.Capacity}"); // 0, 100

        // 3. Быстрое добавление в конец - O(1)
        numbers.Add(42);

        // 4. Медленная вставка в начало - O(n), требует сдвига всех элементов в памяти
        fruits.Insert(0, "Mango");

        // 5. Поиск индекса элемента - O(n) (линейный перебор)
        int index = fruits.IndexOf("Banana");

        // 6. Получение защищенного представления только для чтения
        IReadOnlyList<string> readOnlyFruits = fruits.AsReadOnly();
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Избыточные аллокации (GC Pressure)
> Если вы создаете пустой `List<T>` и в цикле добавляете туда 10 000 элементов, список расширится более 11 раз (4 -> 8 -> 16 ... -> 16384). Это создаст 11 временных массивов в куче, которые нагрузят сборщик мусора. 
> **Решение:** Всегда указывайте примерную емкость в конструкторе, если размер известен заранее: `new List<T>(capacity: 10000)`.

> [!WARNING] Ошибка: Изменение списка во время итерации
> Изменение состава списка (`Add`, `Remove`) внутри цикла `foreach` выбросит `InvalidOperationException`. Итератор отслеживает версию списка (внутреннее поле `_version`) и падает, если она изменилась во время обхода. Для изменения списка используйте цикл `for` в обратном направлении (с конца к началу).

**Best Practices:**
1.  **Абстракция в API:** Не возвращайте сырой `List<T>` из публичных методов или свойств класса, если внешнему коду не нужно его модифицировать. Возвращайте `IReadOnlyList<T>` или `IEnumerable<T>`. Это защитит ваши данные от несанкционированных изменений.
2.  **Эффективное удаление:** Если нужно удалить элементы, соответствующие условию, используйте метод `RemoveAll(Predicate<T>)` вместо удаления в цикле `foreach`. Он выполняет сдвиг элементов в памяти за один проход, что намного быстрее.
3.  **TrimExcess:** Используйте метод `TrimExcess()`, если список сильно уменьшился в размере (например, было 10 000 элементов, осталось 10) и вы хотите освободить неиспользуемую память внутреннего массива.
4.  **Избегайте частых вставок в начало:** Если специфика задачи требует постоянного добавления элементов в начало коллекции, рассмотрите использование `LinkedList<T>` или `Queue<T>`.

> [!TIP] Метод коллекции Exists vs LINQ Any
> Метод экземпляра `list.Exists(x => x == val)` работает немного быстрее, чем метод расширения LINQ `list.Any(x => x == val)`, так как избегает выделения памяти под объект-делегат и вызовов через интерфейс `IEnumerable`.
