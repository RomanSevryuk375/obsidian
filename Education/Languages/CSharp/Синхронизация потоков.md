### Суть (The "What")
**Синхронизация потоков** — это набор механизмов для координации совместного доступа нескольких потоков к разделяемым ресурсам (памяти, файлам, портам). Без синхронизации возникает **состояние гонки (Race Condition)**, когда итоговый результат зависит от непредсказуемого порядка переключения контекста процессором.

Синхронизация решает задачу обеспечения **потокобезопасности (Thread Safety)**, гарантируя, что критический участок кода выполняется атомарно или в строго определенном порядке, сохраняя целостность данных.

---

### Как это работает (The "How" / Nutshell style)

Синхронизация в .NET делится на три уровня по степени нагрузки на систему:

#### 1. Блокирующие конструкции (Exclusive Locking)
Самый распространенный механизм — оператор `lock`.
*   **Механика:** [[lock]] является синтаксическим сахаром для класса [[Monitor]]. Он использует **Sync Block Index** в заголовке объекта в куче.
*   **Низкий уровень:** Когда поток входит в `lock`, он захватывает "монитор" объекта. Если другой поток пытается войти, он переходит в состояние `WaitSleepJoin` и блокируется ОС, не потребляя ресурсы CPU, но вызывая дорогой **контекстный переход (Context Switch)**.

#### 2. Сигнальные конструкции (Signaling)
Используются, когда один поток должен ждать уведомления от другого (например, данные загружены).
*   **Event Wait Handles:** [[AutoResetEvent]], `ManualResetEventSlim`.
*   **Slim-версии:** .NET предоставляет версии с суффиксом `Slim` (напр. [[SemaphoreSlim]]). Они сначала "крутятся" в цикле (spin) в пользовательском режиме, надеясь на быстрый ответ, и только потом запрашивают дорогую блокировку у ядра ОС.

#### 3. Неблокирующие конструкции (Non-blocking)
Самые быстрые механизмы, работающие на уровне инструкций процессора.
*   **Interlocked:** Выполняет атомарные операции (Increment, Add, Exchange). Работает на базе инструкции процессора **CAS (Compare-And-Swap)**.
*   **Memory Barriers:** Инструкция `Thread.MemoryBarrier` (или ключевое слово `volatile`) предотвращает переупорядочивание инструкций компилятором или процессором, гарантируя, что запись в память будет видна всем ядрам немедленно.

> [!INFO] Нюанс: Spinning vs Blocking
> Если ожидание ресурса длится микросекунды, выгоднее заставить поток "бежать на месте" (Spinning), чем усыплять его. Усыпление и пробуждение потока занимает тысячи циклов процессора. `Monitor` и `SemaphoreSlim` используют гибридный подход: сначала недолгое вращение, затем блокировка.

### Пример кода ([[MOC|C#]] 12)

```csharp
using System;
using System.Threading;

public class BankAccount(decimal initialBalance)
{
    private decimal _balance = initialBalance;
    private readonly object _lockObject = new(); // Рекомендуемый объект для блокировки
    private int _operationCount = 0;

    public void Deposit(decimal amount)
    {
        // 1. Эксклюзивная блокировка (Monitor под капотом)
        lock (_lockObject)
        {
            _balance += amount;
        }

        // 2. Атомарная операция (быстрее чем lock для простых типов)
        Interlocked.Increment(ref _operationCount);
    }

    public decimal GetBalance()
    {
        // Чтение должно быть синхронизировано, если запись не атомарна
        lock (_lockObject)
        {
            return _balance;
        }
    }
}

public class Program
{
    public static void Main()
    {
        var account = new BankAccount(1000);
        
        // Массовый параллельный запуск
        Parallel.For(0, 1000, _ => account.Deposit(10));
        
        Console.WriteLine($"Final balance: {account.GetBalance()}");
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Блокировка на публичных объектах
> Никогда не используйте `lock(this)`, `lock(typeof(MyClass))` или `lock("string")`. Эти объекты доступны извне. Если сторонний код тоже сделает `lock` на вашем объекте, возникнет **Deadlock** (взаимная блокировка), которую невозможно отладить. Всегда создавайте приватное поле `private readonly object _locker = new();`.

> [!WARNING] Ошибка: Чрезмерная блокировка (Lock Convoys)
> Слишком длинные блоки `lock` превращают многопоточное приложение в однопоточное. Держите внутри `lock` только те строки, которые непосредственно меняют общее состояние.

**Best Practices:**
1.  **Порядок захвата:** Если нужно захватить два лока, всегда делайте это в одном и том же порядке во всех потоках (сначала A, потом B), чтобы избежать дедлоков.
2.  **SemaphoreSlim:** Используйте его вместо `lock` в асинхронном коде, так как `lock` не поддерживает `await` внутри своего блока.
3.  **ReaderWriterLockSlim:** Если у вас много читающих потоков и мало пишущих, этот механизм позволит "читателям" работать параллельно, блокируя систему только на время записи.
4.  **Атомарность:** Для простых типов (`int`, `long`, `bool`) всегда отдавайте приоритет классу `Interlocked`. Это на порядок быстрее, чем `lock`.

> [!TIP] Нюанс: volatile
> Пометка поля как `volatile` говорит компилятору не кэшировать значение в регистрах процессора. Однако это не делает сложные операции (вроде `x++`) атомарными. Это лишь гарантирует актуальность значения при чтении.
