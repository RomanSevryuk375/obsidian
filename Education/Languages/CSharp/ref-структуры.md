### Суть (The "What")
**ref [[Структуры|struct]]** — это специализированный вид значимого типа, который имеет жесткое ограничение: он может существовать **только в стеке**. Экземпляры таких структур никогда не могут попасть в управляемую кучу (Heap).

Эта возможность была введена в [[MOC#/MOC|C#]] 7.2 (и расширена в последующих версиях) для обеспечения безопасной и высокопроизводительной работы с памятью. Основная цель `ref struct` — [[Инкапсуляция|инкапсуляция]] типов, которые сами содержат ссылки на стек или специфические области памяти, таких как `Span<T>` или `ReadOnlySpan<T>`.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Гарантия стека (Stack-only constraint)
[[CLR]] и компилятор совместно обеспечивают невозможность перемещения `ref struct` в кучу. Это накладывает ряд строгих ограничений на их использование:
*   **Запрет упаковки ([[Boxing Unboxing]]):** `ref struct` невозможно привести к типу `object`, `ValueType` или любому [[Интерфейсы|интерфейсу]].
*   **Запрет полей в классах:** `ref struct` не может быть полем обычного [[Классы и объекты|класса]] или обычной структуры. Она может быть полем только в другой `ref struct`.
*   **Запрет массивов:** Нельзя создать массив `ref struct[]`, так как массивы всегда живут в куче.
*   **Запрет в асинхронных методах:** Нельзя использовать внутри `async` методов, итераторов (`yield`) или [[Лямбда-выражения|лямбда-выражений]], которые захватывают переменные. Это связано с тем, что компилятор генерирует для них классы-состояния (State Machines), которые живут в куче.

#### 2. Interior Pointers и безопасность
`ref struct` часто используются для хранения "внутренних указателей" (interior pointers). Обычные структуры в куче могут перемещаться сборщиком мусора, что сделало бы такие указатели невалидными. Поскольку `ref struct` всегда в стеке, указатели внутри них остаются стабильными относительно жизненного цикла кадра стека.

#### 3. readonly ref struct (C# 12+)
В современных версиях C# рекомендуется комбинировать `readonly` и `ref`, если структура не предполагает изменения своих полей. Это позволяет компилятору избегать создания "защитных копий" при передаче структуры в методы.

#### 4. Dispose в ref-структурах
Хотя `ref struct` не может реализовать интерфейс `IDisposable` (из-за риска упаковки), компилятор распознает в них метод `public void Dispose()`. Это позволяет использовать их в блоках `using`.

### Пример кода (C# 12)

Пример высокопроизводительного парсера, который работает с буфером в стеке и не создает объектов в куче.

```csharp
using System;

public ref struct StackBufferParser(ReadOnlySpan<char> buffer)
{
    private readonly ReadOnlySpan<char> _buffer = buffer;
    private int _position = 0;

    // Метод парсинга, работающий исключительно со стеком
    public ReadOnlySpan<char> ReadNextToken(char delimiter)
    {
        int nextDelimiter = _buffer[_position..].IndexOf(delimiter);
        
        if (nextDelimiter == -1)
        {
            var remaining = _buffer[_position..];
            _position = _buffer.Length;
            return remaining;
        }

        var token = _buffer.Slice(_position, nextDelimiter);
        _position += nextDelimiter + 1;
        return token;
    }

    // "Утиная типизация" для поддержки блока using
    public void Dispose()
    {
        // Логика очистки, если необходима
    }
}

public class Program
{
    public static void Main()
    {
        // Выделяем память в стеке (stackalloc)
        Span<char> rawMemory = stackalloc char[64];
        "user_id:12345:active".AsSpan().CopyTo(rawMemory);

        // Используем ref-структуру для обработки данных без аллокаций в Heap
        using var parser = new StackBufferParser(rawMemory);
        
        while (true)
        {
            var token = parser.ReadNextToken(':');
            if (token.IsEmpty) break;
            Console.WriteLine($"Token: {token.ToString()}");
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Попытка использования в Task/Async
> Если вам нужно передать данные из `ref struct` в асинхронный метод, вы должны сначала скопировать их в обычный тип (например, `string` или массив), иначе код не скомпилируется. `ref struct` и `async` — физически несовместимые концепции в .NET.

> [!WARNING] Ошибка: Передача по значению больших ref-структур
> Несмотря на то что `ref struct` живет в стеке, если она содержит много полей (например, несколько `Span`), её передача по значению может быть дорогой. Используйте модификатор `in` для передачи "по ссылке только для чтения".

**Best Practices:**
1.  **Инструмент для Span:** Используйте `ref struct` в первую очередь тогда, когда вам нужно создать тип, хранящий внутри себя `Span<T>` или `ReadOnlySpan<T>`.
2.  **Производительность:** Применяйте их в "горячих" участках кода (парсинг, сериализация, работа с сетью), где аллокации в куче создают критическую нагрузку на Garbage Collector.
3.  **Readonly:** Всегда помечайте `ref struct` как `readonly`, если это возможно. В C# 12 это дает значительные оптимизации при работе с `in`-параметрами.
4.  **Локальность:** Помните, что `ref struct` — это кратковременный объект. Он идеально подходит для паттерна "Unit of Work" внутри одного метода или небольшой цепочки синхронных вызовов.

> [!TIP] Нюанс: ref fields (C# 11+)
> Начиная с C# 11, внутри `ref struct` можно объявлять настоящие `ref` поля (`public ref int Field`), что позволяет создавать еще более гибкие низкоуровневые обертки над данными в стеке.
