### Суть (The "What")
Помимо базовых столпов ([[Инкапсуляция|инкапсуляция]], [[Наследование|наследование]], [[Полиморфизм|полиморфизм]]), [[MOC#/MOC|C#]] предлагает продвинутые механизмы, которые расширяют классическую объектную модель. К ним относятся **методы расширения**, **частичные типы**, **анонимные типы** и **рекорды ([[records]])**.

Эти возможности решают задачи снижения синтаксического шума, поддержки кодогенерации, обеспечения неизменяемости (immutability) данных и расширения функционала закрытых типов без их модификации.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Методы расширения ([[Extension Methods]])
Позволяют "добавлять" методы в существующие типы без создания производного типа или изменения оригинала.
*   **Механика:** Это синтаксический сахар для статических методов. Компилятор ищет статический метод, где первый параметр помечен ключевым словом `this`.
*   **[[CLR]]:** На низком уровне это обычный вызов статического метода. Он не имеет доступа к `private` членам расширяемого типа.

#### 2. Рекорды ([[Records]]) — C# 9.0+
Специальный тип (может быть `class` или `struct`), ориентированный на хранение данных, а не на поведение.
*   **Value Equality:** В отличие от обычных классов, рекорды сравниваются по значениям полей, а не по ссылке. Компилятор автоматически генерирует переопределения `Equals`, `GetHashCode` и оператора `==`.
*   **Неизменяемость:** С помощью `with`-выражений реализуется "неразрушающее изменение" (создание копии с измененными полями).

#### 3. Частичные типы и методы (Partial)
Позволяют разделять определение класса, структуры или интерфейса на несколько файлов.
*   **Механика:** Все части объединяются компилятором в один тип **до** генерации [[IL]]-кода. 
*   **Применение:** Идеально для кодогенерации (например, дизайнеры UI или мапперы БД), чтобы пользовательский код и генерируемый код не мешали друг другу.

#### 4. Вложенные типы (Nested Types)
Тип, объявленный внутри другого класса.
*   **Инкапсуляция:** Вложенный тип имеет доступ к приватным членам внешнего типа.
*   **Область видимости:** Часто используются для реализации вспомогательной логики (например, итераторов), которая не должна быть видна вовне.

> [!INFO] Нюанс: Разрешение имен в Extension Methods
> Если сигнатура метода расширения совпадает с сигнатурой метода самого класса, приоритет всегда отдается **методу класса**. Методы расширения имеют самый низкий приоритет при поиске компилятором подходящего вызова.

### Пример кода (C# 12)

```csharp
using System;

namespace AdvancedOop;

// 1. Record: краткая запись (Primary Constructor), неизменяемость и Value Equality
public record User(string Name, int Age);

// 2. Частичный класс (Файл 1)
public partial class ComplexProcessor
{
    public partial void LogStep(string message); // Частичный метод
}

// 3. Частичный класс (Файл 2)
public partial class ComplexProcessor
{
    public partial void LogStep(string message) => Console.WriteLine($"[LOG]: {message}");
}

// 4. Методы расширения
public static class StringExtensions
{
    public static bool IsLong(this string str) => str.Length > 10;
}

public class Program
{
    public static void Main()
    {
        // Работа с Record
        var user1 = new User("Artyom", 30);
        var user2 = user1 with { Name = "Ivan" }; // Неразрушающее изменение
        
        var user3 = new User("Artyom", 30);
        Console.WriteLine(user1 == user3); // True (сравнение по значениям)

        // Использование метода расширения
        string text = "Hello World!";
        if (text.IsLong()) Console.WriteLine("Very long string");

        // Анонимный тип
        var anon = new { Id = 1, Temp = 36.6 };
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Загрязнение глобального пространства методов
> Не создавайте слишком много методов расширения для базовых типов вроде `object` или `string` в глобальных пространствах имен. Это приведет к тому, что IntelliSense будет предлагать десятки ненужных методов в каждом месте программы. Ограничивайте их видимость через `namespace`.

> [!WARNING] Ошибка: Изменяемые рекорды
> Хотя технически можно создать `record` с изменяемыми полями (`set;`), это убивает саму идею типа. Рекорды должны быть неизменяемыми (`init;`), чтобы гарантировать стабильность хеш-кода при использовании в качестве ключей в Dictionary.

**Best Practices:**
1.  **Partial для логики:** Используйте `partial`, если ваш класс становится слишком большим из-за разных аспектов (например, обработка данных + логика UI), но только если это действительно оправдано архитектурно.
2.  **Extension Methods для интерфейсов:** Это отличный способ добавить "стандартную реализацию" к интерфейсу без использования DIM (Default Interface Methods). Так сделан LINQ (расширения для `IEnumerable<T>`).
3.  **Nested Types для Helper-ов:** Если класс `B` нужен только внутри класса `A`, сделайте его `private nested`. Это усилит инкапсуляцию.
4.  **with-выражения:** Всегда используйте `with` для рекордов вместо ручного копирования полей. Это безопаснее и учитывает все поля, включая скрытые.

> [!TIP] Нюанс: Анонимные типы
> Анонимные типы в C# являются **ссылочными типами** (классами). Они поддерживают сравнение по значениям (как рекорды), но их нельзя передавать за пределы метода (кроме как через `object` или `dynamic`), так как у них нет имени. Для возврата нескольких значений из метода используйте `ValueTuple`.
