### Суть (The "What")
Ключевое слово `throw` используется для явной генерации (инициирования) исключения во время выполнения программы. Оно сигнализирует системе, что возникла аномальная ситуация, которую текущий блок кода не может обработать самостоятельно.

`throw` позволяет реализовать механизмы защиты кода (Guard Clauses), уведомляя вызывающий код о нарушении контракта метода или недопустимом состоянии данных.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Механика работы [[CLR]]
Когда выполняется инструкция `throw`:
*   Среда выполнения (CLR) приостанавливает обычное выполнение инструкций.
*   Начинается поиск подходящего блока `catch` в текущем методе, а затем вверх по стеку вызовов (процесс **Stack Unwinding**).
*   В момент создания объекта исключения (через `new`) фиксируется **StackTrace** — снимок всех методов, активных в текущем потоке.

#### 2. Throw как выражение (Throw Expressions)
Начиная с [[MOC#/MOC|C#]] 7, `throw` может использоваться не только как отдельная инструкция, но и как выражение. Это позволяет встраивать генерацию исключений в тернарные операторы, выражения объединения с null (`??`) и лямбда-выражения.

#### 3. Переброс исключения (Re-throwing)
Существует критическая разница между двумя способами переброса исключения внутри блока `catch`:
*   **`throw;`**: Перебрасывает то же самое исключение, полностью сохраняя оригинальный StackTrace. Это предпочтительный способ.
*   **`throw ex;`**: CLR воспринимает это как "новое" исключение. Стек вызовов перезаписывается, и информация о том, где ошибка возникла изначально (до попадания в catch), теряется.

> [!INFO] Нюанс: Стоимость производительности
> Основная нагрузка при использовании `throw` ложится на сбор данных о стеке (StackTrace) и размотку стека. На низком уровне это на порядки медленнее, чем обычный возврат значения или использование логических проверок.

### Пример кода (C# 12)

```csharp
using System;

public class UserProfile(string username) // Primary constructor
{
    // Использование throw expression для валидации при инициализации
    public string Username { get; } = username ?? throw new ArgumentNullException(nameof(username));

    public void SetAge(int age)
    {
        // Классический Guard Clause
        if (age is < 0 or > 150)
        {
            throw new ArgumentOutOfRangeException(nameof(age), "Возраст должен быть в диапазоне от 0 до 150.");
        }
    }

    public void ProcessData()
    {
        try 
        {
            // Имитация логики
        }
        catch (Exception ex)
        {
            // Логируем и перебрасываем, сохраняя стек вызовов
            Console.WriteLine($"Ошибка в {nameof(ProcessData)}: {ex.Message}");
            throw; 
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Использование throw для управления логикой
> Никогда не используйте исключения для обычных переходов в программе (например, выход из цикла). Исключения должны быть исключительными. Для ожидаемых ошибок используйте паттерны `Try-Parse` или возвращайте объект результата.

> [!WARNING] Ошибка: Потеря стека при throw ex
> Избегайте конструкции `catch (Exception ex) { ... throw ex; }`. Это затрудняет отладку, так как в логах вы увидите местом возникновения ошибки строку с `throw ex`, а не реальную причину глубоко внутри логики.

**Best Practices:**
1. **Будьте конкретны:** Выбрасывайте максимально специфичные исключения (`ArgumentNullException`, `InvalidOperationException`) вместо базового `Exception`.
2. **Inner Exceptions:** Если вы оборачиваете одно исключение в другое, всегда передавайте оригинальное исключение в конструктор нового: `throw new CustomException("message", ex);`. Это сохранит всю цепочку причин.
3. **Null-coalescing throw:** Используйте краткий синтаксис для проверки на null: `this.data = input ?? throw new ArgumentException(nameof(input));`.
4. **Custom Exceptions:** Создавайте свои классы исключений только если вы планируете их перехватывать отдельно от других. В остальных случаях стандартных типов .NET достаточно.

> [!TIP] Нюанс: ExceptionDispatchInfo
> Если вам нужно сохранить исключение и пробросить его позже (например, из другого потока) без изменения стека, используйте класс `System.Runtime.ExceptionServices.ExceptionDispatchInfo.Throw(ex)`. Это позволяет "телепортировать" исключение между контекстами без потери диагностических данных.
