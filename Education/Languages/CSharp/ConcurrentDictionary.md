### Суть (The "What")
`ConcurrentDictionary<TKey, TValue>` — это высокопроизводительная потокобезопасная хеш-таблица (словарь), предназначенная для одновременного чтения и записи данных из множества параллельных потоков.

В отличие от стандартного [[Dictionary]]`<TKey, TValue>`, требующего блокировки всей таблицы ([[lock]]) при любой записи, `ConcurrentDictionary` минимизирует конкуренцию за ресурсы. Он позволяет разным потокам одновременно обновлять разные части словаря без взаимных блокировок.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Механика расщепления блокировок (Lock Striping)
`ConcurrentDictionary` решает проблему производительности за счет разбиения внутренней структуры данных на сегменты:
*   **Массив блокировок:** Внутри словаря создается массив объектов синхронизации (замок-сегментов). По умолчанию размер этого массива вычисляется на основе количества ядер процессора.
*   **Хеширование:** При обращении по ключу [[CLR]] вычисляет его хеш-код, сопоставляет его с конкретной корзиной (bucket) и захватывает блокировку только для того сегмента, к которому эта корзина привязана.
*   **Параллельная запись:** Если Поток 1 пишет в Сегмент А, а Поток 2 пишет в Сегмент Б, они выполняются параллельно без ожидания.

#### 2. Конструктор: ConcurrencyLevel и Capacity
При создании словаря крайне важно обращать внимание на два параметра конструктора:
*   `concurrencyLevel`: Ожидаемое количество потоков, которые будут одновременно писать в словарь. Это число определяет размер массива внутренних блокировок (сегментов).
*   `capacity`: Начальный размер хеш-таблицы (количество корзин).

#### 3. Атомарные операции (GetOrAdd, AddOrUpdate, TryUpdate)
Словарь предоставляет методы, объединяющие проверку существования ключа и его модификацию в единую операцию:
*   **`GetOrAdd`:** Возвращает значение, если ключ есть, или атомарно создает и добавляет его, если ключа нет.
*   **`AddOrUpdate`:** Атомарно вставляет значение или обновляет его на основе текущего значения.
*   **`TryUpdate`:** Сравнивает текущее значение с ожидаемым и обновляет его только в случае совпадения. Это логический аналог процессорной инструкции CAS (Compare-And-Swap).

> [!DANGER] Неатомарность фабричных делегатов
> [[Делегаты]] (лямбды), передаваемые в `GetOrAdd` (например, `valueFactory`), выполняются **вне внутренней блокировки** сегмента. Это сделано для предотвращения дедлоков при выполнении стороннего кода. Если несколько потоков одновременно вызовут `GetOrAdd` для отсутствующего ключа, их лямбды **выполнятся параллельно несколько раз**, хотя в словарь будет записан результат только одной из них.

> [!WARNING] Накладные расходы на снимки (Snapshots)
> Операции, требующие анализа всего словаря (`Keys`, `Values`, `Count`, `Clear`, `ToArray`), являются чрезвычайно "дорогими". Чтобы получить согласованное состояние, `ConcurrentDictionary` вынужден последовательно **захватить все блокировки всех сегментов** (заморозить коллекцию). Избегайте вызова этих методов в нагруженных циклах.

### Пример кода ([[C#]] 12)

```csharp
using System;
using System.Collections.Concurrent;
using System.Threading.Tasks;

public class CacheService
{
    // Инициализируем словарь с настройками под текущее железо
    private readonly ConcurrentDictionary<string, string> _cache = new(
        concurrencyLevel: Environment.ProcessorCount * 2, 
        capacity: 101
    );

    public string GetOrAddData(string key)
    {
        // GetOrAdd гарантирует потокобезопасность
        // Нюанс: лямбда-экспрешн может выполниться более одного раза при высокой конкуренции
        return _cache.GetOrAdd(key, k => FetchFromDatabase(k));
    }

    public void UpdateData(string key, string newValue)
    {
        // AddOrUpdate атомарно обновит или добавит значение
        _cache.AddOrUpdate(key, 
            newValue, 
            (k, oldValue) => $"{oldValue} -> {newValue}");
    }

    public bool CompareAndSwap(string key, string expectedValue, string newValue)
    {
        // Пытаемся обновить только если текущее значение совпадает с ожидаемым (CAS)
        return _cache.TryUpdate(key, newValue, expectedValue);
    }

    private string FetchFromDatabase(string key)
    {
        Console.WriteLine($"[DB Query] Получение данных для: {key}");
        return $"DataFor_{key}";
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Проверка ContainsKey перед добавлением
> Код `if (!dict.ContainsKey(key)) dict.TryAdd(key, value);` — классическая ошибка Race Condition. Между проверкой и добавлением другой поток успеет вставить ключ. Всегда используйте атомарный метод `GetOrAdd` или просто `TryAdd`.

> [!WARNING] Ошибка: Вызовы тяжелых методов в фабриках GetOrAdd
> Никогда не пишите `dict.GetOrAdd(key, k => LongRunningDatabaseQuery())`. Хотя словарь запишет результат только один раз, метод запроса к БД может быть выполнен параллельно несколькими потоками. Выполняйте тяжелые вычисления *до* обращения к словарю или используйте `Lazy<T>` в качестве значения словаря: `ConcurrentDictionary<string, Lazy<Value>>`.

**Best Practices:**
1.  **Тюнинг производительности:** Если вы знаете максимальное количество пишущих потоков и примерный объем данных, всегда передавайте `concurrencyLevel` и `capacity` в конструктор. Это избавит коллекцию от дорогостоящих перестроений (Rehash) всех сегментов.
2.  **Проверка на пустоту:** Используйте свойство `IsEmpty` вместо `Count == 0`, так как `IsEmpty` проверяет наличие элементов без захвата всех блокировок словаря.
3.  **Использования lazy-значений:** Чтобы гарантировать, что конструктор значения выполнится строго один раз при параллельном `GetOrAdd`, используйте `ConcurrentDictionary<TKey, Lazy<TValue>>`. Метод словаря создаст `Lazy<T>` (это быстро и дешево), а реальное создание тяжелого объекта произойдет внутри `Lazy` под его собственным потокобезопасным флагом.

> [!TIP] Очистка коллекции
> Начиная с .NET Core, метод `Clear()` оптимизирован. Он захватывает блокировки сегментов поочередно, а не все сразу, чтобы минимизировать влияние на читающие потоки, работающие с другими частями словаря.
