### Суть (The "What")
**Виртуальные члены** — это методы, свойства, индексаторы или события, объявленные в базовом классе с модификатором `virtual`, реализация которых может быть переопределена в производных классах с помощью модификатора `override`.

Они решают задачу **[[Динамический полиморфизм|динамического полиморфизма]]**: позволяют коду работать с объектом через ссылку на базовый тип, при этом вызывая реализацию, специфичную для фактического типа объекта во время выполнения (Runtime).

---

### Как это работает (The "How" / Nutshell style)

#### 1. Таблица виртуальных методов ([[VMT]] / vtable)
На низком уровне [[Полиморфизм]] реализуется через **Method Table** (таблицу методов) типа в памяти [[CLR]]:
*   Для каждого типа, содержащего виртуальные члены, создается таблица указателей на адреса исполняемого кода этих членов.
*   Каждый экземпляр объекта в куче содержит в своем заголовке указатель на эту таблицу (`Type Object Pointer`).
*   Когда вызывается виртуальный метод, CLR не переходит по фиксированному адресу (как в обычных методах), а совершает "прыжок" по адресу, найденному в таблице методов фактического типа объекта. Этот процесс называется **поздним связыванием (late binding)**.

#### 2. Свойства как методы
Поскольку свойства в [[MOC#/MOC|C#]] компилируются в методы доступа (`get_PropertyName` и `set_PropertyName`), они подчиняются тем же правилам виртуализации. Переопределение свойства позволяет изменить логику вычисления значения или валидации в наследниках.

#### 3. Модификаторы и цепочки
*   **`virtual`**: Разрешает переопределение. По умолчанию члены не виртуальны (в отличие от Java).
*   **`override`**: Явно указывает, что метод заменяет реализацию из базового класса.
*   **`sealed override`**: Прерывает цепочку виртуализации; дальнейшие наследники не смогут переопределить этот член.
*   **`base`**: Позволяет вызвать реализацию из родительского класса внутри переопределенного члена.

> [!INFO] Нюанс: Стоимость вызова
> Вызов виртуального метода стоит примерно на 2-3 процессорных цикла дороже обычного из-за дополнительного разыменования указателя (индирекции). Однако [[JIT]]-компилятор может оптимизировать этот процесс (devirtualization), если видит, что тип объекта не меняется.

### Пример кода (C# 12)

```csharp
using System;

public class Document(string title)
{
    // Виртуальное свойство
    public virtual string Title { get; set; } = title;

    // Виртуальный метод
    public virtual void Export() => Console.WriteLine($"Exporting document: {Title}");
}

public class PdfDocument(string title, string version) : Document(title)
{
    public string Version { get; } = version;

    // Переопределение свойства с дополнительной логикой
    public override string Title 
    { 
        get => base.Title.ToUpper(); 
        set => base.Title = value; 
    }

    // Переопределение метода
    public override void Export()
    {
        Console.WriteLine($"Exporting PDF {Version}: {Title}");
    }
}

public class Program
{
    public static void Main()
    {
        // Ссылка базового типа, объект производного
        Document doc = new PdfDocument("Annual Report", "v1.2");

        // Вызовется версия из PdfDocument благодаря VMT
        doc.Export(); 
        
        Console.WriteLine(doc.Title); // Выведет заглавными буквами
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Вызов виртуальных членов в конструкторе
> Это одна из самых опасных ошибок в C#. Конструктор базового класса выполняется раньше конструктора производного. Если базовый конструктор вызывает виртуальный метод, который переопределен в потомке, этот метод обратится к полям потомка до того, как они будут инициализированы. Результат: непредсказуемое поведение или `NullReferenceException`.

> [!WARNING] Ошибка: Скрытие (new) вместо переопределения (override)
> Если забыть `override` и написать метод с той же сигнатурой, используя `new`, полиморфизм сломается. При вызове через ссылку базового типа сработает старая логика, а при вызове через производный — новая. Всегда используйте `override`, если намерены изменить поведение.

**Best Practices:**
1.  **Защита архитектуры:** Если метод не предназначен для изменения логики в будущем, не делайте его `virtual`.
2.  **Модификатор sealed:** Если вы переопределяете метод в промежуточном классе и не хотите, чтобы его меняли дальше, используйте `public sealed override`.
3.  **Принцип "Base Call":** При переопределении метода решайте, нужно ли дополнять базовое поведение (`base.Method()`) или полностью заменять его. Ошибка в этом выборе — частая причина багов в иерархиях.
4.  **Абстрактные члены:** Если реализации по умолчанию в базовом классе нет и быть не может, используйте `abstract`. Это обяжет наследников предоставить свою логику.

> [!TIP] Нюанс: Доступность
> Переопределенный метод должен иметь тот же уровень доступа, что и базовый. Вы не можете сделать `public virtual` метод `private override` в наследнике.
