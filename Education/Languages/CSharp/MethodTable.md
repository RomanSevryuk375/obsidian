### Суть (The "What")
**MethodTable (MT)** — это основная структура данных внутри [[CLR]], которая представляет тип в памяти во время выполнения. Если класс — это "чертеж" в коде, то MethodTable — это "паспорт" и "навигационная карта" типа в оперативной памяти.

Каждый экземпляр ссылочного типа в [[Куча|куче]] (Heap) содержит в своем заголовке указатель на соответствующую MethodTable. Это позволяет среде выполнения точно знать, к какому типу относится объект, каков его размер и где находятся его методы.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Место в памяти и жизненный цикл
MethodTable создается загрузчиком типов (ClassLoader) при первом обращении к типу. 
*   **Где хранится:** Вне управляемой кучи, в области под названием **Loader Heap** (точнее, в High Frequency Heap). Эта память не управляется Garbage Collector-ом и освобождается только при выгрузке домена приложения (AppDomain).
*   **Единственность:** На каждый тип (например, `User`) существует ровно одна структура MethodTable, независимо от количества созданных объектов этого типа.

#### 2. Анатомия MethodTable
Структура MethodTable включает в себя:
*   **Flags:** Метаданные о типе (является ли он интерфейсом, массивом, содержит ли финализатор и т.д.).
*   **Component Size:** Размер экземпляра объекта в байтах.
*   **Interface Map:** Указатели на реализации интерфейсов (Interface Dispatch Table).
*   **[[VMT]] (Virtual Method Table):** Список указателей на исполняемый код виртуальных методов. Первые слоты всегда занимают методы `System.Object`.
*   **Parent Pointer:** Указатель на MethodTable базового класса (так выстраивается цепочка наследования).
*   **EEClass:** Ссылка на дополнительную структуру с "холодными" метаданными, которые редко нужны при обычном выполнении (имена полей, атрибуты).

#### 3. Связь с объектом (TypeHandle)
Когда вы вызываете `obj.GetType()`, CLR извлекает адрес MethodTable из заголовка объекта. Этот адрес в .NET также называют **RuntimeTypeHandle**.

> [!INFO] Нюанс: Накладные расходы
> На x64-системах указатель на MethodTable занимает 8 байт. Вместе с Sync Block Index (8 байт) это составляет 16 байт накладных расходов на каждый объект в куче. Даже "пустой" объект будет занимать минимум 24-32 байта из-за этих метаданных и выравнивания памяти.

### Пример кода ([[MOC#/MOC|C#]] 12)

Для демонстрации того, что разные объекты ссылаются на одну MethodTable, можно использовать `RuntimeHelpers`.

```csharp
using System;
using System.Runtime.CompilerServices;

public class User(string name) 
{
    public string Name { get; } = name;
}

public class Program
{
    public static void Main()
    {
        User u1 = new("Alice");
        User u2 = new("Bob");

        // Получаем RuntimeTypeHandle - это и есть обертка над указателем на MethodTable
        RuntimeTypeHandle handle1 = u1.GetType().TypeHandle;
        RuntimeTypeHandle handle2 = u2.GetType().TypeHandle;

        // Адреса в Loader Heap будут идентичны
        IntPtr mtAddress1 = handle1.Value;
        IntPtr mtAddress2 = handle2.Value;

        Console.WriteLine($"MethodTable Address 1: {mtAddress1}");
        Console.WriteLine($"MethodTable Address 2: {mtAddress2}");
        Console.WriteLine($"Same MT: {mtAddress1 == mtAddress2}"); // True

        // Значимые типы тоже имеют MT, но ссылка на них появляется только при boxing
        int x = 42;
        Console.WriteLine($"Int32 MT: {typeof(int).TypeHandle.Value}");
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Попытка изменения MethodTable
> MethodTable — это системная структура CLR. Попытки изменить её через `unsafe` код и прямые манипуляции с указателями приведут к немедленному краху процесса (Access Violation) или повреждению всей среды выполнения.

> [!WARNING] Нюанс: Статические поля
> В старых версиях .NET статические поля хранились прямо внутри MethodTable. В современных реализациях (.NET Core / .NET 5+) статические данные вынесены в отдельный блок памяти, на который MethodTable только ссылается, чтобы улучшить локальность данных и работу GC.

**Best Practices:**
1.  **Понимать стоимость объектов:** Помните, что создание миллионов мелких объектов нагружает память не только вашими данными, но и миллионами указателей на MethodTable.
2.  **Избегать лишних типов:** Каждый новый созданный тип (особенно динамически сгенерированные через `Reflection.Emit`) увеличивает размер Loader Heap, которая не очищается до конца работы приложения.
3.  **Sealed классы:** Пометка класса как `sealed` позволяет [[JIT]]-компилятору игнорировать поиск в VMT внутри MethodTable и вызывать методы напрямую (devirtualization), что работает быстрее.

> [!TIP] Нюанс: Generics
> Для обобщенных типов со ссылочными параметрами (например, `List<string>` и `List<object>`) CLR может использовать **одну и ту же MethodTable** (Generic Code Sharing) для экономии памяти, так как все ссылки имеют одинаковый размер. Для значимых типов (например, `List<int>` и `List<double>`) всегда создаются разные MethodTable.
