### Суть (The "What")
**ValueTask\<T\>** — это структура ([[Структуры|struct]]), представляющая собой объединение (discriminated union) уже готового значения типа `T` или объекта [[Task]]T. 

Она решает проблему **избыточных аллокаций в куче**. В высоконагруженных системах многие асинхронные методы часто завершаются синхронно (например, если данные уже есть в кэше). Использование `Task<T>` в таких случаях заставляет [[CLR]] каждый раз создавать объект в куче, что нагружает сборщик мусора ([[GC]]). `ValueTask` позволяет вернуть значение без аллокации, если операция завершилась мгновенно.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Анатомия структуры
`ValueTask<T>` — это значимый тип. Он содержит либо поле с результатом `T`, либо ссылку на `Task<T>`, либо ссылку на интерфейс `IValueTaskSource<T>`.
*   **Синхронный путь:** Если результат готов, свойство `IsCompleted` возвращает `true`, и значение извлекается напрямую из структуры без обращения к куче.
*   **Асинхронный путь:** Если операция требует ожидания, структура оборачивает реальный `Task<T>` или использует `IValueTaskSource` для управления состоянием.

#### 2. IValueTaskSource
Это продвинутый механизм, позволяющий реализовывать асинхронные операции с использованием переиспользуемых (pooled) объектов. Это позволяет достичь практически "нулевых аллокаций" даже при реальной асинхронности, так как объект состояния не создается заново для каждого вызова.

#### 3. Стоимость использования
По заветам Албахари, `ValueTask` не является "бесплатной" заменой `Task`.
*   Поскольку это структура, она занимает больше места в памяти, чем простая ссылка на `Task` (которая занимает 8 байт на x64).
*   Копирование структуры при передаче между методами стоит дороже, чем копирование ссылки.
*   Если метод **гарантированно** выполняется асинхронно, обычный `Task<T>` будет эффективнее.

> [!IMPORTANT] Ограничение: Однократное ожидание
> В отличие от `Task`, экземпляр `ValueTask` можно **ожидать (await) только один раз**. Также нельзя вызывать `.Result` или `.GetAwaiter().GetResult()` несколько раз. После первого использования объект может быть переиспользован рантаймом (если задействован пул), и повторное обращение приведет к неопределенному поведению или ошибке.

### Пример кода ([[MOC|C#]] 12)

Реализация кэширующего сервиса, где `ValueTask` дает максимальный профит.

```csharp
using System;
using System.Collections.Generic;
using System.Threading.Tasks;

public class DataService(IDataProvider provider) // Primary constructor
{
    private readonly Dictionary<int, string> _cache = [];
    private readonly IDataProvider _provider = provider;

    public async ValueTask<string> GetDescriptionAsync(int id)
    {
        // Синхронный путь: данные в кэше, аллокации Task не происходит
        if (_cache.TryGetValue(id, out var cachedData))
        {
            return cachedData;
        }

        // Асинхронный путь: происходит аллокация или использование IValueTaskSource
        string data = await _provider.FetchFromDatabaseAsync(id);
        _cache[id] = data;
        
        return data;
    }
}

public interface IDataProvider
{
    Task<string> FetchFromDatabaseAsync(int id);
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Использование Task.WhenAll / WhenAny
> `ValueTask` не поддерживается методами `Task.WhenAll` или `Task.WhenAny`. Если вам нужно сгруппировать несколько задач, вы должны сначала преобразовать их в обычные задачи методом `.AsTask()`.

> [!WARNING] Ошибка: Многократное ожидание
> Никогда не сохраняйте `ValueTask` в переменную для повторного ожидания. 
> Плохо: `var vt = DoWork(); await vt; await vt;`
> Если нужно сохранить задачу для множественного доступа, сразу вызывайте `.AsTask()`.

**Best Practices:**
1.  **Тип возвращаемого значения:** Используйте `ValueTask<T>` вместо `Task<T>`, если:
    *   Метод вызывается очень часто (высокая частота вызовов).
    *   Значительный процент вызовов (например, >25%) завершается синхронно.
    *   Операция работает с пулом объектов.
2.  **Для void-методов:** Существует также нетипизированный `ValueTask`. Он полезен реже, чем `ValueTask<T>`, так как для `Task` (без результата) в .NET уже есть закэшированный `Task.CompletedTask`.
3.  **Не блокируйте:** Вызов `.Result` у `ValueTask` так же опасен (deadlocks), как и у `Task`, если операция еще не завершена.
4.  **Синтаксис await:** Самый безопасный способ работы — не сохранять `ValueTask` в переменную, а сразу делать `await GetValueAsync();`.

> [!TIP] Нюанс производительности
> Если ваш метод возвращает `ValueTask` и вы вызываете его из другого `async` метода, конечный автомат (State Machine) компилятора умеет работать с `ValueTask` напрямую, не упаковывая его без необходимости. Это делает современные асинхронные цепочки в .NET очень эффективными.
