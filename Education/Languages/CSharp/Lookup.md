### Суть (The "What")
**Lookup** — это специализированная структура данных "один ко многим", которая сопоставляет один ключ с последовательностью ([[Коллекции|коллекцией]]) значений. В отличие от [[Dictionary]]\<TKey, TValue>, где ключ связан строго с одним значением, `ILookup` представляет собой неизменяемую коллекцию сгруппированных данных.

Этот тип решает задачу эффективного поиска групп элементов по общему признаку. Он избавляет от необходимости вручную создавать структуры вида `Dictionary<TKey, List<TValue>>` и управлять инициализацией списков для новых ключей.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Неизменяемость (Immutability)
`ILookup<TKey, TElement>` спроектирован как коллекция **только для чтения**. В публичном API .NET нет прямого конструктора или методов `Add`/`Remove` для этого типа. Единственный стандартный способ создать его — использовать метод расширения [[LINQ]] `ToLookup()`.

#### 2. Внутреннее устройство
На низком уровне стандартная реализация (находящаяся в пространстве `System.Linq`) ведет себя аналогично хеш-таблице:
*   **Хеширование:** Ключи распределяются по корзинам (buckets) на основе `GetHashCode()`.
*   **Группировка:** Каждая корзина содержит объекты `Grouping`, которые реализуют интерфейс `IGrouping<TKey, TElement>`.
*   **Сложность:** Поиск последовательности по ключу выполняется за **$O(1)$**.

#### 3. Отказоустойчивость
В отличие от [[Dictionary]], индексатор `ILookup` никогда не выбрасывает исключение `KeyNotFoundException`.
*   Если ключ отсутствует в коллекции, `Lookup` просто вернет **пустую последовательность** (`Enumerable.Empty<TElement>`).

#### 4. Группировка против Словаря
Метод `ToLookup()` вычисляется **немедленно** (eager evaluation). Это отличает его от метода `GroupBy()`, который возвращает ленивую последовательность. `Lookup` кэширует все данные в памяти в момент вызова.

> [!INFO] Нюанс: null-ключи
> `Lookup` поддерживает `null` в качестве ключа. При вызове `ToLookup` элементы с `null`-значением ключа будут сгруппированы в отдельную последовательность под этим ключом.

### Пример кода ([[MOC|C#]] 12)

```csharp
using System;
using System.Collections.Generic;
using System.Linq;

public record Product(string Category, string Name, decimal Price);

public class LookupDemo
{
    public static void Main()
    {
        // 1. Исходные данные (Collection Expression)
        List<Product> products = 
        [
            new("Electronics", "Phone", 500),
            new("Electronics", "Laptop", 1000),
            new("Food", "Apple", 1),
            new("Food", "Bread", 2)
        ];

        // 2. Создание Lookup (немедленное выполнение)
        // Ключ - категория, значение - название продукта
        ILookup<string, string> categoryLookup = products.ToLookup(
            p => p.Category, 
            p => p.Name
        );

        // 3. Безопасный доступ
        // Если категории "Books" нет, вернется пустой список, а не ошибка
        var electronics = categoryLookup["Electronics"];
        var books = categoryLookup["Books"]; 

        Console.WriteLine($"Electronics count: {electronics.Count()}"); // 2
        Console.WriteLine($"Books count: {books.Count()}");             // 0

        // 4. Итерация по группам
        foreach (IGrouping<string, string> group in categoryLookup)
        {
            Console.WriteLine($"Category: {group.Key}");
            foreach (string name in group)
            {
                Console.WriteLine($" - {name}");
            }
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: ToLookup для динамических данных
> Не используйте `ToLookup`, если вам нужно постоянно добавлять или удалять элементы из групп. Поскольку он неизменяем, при каждом изменении данных вам придется пересоздавать весь `Lookup` заново через LINQ, что крайне неэффективно по памяти. В таких случаях используйте `Dictionary<TKey, List<TValue>>`.

> [!WARNING] Ошибка: Потребление памяти
> Помните, что `ToLookup` — это терминальный оператор LINQ. Он немедленно вычитывает всю входную последовательность и строит структуру в оперативной памяти. Для очень больших наборов данных (миллионы строк из БД) лучше использовать `GroupBy`, чтобы обрабатывать группы потоково (лениво).

**Best Practices:**
1.  **Замена вложенным циклам:** Используйте `Lookup` для оптимизации поиска в сценариях "Join на клиенте". Поиск в `Lookup` за $O(1)$ гораздо быстрее, чем поиск в списке через `.Where(...)` внутри другого цикла (что дает $O(N^2)$).
2.  **Имена параметров:** При вызове `ToLookup` всегда явно указывайте селектор элементов (`elementSelector`), если вам не нужен весь объект целиком — это снижает потребление памяти.
3.  **Count vs Any:** Если вам нужно проверить существование ключа, используйте `lookup.Contains(key)`. Это быстрее и понятнее, чем проверять `lookup[key].Any()`.

> [!TIP] Нюанс: IGrouping
> Каждый элемент внутри `ILookup` — это `IGrouping`, который сам является `IEnumerable`. Это позволяет применять дополнительные методы LINQ к каждой отдельной группе сразу после извлечения.
