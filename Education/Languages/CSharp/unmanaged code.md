### Суть (The "What")
**Неуправляемые ресурсы** — это ресурсы, которые создаются и управляются операционной системой вне контроля среды выполнения .NET ([[CLR]]). К ним относятся дескрипторы файлов (File Handles), сетевые сокеты, оконные дескрипторы, соединения с базами данных и блоки памяти, выделенные через нативный код (например, `Marshal.AllocHGlobal`).

Главная проблема: **Garbage Collector ([[GC]]) ничего не знает об этих ресурсах**. Он видит только небольшой управляемый объект-обертку в куче, но не видит тяжеловесный ресурс ОС, который за ним стоит. Если не освободить такой ресурс явно, возникнет утечка ресурсов (resource leak), которая может привести к исчерпанию лимитов ОС задолго до того, как сработает сборщик мусора.

### Как это работает (The "How" / Nutshell style)

#### 1. Механизмы детерминированной очистки
Поскольку GC работает недетерминированно (мы не знаем, когда он запустится), в .NET принят паттерн **[[IDisposable]]**.
*   Объект, владеющий неуправляемым ресурсом, реализует метод `Dispose()`.
*   Вызывающий код обязан вызвать этот метод вручную или через конструкцию `using`.

#### 2. Финализаторы ([[Finalizers]])
Это "последняя линия обороны". Если разработчик забыл вызвать `Dispose()`, CLR может попытаться освободить ресурс в финальном деструкторе (`~ClassName`).
*   **Низкий уровень:** Объекты с финализаторами помещаются в специальную очередь ([[Finalization Queue]]).
*   Когда GC обнаруживает, что такой объект мертв, он перемещает его в очередь **F-Reachable**.
*   Специальный поток финализации ([[Finalizer Thread]]) проходит по этой очереди и вызывает деструкторы.
*   **Цена:** Объекты с финализаторами живут как минимум на одну сборку мусора дольше (переходят в следующее поколение), что увеличивает давление на память.

#### 3. SafeHandle
В современном .NET (начиная с 2.0, но критично сейчас) вместо прямой работы с `IntPtr` для дескрипторов используется класс `System.Runtime.InteropServices.SafeHandle`.
*   Он инкапсулирует критическую логику финализации.
*   Предотвращает ошибки типа "race condition" при переработке дескрипторов.
*   CLR гарантирует, что дескриптор будет закрыт, даже если при выполнении метода произойдет исключение `ThreadAbortException` или `AppDomainUnload`.

> [!INFO] Нюанс: Стоимость финализации
> По заветам Албахари, финализаторы следует реализовывать только в тех классах, которые напрямую владеют нативными ресурсами через `IntPtr`. Если ваш класс владеет другими `IDisposable` объектами, финализатор ему **не нужен** — достаточно реализации `Dispose`.

### Пример кода ([[MOC|C#]] 12)

Реализация паттерна Disposable с использованием `SafeHandle` и преимуществ C# 12.

```csharp
using System.Runtime.InteropServices;
using Microsoft.Win32.SafeHandles;

public class UnmanagedWrapper : IDisposable
{
    // SafeHandle — лучший способ работы с неуправляемыми дескрипторами
    private SafeHandle? _handle = new SafeFileHandle(IntPtr.Zero, true);
    private bool _disposed = false;

    public UnmanagedWrapper(IntPtr existingHandle)
    {
        _handle = new SafeFileHandle(existingHandle, true);
    }

    public void DoWork()
    {
        ObjectDisposedException.ThrowIf(_disposed, this);
        // Логика работы с дескриптором...
    }

    // Реализация IDisposable
    public void Dispose()
    {
        Dispose(true);
        // Убираем объект из очереди финализации — он уже чист
        GC.SuppressFinalize(this);
    }

    protected virtual void Dispose(bool disposing)
    {
        if (_disposed) return;

        if (disposing)
        {
            // Освобождаем управляемые ресурсы (другие IDisposable объекты)
            _handle?.Dispose();
        }

        // Освобождаем собственные неуправляемые ресурсы (если есть IntPtr)
        // В данном примере SafeHandle сам об этом позаботится
        
        _disposed = true;
    }

    // Финализатор нужен только если есть прямой IntPtr
    // ~UnmanagedWrapper() => Dispose(false);
}

// Использование в C# 12
public class Program
{
    public static void Main()
    {
        // using var обеспечивает вызов Dispose автоматически
        using var wrapper = new UnmanagedWrapper(IntPtr.Zero);
        wrapper.DoWork();
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Вызов Dispose внутри финализатора
> Внутри финализатора (`~ClassName`) или метода `Dispose(false)` **нельзя** обращаться к другим управляемым объектам. В этот момент они уже могут быть уничтожены сборщиком мусора. Допустимо только освобождение нативных ресурсов через `IntPtr`.

> [!WARNING] Ошибка: Игнорирование SuppressFinalize
> Если вы реализовали финализатор и `Dispose`, но забыли вызвать `GC.SuppressFinalize(this)` в методе `Dispose`, объект всё равно попадет в очередь финализации. Это лишняя работа для GC и задержка освобождения памяти.

**Best Practices:**
1.  **using var:** Всегда используйте `using var` (или блок `using`) для объектов, реализующих `IDisposable`. Это гарантирует вызов очистки даже при исключениях.
2.  **SafeHandle по умолчанию:** Избегайте хранения сырых `IntPtr`. Оборачивайте их в наследников `SafeHandle` сразу при получении из нативного кода.
3.  **Идемпотентность:** Метод `Dispose` должен быть написан так, чтобы его можно было вызывать многократно без ошибок.
4.  **ThrowIfDisposed:** В публичных методах класса, который был "закрыт", выбрасывайте `ObjectDisposedException`.

> [!TIP] Нюанс: IAsyncDisposable
> Для ресурсов, требующих асинхронного закрытия (например, потоки записи в облако или сокеты), в C# 8.0+ реализован интерфейс `IAsyncDisposable`. Используйте `await using var`.
