### Суть (The "What")
**Интерфейс** — это полностью абстрактный тип данных, который определяет контракт (набор обязательств) без указания того, как эти обязательства должны быть реализованы. Он описывает возможности типа: "что он умеет делать", а не "чем он является".

Интерфейсы решают задачу **слабой связанности (loose coupling)** и позволяют реализовать **множественное наследование поведения**, которое запрещено для классов. Они являются фундаментом для модульной архитектуры и тестирования (через моки/заглушки).

---

### Как это работает (The "How" / Nutshell style)

#### 1. Механика диспетчеризации (Virtual Stub Dispatch)
В отличие от классов, где вызов метода идет через простую таблицу [[VMT]], вызов через интерфейс технически сложнее:
*   **ITable (Interface Table):** Каждый [[Классы и объекты|класс]], реализующий интерфейсы, имеет дополнительную таблицу, сопоставляющую методы интерфейса с конкретными реализациями в MethodTable класса.
*   **[[VSD]] (Virtual Stub Dispatch):** Современная [[CLR]] использует механизм "заглушек", который кэширует результаты поиска метода. Если тип объекта в точке вызова не меняется, вызов интерфейса по скорости практически идентичен вызову виртуального метода класса.

#### 2. Неявная и явная реализация
*   **Неявная (Implicit):** Метод интерфейса объявляется как `public` в классе. К нему можно обратиться и через ссылку на класс, и через ссылку на интерфейс.
*   **Явная (Explicit):** Метод объявляется с указанием имени интерфейса (`IMyInterface.Method`). Он **невидим** через ссылку на класс и доступен только при приведении к типу интерфейса. Это используется для разрешения конфликтов имен и скрытия деталей реализации.

#### 3. Default Interface Methods (DIM) — [[MOC#/MOC|C#]] 8.0+
Интерфейсы теперь могут содержать реализацию методов по умолчанию. 
*   Это позволяет расширять интерфейс, не ломая уже существующие реализации в сторонних библиотеках.
*   Методы с реализацией по умолчанию ведут себя как "виртуальные методы интерфейса", но не наследуются классами автоматически (доступны только через ссылку на интерфейс).

#### 4. Static Abstract члены — C# 11+
Интерфейсы могут объявлять `static abstract` или `static virtual` члены. Это позволяет использовать [[Полиморфизм]] для статических методов, что критично для обобщенной математики (Generic Math) и фабричных методов.

### Пример кода (C# 12)

```csharp
using System;

// Интерфейс с контрактом и статической абстракцией
public interface IValidator<T> where T : IValidator<T>
{
    // Статический абстрактный метод (C# 11)
    static abstract bool TryParse(string input, out T? result);
    
    // Обычный контракт
    bool IsValid { get; }

    // Метод с реализацией по умолчанию (DIM)
    void PrintStatus() => Console.WriteLine($"Validation status: {IsValid}");
}

public class UserCode(string code) : IValidator<UserCode>
{
    public string Code { get; } = code;

    // Неявная реализация свойства
    public bool IsValid => !string.IsNullOrWhiteSpace(Code) && Code.Length > 3;

    // Реализация статического метода
    public static bool TryParse(string input, out UserCode? result)
    {
        result = new UserCode(input);
        return result.IsValid;
    }

    // Явная реализация (скрыта от публичного API класса)
    void IDisposable.Dispose() => Console.WriteLine("Explicit dispose");
}

public class Program
{
    public static void Main()
    {
        string input = "USR-123";

        // Вызов статического полиморфного метода
        if (UserCode.TryParse(input, out var user))
        {
            // Вызов метода по умолчанию (требуется приведение к интерфейсу)
            ((IValidator<UserCode>)user).PrintStatus();
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Толстые интерфейсы (Fat Interfaces)
> Нарушение принципа **Interface Segregation (ISP)**. Не создавайте один гигантский интерфейс `IServer` с 50 методами. Разделите его на `IReader`, `IWriter`, `IAuthenticator`. Класс может реализовать все три, но потребители будут зависеть только от того, что им нужно.

> [!WARNING] Ошибка: Злоупотребление DIM
> Не используйте реализации по умолчанию (DIM) как замену базовым классам. DIM предназначены для обеспечения обратной совместимости API, а не для проектирования сложной иерархии бизнес-логики.

**Best Practices:**
1.  **Префикс 'I':** Всегда начинайте имя интерфейса с заглавной буквы `I`. Это стандарт индустрии.
2.  **Проектирование через интерфейсы:** Принимайте в методах интерфейсы (`IEnumerable<T>`, `IList<T>`), а не конкретные типы (`List<T>`). Это делает ваш код гибким.
3.  **Явная реализация для чистоты API:** Если класс реализует интерфейс для внутренних нужд инфраструктуры (например, `IDisposable` или `ISerializable`), используйте явную реализацию, чтобы не засорять публичное пространство имен класса.
4.  **[[Value Types]] и Интерфейсы:** Помните, что приведение структуры к интерфейсу вызывает **упаковку ([[Boxing]])**. Если производительность критична, используйте generic-ограничения: `void Process<T>(T item) where T : IMyInterface`.

> [!TIP] Нюанс: Интерфейс без методов
> Интерфейсы-маркеры (пустые интерфейсы без методов) в современном C# считаются антипаттерном. Вместо них лучше использовать атрибуты, так как они не влияют на иерархию типов и не создают лишних записей в ITable.
