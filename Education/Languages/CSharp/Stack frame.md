### Суть (The "What")
**Кадр стека (Stack Frame / Activation Record)** — это структурированный блок памяти, который выделяется в [[Стек|стеке]] потока при каждом вызове метода. Он содержит всю контекстную информацию, необходимую для выполнения этого метода и последующего безопасного возврата управления вызывающему коду.

Кадр стека обеспечивает **изоляцию данных метода**. Благодаря ему локальные переменные разных методов не пересекаются в памяти, даже если методы вызывают друг друга рекурсивно.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Анатомия кадра стека
Кадр стека физически ограничивается двумя регистрами процессора:
*   **Stack Pointer (RSP / ESP):** Указывает на вершину стека (текущий конец кадра).
*   **Base/Frame Pointer (RBP / EBP):** Указывает на фиксированную базу (начало) текущего кадра стека. Относительно этого регистра вычисляются адреса локальных переменных и аргументов.

Каждый кадр стека содержит:
1.  **Аргументы метода:** Значения (для значимых типов) или ссылки (для ссылочных типов). На x64 первые 4 аргумента обычно передаются через быстрые регистры процессора (RCX, RDX, R8, R9), а остальные складываются в стек.
2.  **Адрес возврата:** Указатель на инструкцию в коде, которая должна быть выполнена сразу после завершения текущего метода.
3.  **Ссылка на предыдущий кадр стека:** Сохраненное значение регистра `RBP` вызывающего метода.
4.  **Локальные переменные:** Память под все объявленные в методе переменные.

#### 2. Жизненный цикл кадра стека
1.  **Создание (Prologue):** При вызове метода процессор пушит адрес возврата в стек, сохраняет текущий `RBP`, устанавливает `RBP` на новое место и сдвигает `RSP` для выделения памяти под локальные переменные.
2.  **Выполнение:** Метод работает со своими переменными, обращаясь к ним через смещения от `RBP` (например, `[RBP - 8]`).
3.  **Уничтожение (Epilogue):** При выходе из метода (`return`) указатель `RSP` возвращается на место `RBP`, восстанавливается старый `RBP` из стека, и процессор совершает переход по адресу возврата (`ret`). Кадр стека мгновенно исчезает.

#### 3. Встраивание методов ([[JIT]] Inlining)
JIT-компилятор .NET оптимизирует выполнение программы:
*   Если метод маленький (например, простое свойство-геттер), JIT может выполнить **инлайнинг (встраивание)** — заменить вызов метода непосредственно его телом в месте вызова.
*   При инлайнинге **кадр стека для этого метода не создается**, что экономит время процессора и память стека.

#### 4. Диагностика (StackTrace API)
Рантайм позволяет исследовать кадры стека во время выполнения с помощью классов `System.Diagnostics.StackTrace` и `System.Diagnostics.StackFrame`. Это используется для логирования ошибок и построения отладочных отчетов.

### Пример кода ([[MOC|C#]] 12)

Пример программного исследования кадров стека.

```csharp
using System;
using System.Diagnostics;
using System.Reflection;
using System.Runtime.CompilerServices;

public class DiagnosticsDemo
{
    public static void Main()
    {
        OuterMethod();
    }

    [MethodImpl(MethodImplOptions.NoInlining)] // Запрещаем JIT встраивать метод, чтобы сохранить кадр стека
    private static void OuterMethod()
    {
        InnerMethod();
    }

    [MethodImpl(MethodImplOptions.NoInlining)]
    private static void InnerMethod()
    {
        // Получаем снимок текущего стека вызовов
        StackTrace stackTrace = new StackTrace();

        Console.WriteLine($"Количество кадров стека: {stackTrace.FrameCount}\n");

        for (int i = 0; i < stackTrace.FrameCount; i++)
        {
            // Получаем конкретный кадр стека
            StackFrame? frame = stackTrace.GetFrame(i);
            if (frame == null) continue;

            // Извлекаем метод, которому принадлежит этот кадр
            MethodBase? method = frame.GetMethod();
            
            Console.WriteLine($"Кадр [{i}]: {method?.DeclaringType?.Name}.{method?.Name}");
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Использование StackTrace в продакшене для бизнес-логики
> Сбор информации о кадрах стека через `new StackTrace()` — чрезвычайно дорогая операция. Рантайму приходится приостанавливать поток, обходить стек в памяти и сопоставлять адреса с таблицами метаданных (что требует рефлексии). Никогда не используйте `StackTrace` в основной логике приложения (например, для принятия решений о правах доступа).

> [!WARNING] Ошибка: "Исчезновение" кадров в Release-сборках
> В режиме `Release` JIT-компилятор агрессивно оптимизирует код и встраивает (инлайнит) методы. Из-за этого при анализе `StackTrace` вы можете не обнаружить кадры некоторых методов. Если вам критически важно наличие кадра метода в стеке (например, для систем безопасности), помечайте его атрибутом `[MethodImpl(MethodImplOptions.NoInlining)]`.

**Best Practices:**
1.  **Caller Information Attributes:** Вместо тяжелого `StackTrace` для логирования имени вызывающего метода используйте атрибуты компилятора: `[CallerMemberName]`, `[CallerFilePath]`, `[CallerLineNumber]`. Они подставляют данные на этапе компиляции в виде константных строк, что абсолютно бесплатно в рантайме.
2.  **Агрессивное встраивание:** Для мелких, часто вызываемых методов помогайте JIT-компилятору оптимизировать кадры стека, помечая их как `[MethodImpl(MethodImplOptions.AggressiveInlining)]`.
3.  **Рекурсия:** Помните, что каждый шаг рекурсии создает новый кадр стека. Если глубина рекурсии не контролируется, стек переполнится (`StackOverflowException`).

> [!TIP] Свойства Exception.StackTrace
> Свойство `Exception.StackTrace` возвращает строковое представление стека на момент генерации (`throw`) ошибки. Это свойство кэшируется внутри объекта исключения, поэтому повторное обращение к нему не вызывает повторного обхода стека.
