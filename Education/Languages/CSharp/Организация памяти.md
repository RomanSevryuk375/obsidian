# Организация памяти в .NET

### Суть (The "What")
Организация памяти в .NET — это системная архитектура распределения оперативной памяти процесса, управляемая средой [[CLR]] (Common Language Runtime). Память в .NET строго сегментирована на несколько независимых областей, каждая из которых имеет свои правила выделения, времени жизни и очистки данных.

Она решает задачу **автоматического и безопасного управления ресурсами**. Разработчик избавляется от необходимости вручную вызывать деструкторы и управлять указателями, а система защищена от утечек памяти и несанкционированного доступа к чужим адресным пространствам.

---

### Как это работает (The "How" / Nutshell style)

Память работающего .NET-приложения делится на четыре основных сегмента:

```
[ Процесс .NET в ОС ]
 ├── Стек (Stack) — локальный для каждого потока
 ├── Управляемая куча (Managed Heap) — общая, управляется GC
 │    ├── SOH (Small Object Heap): Gen 0, Gen 1, Gen 2
 │    ├── LOH (Large Object Heap): объекты > 85 000 байт
 │    └── POH (Pinned Object Heap): закрепленные объекты
 ├── Куча загрузчика (Loader/Metadata Heap) — метаданные типов
 └── Неуправляемая память (Unmanaged Memory) — системные ресурсы ОС
```

#### 1. [[Стек]] (Stack)
*   **Управление:** Локальный для каждого [[Многопоточность|потока]]. Размер по умолчанию — 1 МБ.
*   **Что хранится:** Локальные переменные, аргументы методов, адреса возврата, значимые типы (`struct`, `int` и т.д.) и указатели на объекты в куче.
*   **Механика:** Память выделяется и освобождается мгновенно по принципу LIFO (Last-In, First-Out) простым сдвигом указателя стека при входе/выходе из метода ([[Stack frame]]).

#### 2. Управляемая куча ([[Managed Heap]])
Единое пространство памяти для всех потоков. Здесь живут ссылочные типы (`class`, `string`, массивы). Управляется Сборщиком мусора ([[GC]]) и разделена на три зоны:
*   **[[SOH]] (Small Object Heap):** Куча малых объектов. Разделена на поколения (Gen 0, 1, 2) для оптимизации сборки. При очистке GC выполняет **компактизацию** (сжимает память, сдвигая живые объекты и обновляя ссылки на них).
*   **[[LOH]] (Large Object Heap):** Сюда попадают объекты размером **более 85 000 байт** (обычно массивы). GC не сжимает LOH по умолчанию, так как копирование больших блоков данных слишком дорого. Это может приводить к фрагментации памяти.
*   **[[POH]] (Pinned Object Heap — .NET 5+):** Специальная куча для объектов, которые "приколоты" в памяти (их нельзя перемещать, например, при передаче в нативный код). Предотвращает фрагментацию SOH.

#### 3. Куча загрузчика (Loader / [[Metadata Heap]])
Область памяти вне контроля GC.
*   **Что хранится:** [[Метаданные]] типов, скомпилированный [[JIT]]-компилятором нативный код, таблицы методов (`MethodTable`).
*   **Время жизни:** Живет до тех пор, пока жив соответствующий домен приложения (`AssemblyLoadContext`) или сам процесс.

#### 4. Неуправляемая память ([[Unmanaged Memory]])
Память за пределами виртуальной машины .NET. Выделяется напрямую через вызовы ОС (например, `Marshal.AllocHGlobal`) или при работе с нативными библиотеками на C/C++. GC полностью слеп к этой памяти — разработчик обязан очищать её вручную.

### Пример кода ([[MOC|C#]] 12)

```csharp
using System;
using System.Runtime.InteropServices;

public class SampleClass { }

public class MemoryDemo
{
    public static void Main()
    {
        // 1. Локальная переменная и указатель - в СТЕКЕ.
        // Сам объект - в КУЧЕ (SOH).
        SampleClass heapObject = new SampleClass();

        // 2. Выделение памяти прямо в СТЕКЕ (stackalloc)
        // Исключает аллокации в куче, живет до конца метода.
        Span<int> stackMemory = stackalloc int[10];
        stackMemory[0] = 42;

        // 3. Выделение памяти в LOH (Large Object Heap)
        // Объект больше 85 000 байт
        byte[] largeArray = new byte[90_000];

        // 4. Выделение НЕУПРАВЛЯЕМОЙ памяти
        IntPtr nativeMemory = Marshal.AllocHGlobal(100); // Выделение
        try
        {
            // Работа с нативной памятью...
        }
        finally
        {
            // Обязательное ручное освобождение!
            Marshal.FreeHGlobal(nativeMemory); 
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: StackOverflowException
> Системный стек потока мал (всего 1 МБ). Бесконечная рекурсия или попытка выделить слишком много памяти в стеке через `stackalloc` (например, более 100 Кб за раз) приведет к немедленному краху всего процесса без возможности перехватить эту ошибку через `try-catch`.

> [!WARNING] Ошибка: Фрагментация LOH
> Частая аллокация и удаление больших массивов (свыше 85 Кб) приводит к фрагментации LOH. Память приложения разрастается, хотя физически она полупустая.
> **Решение:** Для повторно используемых больших буферов используйте пул массивов: `System.Buffers.ArrayPool<T>.Shared.Rent(size)`.

**Best Practices:**
1.  **Используйте [[Span и ReadOnlySpan]]\<T\>:** Типы `Span<T>` и `ReadOnlySpan<T>` позволяют безопасно и единообразно работать со стековой, управляемой и нативной памятью без копирования данных.
2.  **Минимизируйте Pinning:** "Прикалывание" объектов в куче мешает сборщику мусора сжимать память (дефрагментировать SOH). Если нужно приколоть объект, делайте это на минимально короткое время.
3.  **Очистка неуправляемой памяти:** Любой класс, работающий с неуправляемой памятью или нативными дескрипторами ОС, **обязан** реализовывать интерфейс [[IDisposable]] и шаблон очистки ресурсов (Disposable Pattern).
4.  **Сжатие LOH (C# / .NET Core):** При необходимости вы можете попросить GC принудительно сжать LOH при следующей сборке: `GCSettings.LargeObjectHeapCompactionMode = GCLargeObjectHeapCompactionMode.CompactOnce;`.

> [!TIP] Нюанс: Структуры в куче
> Значимый тип (структура) живет в стеке только если является локальной переменной. Если структура объявлена как поле класса `class User { public MyStruct Info; }`, она будет находиться целиком в куче в составе памяти этого класса.
