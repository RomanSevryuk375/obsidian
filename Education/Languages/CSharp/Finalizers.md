### Суть (The "What")
**Финализатор** (исторически называемый деструктором) — это специальный метод класса, который вызывается сборщиком мусора ([[GC]]) перед тем, как объект будет удален из памяти. Его синтаксис: `~ИмяКласса()`.

Финализатор решает задачу **недетерминированной очистки ресурсов**. Это "последняя линия обороны": если разработчик забыл вызвать `Dispose()` для объекта, владеющего неуправляемыми ресурсами (указатели на память, дескрипторы файлов/сокетов), финализатор позволит операционной системе закрыть эти ресурсы, когда объект станет недостижимым.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Механика двухфазной сборки мусора
Наличие финализатора значительно усложняет жизнь GC и замедляет освобождение памяти:
*   **Регистрация:** При создании объекта (через `new`) ссылка на него помещается в специальную **Очередь финализации ([[Finalization Queue]])**.
*   **Первая фаза GC:** Когда объект становится недостижимым, GC видит его в очереди финализации и вместо удаления перемещает его в очередь **[[F-Reachable]]**. В этот момент объект считается "живым" и переживает сборку мусора, переходя в следующее поколение.
*   **Поток финализации:** Специальный высокоприоритетный поток [[CLR]] последовательно вызывает методы `Finalize` для всех объектов из очереди F-Reachable.
*   **Вторая фаза GC:** Только после того, как финализатор отработал, объект становится "по-настоящему" мертвым. При следующей сборке мусора его память будет освобождена.

#### 2. Представление в [[IL]]-коде
Компилятор [[MOC|C#]] преобразует синтаксис `~ClassName()` в переопределение метода `Object.Finalize()`. Код оборачивается в блок `try-finally`, где в секции `finally` вызывается `base.Finalize()`. Это гарантирует, что цепочка финализаторов будет выполнена вверх до `System.Object`.

#### 3. Особенности выполнения
*   Финализаторы вызываются **недетерминированно**: вы не знаете, когда именно это произойдет.
*   Порядок выполнения финализаторов для разных объектов не гарантирован.
*   Все финализаторы выполняются в одном потоке. Если один финализатор "зависнет", он заблокирует очистку всех остальных объектов в очереди.

> [!INFO] Нюанс: Пустые финализаторы
> Никогда не оставляйте в коде пустые финализаторы `~MyClass() {}`. Даже пустой финализатор заставляет GC регистрировать объект в очередях и задерживает освобождение памяти на один или более циклов сборки.

### Пример кода (C# 12)

Стандартная реализация паттерна Dispose, включающая финализатор как страховку.

```csharp
using System.Runtime.InteropServices;

public class UnmanagedResourceWrapper : IDisposable
{
    // Указатель на неуправляемую память
    private IntPtr _buffer;
    private bool _disposed = false;

    public UnmanagedResourceWrapper(int size)
    {
        _buffer = Marshal.AllocHGlobal(size);
    }

    // Финализатор
    ~UnmanagedResourceWrapper()
    {
        // Вызываем очистку только неуправляемых ресурсов
        Dispose(false);
    }

    public void Dispose()
    {
        Dispose(true);
        // Сообщаем GC, что финализатор вызывать не нужно
        GC.SuppressFinalize(this);
    }

    protected virtual void Dispose(bool disposing)
    {
        if (_disposed) return;

        if (disposing)
        {
            // Здесь освобождаем управляемые ресурсы (другие IDisposable объекты)
        }

        // Освобождаем неуправляемые ресурсы
        if (_buffer != IntPtr.Zero)
        {
            Marshal.FreeHGlobal(_buffer);
            _buffer = IntPtr.Zero;
        }

        _disposed = true;
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Доступ к управляемым объектам
> Внутри финализатора (когда `disposing == false`) категорически запрещено обращаться к полям класса, которые являются ссылочными типами. К моменту вызова финализатора эти объекты уже могли быть финализированы и уничтожены сборщиком мусора. Допустимо только освобождение ресурсов по "сырым" указателям (`IntPtr`).

> [!WARNING] Ошибка: Блокировка потока
> Не выполняйте в финализаторе длительных операций или блокировок (lock, ожидание задач). Это парализует работу всего механизма финализации в приложении.

**Best Practices:**
1.  **Избегайте финализаторов:** Используйте их только в классах, которые *напрямую* владеют неуправляемыми ресурсами через `IntPtr`. В 99% случаев достаточно реализации `IDisposable`.
2.  **Используйте SafeHandle:** Вместо написания финализатора для работы с дескрипторами ОС, используйте классы, производные от `System.Runtime.InteropServices.SafeHandle`. Они уже реализуют необходимую логику финализации максимально безопасно.
3.  **GC.SuppressFinalize:** Всегда вызывайте этот метод в `Dispose()`, если у класса есть финализатор. Это сэкономит ресурсы процессора и ускорит освобождение памяти.
4.  **Exceptions:** Исключение, выброшенное в финализаторе и не перехваченное внутри него, обычно приводит к немедленному падению всего приложения (Process Termination).

> [!TIP] Нюанс: Воскрешение объекта (Resurrection)
> Технически внутри финализатора можно присвоить ссылку `this` какому-нибудь статическому полю. Это "воскресит" объект, сделав его снова достижимым. Это считается крайне плохой практикой и вносит хаос в управление памятью, так как финализатор для этого объекта второй раз вызван не будет (если не вызвать `GC.ReRegisterForFinalize`).
