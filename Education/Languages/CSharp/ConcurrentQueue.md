### Суть (The "What")
`ConcurrentQueue<T>` — это высокопроизводительная, потокобезопасная коллекция, реализующая структуру данных **FIFO (First-In, First-Out — «первым пришел, первым ушел»)**. Она позволяет множеству потоков параллельно и безопасно добавлять элементы в конец очереди и извлекать их из её начала.

Она решает задачу **надежной многопоточной последовательной обработки задач**. Это идеальный выбор для классических конвейеров данных (например, логгеры, очереди фоновых задач, брокеры сообщений внутри приложения), где одни потоки только генерируют задачи, а другие только обрабатывают их.

### Как это работает (The "How" / Nutshell style)

#### 1. Сегментированная архитектура (Segmented Array)
В отличие от стандартной [[Queue]]`<T>`, использующей один круговой массив, `ConcurrentQueue<T>` устроена гораздо сложнее:
*   **Связанный список сегментов:** Внутри коллекция представлена в виде связанного списка сегментов (локальных массивов фиксированного размера, обычно по 32 элемента).
*   **Локальность и расширение:** Когда текущий сегмент заполняется, создается новый сегмент и привязывается к концу списка. Это позволяет избегать дорогостоящего копирования всех элементов в один новый массив при расширении (в отличие от [[List]]`<T>` или обычной `Queue<T>`).

#### 2. Lock-Free структура данных (CAS-алгоритмы)
Большинство операций в `ConcurrentQueue<T>` реализованы без использования тяжелых системных блокировок ([[Monitor|lock]] / [[Monitor]]):
*   **Атомарные операции:** Добавление элементов и перемещение указателей `head` и `tail` внутри сегментов происходят с помощью атомарных инструкций процессора (Compare-And-Swap / CAS) через класс `Interlocked`.
*   **Пропускная способность:** Потоки практически не блокируют друг друга при одновременной записи, что обеспечивает колоссальную пропускную способность под высокой нагрузкой.

#### 3. Безопасность итерации и снимки (Snapshots)
Методы `ToArray()`, `Count`, а также обход через `foreach` создают **моментальный снимок (snapshot)** состояния очереди на момент вызова.
*   Вы можете безопасно перебирать элементы очереди через `foreach`, пока другие потоки параллельно добавляют или удаляют данные. Исключение `InvalidOperationException` выброшено не будет.

### Пример кода ([[C#]] 12)

```csharp
using System;
using System.Collections.Concurrent;
using System.Threading.Tasks;

public class JobQueueProcessor
{
    private readonly ConcurrentQueue<string> _jobs = new();

    public async Task StartProcessingAsync()
    {
        // 1. Поток-производитель (наполняет очередь)
        Task producer = Task.Run(() =>
        {
            for (int i = 1; i <= 10; i++)
            {
                _jobs.Enqueue($"Job #{i}"); // Добавление всегда безусловно
                Thread.Sleep(50); // Имитация генерации задач
            }
        });

        // 2. Поток-потребитель (обрабатывает очередь)
        Task consumer = Task.Run(() =>
        {
            int processedCount = 0;
            while (processedCount < 10)
            {
                // Извлечение всегда условное через Try-паттерн
                if (_jobs.TryDequeue(out string? job))
                {
                    Console.WriteLine($"Processed: {job}");
                    processedCount++;
                }
                else
                {
                    // Если очередь пуста, даем процессору отдохнуть
                    Thread.Sleep(10); 
                }
            }
        });

        await Task.WhenAll(producer, consumer);
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Антипаттерн "Check-then-Act"
> Никогда не пишите код вида: `if (!_queue.IsEmpty) { _queue.TryDequeue(out var item); }`. Между проверкой свойства `IsEmpty` и вызовом `TryDequeue` другой поток может успеть забрать последний элемент, и ваш вызов вернет `false`. Всегда вызывайте `TryDequeue` напрямую и проверяйте возвращаемое им `bool` значение.

> [!WARNING] Ошибка: Неконтролируемый рост памяти
> `ConcurrentQueue<T>` по умолчанию является неограниченной (unbounded). Если скорость генерации задач производителями выше, чем скорость их обработки потребителями, очередь будет бесконечно разрастаться, выделяя новые сегменты в [[Куча|куче]], что приведет к `OutOfMemoryException`.
> **Решение:** Оборачивайте очередь в `BlockingCollection<T>` с указанием `boundedCapacity`, чтобы временно блокировать производителей при переполнении буфера.

**Best Practices:**
1.  **Используйте для FIFO конвейеров:** Это лучшая коллекция для сценариев, где строго разделены роли производителей и потребителей данных (в отличие от `ConcurrentBag`, который в этом случае будет работать медленно из-за постоянной "кражи" элементов).
2.  **Избегайте Count:** Для проверки наличия элементов всегда используйте свойство `IsEmpty`. Хотя в очереди вызов `Count` оптимизирован лучше, чем в `ConcurrentBag`, он все равно требует обхода цепочки сегментов.
3.  **Безопасная очистка:** Если нужно полностью очистить очередь, используйте метод `Clear()` (доступен в современных версиях .NET). Он эффективно обнуляет ссылки во всех сегментах.

> [!TIP] Нюанс: Сегменты и GC
> Поскольку память в `ConcurrentQueue<T>` выделяется сегментами (массивами по 32 элемента), удаленные сегменты со временем утилизируются [[GC|сборщиком мусора]]. Это делает управление памятью более гибким по сравнению с классической `Queue<T>`, которая никогда автоматически не уменьшает размер своего внутреннего массива после вызовов `Dequeue`.
