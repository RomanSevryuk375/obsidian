### Суть (The "What")
`HashSet<T>` — это обобщенная коллекция, представляющая собой неупорядоченное множество уникальных элементов. Она исключает наличие дубликатов и оптимизирована для сверхбыстрой проверки нахождения элемента в множестве, а также для выполнения классических математических операций над множествами (объединение, пересечение, разность).

`HashSet<T>` решает задачу **высокопроизводительного контроля уникальности данных**. Поиск, добавление и удаление элементов в нем выполняются за константное время **$O(1)$**, в то время как поиск в обычном списке [[List]]\<T> требует линейного перебора ($O(n)$).

### Как это работает (The "How" / Nutshell style)

#### 1. Внутреннее устройство на базе хеш-таблицы
Конструктивно `HashSet<T>` устроен практически так же, как [[Dictionary]]\<TKey, TValue>:
*   **Отсутствие значений:** `HashSet<T>` — это, по сути, словарь, у которого есть только ключи, а сопоставленные им значения отсутствуют (нет накладных расходов памяти на хранение `TValue`).
*   **Массивы `buckets` и `slots`:** Как и в словаре, здесь используются два внутренних массива. Хеш-код элемента определяет индекс корзины (`bucket`), которая указывает на реальный слот в массиве элементов.
*   **Разрешение коллизий:** Если разные элементы имеют одинаковый хеш-код, они выстраиваются в односвязный список внутри массива слотов (метод цепочек).

#### 2. Зависимость от GetHashCode и Equals
Для корректной работы `HashSet<T>` критически важно поведение методов хранимого типа:
1.  `GetHashCode()` — определяет, в какую корзину попадет элемент.
2.  `Equals()` — проверяет реальное равенство объектов, попавших в одну корзину при коллизии.
Если вы используете кастомные классы, вы должны переопределить оба этих метода (или передать `IEqualityComparer<T>` в конструктор), иначе сравнение будет идти по ссылочной идентичности.

#### 3. Математические операции
`HashSet<T>` предоставляет встроенные методы для работы с множествами (модифицируют исходное множество):
*   `UnionWith` — объединение (добавляет отсутствующие элементы из другой коллекции).
*   `IntersectWith` — пересечение (оставляет только те элементы, которые есть в обеих коллекциях).
*   `ExceptWith` — разность (удаляет элементы, которые есть в другой коллекции).
*   `SymmetricExceptWith` — симметрическая разность (оставляет элементы, которые есть только в одной из коллекций).

### Пример кода ([[C#]] 12)

```csharp
using System;
using System.Collections.Generic;

public class HashSetDemo
{
    public static void Main()
    {
        // 1. Инициализация множества уникальных тегов (C# 12)
        HashSet<string> tags = ["dotnet", "csharp", "architecture"];

        // 2. Попытка добавить дубликат - O(1)
        // Метод Add возвращает false, если элемент уже существует, и ничего не добавляет
        bool addedAgain = tags.Add("csharp");
        Console.WriteLine($"Добавлен повторно? {addedAgain}"); // False

        // 3. Мгновенная проверка наличия - O(1)
        if (tags.Contains("dotnet"))
        {
            Console.WriteLine("Тег 'dotnet' присутствует в множестве.");
        }

        // 4. Математические операции над множествами
        HashSet<string> azureTags = ["cloud", "azure", "dotnet"];

        // Пересечение: оставляем только общие элементы
        tags.IntersectWith(azureTags); // Теперь в tags остался только "dotnet"

        Console.WriteLine("\nЭлементы после пересечения:");
        foreach (var tag in tags)
        {
            Console.WriteLine(tag);
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Изменение объекта после добавления в HashSet
> Если вы добавили изменяемый объект класса в `HashSet<T>`, а затем изменили свойства этого объекта, влияющие на расчет его `GetHashCode()`, этот объект станет "потерянным" в коллекции. `HashSet` будет искать его в старой корзине, а объект логически должен находиться в новой. **Элементы множества должны быть неизменяемыми (Immutable/ReadOnly).**

> [!WARNING] Ошибка: Отсутствие Concurrent-версии
> В .NET нет класса `ConcurrentHashSet<T>`. Использование стандартного `HashSet<T>` в многопоточном режиме без ручной синхронизации приведет к порче данных. В качестве альтернативы используйте `ConcurrentDictionary<T, byte>` (где ключи выступают в роли множества).

**Best Practices:**
1.  **Задавайте Capacity:** Если размер уникального множества предсказуем, передавайте его в конструктор `new HashSet<T>(capacity)`, чтобы избежать дорогостоящих внутренних перестроений хеш-таблицы (Rehash) при заполнении.
2.  **Быстрая фильтрация дубликатов:** Чтобы получить уникальный список элементов, быстрее и дешевле передать коллекцию в конструктор `new HashSet<T>(collection)`, чем писать сложные цепочки LINQ-методов `.Distinct().ToList()`.
3.  **Используйте IEquatable\<T\>:** Для пользовательских структур всегда реализовывайте `IEquatable<T>`, чтобы избежать неявной упаковки ([[Boxing Unboxing|boxing]]) при сравнении элементов внутри `HashSet<T>`.
4.  **Кастомные компараторы:** Для сравнения строк без учета регистра передавайте в конструктор стандартные компараторы: `new HashSet<string>(StringComparer.OrdinalIgnoreCase)`.

> [!TIP] Нюанс: Порядок перебора
> В отличие от `SortedSet<T>`, класс `HashSet<T>` **не гарантирует** сохранение порядка элементов при переборе через `foreach`. Если вам важен строгий порядок сортировки элементов — пожертвуйте частью производительности и используйте `SortedSet<T>` (он работает на красно-черных деревьях со сложностью операций $O(\log n)$).
