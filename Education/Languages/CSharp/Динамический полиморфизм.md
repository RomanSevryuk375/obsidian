### Суть (The "What")
**Динамический полиморфизм** — это механизм, позволяющий определять конкретную реализацию метода непосредственно во время выполнения программы (Runtime), а не на этапе компиляции. Это реализуется через наследование и переопределение [[Виртуальные методы и свойства|виртуальных]] членов.

Он решает задачу **взаимозаменяемости компонентов**: вызывающий код может работать с абстрактным базовым типом, не зная о деталях реализации конкретных наследников. Это основа принципа открытости/закрытости ([[Open Closed Principle|Open/Cposed]]) из [[SOLID]].

---

### Как это работает (The "How" / Nutshell style)

#### 1. Механизм позднего связывания (Late Binding)
В отличие от[[Статический полиморфизм|статического полиморфизма]] (перегрузки), где адрес метода известен заранее, динамический полиморфизм использует **индирекцию** (обращение через указатель).

*   **Method Table:** [[CLR]] создает для каждого типа структуру данных в памяти (Loader Heap), описывающую этот тип.
*   **[[VMT]] (Virtual Method Table):** Внутри таблицы типа находится массив указателей на реализации виртуальных методов. Если класс `Derived` переопределяет метод класса `Base`, в его VMT адрес базового метода заменяется на адрес переопределенного.
*   **Object Header:** Каждый объект в куче содержит указатель на таблицу своего типа (Type Handle).

#### 2. Инструкция callvirt
При вызове виртуального метода компилятор генерирует [[IL]]-инструкцию `callvirt`. В отличие от обычной `call`, она выполняет следующие шаги:
1.  Проверяет экземпляр на `null` (генерируя `NullReferenceException`).
2.  Обращается к заголовку объекта, чтобы найти его таблицу методов.
3.  Ищет адрес метода в VMT по индексу слота.
4.  Переходит по этому адресу.

#### 3. Ключевые слова: virtual, override, sealed
*   **virtual:** Резервирует "слот" в VMT для возможности переопределения.
*   **override:** Записывает новый адрес метода в существующий слот базового типа.
*   **sealed:** Сообщает [[JIT]]-компилятору, что дальнейших переопределений не будет, позволяя выполнить **девиртуализацию** (превращение `callvirt` обратно в быстрый `call`).

> [!INFO] Нюанс производительности
> Виртуальный вызов медленнее прямого из-за дополнительного обращения к памяти (разыменования указателя). Однако для 99% бизнес-задач эта разница нивелируется работой JIT-оптимизатора и предсказателя переходов процессора.

### Пример кода ([[MOC#/MOC|C#]] 12)

```csharp
using System;
using System.Collections.Generic;

// Базовая абстракция
public abstract class NotificationSender(string providerName)
{
    public string ProviderName { get; } = providerName;

    // Слот в VMT
    public abstract void Send(string message);

    // Виртуальный метод с реализацией по умолчанию
    public virtual void LogStatus() => Console.WriteLine($"Sender {ProviderName} is ready.");
}

// Конкретная реализация 1
public class EmailSender(string email) : NotificationSender("SMTP")
{
    public override void Send(string message) => 
        Console.WriteLine($"Email sent to {email}: {message}");
}

// Конкретная реализация 2
public class SmsSender(string phone) : NotificationSender("GSM")
{
    public override void Send(string message) => 
        Console.WriteLine($"SMS sent to {phone}: {message}");
    
    // Переопределение виртуального метода
    public override void LogStatus() => 
        Console.WriteLine($"SmsSender ({ProviderName}) check signal strength...");
}

public class Program
{
    public static void Main()
    {
        // Использование полиморфной коллекции (C# 12 collection expression)
        List<NotificationSender> senders = [
            new EmailSender("admin@dev.local"),
            new SmsSender("+79991234567")
        ];

        // Код работает с базовым типом, но поведение определяется объектом
        foreach (var sender in senders)
        {
            sender.LogStatus(); // Динамический вызов
            sender.Send("System alert!"); // Динамический вызов
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Вызов виртуальных методов в конструкторе
> Никогда не вызывайте виртуальные методы в конструкторе класса. В C# конструктор базового класса выполняется *раньше* [[Конструкторы|конструктора]] производного. Если виртуальный метод переопределен в потомке, он выполнится до того, как поля потомка будут проинициализированы, что приведет к непредсказуемому поведению.

> [!WARNING] Ошибка: Использование 'new' для скрытия
> Ключевое слово `new` не является полиморфным. Если вы скроете метод вместо переопределения, то при обращении через ссылку базового типа (`Base b = new Derived()`) вызовется метод базового класса, а не производного. Это классический источник багов.

**Best Practices:**
1.  **Принцип подстановки Лисков ([[Liskov Substitution Principle|LSP]]):** Производные классы должны расширять, а не замещать или ломать поведение базового класса. Вызывающий код не должен чувствовать разницы при замене `Base` на `Derived`.
2.  **Sealed by default:** Если вы не проектируете класс специально для наследования, помечайте его как `sealed`. Это защищает архитектуру и помогает [[JIT]]-оптимизации.
3.  **[[Абстрактные классы и члены классов|Абстрактные методы]] для обязательной логики:** Если у метода нет "разумной" реализации по умолчанию, делайте его `abstract`, а не `virtual` с пустой реализацией или исключением.
4.  **Модификатор base:** Используйте `base.MethodName()` внутри `override`, если вам нужно дополнить, а не полностью заменить логику родителя.

> [!TIP] Нюанс: Interface Dispatch
> Вызов метода через интерфейс технически сложнее вызова через VMT класса, так как один класс может реализовывать множество интерфейсов. [[CLR]] использует механизм **Virtual Stub Dispatch**, который кэширует результаты поиска реализации для повышения скорости.
