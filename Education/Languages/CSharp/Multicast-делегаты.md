### Суть (The "What")
**Multicast-[[Делегаты|делегат]]** — это делегат, который содержит ссылки не на один, а на несколько методов одновременно. При вызове такого делегата все методы, входящие в его список рассылки, выполняются последовательно.

Эта концепция решает задачу реализации паттерна **Observer (Наблюдатель)**: она позволяет одному событию или сигналу оповещать неограниченное количество подписчиков без жесткой привязки отправителя к получателям.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Иерархия и структура [[CLR]]
В [[MOC#/MOC|C#]] все делегаты технически являются "групповыми". Даже если вы объявляете простой делегат, он наследуется от класса `System.MulticastDelegate`. 
Внутри объекта делегата ключевую роль играет скрытое поле **`_invocationList`**:
*   Если делегат указывает на один метод, `_invocationList` равен `null`, а данные метода хранятся в полях `_target` и `_methodPtr`.
*   Если делегат становится групповым (через оператор `+=`), `_invocationList` заполняется массивом объектов делегатов, каждый из которых представляет отдельный метод.

#### 2. Неизменяемость (Immutability)
Делегаты в .NET — это **неизменяемые типы (immutable)**. 
Операции `+` или `+=` не модифицируют существующий объект. Вместо этого вызывается статический метод `Delegate.Combine`, который создает **новый** экземпляр делегата с обновленным списком вызовов. Аналогично `Delegate.Remove` создает новый экземпляр при отписке (`-=`).

#### 3. Порядок и особенности выполнения
*   **Последовательность:** Методы вызываются строго в том порядке, в котором они были добавлены в список.
*   **Возвращаемые значения:** Если делегат возвращает значение (не `void`), то результатом вызова группового делегата будет значение, возвращенное **последним** методом в списке. Результаты предыдущих методов просто теряются.
*   **Прерывание цепи:** Если один из методов в цепочке выбрасывает исключение, выполнение оставшихся методов в списке **прекращается**, а исключение пробрасывается вызывающему коду.

> [!INFO] Нюанс: GetInvocationList()
> Если вам нужно выполнить все методы независимо от исключений в них или собрать результаты всех функций, используйте метод `GetInvocationList()`. Он возвращает массив отдельных делегатов, по которым можно пройти циклом.

### Пример кода (C# 12)

```csharp
using System;
using System.Linq;

public class NotificationService
{
    // Пример с использованием встроенного делегата Action
    public void RunDemo()
    {
        // 1. Инициализация первого обработчика
        Action<string> notify = SendSms;

        // 2. Добавление второго обработчика (групповое преобразование)
        notify += SendEmail;

        // 3. Добавление анонимного метода (лямбда)
        notify += (msg) => Console.WriteLine($"Push: {msg}");

        Console.WriteLine("--- Первый вызов (3 метода) ---");
        notify?.Invoke("System Update");

        // 4. Удаление метода из списка
        notify -= SendSms;

        Console.WriteLine("\n--- Второй вызов (2 метода) ---");
        notify?.Invoke("Security Alert");
    }

    private void SendSms(string msg) => Console.WriteLine($"SMS: {msg}");
    private void SendEmail(string msg) => Console.WriteLine($"Email: {msg}");
}

// Пример ручного обхода для обработки исключений и сбора данных
public class AdvancedDelegateDemo
{
    public void SafeInvoke(Func<int> calculations)
    {
        if (calculations == null) return;

        // Извлекаем все методы из цепочки
        Delegate[] handlers = calculations.GetInvocationList();

        foreach (Func<int> handler in handlers)
        {
            try
            {
                int result = handler();
                Console.WriteLine($"Result: {result}");
            }
            catch (Exception ex)
            {
                Console.WriteLine($"Error in handler: {ex.Message}");
            }
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Зависимость от порядка выполнения
> Никогда не пишите код, который предполагает, что один метод в делегате должен выполниться раньше другого для корректной работы. Групповые делегаты должны использоваться для независимых действий.

> [!WARNING] Ошибка: Утечки памяти при подписке на события
> Если объект `A` подписывается на делегат объекта `B`, объект `B` будет хранить ссылку на `A` в своем `_invocationList`. Это не позволит Garbage Collector удалить объект `A`. Всегда делайте `-=` (отписку), когда объект `A` больше не нужен.

**Best Practices:**
1.  **Проверка на null:** Всегда вызывайте групповые делегаты через null-условный оператор: `handler?.Invoke()`. Если из списка удален последний метод, переменная станет `null`.
2.  **Void для Multicast:** Старайтесь использовать групповые делегаты только с типами, возвращающими `void` (`Action`). Это избавляет от путаницы с "потерянными" результатами вычислений.
3.  **Безопасный вызов:** Если подписчиков много и они нестабильны, используйте `GetInvocationList()` и блок `try-catch` внутри цикла, чтобы один упавший подписчик не "уронил" всю систему оповещения.
4.  **[[События]] (Events):** Для публичных API всегда оборачивайте Multicast-делегаты в ключевое слово `event`. Это запретит внешнему коду случайно затереть весь список вызовов (через `=`), разрешая только `+=` и `-=`.

> [!TIP] Нюанс: Потокобезопасность
> Операция `+=` и `-=` является потокобезопасной в том смысле, что она заменяет ссылку на новый объект атомарно. Однако само выполнение цепочки методов не является потокобезопасным автоматически — если во время выполнения списка один поток изменит переменную делегата, текущий поток все равно доделает старую копию списка.
