### Суть (The "What")
**Потокобезопасные коллекции** (находящиеся в пространстве имен `System.Collections.Concurrent`) — это высокопроизводительные структуры данных, спроектированные для одновременного доступа из нескольких потоков. Они заменяют стандартные коллекции ([[List]]`<T>`, [[Dictionary]]`<TKey, TValue>`), когда требуется избежать состояния гонки (race conditions) и порчи данных.

Главная проблема, которую они решают — устранение необходимости ручной синхронизации через [[Monitor|lock]]. Конкурентные коллекции используют оптимизированные алгоритмы блокировки или вовсе обходятся без них там, где это возможно.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Механизмы синхронизации
В отличие от простых оберток, которые блокируют всю коллекцию на любую операцию, конкурентные коллекции используют более гранулярные подходы:

*   **Lock Striping (Расщепление блокировок):** Применяется в `ConcurrentDictionary`. Словарь разделен на сегменты (buckets), каждый из которых имеет свой объект синхронизации. Это позволяет нескольким потокам одновременно обновлять разные части словаря без взаимной блокировки.
*   **Lock-Free алгоритмы:** Основаны на атомарных инструкциях процессора (CAS — Compare-And-Swap) через класс `Interlocked`. Применяются в `ConcurrentStack` и `ConcurrentQueue`. Если операция не удалась из-за вмешательства другого потока, выполняется повторная попытка в цикле.
*   **Thread-Local Storage (TLS):** Используется в `ConcurrentBag`. Каждый поток имеет свой локальный список элементов. Поток сначала пытается взять элемент из своего списка (без блокировки). Если он пуст, поток "крадет" (steals) элемент из списка другого потока (с синхронизацией).

#### 2. Семантика итерации
Итераторы (`IEnumerable`) конкурентных коллекций предоставляют **моментальный снимок (snapshot)** данных или работают по принципу "грязного чтения".
*   Они не выбрасывают `InvalidOperationException`, если коллекция изменилась во время обхода.
*   Изменения, сделанные после начала итерации, могут быть видны или не видны в зависимости от конкретного типа коллекции.

#### 3. Паттерн Try-Receive
Поскольку состояние коллекции в многопоточной среде может измениться мгновенно, конкурентные коллекции используют методы возврата `bool` вместо прямого извлечения: `TryGetValue`, `TryDequeue`, `TryPop`.

> [!IMPORTANT] Нюанс производительности: свойство Count
> Для многих конкурентных коллекций (особенно `ConcurrentBag` и `ConcurrentStack`) обращение к свойству `Count` является чрезвычайно дорогой операцией $O(N)$. Чтобы вычислить точное количество, рантайму приходится временно "замораживать" или суммировать состояния всех внутренних сегментов/потоков. Используйте свойство `IsEmpty` для проверки наличия элементов.

### Пример кода ([[MOC|C#]] 12)

```csharp
using System;
using System.Collections.Concurrent;
using System.Threading.Tasks;

public class DataStorage
{
    // Использование ConcurrentDictionary для хранения результатов
    private readonly ConcurrentDictionary<int, string> _cache = new();

    public async Task ProcessDataAsync()
    {
        // 1. Атомарное добавление или обновление
        _cache.AddOrUpdate(1, "Initial", (key, oldVal) => oldVal + " + Updated");

        // 2. Использование BlockingCollection для паттерна Producer-Consumer
        // boundedCapacity: 10 ограничивает размер очереди для предотвращения переполнения памяти
        using BlockingCollection<int> pipeline = new(boundedCapacity: 10);

        var consumer = Task.Run(() =>
        {
            // GetConsumingEnumerable блокирует поток до появления данных
            foreach (var item in pipeline.GetConsumingEnumerable())
            {
                Console.WriteLine($"Consumed: {item}");
            }
        });

        var producer = Task.Run(() =>
        {
            for (int i = 0; i < 20; i++)
            {
                pipeline.Add(i); // Заблокирует поток, если в очереди уже 10 элементов
            }
            pipeline.CompleteAdding(); // Сигнал о завершении
        });

        await Task.WhenAll(producer, consumer);
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Логика Check-then-Act
> Код вида `if (!_dict.ContainsKey(key)) _dict.Add(key, value);` **не является потокобезопасным**, так как между проверкой и добавлением другой поток может вставить этот же ключ. Всегда используйте атомарные методы: `GetOrAdd`, `TryAdd` или `AddOrUpdate`.

> [!WARNING] Ошибка: Блокировки внутри делегатов
> Делегаты, передаваемые в `GetOrAdd` или `AddOrUpdate`, могут выполняться **вне блокировки** (в зависимости от реализации) или выполняться несколько раз. Никогда не делайте тяжелых вычислений или блокирующих вызовов внутри этих лямбда-выражений.

**Best Practices:**
1.  **Выбор коллекции:** 
    *   `ConcurrentQueue<T>` — строгий порядок FIFO.
    *   `ConcurrentStack<T>` — строгий порядок LIFO.
    *   `ConcurrentBag<T>` — когда порядок не важен и работа идет преимущественно внутри одного потока.
2.  **BlockingCollection:** Используйте её в качестве обертки над любой конкурентной коллекцией, если вам нужен функционал блокировки потока-потребителя при пустой очереди (паттерн Производитель-Потребитель).
3.  **Минимизация Count:** Вместо `if (bag.Count > 0)` всегда пишите `if (!bag.IsEmpty)`.
4.  **Try-методы:** Всегда проверяйте результат `TryDequeue` или `TryPop` в цикле `while`, если ваша логика предполагает ожидание данных (или используйте `BlockingCollection`).

> [!TIP] Нюанс Memory Barrier
> Конкурентные коллекции гарантируют полную видимость изменений между ядрами процессора (Full Memory Barrier). Вам не нужно использовать ключевое слово `volatile` для данных, извлекаемых из этих коллекций.
