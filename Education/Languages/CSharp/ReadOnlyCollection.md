### Суть (The "What")
**ReadOnlyCollection`<T>`** — это класс-обертка (wrapper) из пространства имен `System.Collections.ObjectModel`, который предоставляет доступ к существующей коллекции только для чтения. Он не является самостоятельным хранилищем данных, а лишь ограничивает интерфейс взаимодействия с уже созданным списком.

Этот класс решает задачу **инкапсуляции внутреннего состояния объекта**. С его помощью разработчик может предоставить внешним потребителям доступ к данным для чтения, гарантируя, что они не смогут изменить состав коллекции (добавить или удалить элементы) в обход логики класса-владельца.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Паттерн "Прокси" (Wrapper)
`ReadOnlyCollection<T>` реализует паттерн «Заместитель». При создании экземпляра в конструктор передается ссылка на оригинальный [[Интерфейс IList|IList]]`<T>`. 
*   **Отсутствие копирования:** Обертка не создает копию элементов. Это делает создание `ReadOnlyCollection` операцией **O(1)** по времени и памяти.
*   **Живое представление (Live View):** Поскольку обертка хранит ссылку на оригинал, любые изменения, внесенные в исходную коллекцию (внутри класса-владельца), будут немедленно отражены в `ReadOnlyCollection`.

#### 2. Реализация интерфейсов
Класс реализует `IList<T>`, `ICollection<T>` и `IEnumerable<T>`. 
*   Методы модификации (такие как `Add`, `Insert`, `Remove`, `Clear`) реализованы **явно** (explicitly) и выбрасывают исключение `NotSupportedException`.
*   Свойство `IsReadOnly` всегда возвращает `true`.

#### 3. Отличие от ImmutableCollections
Важно не путать "защищенную" коллекцию с "неизменяемой":
*   **ReadOnlyCollection:** Если вы измените базовый список, "защищенный" список тоже изменится. Он лишь запрещает изменения *через себя*.
*   **ImmutableList:** Физически не может быть изменен никем. Любая попытка изменения порождает новый объект.

> [!INFO] Нюанс: Производительность индексатора
> Доступ по индексу в `ReadOnlyCollection<T>` практически так же эффективен, как и в обычном `List<T>`, так как вызов просто перенаправляется внутреннему списку. Однако использование интерфейса `IReadOnlyList<T>` (который реализует этот класс) позволяет JIT-компилятору выполнять дополнительные оптимизации.

### Пример кода ([[MOC|C#]] 12)

```csharp
using System;
using System.Collections.Generic;
using System.Collections.ObjectModel;

public class Classroom
{
    // Внутренний изменяемый список
    private readonly List<string> _students = [];

    // Публичная обертка только для чтения
    public ReadOnlyCollection<string> Students { get; }

    public Classroom()
    {
        // Students "смотрит" на тот же участок памяти, что и _students
        Students = new ReadOnlyCollection<string>(_students);
    }

    public void AddStudent(string name)
    {
        // Мы можем менять коллекцию внутри класса
        _students.Add(name);
    }
}

public class Program
{
    public static void Main()
    {
        var room = new Classroom();
        room.AddStudent("Artyom");

        // Чтение разрешено
        Console.WriteLine(room.Students[0]); // Artyom

        // room.Students.Add("Ivan"); // Ошибка компиляции (у ReadOnlyCollection нет метода Add)
        
        // Попытка обмануть систему через приведение к интерфейсу
        IList<string> list = room.Students;
        try 
        {
            list.Add("Ivan"); 
        }
        catch (NotSupportedException)
        {
            Console.WriteLine("Изменение через интерфейс невозможно.");
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Возврат внутреннего List`<T>`
> Возвращать приватный List`<T>` напрямую через свойство — грубое нарушение инкапсуляции. Любой внешний код сможет вызвать `.Clear()`, разрушив состояние вашего объекта. Всегда оборачивайте его в `ReadOnlyCollection` или возвращайте `IEnumerable`.

> [!WARNING] Ошибка: Потокобезопасность
> `ReadOnlyCollection<T>` не делает оригинальный список потокобезопасным. Если один поток читает из `ReadOnlyCollection`, а другой в это же время добавляет элементы в базовый `List`, возникнет исключение при итерации (изменение коллекции во время перечисления).

**Best Practices:**
1.  **Инициализация в конструкторе:** Создавайте экземпляр `ReadOnlyCollection` один раз в конструкторе, передавая туда внутренний список. Не создавайте новую обертку при каждом обращении к свойству (это лишние аллокации).
2.  **Использование IReadOnlyList`<T>`:** В современном C# часто лучше возвращать интерфейс `IReadOnlyList<T>` вместо конкретного класса `ReadOnlyCollection<T>`. Это дает большую гибкость в будущем (вы сможете сменить реализацию на массив или `ImmutableList` без смены сигнатуры метода).
3.  **Защита от null:** Перед передачей списка в конструктор `ReadOnlyCollection` убедитесь, что он проинициализирован.

> [!TIP] Нюанс: Массивы
> Если вы передаете массив `T[]` в конструктор `ReadOnlyCollection<T>`, он также будет работать как живое представление. Однако помните, что массивы имеют фиксированный размер, поэтому "изменения" в них возможны только через замену элементов по индексу, а не через добавление новых.
