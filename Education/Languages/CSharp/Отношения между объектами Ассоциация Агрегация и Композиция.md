### Суть (The "What")
В объектно-ориентированном программировании объекты редко существуют изолированно. Отношения определяют, как объекты взаимодействуют и управляют жизненным циклом друг друга.

*   **Ассоциация:** Самый широкий термин. Объект "А" просто знает об объекте "Б" (использует его).
*   **Агрегация:** Отношение "целое-часть", где часть может существовать независимо от целого (слабая связь).
*   **Композиция:** Строгое отношение "целое-часть", где часть не может существовать без целого и управляется им (сильная связь).

Эти концепции решают задачу **управления связностью (coupling)**: чем слабее связь, тем легче тестировать и изменять систему.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Уровень реализации в .NET
На уровне [[CLR]] все эти отношения реализуются одинаково: через хранение ссылки (указателя на адрес в куче) в поле класса. Разница заключается исключительно в **логике владения** и **управлении временем жизни**.

#### 2. Жизненный цикл и [[GC]]
*   **При Агрегации:** Объект-часть создается вне объекта-целого. Когда Garbage Collector удаляет объект-целое, объект-часть остается в памяти, так как на него, скорее всего, есть внешняя ссылка.
*   **При Композиции:** Объект-часть обычно инстанцируется внутри конструктора объекта-целого. Если объект-целое удаляется из кучи, и на объект-часть нет других ссылок (что предполагается логикой композиции), он также становится доступным для GC.

#### 3. Иерархия владения
*   **Ассоциация (Association):** "Я использую тебя". Часто реализуется через передачу объекта в метод в качестве аргумента.
*   **Агрегация (Aggregation):** "У меня есть ты, но ты принадлежишь не только мне". Передается через конструктор (Dependency Injection).
*   **Композиция (Composition):** "Ты — моя неотъемлемая часть". Создается внутри класса.

> [!INFO] Нюанс: Направленность
> Ассоциация может быть однонаправленной (А знает о Б) или двунаправленной (А и Б знают друг о друге). Композиция и агрегация почти всегда направлены от целого к части.

### Пример кода ([[MOC#/MOC|C#]] 12)

```csharp
using System;

// 1. Ассоциация (Простое использование)
public class GasStation
{
    public void Refill(Car car) => Console.WriteLine($"Refilling {car.Model}");
}

// 2. Часть для Агрегации и Композиции
public class Engine(string power) 
{
    public string Power { get; } = power;
}

public class Driver(string name)
{
    public string Name { get; } = name;
}

public class Car
{
    public string Model { get; }
    
    // 3. Композиция (Engine создается внутри Car)
    // Car полностью владеет жизненным циклом этого Engine
    private readonly Engine _engine;

    // 4. Агрегация (Driver передается извне)
    // Если Car будет уничтожен, Driver продолжит существовать
    public Driver? CurrentDriver { get; set; }

    public Car(string model, string enginePower)
    {
        Model = model;
        _engine = new Engine(enginePower); // Реализация композиции
    }

    public void Drive(GasStation station)
    {
        // Пример ассоциации: Car использует GasStation через метод
        station.Refill(this);
    }
}

public class Program
{
    public static void Main()
    {
        Driver driver = new("Ivan"); // Объект живет в Main
        GasStation station = new();

        {
            Car myCar = new("Tesla", "500hp");
            myCar.CurrentDriver = driver; // Агрегация
            myCar.Drive(station);         // Ассоциация
        } 
        // Здесь myCar и его Engine (Композиция) могут быть удалены GC.
        // Но driver и station (Агрегация/Ассоциация) остаются в памяти.
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Скрытая композиция (Hardcoding)
> Создание всех зависимостей внутри класса через `new` (жесткая композиция) делает код нетестируемым. Вы не сможете подменить реальный объект "заглушкой" (Mock) в Unit-тестах.

> [!WARNING] Ошибка: Утечка ссылок при композиции
> Если вы реализуете композицию, не возвращайте прямую ссылку на внутренний объект-часть через публичное свойство. Это позволит внешнему коду сохранить ссылку на него, превратив композицию в агрегацию и нарушив [[Инкапсуляция|инкуапсуляцию]] владения.

**Best Practices:**
1.  **Предпочитайте Агрегацию (DI):** В современной разработке (ASP.NET Core и др.) агрегация через внедрение зависимостей в конструктор является стандартом. Это делает систему гибкой.
2.  **Композиция для инкапсуляции:** Используйте композицию только тогда, когда компонент является "деталью реализации" и не имеет смысла вне контекста владельца (например, `Node` внутри `LinkedList`).
3.  **Принцип единственной ответственности ([[SRP]]):** Если класс начинает агрегировать слишком много объектов, это сигнал о том, что он берет на себя слишком много задач и его пора разделять.
4.  **Immutability в Агрегации:** При агрегации данных старайтесь использовать `readonly` поля или `init` свойства, чтобы связь между объектами не менялась непредсказуемо во время работы.

> [!TIP] Нюанс: Records и владение
> Использование `record` в C# 12 упрощает создание агрегатов данных. Однако помните, что при копировании рекорда (`with` выражение) копируются только ссылки на агрегированные объекты (Shallow Copy), а не сами объекты.
