# Значимые типы (Value Types)

### Суть (The "What")
**Значимые типы** — это категории типов, переменные которых содержат свои данные непосредственно. В отличие от ссылочных типов, здесь отсутствует уровень косвенности (индирекции): имя переменной указывает прямо на область памяти, где лежат байты значения.

Основная задача значимых типов — минимизировать нагрузку на сборщик мусора ([[GC|Garbage Collector]]) и обеспечить высокую локальность данных в памяти, что критично для производительности.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Размещение в памяти
Существует распространенное заблуждение, что "значимые типы всегда живут в стеке". Это не совсем так. Местоположение зависит от контекста:
- **Локальные переменные:** Обычно размещаются в **стеке**. Это крайне дешевая операция: память выделяется простым сдвигом указателя стека.
- **Поля классов:** Если значимый тип является полем `class`, он размещается в **куче ([[heap]])** как часть этого объекта.
- **Статические поля:** Размещаются в специальной области [[Куча|кучи]] для статических данных.

#### 2. Копирование при присваивании
При присваивании одной переменной значимого типа другой (`a = b`) происходит **побитовое копирование** содержимого. Это означает, что изменения в `a` никак не затронут `b`.

#### 3. Наследование и System.ValueType
Все значимые типы неявно наследуются от класса `System.ValueType`, который в свою очередь наследуется от `System.Object`. 
- Значимые типы являются **запечатанными ([[sealed]])**: вы не можете наследоваться от `struct` или `int`.
- `System.ValueType` переопределяет метод `Equals()`, используя [[Рефлексия|рефлексию]] для сравнения полей. 

> [!WARNING] Нюанс производительности: Equals
> Стандартная реализация `ValueType.Equals` медленная, так как использует рефлексию для обхода полей структуры. Для высокопроизводительного кода всегда переопределяйте `Equals` и `GetHashCode` вручную или используйте `readonly record struct`.

#### 4. Специальные категории ([[MOC|C#]] 7.2 - 12+)
- **readonly struct:** Гарантирует неизменяемость полей. Позволяет компилятору оптимизировать передачу структуры, избегая создания защитных копий (defensive copies).
- **[[ref struct]]:** Структуры, которые гарантированно живут только в стеке (например, `Span<T>`). <font color="#c00000">Они не могут быть упакованы в кучу или быть полями обычного класса</font>.

### Пример кода (C# 12)

```csharp
using System;

// 1. Определение структуры с использованием Primary Constructor (C# 12)
public readonly struct Vector3(double x, double y, double z)
{
    public double X { get; init; } = x;
    public double Y { get; init; } = y;
    public double Z { get; init; } = z;

    // Переопределение для исключения рефлексии
    public override bool Equals(object? obj) => 
        obj is Vector3 v && v.X == X && v.Y == Y && v.Z == Z;
        
    public override int GetHashCode() => HashCode.Combine(X, Y, Z);
}

public class Program
{
    public static void Main()
    {
        // Размещение в стеке
        Vector3 v1 = new(1.0, 2.0, 3.0);
        
        // Копирование значения
        Vector3 v2 = v1; 
        
        // v2 — это независимая копия v1 в памяти стека
        Console.WriteLine(v1.Equals(v2)); // True
        
        // Пример работы с ref для исключения копирования тяжелых структур
        UpdateVector(ref v1);
    }

    static void UpdateVector(ref Vector3 v)
    {
        // Работаем напрямую с оригинальной памятью через указатель (ref)
        Console.WriteLine(v.X);
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Mutable Structs (Изменяемые структуры)
> Никогда не создавайте структуры, поля которых можно изменить после инициализации. Изменяемые структуры ведут к непредсказуемому поведению при передаче в методы (вы будете менять копию, а не оригинал) и ошибкам при работе со свойствами. Используйте `readonly struct`.

> [!INFO] Правило размера
> Как правило (согласно Nutshell), если размер структуры превышает **16-24 байта**, её передача в методы становится дороже, чем передача ссылки. В таких случаях стоит рассмотреть использование `class` или передачу через `in` / `ref`.

**Best Practices:**
1. **Неизменяемость:** Делайте все структуры `readonly`.
2. **Boxing Awareness:** Избегайте передачи значимых типов в методы, принимающие `object` или интерфейсы (например, `IComparable`), если это происходит в часто выполняемых циклах. Это вызывает выделение памяти в куче ([[Boxing]]).
3. **Record Structs:** В современном [[MOC#/MOC|C#]] используйте `public readonly record struct Name(...)`, чтобы автоматически получить оптимизированные методы сравнения и лаконичный синтаксис.
4. **Default Constructor:** Помните, что структуры всегда имеют неявный конструктор без параметров, который зануляет все поля (`default`). С C# 10 можно определять собственный конструктор без параметров, но делайте это с осторожностью.
