### Суть (The "What")
**Неизменяемые коллекции** — это структуры данных из пространства имен `System.Collections.Immutable`, состояние которых невозможно изменить после их создания. Любая операция, которая логически "модифицирует" такую [[Коллекции|коллекцию]] (добавление, удаление), на самом деле возвращает **новую версию** коллекции, оставляя оригинал нетронутым.

Они решают задачу **безопасного совместного использования данных** в многопоточных средах без использования блокировок ([[Monitor|lock]]). Это исключает целый класс багов, связанных с состоянием гонки (race conditions) и побочными эффектами при передаче коллекций между методами.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Неизменяемость против Read-Only
Важно различать `ReadOnlyCollection<T>` и `ImmutableList<T>`:
*   **[[ReadOnlyCollection]]:** Это лишь "обертка" (вью) над обычной коллекцией. Если кто-то изменит оригинальный список под оберткой, данные в ReadOnly-версии тоже изменятся.
*   **Immutable Collection:** Гарантирует абсолютную неизменность. Данные не могут быть изменены никем и никогда после инициализации.

#### 2. Внутренняя реализация: Сбалансированные деревья
В отличие от [[List]]`<T>`, который использует массив, большинство неизменяемых коллекций (например, `ImmutableList<T>`, `ImmutableDictionary<K,V>`) внутри реализованы как **AVL-деревья (сбалансированные бинарные деревья поиска)**.
*   **Механика:** При "добавлении" элемента дерево не копируется целиком. Создаются новые узлы только для измененной ветви дерева, а остальные узлы переиспользуются (структурное разделение — *structural sharing*).
*   **Сложность:** Операции добавления и поиска в таких коллекциях имеют сложность **$O(\log n)$**, в то время как у обычного списка добавление — $O(1)$.

#### 3. Специальный случай: ImmutableArray`<T>`
`ImmutableArray<T>` — это тонкая обертка над обычным массивом.
*   **Память:** Она занимает ровно столько же места, сколько и массив.
*   **Производительность:** Доступ по индексу — $O(1)$. Но любая модификация (Add/Remove) требует **полного копирования** массива ($O(n)$).
*   **Применение:** Идеально подходит для данных, которые почти никогда не меняются, но часто читаются.

#### 4. Паттерн Builder
Для массового наполнения коллекции используется вспомогательный класс `Builder`. Он временно позволяет выполнять мутации без создания промежуточных копий, а затем "замораживает" состояние в неизменяемый объект методом `ToImmutable()`.

> [!INFO] Нюанс: Поверхностная неизменяемость
> Неизменяемые коллекции гарантируют неизменность **списка ссылок**, но не самих объектов внутри него. Если вы положите изменяемый класс в `ImmutableList<User>`, вы все еще сможете изменить поле `User.Name`. Для полной безопасности объекты внутри коллекции тоже должны быть неизменяемыми (например, [[Record's]]).

### Пример кода ([[MOC|C#]] 12)

```csharp
using System;
using System.Collections.Immutable;

public class ImmutableDemo
{
    public void Run()
    {
        // 1. Создание через Collection Expressions (C# 12)
        // Требуется явное указание типа для вызова метода расширения
        ImmutableList<int> original = [1, 2, 3];

        // 2. "Модификация" возвращает новый объект
        ImmutableList<int> modified = original.Add(4);

        Console.WriteLine($"Original count: {original.Count}"); // 3
        Console.WriteLine($"Modified count: {modified.Count}"); // 4

        // 3. Использование Builder для эффективного наполнения
        var builder = ImmutableDictionary.CreateBuilder<int, string>();
        for (int i = 0; i < 100; i++)
        {
            builder.Add(i, $"Value {i}");
        }
        // Превращаем в неизменяемую коллекцию
        ImmutableDictionary<int, string> dict = builder.ToImmutable();

        // 4. ImmutableArray для константных справочников
        ImmutableArray<string> roles = ["Admin", "User", "Guest"];
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Модификация в цикле без Builder
> Выполнение `list = list.Add(i)` внутри длинного цикла — катастрофа для производительности. Каждая итерация создает новое дерево или массив. Для массовых операций всегда используйте `ToBuilder()` или `ImmutableList.CreateRange()`.

> [!WARNING] Ошибка: Передача через интерфейс IEnumerable
> Если ваш метод принимает `IEnumerable<T>`, он не знает, что коллекция неизменяема. Чтобы явно требовать потокобезопасные данные, используйте в параметрах конкретные типы вроде `IImmutableList<T>`.

**Best Practices:**
1. **Выбор типа:** Используйте `ImmutableArray<T>` для редко меняющихся данных (конфигурации, справочники). Используйте `ImmutableList<T>`, если вставки происходят чаще.
2. **Типобезопасность:** Используйте `ImmutableArray<T>.Optional` или `default` с осторожностью. Пустой `ImmutableArray` ведет себя как `null`-ссылка, если он не инициализирован (в отличие от `ImmutableList`).
3. **Методы расширения:** Используйте методы `.ToImmutableList()`, `.ToImmutableArray()` для быстрого преобразования из обычных `IEnumerable`.
4. **Параметры:** Принимайте в конструкторах классов неизменяемые коллекции, если хотите гарантировать, что состояние объекта не будет подменено извне после инициализации.

> [!TIP] Нюанс: LINQ
> Методы LINQ (Where, Select) работают с неизменяемыми коллекциями так же, как и с обычными, но они всегда возвращают `IEnumerable<T>`. Чтобы снова получить неизменяемую коллекцию, нужно в конце вызвать `.ToImmutableList()`.
