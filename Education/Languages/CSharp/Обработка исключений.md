### Суть (The "What")
**Исключение** — это объект, представляющий возникновение аномальной ситуации или ошибки во время выполнения программы. Система обработки исключений в [[MOC#/MOC|C#]] построена на блоках `try`, `catch`, `finally` и `throw`.

Она решает задачу **отделения основного алгоритма от логики обработки ошибок**. В отличие от кодов возврата (error codes), исключения нельзя "тихо проигнорировать": если ошибка не обработана, [[CLR]] прервет выполнение потока.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Механика CLR и Stack Unwinding
Когда вызывается [[throw]]:
1.  CLR создает объект исключения, содержащий метаданные и **Stack Trace** (снимок текущего состояния стека).
2.  Начинается процесс **раскрутки стека (Stack Unwinding)**: рантайм ищет ближайший подходящий блок `catch` вверх по иерархии вызовов.
3.  Если поиск достигает вершины стека (Main) и обработчик не найден, происходит завершение процесса (Uncaught Exception).

#### 2. Фильтры исключений (Exception Filters) — [[MOC#/MOC|C#]] 6+
Использование ключевого слова `when` позволяет ловить исключение только при соблюдении условия.
> [!IMPORTANT] Нюанс: Состояние стека
> Фильтры `when` выполняются до того, как стек будет "раскручен". Это критически важно для отладки: если условие в `when` не выполняется, управление передается дальше, а точка возникновения ошибки (dump) сохраняет исходное состояние. Если же вы поймали исключение и сделали `if` внутри `catch`, состояние стека в точке ошибки будет уже частично потеряно.

#### 3. Стоимость исключений (Performance)
Создание исключения — одна из самых дорогих операций в .NET. Основные накладные расходы приходятся на формирование Stack Trace. 
*   Используйте исключения для **исключительных** ситуаций (например, обрыв сетевого соединения).
*   Для ожидаемых ситуаций (неверный ввод пользователя) используйте паттерны `Try-Parse` или возвращайте специализированные объекты результата.

#### 4. throw vs throw ex
*   `throw;`: Перебрасывает текущее исключение, **сохраняя** исходный Stack Trace.
*   `throw ex;`: Сбрасывает стек и начинает его заново от текущего метода. Оригинальная точка ошибки теряется.

### Пример кода (C# 12)

```csharp
using System;
using System.Net.Http;
using System.Threading.Tasks;

public class ExceptionService
{
    public async Task ProcessDataAsync(string url)
    {
        try
        {
            using var client = new HttpClient();
            var response = await client.GetStringAsync(url);
            
            // Throw expression (C# 7+)
            _ = response ?? throw new InvalidOperationException("Empty response");
        }
        catch (HttpRequestException ex) when (ex.StatusCode == System.Net.HttpStatusCode.NotFound)
        {
            // Обработка только конкретного кода ошибки через фильтр
            Console.WriteLine("Ресурс не найден (404).");
        }
        catch (Exception ex)
        {
            // Логирование и повторный проброс с сохранением стека
            Log(ex);
            throw; 
        }
        finally
        {
            // Блок выполняется всегда, независимо от того, было ли исключение
            Cleanup();
        }
    }

    private void Log(Exception ex) => Console.WriteLine(ex.Message);
    private void Cleanup() => Console.WriteLine("Cleanup done.");
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Пустой catch (Swallowing Exceptions)
> Никогда не используйте `catch (Exception) {}`. Это скрывает ошибки в логике, которые могут привести к повреждению данных. Программа продолжит работу в неопределенном состоянии, что на порядки сложнее отладить.

> [!WARNING] Ошибка: Использование исключений для управления логикой
> Исключения не должны заменять операторы `if` или `switch`. Например, выход из цикла через `IndexOutOfRangeException` — это катастрофическая ошибка проектирования, снижающая производительность и читаемость.

**Best Practices:**
1.  **Ловите только то, что можете обработать.** Если метод не знает, как исправить ошибку, он должен позволить ей подняться выше по стеку.
2.  **Специфичность.** Всегда старайтесь ловить максимально конкретные типы исключений (`IOException` вместо `Exception`).
3.  **Использование throw expressions.** Делайте код чище, используя `throw` в выражениях: `var x = arg ?? throw new ArgumentNullException(nameof(arg));`.
4.  **Custom Exceptions.** Создавайте свои классы исключений только если вы планируете обрабатывать их программно иначе, чем стандартные. Они должны наследоваться от `System.Exception` и иметь суффикс `Exception`.

> [!TIP] Нюанс: ExceptionDispatchInfo
> Если вам нужно сохранить исключение и пробросить его позже (например, из другого потока) без повреждения стека, используйте класс `ExceptionDispatchInfo`. Это позволяет избежать проблем, которые создает обычный `throw ex`.
