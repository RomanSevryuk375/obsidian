### Суть (The "What")
**Абстрактный класс** — это неполный чертеж типа, который не может быть инстанцирован (нельзя создать объект через `new`). Он служит исключительно базовым классом для других типов.
**Абстрактные члены** (`abstract`) — это методы, свойства или события, которые не имеют реализации в базовом классе. Они лишь объявляют контракт, который **обязан** реализовать первый же неабстрактный наследник.

Абстрактные классы решают задачу создания фундаментальных основ для иерархий типов, где базовое понятие слишком общее, чтобы иметь смысл само по себе (например, "Фигура" или "Платежное средство").

### Как это работает (The "How" / Nutshell style)

#### 1. Механика в [[CLR]] и [[VMT]]
*   **Флаг в метаданных:** На уровне метаданных сборки абстрактный класс помечается флагом `abstract`. При попытке выполнить `new AbstractClass()` CLR выбросит исключение еще на этапе компиляции или верификации.
*   **Слоты в VMT:** Абстрактный метод резервирует слот в Таблице виртуальных методов (VMT), но вместо адреса исполняемого кода в этом слоте находится специальная "заглушка". Компилятор гарантирует, что любой конкретный (concrete) наследник заполнит этот слот через `override`.
*   **Конструкторы:** Абстрактные классы могут иметь конструкторы. Хотя вы не можете создать экземпляр класса напрямую, его конструктор вызывается из производных классов через `base()` для инициализации общих полей.

#### 2. Абстрактный класс vs [[Интерфейс]]
Согласно Албахари, выбор между ними критичен:
*   **Абстрактный класс** определяет **идентичность** ("is-a"). Может содержать состояние (поля) и реализацию по умолчанию. Поддерживает модификаторы доступа для своих членов.
*   **Интерфейс** определяет **способность** ("can-do"). До [[MOC#/MOC|C#]] 8.0 не мог содержать реализации. Не может содержать поля экземпляра (состояние).

#### 3. Правила и ограничения
*   Если класс содержит хотя бы один абстрактный член, сам класс **обязан** быть помечен как `abstract`.
*   Абстрактные члены не могут быть `private` (так как их невозможно будет переопределить) или `sealed`.
*   Абстрактный метод неявно является виртуальным (`virtual`), но использовать ключевое слово `virtual` вместе с `abstract` запрещено.

> [!INFO] Нюанс: Частичная реализация
> Абстрактный класс может наследоваться от другого абстрактного класса. При этом он не обязан реализовывать абстрактные методы родителя — эта ответственность перекладывается на первый "конкретный" класс в цепочке наследования.

### Пример кода (C# 12)

```csharp
using System;

// Абстрактный базовый класс с первичным конструктором (C# 12)
public abstract class Device(string serialNumber)
{
    public string SerialNumber { get; } = serialNumber;

    // Обычное свойство (состояние)
    public bool IsPowerOn { get; protected set; }

    // Абстрактный метод - обязателен к реализации
    public abstract void TurnOn();

    // Виртуальный метод - реализация по умолчанию
    public virtual void Reboot()
    {
        Console.WriteLine("System rebooting...");
    }
}

public class Smartphone(string serialNumber, string os) : Device(serialNumber)
{
    public string OS { get; } = os;

    // Обязательная реализация абстрактного члена
    public override void TurnOn()
    {
        IsPowerOn = true;
        Console.WriteLine($"Smartphone {OS} initialized (SN: {SerialNumber})");
    }
}

public class Program
{
    public static void Main()
    {
        // Device dev = new Device("123"); // Ошибка компиляции

        Device myPhone = new Smartphone("SN-990", "Android 14");
        myPhone.TurnOn();
        myPhone.Reboot();
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Замена интерфейсов абстрактными классами
> Не используйте абстрактные классы только для описания поведения. Из-за отсутствия множественного наследования классов в C#, вы заставляете наследника тратить его "единственную возможность наследоваться" на ваш класс. Если состояние не требуется — используйте интерфейс.

> [!WARNING] Ошибка: Публичные поля
> Даже в абстрактных классах поля должны быть `private`. Для доступа наследников к состоянию используйте `protected` свойства. Это сохраняет инкапсуляцию и позволяет менять внутреннюю структуру базового типа, не ломая код потомков.

**Best Practices:**
1.  **Конструкторы:** Всегда делайте конструкторы в абстрактных классах `protected`. Это явно указывает на то, что они предназначены только для вызова из производных классов.
2.  **Template Method Pattern:** Абстрактные классы идеально подходят для этого паттерна. Базовый класс определяет скелет алгоритма (в обычном методе), вызывая внутри него абстрактные методы, которые "заполняются" наследниками.
3.  **Required члены (C# 11):** Если абстрактный класс имеет `required` свойства, наследники обязаны учитывать их при инициализации через инициализатор объектов.
4.  **Синхронизация абстракции:** Старайтесь делать абстрактными только те члены, которые действительно уникальны для каждого наследника. Если 80% реализации совпадает, сделайте метод виртуальным с базовой логикой.

> [!TIP] Нюанс: Static Abstract (C# 11+)
> В современных версиях .NET можно объявлять `static abstract` члены в интерфейсах, но не в классах. Это позволяет использовать полиморфизм для статических членов (например, операторов или фабрик) в generic-коде. Для абстрактных классов это пока недоступно.
