### Суть (The "What")
**ООП** — это методология программирования, основанная на представлении программы как совокупности объектов, каждый из которых является экземпляром определенного класса. 

ООП решает проблему **сложности (complexity management)**. Оно позволяет декомпозировать тяжелую логику на изолированные модули, которые легче тестировать, поддерживать и расширять. В основе лежат четыре столпа: [[Инкапсуляция]], [[Наследование]], [[Полиморфизм]] и [[Абстракция]].

---

### Как это работает (The "How" / Nutshell style)

#### 1. Инкапсуляция и сокрытие данных
Инкапсуляция в .NET реализуется через уровни доступа ([[Модификаторы доступа|Access Modifiers]]). Она гарантирует, что внутреннее состояние объекта (поля) остается валидным, а доступ к нему осуществляется только через строго определенный интерфейс (свойства и методы).
*   `private protected`: Доступ в текущей сборке И только в производных классах.
*   `protected internal`: Доступ в текущей сборке ИЛИ в производных классах в других сборках.

#### 2. Наследование и структура памяти
[[MOC#/MOC|C#]] поддерживает **одиночное наследование классов** и множественное наследование интерфейсов. 
*   При создании объекта производного класса (`Derived`), [[CLR]] сначала выделяет память под поля базового класса (`Base`), а затем добавляет к ним поля `Derived`. 
*   Все объекты в C# неявно наследуются от `System.Object`.

#### 3. Полиморфизм и [[VMT (Virtual Method Table)]]
Динамический полиморфизм (переопределение методов) реализуется через таблицу виртуальных методов.
*   Когда класс объявляет `virtual` или `abstract` метод, CLR создает для этого типа **VMT**.
*   Каждая запись в VMT указывает на конкретную реализацию метода в памяти (адрес кода).
*   Объект в куче хранит указатель на таблицу типов (Method Table), через которую происходит вызов нужной реализации во время выполнения.

> [!INFO] Нюанс: Стоимость виртуального вызова
> Виртуальные вызовы обходятся чуть дороже обычных, так как требуют дополнительного шага — поиска адреса метода в VMT (indirect call). Однако современные [[JIT]]-компиляторы часто оптимизируют это через девиртуализацию (devirtualization), если могут доказать, что тип объекта не изменится.

#### 4. Абстракция: Abstract vs Interface
*   **[[Abstract Class]]:** Определяет идентичность ("is-a"). Может содержать состояние (поля) и реализацию методов.
*   **[[Interface]]:** Определяет контракт ("can-do"). В C# 8+ интерфейсы могут содержать реализации по умолчанию (DIM), но не могут содержать поля экземпляра.

### Пример кода (C# 12)

```csharp
using System;

// 1. Абстракция и Инкапсуляция
public abstract class PaymentProcessor(decimal amount) // Primary constructor
{
    // Защищенное поле - инкапсуляция данных
    protected decimal Amount = amount;

    // Абстрактный метод - полиморфное поведение
    public abstract void Process();

    public void PrintReceipt() => Console.WriteLine($"Processed: {Amount:C}");
}

// 2. Наследование
public class CreditCardProcessor(decimal amount, string cardNumber) 
    : PaymentProcessor(amount)
{
    private readonly string _cardNumber = cardNumber;

    // 3. Полиморфизм (переопределение)
    public override void Process()
    {
        Console.WriteLine($"Charging {_cardNumber} for {Amount}");
    }
}

public class Program
{
    public static void Main()
    {
        // Полиморфизм в действии: ссылка базового типа, объект производного
        PaymentProcessor processor = new CreditCardProcessor(150.00m, "1234-5678");
        
        processor.Process(); // Вызовется реализация CreditCardProcessor
        processor.PrintReceipt();
    }
}
```

### Ошибки и Best Practices

> [!WARNING] Ошибка: Наследование ради кода (Fragile Base Class)
> Избегайте глубоких иерархий наследования (более 3 уровней). Изменение в базовом классе может неявно сломать логику в десятках потомков. Предпочитайте **композицию наследованию** (Composition over Inheritance): вкладывайте один объект в другой вместо расширения.

> [!DANGER] Anti-pattern: Сокрытие методов через 'new'
> Использование ключевого слова `new` для скрытия метода базового класса вместо `override` приводит к путанице. Вызываемый метод будет зависеть от типа переменной (ссылки), а не от типа объекта, что нарушает интуитивный полиморфизм.

**Best Practices:**
1.  **Liskov Substitution Principle ([[Liskov Substitution Principle|LSP]]):** Производный класс должен быть пригоден для использования везде, где используется базовый, не ломая логику программы.
2.  **Интерфейсы для расширяемости:** Модули должны зависеть от абстракций (интерфейсов), а не от конкретных реализаций ([[Dependency Inversion Principle|DIP]]).
3.  **Sealed по умолчанию:** Если вы не проектируете класс специально для наследования, помечайте его как `sealed`. Это защищает архитектуру и помогает [[JIT]]-оптимизации.
4.  **Virtual в конструкторе:** Никогда не вызывайте виртуальные методы в конструкторе класса. В C# конструктор базового класса выполняется *раньше* конструктора производного, поэтому виртуальный метод может попытаться обратиться к еще не инициализированным полям потомка.

> [!TIP] Нюанс: Interface Dispatch
> Вызов метода через интерфейс исторически был медленнее, чем через виртуальный метод класса, из-за необходимости поиска в таблице интерфейсов (ITable). В современных версиях .NET (Core / .NET 5+) разница практически нивелирована благодаря технике **Interface Dispatch Optimization**.
