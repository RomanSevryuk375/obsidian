### Суть (The "What")
`IList<T>` — это центральный [[Интерфейсы|интерфейс]] для [[Коллекции|коллекций]], которые поддерживают доступ к элементам по индексу. Он расширяет `ICollection<T>` и `IEnumerable<T>`, добавляя возможность вставки, удаления и модификации элементов в конкретных позициях.

Интерфейс решает задачу **упорядоченного хранения данных**. Если `ICollection` просто гарантирует, что данные можно добавить или удалить, то `IList` гарантирует, что каждый элемент имеет свой порядковый номер (индекс), и этот порядок детерминирован.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Индексатор (The Indexer)
Ключевой член интерфейса — свойство-индексатор `this[int index]`. 
*   **Чтение:** Позволяет получить элемент по индексу. Для массивов и `List<T>` это операция **O(1)** (прямой доступ к памяти).
*   **Запись:** Позволяет заменить элемент по индексу.

#### 2. Манипуляция по индексу
Интерфейс вводит три критически важных метода:
*   `IndexOf(T item)`: Поиск индекса первого вхождения элемента ($O(n)$).
*   `Insert(int index, T item)`: Вставка элемента в середину. Для динамических массивов (`List<T>`) это дорогая операция ($O(n)$), так как все последующие элементы должны быть сдвинуты вправо.
*   `RemoveAt(int index)`: Удаление по позиции (также требует сдвига элементов влево — $O(n)$).

#### 3. IList (Legacy) против IList\<T\>
Существует нетипизированный интерфейс `IList` из пространства `System.Collections`. 
*   Он работает с типом `object`.
*   Содержит свойства `IsFixedSize` и `IsReadOnly`. 
*   Массивы (`System.Array`) реализуют именно этот интерфейс как "фиксированный по размеру". Вы можете менять элементы в массиве через индексатор `IList`, но вызов `Add` или `Remove` выбросит `NotSupportedException`.

> [!INFO] Нюанс: Явная реализация (Explicit Implementation)
> Большинство современных коллекций реализуют нетипизированный `IList` явно. Это означает, что метод `Add(object value)` не будет виден у `List<int>`, пока вы не приведете его к типу `IList`. Это защищает систему типов от случайного добавления некорректных объектов (например, строки в список чисел).

### Пример кода ([[MOC|C#]] 12)

```csharp
using System;
using System.Collections.Generic;

public class ListDemo
{
    public static void Main()
    {
        // 1. Использование интерфейса для абстракции
        // Слева интерфейс, справа конкретная реализация
        IList<string> fruits = ["Apple", "Banana", "Orange"];

        // 2. Доступ и модификация по индексу
        fruits[1] = "Blueberry"; 

        // 3. Вставка в конкретную позицию
        fruits.Insert(0, "Mango"); // ["Mango", "Apple", "Blueberry", "Orange"]

        // 4. Поиск индекса
        int orangeIndex = fruits.IndexOf("Orange");
        
        // 5. Удаление по индексу
        if (orangeIndex != -1)
        {
            fruits.RemoveAt(orangeIndex);
        }

        PrintList(fruits);
    }

    static void PrintList(IList<string> list)
    {
        for (int i = 0; i < list.Count; i++)
        {
            Console.WriteLine($"[{i}]: {list[i]}");
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: IndexOutOfRangeException
> Обращение к индексу, который меньше 0 или больше либо равен `Count`, приведет к немедленному падению программы. Всегда проверяйте границы, если индекс вычисляется динамически.

> [!WARNING] Ошибка: Вставка в цикле
> Выполнение `Insert(0, item)` или `RemoveAt(0)` внутри цикла по большой коллекции (`List<T>`) приводит к квадратичной сложности $O(n^2)$, так как каждое действие заставляет процессор копировать огромные блоки памяти. Для таких задач рассмотрите `LinkedList<T>` или `Queue<T>`.

**Best Practices:**
1.  **Принцип минимальных привилегий:** Если методу нужно только перебрать элементы, используйте `IEnumerable<T>`. Если нужно только знать количество — `IReadOnlyCollection<T>`. Используйте `IList<T>` только тогда, когда вам действительно нужен произвольный доступ по индексу или вставка/удаление.
2.  **Типизация:** Избегайте использования нетипизированного `IList`. Это приводит к упаковке ([[Boxing Unboxing]]) значимых типов и потере типобезопасности.
3.  **Массивы как IList:** Помните, что массив — это `IList` с фиксированным размером. Если вы передаете массив в метод, принимающий `IList`, этот метод не должен вызывать `Add` или `Clear`.
4.  **Foreach против For:** Хотя `IList` позволяет использовать `for` с индексом, `foreach` часто предпочтительнее для читаемости. Однако для `List<T>` обычный цикл `for` работает чуть быстрее, так как не создает объект-итератор в куче.

> [!TIP] Нюанс: Поиск
> Метод `IndexOf` использует стандартное сравнение `Equals`. Если вы работаете со сложными объектами, убедитесь, что в них правильно переопределена эквивалентность, иначе поиск будет работать только по ссылочной идентичности.
