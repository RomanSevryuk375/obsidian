### Суть (The "What")
**State Machine (Конечный автомат)** — это низкоуровневая структура (обычно [[Структуры|struct]]), которую компилятор [[MOC|C#]] генерирует автоматически при использовании ключевых слов `async/await` или [[yield return]]. 

Она решает задачу **сохранения контекста выполнения**: когда метод "прерывается" (натыкается на `await` незавершенной задачи или на `yield return`), его состояние (локальные переменные, параметры и точка выполнения) должно быть где-то сохранено, чтобы при возобновлении метод мог продолжить работу с того же места, а не сначала.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Трансформация метода (Lifting)
Когда вы помечаете метод как `async`, компилятор полностью переписывает его тело:
*   Оригинальный метод становится "оберткой", которая инициализирует структуру автомата и запускает её.
*   **Локальные переменные и параметры:** "поднимаются" (lifting) в поля структуры. Это необходимо, так как стек метода очищается при каждом `await`, и данные должны храниться в памяти (в куче, если автомат упакован), чтобы быть доступными при возобновлении.

#### 2. Интерфейс IAsyncStateMachine
Генерируемая структура реализует [[Интерфейсы|интерфейс]] `IAsyncStateMachine`, имеющий два ключевых метода:
*   **MoveNext():** Главный метод, содержащий логику вашего метода, обернутую в большой блок `switch` или серию `if`. Каждый `await` разделяет код на "состояния" (states).
*   **SetStateMachine():** Связывает автомат с билдером.

#### 3. Состояния автомата
*   **-1 (Start/Running):** Начальное выполнение.
*   **0, 1, 2... (Suspended):** Метод приостановлен на конкретном `await`. Номер указывает, на каком именно.
*   **-2 (Completed):** Выполнение завершено (успешно или с ошибкой).

#### 4. AsyncTaskMethodBuilder
Это вспомогательная структура, которая управляет созданием [[Task]], передачей исключений и контекста синхронизации. Именно она "склеивает" работу автомата с инфраструктурой [[TPL]] (Task Parallel Library).

> [!INFO] Нюанс: Структура или Класс?
> Компилятор генерирует автомат как `struct` для минимизации аллокаций. Однако, если метод не завершается синхронно (что происходит почти всегда), этот стэк-объект упаковывается (boxing) в кучу, чтобы сохранить состояние между переключениями потоков.

### Пример кода: Что видит компилятор

**Ваш код (C# 12):**
```csharp
public async Task<int> CalculateAsync(int delay)
{
    int result = 42;
    await Task.Delay(delay); // Точка прерывания
    return result + 10;
}
```

**Приблизительная логика генерируемого автомата (упрощенно):**
```csharp
// Это ПРИМЕР того, во что превращается метод
public struct CalculateAsync_StateMachine : IAsyncStateMachine
{
    public int state;
    public AsyncTaskMethodBuilder<int> builder;
    public int delay; // Параметр
    private int result; // Локальная переменная
    private TaskAwaiter awaiter;

    public void MoveNext()
    {
        try
        {
            if (state != 0) // Начало выполнения
            {
                result = 42;
                awaiter = Task.Delay(delay).GetAwaiter();
                if (!awaiter.IsCompleted)
                {
                    state = 0; // Переходим в состояние ожидания
                    builder.AwaitUnsafeOnCompleted(ref awaiter, ref this);
                    return; // Выход из потока, поток свободен
                }
            }
            
            // Возобновление после ожидания
            awaiter.GetResult();
            int finalResult = result + 10;
            state = -2;
            builder.SetResult(finalResult);
        }
        catch (Exception ex)
        {
            state = -2;
            builder.SetException(ex);
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Огромное количество локальных переменных
> Каждая локальная переменная в `async` методе становится полем в объекте автомата. Если в методе много тяжелых локальных переменных (большие структуры, много ссылок), объект автомата раздувается, увеличивая нагрузку на Garbage Collector при каждом переходе в асинхронный режим.

> [!WARNING] Ошибка: Использование Span в асинхронных методах
> Вы не можете использовать `Span` или `stackalloc` внутри `async` методов. Почему? Потому что `Span` — это `ref struct` (только стек), а автомат при паузе может быть упакован в кучу. Это физически несовместимо.

**Best Practices:**
1.  **[[ValueTask]]:** Если ваш метод часто завершается синхронно (данные уже в кэше), возвращайте `ValueTask`. Это позволит автомату избежать аллокации задачи в куче в "быстром" (синхронном) пути выполнения.
2.  **Декомпозиция:** Если асинхронный метод слишком длинный, разбейте его на части. Это упростит структуру конечного автомата и облегчит отладку.
3.  **ConfigureAwait(false):** В библиотечном коде всегда используйте `ConfigureAwait(false)`. Это позволяет автомату возобновиться в любом потоке пула, не пытаясь вернуться в исходный контекст (UI/ASP.NET Legacy), что упрощает работу планировщика.
4.  **Static Lambdas в Async:** Если вы используете локальные функции внутри async-метода, помечайте их как `static`, чтобы они не захватывали контекст автомата без необходимости.

> [!TIP] Нюанс: Итераторы (yield)
> Конечный автомат для `yield return` работает иначе: он реализует `IEnumerator`, а не `IAsyncStateMachine`. Он не "возвращает управление ОС", а просто приостанавливает цикл до следующего вызова `MoveNext()`. Потоки при этом не переключаются.
