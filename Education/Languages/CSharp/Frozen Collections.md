### Суть (The "What")
**Замороженные коллекции** (`FrozenDictionary<TKey, TValue>` и `FrozenSet<T>`) — это специализированные структуры данных, представленные в .NET 8, которые оптимизированы исключительно для операций чтения. После создания такая [[Коллекции|коллекция]] не может быть изменена (даже через механизмы копирования, как в `ImmutableCollections`).

Они решают задачу **максимальной производительности поиска** в сценариях, где коллекция создается один раз (например, при старте приложения) и затем используется миллионы раз для поиска данных ([[Lookup]]-таблицы, конфигурации, справочники).

---

### Как это работает (The "How" / Nutshell style)

#### 1. Агрессивная оптимизация при создании
В отличие от обычного [[Dictionary]], который должен поддерживать возможность добавления элементов и разрешения коллизий в динамике, `FrozenDictionary` при создании выполняет глубокий анализ набора ключей:
*   **Идеальное хеширование:** Рантайм может подобрать наиболее эффективную хеш-функцию специально под конкретный набор ключей, чтобы свести количество коллизий к абсолютному минимуму.
*   **Специализированные реализации:** В зависимости от типа ключей (например, `string`, `int`, `Guid`) и их количества, .NET выбирает одну из десяти внутренних реализаций (например, оптимизированную специально для коротких строк).

#### 2. Стоимость создания vs Скорость чтения
*   **Creation Cost:** Создание замороженной коллекции обходится значительно дороже (по времени и CPU), чем создание обычного словаря, из-за этапа анализа и перестроения таблиц.
*   **Read Speed:** Операции `TryGetValue` и `Contains` выполняются быстрее, чем в любых других коллекциях .NET, за счет меньшего количества проверок и оптимизированного расположения данных в памяти.

#### 3. Отличие от [[Immutable collections]]
*   `ImmutableList/Dictionary` спроектированы для сценариев, где вы *хотите* получать модифицированные версии коллекции (через создание новых узлов дерева). Сложность поиска там — $O(\log n)$.
*   `FrozenCollections` не поддерживают модификацию вообще. Сложность поиска — стабильная $O(1)$ с очень малым коэффициентом.

> [!INFO] Нюанс: Размещение в памяти
> Замороженные коллекции стремятся к максимальной локальности данных. Внутренние массивы ключей и значений располагаются максимально компактно, что улучшает работу кэша процессора (L1/L2 cache hit ratio) при частых обращениях.

### Пример кода ([[MOC|C#]] 12)

```csharp
using System;
using System.Collections.Frozen;
using System.Collections.Generic;

public class FrozenDemo
{
    private static readonly FrozenDictionary<string, string> _config;

    static FrozenDemo()
    {
        // 1. Сначала создаем обычный словарь или коллекцию
        var sourceData = new Dictionary<string, string>
        {
            ["ApiUrl"] = "https://api.example.com",
            ["Timeout"] = "00:00:30",
            ["RetryCount"] = "3"
        };

        // 2. "Замораживаем" её (происходит анализ ключей и оптимизация)
        _config = sourceData.ToFrozenDictionary();
    }

    public void ProcessRequest()
    {
        // 3. Сверхбыстрый поиск в рантайме
        if (_config.TryGetValue("ApiUrl", out var url))
        {
            Console.WriteLine($"Target URL: {url}");
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Замораживание короткоживущих коллекций
> Никогда не вызывайте `.ToFrozenDictionary()` на коллекциях, которые вы собираетесь использовать всего несколько раз или которые живут в рамках одного HTTP-запроса. Затраты на анализ ключей при создании перекроют всю выгоду от быстрого чтения.

> [!WARNING] Ошибка: Ожидание мутабельности
> Помните, что у Frozen-коллекций нет методов `Add` или `Remove`. Если вам нужно обновить справочник в рантайме, вам придется создать новый `Dictionary`, обновить его и вызвать `.ToFrozenDictionary()` заново, заменив ссылку.

**Best Practices:**
1.  **Static Read-Only Fields:** Идеальное место для Frozen-коллекций — статические поля классов, представляющие собой неизменяемые справочники (например, маппинг статус-кодов или типов валют).
2.  **String Keys:** Frozen-коллекции особенно эффективны при использовании строковых ключей, так как рантайм применяет специфические для строк оптимизации хеширования.
3.  **Dependency Injection:** Регистрируйте Frozen-коллекции как `Singleton` в DI-контейнере для глобального доступа к оптимизированным данным.
4.  **Размер имеет значение:** Наибольший прирост производительности по сравнению с обычным `Dictionary` наблюдается на коллекциях среднего и большого размера (от 50-100 элементов и выше).

> [!TIP] Нюанс: Потокобезопасность
> Поскольку замороженная коллекция абсолютно неизменяема, она является потокобезопасной по определению (Thread-Safe). Любое количество потоков может одновременно выполнять чтение без каких-либо блокировок ([[Monitor|lock]])。
