### Суть (The "What")
**Метод** — это именованный блок кода, который реализует определенное поведение или вычисления внутри [[Классы и объекты|класса]] или [[Структуры|структуры]]. Методы являются основными единицами выполняемого кода в [[MOC#/MOC|C#]].

Методы решают задачу **повторного использования кода**, декомпозиции сложных задач на мелкие управляемые модули и реализации интерфейса взаимодействия с объектом. Они [[Инкапсуляция|инкапсулируют]] логику, скрывая детали реализации от вызывающего кода.

---

### Как это работает (The "How" / Nutshell style)

#### 1. [[Стэк]] и кадры (Stack Frames)
При вызове метода в стеке текущего потока создается **кадр стека (Stack Frame)**. Он содержит:
*   Адрес возврата (куда передать управление после завершения).
*   Аргументы, переданные методу.
*   Локальные переменные.
*   Указатель на текущий объект (`this`), если метод не статический.

После завершения метода кадр стека "схлопывается", и память мгновенно освобождается.

#### 2. [[JIT]]-компиляция и исполнение
Методы не хранятся в памяти как машинный код до первого вызова.
*   При первом обращении к методу **JIT-компилятор (Just-In-Time)** транслирует его [[IL]]-код в нативные инструкции процессора.
*   Адрес скомпилированного кода записывается в таблицу методов ([[MethodTable]]) типа.
*   Последующие вызовы происходят напрямую по адресу в памяти без участия компилятора.

#### 3. Перегрузка (Overloading)
C# поддерживает перегрузку методов — возможность иметь несколько методов с одним именем, но разными сигнатурами. 
*   **Сигнатура** включает: имя метода, количество параметров, типы параметров и их [[Модификаторы входных параметров|модификаторы]] (`ref`, `out`).
*   Тип возвращаемого значения **не является** частью сигнатуры.

#### 4. Expression-bodied members
Для методов, состоящих из одного выражения, C# предлагает сокращенный синтаксис `=>` (lambda-style), который компилируется в идентичный IL-код, но улучшает читаемость.

> [!INFO] Нюанс: call vs callvirt
> На уровне IL обычные методы вызываются инструкцией `call`, а виртуальные — `callvirt`. Интересно, что компилятор C# использует `callvirt` даже для невиртуальных методов экземпляра, чтобы обеспечить автоматическую проверку на `null` (инструкция `callvirt` выбрасывает `NullReferenceException`, если целевой объект пуст).

### Пример кода ([[MOC|C#]] 12)

```csharp
using System;
using System.Runtime.CompilerServices;

public class MathService(string name) // Primary constructor
{
    public string Name { get; } = name;

    // 1. Expression-bodied метод
    public int Add(int a, int b) => a + b;

    // 2. Перегрузка метода
    public double Add(double a, double b) => a + b;

    // 3. Метод с необязательными параметрами
    public void LogInfo(string message, bool timestamp = true)
    {
        var time = timestamp ? $"[{DateTime.Now:T}] " : "";
        Console.WriteLine($"{time}{Name}: {message}");
    }

    // 4. Локальная функция внутри метода
    public int CalculateComplex(int factor)
    {
        return factor * GetMultiplier();

        // Инкапсуляция логики, нужной только здесь
        static int GetMultiplier() => 42;
    }

    // 5. Подсказка JIT-компилятору (агрессивное встраивание)
    [MethodImpl(MethodImplOptions.AggressiveInlining)]
    public int FastMultiply(int x) => x * x;
}

public class Program
{
    public static void Main()
    {
        var service = new MathService("Calculator");
        
        // Именованные аргументы для ясности
        service.LogInfo(message: "Started", timestamp: false);
        
        int result = service.Add(10, 20);
        Console.WriteLine(result);
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Слишком длинные методы (God Methods)
> Метод должен выполнять одну задачу (Принцип единственной ответственности — [[SRP]]). Если метод занимает более 20–30 строк, это сигнал к рефакторингу и выделению локальных функций или приватных методов.

> [!WARNING] Ошибка: Чрезмерное количество параметров
> Методы с более чем 4–5 параметрами трудно читать и тестировать. Если данных много, объедините их в структуру или класс (Parameter Object).

**Best Practices:**
1.  **Чистота функций:** Старайтесь делать методы "чистыми" (результат зависит только от входных данных, нет побочных эффектов). Это упрощает тестирование.
2.  **PascalCase:** Всегда называйте методы с большой буквы согласно соглашениям .NET.
3.  **Использование static:** Если метод не обращается к полям и свойствам экземпляра класса, помечайте его как `static`. Это явно указывает на отсутствие зависимости от состояния объекта и немного ускоряет вызов.
4.  **Валидация на входе:** Проверяйте аргументы (Guard Clauses) в самом начале метода. Это предотвращает выполнение логики с некорректными данными.

> [!TIP] Нюанс: Опциональные параметры
> Значения по умолчанию для опциональных параметров подставляются в месте **вызова** во время компиляции. Если вы измените значение по умолчанию в библиотеке, но не пересоберете вызывающий проект, он продолжит использовать старое значение.
