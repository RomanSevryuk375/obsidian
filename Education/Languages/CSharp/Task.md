### Суть (The "What")
**Task** — это объект, представляющий асинхронную операцию. В отличие от низкоуровневого [[Многопоточность|Thread]], который является прямой проекцией потока ОС, `Task` — это абстракция более высокого уровня, входящая в состав **[[TPL]] (Task Parallel Library)**.

Он решает задачу эффективного управления параллелизмом: позволяет не заботиться о создании и уничтожении потоков вручную, предоставляя удобный API для обработки результатов, цепочек вызовов (continuations) и обработки исключений.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Пул потоков (ThreadPool)
Большинство задач по умолчанию выполняются на потоках из **ThreadPool**. Пул потоков оптимизирует ресурсы, переиспользуя уже созданные потоки вместо дорогостоящей аллокации новых для каждой короткой задачи. Планировщик задач (`TaskScheduler`) решает, когда и на каком потоке запустить задачу.

#### 2. Жизненный цикл и состояния
Каждый `Task` имеет свойство `Status`. Основные этапы:
*   **Created**: Задача создана вручную через `new Task()`, но еще не запущена (Cold Task).
*   **WaitingForActivation**: Задача ожидает выполнения инфраструктурой.
*   **Running**: Задача выполняется в потоке.
*   **RanToCompletion / Faulted / Canceled**: Финальные состояния (Hot Task).

#### 3. Task vs Task\<T\>
*   `Task`: Аналог `void`. Используется, когда важен сам факт завершения операции.
*   `Task<T>`: Возвращает результат типа `T`. Результат доступен через свойство `.Result`, которое является блокирующим (лучше использовать `await`).

#### 4. TaskCompletionSource\<T\>
Это способ создать "обещание" (promise). Вы создаете `Task`, который не привязан к конкретному методу в пуле потоков, а завершается вручную через методы `SetResult` или `SetException`. Это фундамент для превращения старых событийных API (EAP) в современные асинхронные.

> [!INFO] Нюанс: "Холодные" и "Горячие" задачи
> Методы `Task.Run` и `Task.Factory.StartNew` создают и сразу запускают задачу ("горячая" задача). Если создать задачу через конструктор `new Task()`, она будет "холодной", и её нужно запустить методом `.Start()`. В современной практике "холодные" задачи используются крайне редко.

### Пример кода ([[MOC|C#]] 12)

```csharp
using System;
using System.Threading;
using System.Threading.Tasks;

public class TaskDemo
{
    public static async Task Main()
    {
        // 1. Создание и запуск задачи (CPU-bound операция)
        Task<int> calculationTask = Task.Run(() => 
        {
            int sum = 0;
            for (int i = 0; i < 100; i++) sum += i;
            return sum;
        });

        // 2. Использование CancellationToken
        using var cts = new CancellationTokenSource();
        Task longTask = DoWorkAsync(cts.Token);

        // Имитация отмены через 1 секунду
        cts.CancelAfter(1000);

        // 3. Ожидание завершения (неблокирующее)
        try 
        {
            int result = await calculationTask;
            Console.WriteLine($"Result: {result}");
            await longTask;
        }
        catch (OperationCanceledException)
        {
            Console.WriteLine("Task was canceled.");
        }
    }

    static async Task DoWorkAsync(CancellationToken token)
    {
        for (int i = 0; i < 10; i++)
        {
            token.ThrowIfCancellationRequested();
            await Task.Delay(500, token); // Асинхронная пауза
            Console.WriteLine($"Step {i} completed");
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Использование .Result и .Wait()
> Обращение к `.Result` или вызов `.Wait()` у незавершенной задачи блокирует текущий поток до момента завершения задачи. В средах с `SynchronizationContext` (WPF, старый ASP.NET) это ведет к гарантированному **Deadlock**. Всегда используйте `await`.

> [!WARNING] Ошибка: Исключения в Task.Run
> Если внутри `Task.Run` возникло исключение и вы не "ожидаете" (await) эту задачу, исключение будет проглочено до момента сборки мусора или останется незамеченным, что приведет к трудноотлавливаемым багам.

**Best Practices:**
1.  **Task.Run для CPU-bound:** Используйте `Task.Run` только для тяжелых вычислений. Для I/O (сеть, файлы) используйте встроенные асинхронные методы API (напр. `ReadAsync`).
2.  **Task.WhenAll:** Если нужно запустить несколько независимых задач, используйте `await Task.WhenAll(tasks)`. Это эффективнее, чем ждать каждую задачу по очереди.
3.  **Избегайте Task.Factory.StartNew:** Этот метод сложнее и опаснее (он не понимает асинхронные делегаты правильно без `Unwrap()`). В 99% случаев `Task.Run` — лучший выбор.
4.  **[[ValueTask]]:** Если метод часто возвращает результат синхронно (например, данные из кэша), используйте `ValueTask<T>`, чтобы избежать лишних аллокаций в куче.

> [!TIP] Нюанс: Исключения и AggregateException
> Когда вы ждете задачу через `await`, выбрасывается первое возникшее исключение. Однако если вы используете `Task.WaitAll` или `.Result`, все исключения упаковываются в `AggregateException`. Для правильной обработки нескольких исключений в `Task.WhenAll` следует обращаться к свойству `task.Exception` после завершения.
