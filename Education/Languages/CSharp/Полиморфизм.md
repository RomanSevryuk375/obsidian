# Полиморфизм (Polymorphism)

### Суть (The "What")
**Полиморфизм** (от греч. "много форм") — это способность системы использовать объекты с одинаковым интерфейсом без информации о конкретном типе этого объекта. В [[MOC#/MOC|C#]] это означает, что производный класс может рассматриваться как его базовый класс или реализуемый им интерфейс.

Полиморфизм решает задачу **взаимозаменяемости компонентов**: он позволяет писать код, который работает с абстракциями, а не с конкретными реализациями, что делает систему гибкой и расширяемой (соблюдение принципа Open/Closed из SOLID).

---
### Как это работает (The "How" / Nutshell style)

В .NET полиморфизм разделяют на два основных вида:

#### 1. [[Статический полиморфизм]] (Compile-time)
Реализуется через **перегрузку методов (overloading)**. Решение о том, какая версия метода будет вызвана, принимается компилятором на основе сигнатуры (типов и количества аргументов). 
*   **Нюанс:** Это не полиморфизм в строгом смысле слова "динамического поведения", но он позволяет одному имени метода принимать разные формы.

#### 2. [[Динамический полиморфизм]] (Runtime)
Это "настоящий" полиморфизм, основанный на наследовании и интерфейсах. Решение о вызове конкретного кода принимается [[CLR]] во время выполнения программы.

*   **[[VMT]] (Virtual Method Table):** Для каждого типа, имеющего виртуальные члены, создается таблица методов. Объект в куче хранит указатель на эту таблицу.
*   **Инструкция callvirt:** В отличие от обычной инструкции `call` (которая обращается по фиксированному адресу), компилятор генерирует инструкцию `callvirt`. Она заставляет процессор:
    1. Проверить объект на `null` (генерирует `NullReferenceException`, если объект пуст).
    2. Перейти в таблицу типов объекта.
    3. Найти адрес метода по индексу слота в VMT и выполнить его.

> [!INFO] Нюанс: Диспетчеризация интерфейсов
> Вызов метода через интерфейс (Interface Dispatch) технически сложнее вызова через VMT класса. CLR использует механизмы **Virtual Stub Dispatch (VSD)**, которые кэшируют результаты поиска реализации, чтобы минимизировать накладные расходы на поиск в таблицах интерфейсов.

#### 3. Приведение типов (Casting)
*   **Upcasting:** Приведение к базовому типу (безопасно, неявно).
*   **Downcasting:** Приведение к производному типу (небезопасно, требует явной проверки через `is` или `as`).

### Пример кода (C# 12)

```csharp
using System;
using System.Collections.Generic;

// Интерфейс как контракт
public interface ISoundProducer
{
    void MakeSound();
}

// Базовый класс с первичным конструктором (C# 12)
public abstract class Animal(string name) : ISoundProducer
{
    public string Name { get; } = name;

    // Виртуальный метод для переопределения
    public virtual void Move() => Console.WriteLine($"{Name} перемещается.");

    // Абстрактный метод (обязателен к реализации)
    public abstract void MakeSound();
}

public class Dog(string name) : Animal(name)
{
    public override void MakeSound() => Console.WriteLine($"{Name} лает: Гав-гав!");
    
    public override void Move() => Console.WriteLine($"{Name} бежит на четырех лапах.");
}

public class Bird(string name) : Animal(name)
{
    public override void MakeSound() => Console.WriteLine($"{Name} поет: Чирик-чирик!");

    // Bird использует базовую реализацию Move(), но может быть приведена к ISoundProducer
}

public class Program
{
    public static void Main()
    {
        // Полиморфная коллекция (C# 12 collection expression)
        List<Animal> zoo = [new Dog("Шарик"), new Bird("Кеша")];

        foreach (var animal in zoo)
        {
            // Полиморфный вызов методов Animal
            animal.Move();
            animal.MakeSound();

            // Проверка на принадлежность к контракту (интерфейсу)
            if (animal is ISoundProducer producer)
            {
                producer.MakeSound();
            }
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Скрытие через 'new'
> Избегайте использования ключевого слова `new` для скрытия методов базового класса. Это ломает полиморфизм: при вызове через ссылку базового типа выполнится старый код, а через ссылку производного — новый. Всегда используйте `virtual` и `override`.

> [!WARNING] Ошибка: Нарушение принципа Лисков ([[LSP]])
> Не переопределяйте методы так, чтобы они нарушали логику базового класса (например, выбрасывали `NotImplementedException`). Если метод базового класса не имеет смысла в потомке, значит, иерархия спроектирована неверно.

**Best Practices:**
1.  **Проектируйте от интерфейсов:** Если вам нужно только поведение (can-do), используйте интерфейс. Если нужно общее состояние и базовая логика (is-a), используйте абстрактный класс.
2.  **Запечатывание для оптимизации:** Если вы создаете конечный класс в иерархии, помечайте его как `sealed`. Это позволяет JIT-компилятору выполнить **девиртуализацию** (вызвать метод напрямую без обращения к VMT), что повышает производительность.
3.  **Pattern Matching:** Вместо классического приведения типов `(Dog)animal` используйте современный синтаксис `if (animal is Dog dog)`. Это безопаснее и читаемее.
4.  **Virtual в конструкторе:** Никогда не вызывайте виртуальные методы в конструкторе класса. Объект еще не полностью инициализирован, и вызов переопределенного метода в потомке может привести к обращению к неинициализированным полям.

> [!TIP] Нюанс производительности
> Начиная с .NET 6+, [[JIT]]-компилятор стал намного эффективнее в "инлайнинге" (встраивании) виртуальных вызовов, если он может доказать, что в данной точке кода объект всегда имеет определенный тип. Это делает использование полиморфизма практически бесплатным для большинства бизнес-сценариев.
