### Суть (The "What")
**Рекорд** — это особый тип данных (появившийся в [[MOC#/MOC|C#]] 9), который предоставляет встроенный функционал для инкапсуляции данных с семантикой **значимого равенства (value-based equality)**. По умолчанию рекорды спроектированы как неизменяемые (immutable).

Рекорды решают проблему избыточности кода (boilerplate) при создании простых классов-контейнеров данных (DTO, модели), где требуется сравнение не по ссылке в памяти, а по содержимому полей.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Генерация кода компилятором
На низком уровне `record` остается классом (если не указано `record struct`), но компилятор автоматически генерирует:
*   Методы сравнения `Equals(object)`, `Equals(T)` и перегрузки операторов `==` и `!=`.
*   Переопределение `GetHashCode()`, основанное на значениях полей.
*   Метод `ToString()`, который выводит названия и значения свойств (удобно для дебага).
*   Метод `Deconstruct` для поддержки деконструкции.
*   Специальный метод-клонировщик (обычно скрытый) для работы оператора `with`.

#### 2. Позиционные параметры (Primary Constructors)
Синтаксис `public record Person(string Name, int Age);` автоматически создает свойства с модификатором `init`. Это означает, что их можно установить только при создании объекта или в инициализаторе.

#### 3. Неразрушающая мутация (Non-destructive mutation)
Для изменения данных в неизменяемом рекорде используется оператор `with`. 
*   **Механика:** Создается новая копия объекта в куче, копируются все текущие значения полей, а затем применяются указанные изменения. Оригинальный объект остается нетронутым.

#### 4. Record Class vs Record Struct
*   **record class (или просто record):** Ссылочный тип. Живет в куче. Поддерживает иерархию наследования.
*   **record struct (C# 10+):** Значимый тип. Живет в стеке. По умолчанию **изменяем** (mutable), если не добавить модификатор `readonly`.

> [!INFO] Нюанс: EqualityContract
> В отличие от структур, рекорды-классы генерируют свойство `EqualityContract`. Это гарантирует, что два объекта разных типов рекордов не будут признаны равными, даже если их поля полностью совпадают.

### Пример кода (C# 12)

```csharp
using System;

// 1. Объявление через первичный конструктор (Primary Constructor)
public record User(string FirstName, string LastName);

public class Program
{
    public static void Main()
    {
        // 2. Инициализация
        var user1 = new User("Ivan", "Ivanov");
        
        // 3. Значимое равенство
        var user2 = new User("Ivan", "Ivanov");
        Console.WriteLine(user1 == user2); // True (хотя ссылки разные)

        // 4. Неразрушающее изменение (with)
        var user3 = user1 with { LastName = "Petrov" };
        
        // 5. Деконструкция
        var (name, surname) = user3;
        Console.WriteLine($"{name} {surname}"); // Ivan Petrov

        // 6. Record Struct (Значимый тип)
        Point p1 = new(10, 20);
    }
}

// Позиционная структура (C# 10+)
public readonly record struct Point(int X, int Y);
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Изменяемые свойства в рекордах
> Технически можно объявить рекорд с обычными `set`-свойствами: `public record Bad(string Data) { public string Data { get; set; } }`. Это нарушает основную идею рекордов. Если данные должны меняться на месте, лучше использовать обычный `class`.

> [!WARNING] Ошибка: Сравнение коллекций
> Если поле рекорда является массивом или `List<T>`, значимое равенство "сломается". Рекорды выполняют поверхностное сравнение (shallow comparison) для ссылочных типов внутри. Два рекорда со списками, содержащими одинаковые элементы, будут **не равны**, так как ссылки на списки разные.

**Best Practices:**
1.  **DTO и API:** Используйте `record` для любых объектов, которые служат только для передачи данных.
2.  **Наследование:** Будьте осторожны с наследованием рекордов. Оператор `with` корректно копирует тип наследника, но логика сравнения может стать запутанной.
3.  **Readonly Record Struct:** Если вам нужна структура с поведением рекорда, всегда делайте её `readonly`, чтобы избежать проблем с изменением хеш-кода.
4.  **Валидация:** Если в первичном конструкторе нужна валидация, её можно добавить в тело рекорда, используя свойства с логикой или дополнительные проверки.

> [!TIP] Нюанс производительности
> Поскольку рекорды — это классы, они создают нагрузку на [[GC]]. Если вам нужно хранить миллионы мелких точек с логикой сравнения, используйте `readonly record struct`, чтобы избежать аллокаций в куче.
