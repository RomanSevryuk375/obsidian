# Стек (Call Stack) в .NET

### Суть (The "What")
**Стек вызовов (Stack)** — это локальная для каждого [[Многопоточность|потока]], строго упорядоченная область оперативной памяти, предназначенная для мгновенного выделения ресурсов под локальные переменные, аргументы методов и данные о состоянии выполнения программы.

Стек обеспечивает **детерминированное и сверхбыстрое управление памятью** в рамках лексической области видимости. Память здесь выделяется и освобождается автоматически по принципу LIFO (Last-In, First-Out — «последним пришел, первым ушел») по мере входа в методы и выхода из них.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Устройство стека и кадры ([[Stack frame]])
При запуске нового потока [[CLR]] (или ОС) выделяет под его стек фиксированный объем памяти (по умолчанию 1 МБ). При каждом вызове метода в стеке формируется изолированная ячейка — **кадр стека (Stack Frame)**. Он содержит:
*   Аргументы, переданные в метод.
*   Адрес возврата (инструкцию CPU, к которой нужно вернуться после завершения метода).
*   Локальные переменные (включая значимые типы и указатели/ссылки на объекты в куче).
*   Сохраненные значения регистров процессора.

#### 2. Аппаратная скорость
Выделение памяти в стеке — это фактически одна инструкция процессора. Она заключается в простом изменении значения специального регистра процессора (указателя стека `RSP` / `ESP`). Никакого поиска свободных блоков или уплотнения памяти не происходит. При выходе из метода указатель возвращается назад, мгновенно "уничтожая" весь кадр стека.

#### 3. Значимые типы и ссылки
*   **Локальные [[Value Types|значимые типы**]] (например, `int x = 5;`) полностью хранятся в текущем кадре стека.
*   **[[Reference Types|Ссылочные типы]]** хранятся иначе: в кадре стека лежит только адрес (4 или 8 байт), указывающий на сам объект, который физически расположен в [[Managed Heap|управляемой куче]] (Heap).

#### 4. stackalloc
Ключевое слово `stackalloc` позволяет напрямую выделить блок неструктурированной памяти в текущем кадре стека. В современном [[MOC|C#]] результат `stackalloc` принято оборачивать в [[Span и ReadOnlySpan]] для безопасной и быстрой работы с индексами.

### Пример кода (C# 12)

```csharp
using System;

public class Sample { }

public class StackDemo
{
    public static void Main()
    {
        int limit = 5; // Значимый тип целиком в кадре стека Main
        
        // Ссылка 'obj' лежит в кадре стека Main, а сам объект — в куче (Heap)
        Sample obj = new Sample(); 

        ProcessData(limit);
    } // Кадр стека Main уничтожается вместе со всеми локальными данными

    private static void ProcessData(int count)
    {
        // Создается новый кадр стека для ProcessData.
        // Параметр 'count' скопирован в этот кадр.

        // Выделение памяти прямо в кадре стека под массив из 4 элементов
        Span<int> tempBuffer = stackalloc int[4];
        tempBuffer[0] = count * 2;

        Console.WriteLine($"Buffer value: {tempBuffer[0]}");
    } // Кадр стека ProcessData мгновенно уничтожается, tempBuffer освобождается
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: StackOverflowException
> Объем стека жестко ограничен (всего 1 МБ). Бесконечная рекурсия или выделение слишком большого объема памяти через `stackalloc` мгновенно переполняют стек. CLR обрабатывает это как критическую ошибку и немедленно завершает процесс без возможности перехватить исключение через `try-catch`.

> [!WARNING] Ошибка: Передача тяжелых структур по значению
> Поскольку при передаче параметров значимых типов в методы происходит их полное копирование в новый кадр стека, передача структур размером более 16-24 байт (например, структура с 10 полями типа `double`) может привести к снижению производительности.
> **Решение:** Передавайте тяжелые структуры по ссылке, используя модификатор `in` (только для чтения) или `ref`.

**Best Practices:**
1.  **Безопасный stackalloc:** Всегда оборачивайте `stackalloc` в `Span<T>` или `ReadOnlySpan<T>`. Никогда не используйте сырые указатели (`int*`), если только это не требуется для работы с устаревшим нативным API (Interop).
2.  **Лимит stackalloc:** Используйте выделение памяти в стеке только для небольших массивов (обычно до 1-2 Кб). Если размер буфера динамический и может превысить этот лимит, делайте ветвление:
    `Span<byte> buffer = size < 1024 ? stackalloc byte[size] : new byte[size];`
3.  **Не возвращайте ссылки на стек:** Избегайте возврата `Span<T>`, указывающего на `stackalloc`, наружу из метода. Компилятор C# заблокирует такую попытку, так как это привело бы к появлению висячего указателя (dangling pointer) на очищенную память стека.
4.  **Избегайте глубокой рекурсии:** Всегда предусматривайте базовый случай выхода из рекурсивных методов или переписывайте алгоритмы на итеративный подход с использованием `Queue<T>` или `Stack<T>` в куче.

> [!TIP] Нюанс: Структуры в куче
> Правило "структуры живут в стеке" применимо только к локальным переменным. Если структура является полем класса (`class User { public Point Position; }`), она будет физически размещена в управляемой куче в составе памяти этого класса.
