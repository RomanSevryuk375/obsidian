### Суть (The "What")
`IEnumerable<T>` — это базовый [[Интерфейсы|интерфейс ]] для всех [[Коллекции|коллекций]] в .NET. Он определяет единственный метод `GetEnumerator()`, который возвращает объект `IEnumerator<T>`. 

Интерфейс решает задачу **унифицированного перебора данных**. Он предоставляет абстракцию "последовательности", позволяя коду перебирать элементы массива, списка, файла или результата запроса к БД по одной и той же логике, не зная о внутренней структуре хранения данных.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Паттерн "Итератор"
`IEnumerable` реализует модель "вытягивания" (pull model):
*   `IEnumerator.Current`: Возвращает текущий элемент.
*   `IEnumerator.MoveNext()`: Перемещает указатель на следующий элемент и возвращает `false`, если элементы закончились.
*   `IEnumerator.Reset()`: Сбрасывает указатель (многие современные провайдеры, включая LINQ, этот метод не реализуют и выбрасывают исключение).

#### 2. Магия foreach и Duck Typing
Компилятор [[MOC|C#]] при обработке `foreach` не ищет строго интерфейс `IEnumerable`. Он использует **утиную типизацию**: если у типа есть публичный метод `GetEnumerator()`, который возвращает объект с методом `MoveNext()` и свойством `Current`, такой тип можно перебирать.
Это сделано для оптимизации: например, `List<T>.GetEnumerator()` возвращает **структуру**, а не интерфейс, что исключает упаковку ([[boxing]]) и аллокации в куче при итерации.

#### 3. Отложенное выполнение (Deferred Execution)
Интерфейс представляет не сами данные, а **способ их получения**. 
Когда вы используете `yield return` или [[LINQ]]-запрос, код не выполняется мгновенно. Он "засыпает" в методе `GetEnumerator` и "просыпается" только тогда, когда вызывающий код вызывает `MoveNext()`.

> [!INFO] Нюанс: Состояние итератора
> Компилятор превращает метод, возвращающий `IEnumerable` с `yield return`, в скрытый класс — конечный автомат (State Machine). Этот класс хранит все локальные переменные метода в своих полях, чтобы восстановить состояние при следующем шаге итерации.

### Пример кода (C# 12)

```csharp
using System;
using System.Collections;
using System.Collections.Generic;

public class DataStore : IEnumerable<int>
{
    private int[] _data = [10, 20, 30, 40, 50]; // Collection expression

    // Реализация интерфейса
    public IEnumerator<int> GetEnumerator()
    {
        foreach (var item in _data)
        {
            // Итератор вернет управление вызывающему коду здесь
            yield return item;
        }
    }

    // Явная реализация старого интерфейса (обязательна для совместимости)
    IEnumerator IEnumerable.GetEnumerator() => GetEnumerator();
}

public class Program
{
    public static void Main()
    {
        var store = new DataStore();

        // Компилятор развернет это в while(enumerator.MoveNext())
        foreach (int val in store)
        {
            Console.WriteLine(val);
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Множественное перечисление (Multiple Enumeration)
> Если `IEnumerable` представляет результат тяжелого запроса (например, к БД), то каждый вызов `foreach` или методов `Count()`, `First()` будет выполнять запрос заново. Если данные нужны несколько раз, используйте `.ToList()` или `.ToArray()` для фиксации результата в памяти.

> [!WARNING] Ошибка: Изменение коллекции
> Стандартные итераторы коллекций .NET (List, Dictionary) проверяют версию коллекции. Если внутри `foreach` вызвать `Add()` или `Remove()`, итератор при следующем шаге `MoveNext()` выбросит `InvalidOperationException`.

**Best Practices:**
1.  **IReadOnlyCollection vs IEnumerable:** Если вы возвращаете коллекцию, которая уже находится в памяти и вы знаете её размер, лучше возвращать `IReadOnlyCollection<T>`. Это избавит потребителя от случайного множественного перечисления.
2.  **Использование yield:** Используйте `yield return` для создания ленивых последовательностей (например, чтение строк из огромного файла), чтобы не загружать все данные в оперативную память сразу.
3.  **Covariance:** Интерфейс `IEnumerable<out T>` является ковариантным. Это значит, что вы можете передать `IEnumerable<string>` туда, где ожидается `IEnumerable<object>`.
4.  **Dispose:** Помните, что `IEnumerator<T>` наследует `IDisposable`. Блок `foreach` автоматически вызывает `Dispose()`, что гарантирует выполнение блоков `finally` внутри методов с `yield return`.

> [!TIP] Производительность
> Для массивов и списков `foreach` оптимизирован компилятором и работает максимально быстро. Однако, если вы работаете с `IEnumerable<T>` через LINQ, каждый вызов `MoveNext()` — это виртуальный вызов метода, что может быть накладным в сверхнагруженных циклах (Hot Paths).
