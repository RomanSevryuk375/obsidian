### Суть (The "What")
**Mutex** — это примитив синхронизации, обеспечивающий исключительный доступ к разделяемому ресурсу. В отличие от оператора `lock` ([[Monitor]]), который работает только в рамках одного домена приложения, Mutex является **объектом ядра ОС**.

Главная задача Mutex — обеспечение синхронизации между **разными процессами**. Его часто используют для предотвращения запуска нескольких экземпляров одного и того же приложения (Single Instance Application).

---

### Как это работает (The "How" / Nutshell style)

#### 1. Локальные и Именованные Mutex
*   **Локальный Mutex:** Не имеет имени. Виден только внутри процесса. По функционалу близок к `Monitor`, но медленнее.
*   **Именованный (Глобальный) Mutex:** Имеет строковое имя, видимое во всей операционной системе. Имена с префиксом `Global\` видны всем пользователям, `Local\` — только внутри сессии текущего пользователя.

#### 2. Переход в режим ядра (Kernel Transition)
Mutex является оберткой над нативным объектом Windows.
*   Каждый вызов `WaitOne()` и `ReleaseMutex()` требует переключения контекста из пользовательского режима (User Mode) в режим ядра (Kernel Mode).
*   **Производительность:** Из-за системных вызовов Mutex работает примерно в 50 раз медленнее, чем `Monitor` или `lock`.

#### 3. Потоковая привязка (Thread Affinity)
Mutex обладает строгой привязкой к потоку:
*   Только тот поток, который захватил Mutex через `WaitOne()`, имеет право освободить его через `ReleaseMutex()`.
*   Если поток завершается, не освободив Mutex, объект помечается как **Abandoned (брошенный)**. Следующий поток, пытающийся его захватить, получит исключение `AbandonedMutexException`.

#### 4. Рекурсивность
Mutex является рекурсивным: один и тот же поток может захватывать его несколько раз подряд, не блокируя сам себя. Однако освободить его нужно будет ровно столько же раз.

> [!IMPORTANT] Нюанс: WaitHandle
> Mutex наследуется от `WaitHandle`. Это позволяет использовать его в статических методах `WaitAll` или `WaitAny`, объединяя логику ожидания с другими примитивами (например, `AutoResetEvent`).

### Пример кода ([[MOC|C#]] 12)

Реализация проверки на запуск единственного экземпляра приложения.

```csharp
using System;
using System.Threading;

public class SingleInstanceApp
{
    // Уникальное имя для всей ОС
    private const string AppGuid = "Global\\77B3-4A55-9011-F032B87C2241";

    public static void Main()
    {
        // initiallyOwned: false - не захватывать сразу
        // createdNew - флаг, удалось ли создать новый объект (или он уже был в ОС)
        using var mutex = new Mutex(false, AppGuid, out bool createdNew);

        if (!createdNew)
        {
            Console.WriteLine("Приложение уже запущено в другом процессе.");
            return;
        }

        try
        {
            Console.WriteLine("Приложение запущено. Нажмите Enter для выхода...");
            RunBusinessLogic();
        }
        finally
        {
            // На всякий случай освобождаем, если захватили вручную
            // В данном паттерне это не обязательно, т.к. createdNew просто проверяет наличие
        }
    }

    static void RunBusinessLogic() => Console.ReadLine();
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Использование Mutex для внутрипроцессной синхронизации
> Если вам нужно защитить переменную внутри одного приложения, никогда не используйте Mutex. Используйте `lock` или `Interlocked`. Накладные расходы Mutex на работу с ядром ОС избыточны для таких задач.

> [!WARNING] Ошибка: Забытый ReleaseMutex
> Если поток аварийно завершится без освобождения Mutex, другие процессы могут годами ждать доступа. Хотя ОС пометит Mutex как "брошенный", обработка `AbandonedMutexException` — сложная задача, так как состояние защищаемого ресурса в этот момент может быть повреждено.

**Best Practices:**
1.  **Всегда используйте using:** Mutex владеет неуправляемым дескриптором (Handle), поэтому его уничтожение через `Dispose()` обязательно.
2.  **Обработка исключений:** Оборачивайте `WaitOne()` в блок `try-finally`, чтобы гарантировать вызов `ReleaseMutex()`.
3.  **Таймауты:** В `WaitOne()` всегда полезно передавать таймаут (например, 5000мс), чтобы избежать бесконечного зависания процесса при "мертвых" блокировках.
4.  **Prefix:** Для именованных Mutex всегда используйте префикс `Global\` или `Local\`, чтобы четко определить область видимости объекта в рамках ОС.

> [!TIP] Нюанс: Semaphore vs Mutex
> Mutex можно рассматривать как `Semaphore` с емкостью 1, но с одним важным отличием: у семафора нет привязки к потоку (любой поток может освободить семафор), а у Mutex — есть.
