### Суть (The "What")
Nullable-типы позволяют переменным представлять отсутствие значения (`null`). В [[C#]] эта концепция разделена на две фундаментально разные категории:
1.  **Nullable значимые типы (Nullable [[Value Types]], `T?`):** расширяют обычные структуры (`int`, `bool`), позволяя им принимать значение `null`.
2.  **Nullable ссылочные типы (Nullable [[Reference Types]], NRT):** добавляют статический анализ на уровне компилятора, разделяя обычные ссылки на те, которые могут быть `null` (`string?`), и те, которые не должны ими быть (`string`).

Эта концепция решает задачу **обработки отсутствующих данных** (например, `NULL` полей в базах данных) и предотвращает возникновение самой частой ошибки в мире .NET — `NullReferenceException` (так называемой «ошибки на миллиард долларов»).

### Как это работает (The "How" / Nutshell style)

#### 1. Nullable значимые типы (System.Nullable\<T\>)
На низком уровне `int?` является синтаксическим сахаром для универсальной структуры `System.Nullable<T>`:
*   **Устройство в памяти:** Это обычная структура ([[Структуры|struct]]), содержащая два поля: `T value` и `bool hasValue`. Её размер в стеке равен размеру исходного типа `sizeof(T)` плюс 1 байт для флага присутствия значения (с учетом выравнивания памяти процессором).
*   **Особые правила упаковки ([[Boxing Unboxing|Boxing]]):** 
    *   Если упаковывается `Nullable<T>`, у которого `HasValue == false`, [[CLR]] возвращает физическую **пустую ссылку (`null`)**, а не объект-заглушку в [[Куча|куче]].
    *   Если `HasValue == true`, упаковывается чистое значение типа `T` (в куче выделяется память под тип `T`, а не под `Nullable<T>`).
*   **Лифтинг операторов (Operator Lifting):** Компилятор автоматически "расширяет" стандартные операторы сравнения и математики. Например, `bool?` переходит на **трехзначную логику** (true, false, null), где `true & null` дает `null`.

#### 2. Nullable ссылочные типы (NRT) — C# 8.0+
В отличие от значимых типов, NRT — это **исключительно компиляторная фича**.
*   **CLR не знает о разнице:** На уровне IL-кода и рантайма типы `string` и `string?` компилируются в один и тот же системный тип `System.String`. Никаких накладных расходов в памяти и рантайме нет.
*   **[[Метаданные]]:** Компилятор расставляет в коде атрибуты `[NullableAttribute]` и `[NullableContextAttribute]`. Среда разработки анализирует эти атрибуты и выдает предупреждения (warnings) при попытке разыменовать потенциально пустую ссылку.

> [!INFO] Нюанс: Оператор ! (Null-forgiving)
> Символ `!` в конце выражения (например, `user!.Name`) сообщает компилятору: «Я гарантирую, что это значение здесь не равно null, заглуши предупреждение». На рантайм это никак не влияет — если там окажется `null`, вы все равно получите `NullReferenceException`.

### Пример кода (C# 12)

```csharp
using System;

// Использование первичного конструктора (C# 12)
// Свойство Email помечено как nullable (string?)
public class User(string name, string? email)
{
    public string Name { get; } = name;
    public string? Email { get; } = email;

    public void PrintInfo()
    {
        // 1. Использование Null-conditional оператора (?.) и Null-coalescing (??)
        string emailDisplay = Email?.ToLower() ?? "no email provided";
        Console.WriteLine($"User: {Name}, Email: {emailDisplay}");
    }
}

public class Program
{
    public static void Main()
    {
        // --- 1. Работа с Nullable значимым типом ---
        int? age = null;
        
        // Безопасное извлечение значения
        if (age.HasValue)
        {
            int realAge = age.Value; // Или просто (int)age
            Console.WriteLine($"Age: {realAge}");
        }
        
        // Паттерн-матчинг (современный подход)
        if (age is int actualAge)
        {
            Console.WriteLine($"Age via pattern: {actualAge}");
        }

        // --- 2. Работа с Nullable ссылочным типом ---
        User user1 = new("Alice", null);
        User user2 = new("Bob", "bob@dev.local");

        user1.PrintInfo();
        user2.PrintInfo();
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Обращение к .Value без проверки
> Попытка получить значение значимого типа через свойство `age.Value`, когда оно равно `null`, приведет к генерации `InvalidOperationException`. Всегда делайте предварительную проверку через `age.HasValue`, паттерн-матчинг `age is int` или используйте безопасный метод `age.GetValueOrDefault()`.

> [!WARNING] Ошибка: Игнорирование предупреждений NRT
> Включение Nullable Reference Types в проекте (`<Nullable>enable</Nullable>`) без реальной борьбы с предупреждениями компилятора бессмысленно. Относитесь к предупреждениям о nullability как к ошибкам сборки, настроив `<TreatWarningsAsErrors>true</TreatWarningsAsErrors>` для критических предупреждений.

**Best Practices:**
1.  **GetValueOrDefault:** Вместо тернарных операторов для значимых типов используйте метод `age.GetValueOrDefault(18)`, который вернет значение по умолчанию, если переменная равна `null`.
2.  **Null-forgiveness (!):** Используйте оператор `!` только в крайних случаях, когда статический анализ компилятора бессилен (например, в Unit-тестах или после ручных сложных проверок на `null`), но никогда не используйте его просто для "подавления варнингов".
3.  **[[Конструкторы]] и NRT:** Проектируйте классы так, чтобы все не-nullable ссылочные свойства инициализировались в конструкторе. Если свойство может быть инициализировано позже (например, в [[DI]]), используйте модификатор `required` или помечайте его как nullable.
4.  **Арифметика с null:** Помните, что любая математическая операция, где один из операндов равен `null` (например, `int? x = null; int? y = x + 5;`), всегда возвращает `null`.

> [!TIP] Нюанс: [[Generics|Дженерики]] и nullability (C# 9+)
> Вы можете использовать ограничение `where T : notnull` для обобщенных классов, чтобы гарантировать, что переданный тип `T` (будь то структура или [[Классы и объекты|класс]]) гарантированно не поддерживает значение `null`.
