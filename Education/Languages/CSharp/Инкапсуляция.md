### Суть (The "What")
**Инкапсуляция** — это принцип [[ООП]], заключающийся в объединении данных и [[Методы|методов]], работающих с этими данными, в единый компонент ([[Классы и объекты|класс]]) и ограничении прямого доступа к внутреннему состоянию объекта извне.

Она решает задачу **управления сложностью** и **поддержания целостности (invariants)**: объект сам контролирует правильность своих данных, предоставляя внешнему миру только безопасный публичный интерфейс. Это позволяет изменять внутреннюю реализацию класса, не ломая код, который его использует.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Уровни доступа и границы видимости
[[CLR]] (Common Language Runtime) обеспечивает жесткое соблюдение правил доступа на уровне инструкций метаданных. 
- **Сокрытие данных:** Поля почти всегда должны быть `private`.
- **Публичный контракт:** Методы и свойства, помеченные как `public`, формируют "фасад" объекта.
- **Внутренняя кухня:** Модификатор `internal` ограничивает область видимости текущей сборкой (assembly), что позволяет скрывать детали реализации библиотек от конечного пользователя.

#### 2. Свойства как механизм контроля
В C# инкапсуляция чаще всего реализуется через свойства. На низком уровне свойство — это пара методов (аксессоров) `get_PropertyName` и `set_PropertyName`. Это дает возможность:
- Валидировать данные перед записью.
- Выполнять вычисления при чтении.
- Реализовать доступ "только для чтения".

#### 3. Первичные конструкторы ([[MOC#/MOC|C#]] 12) и захват состояния
В C# 12 параметры первичного конструктора могут использоваться непосредственно в методах класса. Компилятор автоматически создает скрытые поля для хранения этих значений, обеспечивая инкапсуляцию параметров без написания шаблонного кода.

> [!INFO] Нюанс: Инкапсуляция vs Безопасность
> Важно понимать, что инкапсуляция — это архитектурный инструмент для защиты от ошибок программиста, а не механизм безопасности. С помощью **Reflection** (рефлексии) можно обойти ограничения доступа и изменить даже `private readonly` поля в рантайме.

### Пример кода (C# 12)

```csharp
using System;
using System.Collections.Generic;

// Использование первичного конструктора (C# 12)
public class BankAccount(string owner, decimal initialBalance)
{
    // Поле полностью скрыто от внешнего мира
    private decimal _balance = initialBalance > 0 
        ? initialBalance 
        : throw new ArgumentException("Начальный баланс должен быть положителен");

    // Инкапсуляция через свойство с логикой
    public decimal Balance
    {
        get => _balance;
        // Сеттер отсутствует — баланс нельзя изменить напрямую
    }

    public string Owner { get; init; } = owner;

    // Публичный метод — единственный способ изменить состояние
    public void Deposit(decimal amount)
    {
        if (amount <= 0) return;
        
        _balance += amount;
        LogTransaction($"Зачислено {amount}");
    }

    // Внутренняя логика скрыта
    private void LogTransaction(string message) 
        => Console.WriteLine($"[{DateTime.Now:T}] {Owner}: {message}");
}

public class Program
{
    public static void Main()
    {
        var account = new BankAccount("Ivanov", 1000m);
        
        account.Deposit(500m);
        
        // account._balance = 0; // Ошибка компиляции: поле private
        // account.Balance = 1000000; // Ошибка компиляции: нет сеттера
        
        Console.WriteLine($"Текущий баланс: {account.Balance}");
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Анемичная модель (Anemic Domain Model)
> Классы, состоящие только из публичных свойств `{ get; set; }` без логики, нарушают принцип инкапсуляции. Состояние такого объекта может быть бесконтрольно изменено из любой точки программы, что ведет к трудноуловимым багам.

> [!WARNING] Ошибка: Возврат ссылок на изменяемые коллекции
> Если поле класса — это `List<T>`, и вы возвращаете его через публичное свойство, внешний код сможет вызвать `.Clear()` или `.Add()`, обойдя вашу логику. Всегда возвращайте `ReadOnlyCollection<T>` или `IEnumerable<T>`.

**Best Practices:**
1. **Tell, Don't Ask:** Вместо того чтобы запрашивать у объекта данные для принятия решения, просите сам объект выполнить действие.
2. **Модификатор init:** Используйте `init` вместо `set` для свойств, которые не должны меняться после создания объекта. Это форма инкапсуляции во времени.
3. **Private по умолчанию:** Начинайте проектирование с самого строгого уровня доступа, расширяя его до `internal` или `public` только при явной необходимости.
4. **Защитные копии:** Если объект хранит массив или другую изменяемую структуру, при передаче её вовне возвращайте копию, чтобы сохранить контроль над оригиналом.

> [!TIP] Нюанс: Ключевое слово required (C# 11)
> Используйте `required properties`, чтобы гарантировать, что объект будет корректно проинициализирован при использовании объектных инициализаторов, не создавая при этом огромных конструкторов. Это помогает соблюдать контракт инкапсуляции на этапе сборки объекта.
