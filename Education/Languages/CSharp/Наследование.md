### Суть (The "What")
**Наследование** — это механизм [[ООП]], позволяющий создать новый [[Классы и объекты|класс]] (производный) на основе уже существующего (базового). Производный класс перенимает функциональность базового и может расширять её или изменять.

Наследование решает задачу **повторного использования кода** и установления иерархических отношений типа **"is-a" (является)**. Оно служит фундаментом для полиморфизма, позволяя работать с группой разных объектов через их общий базовый тип.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Устройство в памяти (Memory Layout)
Когда создается экземпляр производного класса, [[CLR]] выделяет в куче единый блок памяти, который включает:
*   Поля базового класса.
*   Поля производного класса.
*   Заголовок объекта (Sync Block Index + Type Object Pointer).
**Важно:** Поля базового класса всегда располагаются в начале блока памяти, перед полями производного класса. Это позволяет безопасно приводить ссылку на производный класс к базовому (upcasting) — адрес начала объекта остается тем же.

#### 2. Одиночное наследование
В .NET поддерживается только **одиночное наследование классов**. Класс может иметь только одного непосредственного родителя. Если родитель не указан явно, класс наследуется от `System.Object`. Множественное наследование реализовано только для интерфейсов.

#### 3. Цепочка вызова конструкторов
Конструкторы не наследуются. При создании объекта производного класса выполнение всегда идет "снизу вверх":
1. Сначала вызывается конструктор базового класса (явно через `base()` или неявно).
2. После успешного завершения базового конструктора выполняется код конструктора производного класса.

#### 4. Виртуальные методы и [[VMT]]
*   **[[virtual]]:** Помечает метод как доступный для переопределения.
*   **[[override]]:** Переопределяет реализацию в потомке.
*   **new (Shadowing):** Скрывает метод базового класса. В этом случае метод выбирается по типу ссылки, а не по типу объекта (нарушение [[Полиморфизм|полиморфизма]]).

> [!INFO] Нюанс: Запечатывание (Sealing)
> Использование модификатора `sealed` на классе запрещает наследование от него. Использование `sealed` на методе (в сочетании с `override`) запрещает дальнейшее переопределение этого метода в цепочке наследования. Это дает небольшое преимущество в производительности (devirtualization).

### Пример кода ([[MOC#/MOC|C#]] 12)

```csharp
using System;

// Базовый класс 
public class Machine(string model)
{
    public string Model { get; } = model;

    // Виртуальный метод для полиморфизма
    public virtual void Start() => Console.WriteLine($"{Model} is starting...");
}

// Производный класс
public class Car(string model, int speed) : Machine(model)
{
    public int Speed { get; } = speed;

    // Переопределение базового метода
    public override void Start()
    {
        base.Start(); // Вызов базовой логики
        Console.WriteLine($"Car is moving at {Speed} km/h.");
    }
}

public class Program
{
    public static void Main()
    {
        // Upcasting: ссылка базового типа, объект производного
        Machine myCar = new Car("Tesla Model 3", 200);
        
        // Полиморфный вызов: сработает override версия из Car
        myCar.Start();

        // Downcasting: приведение обратно к Car
        if (myCar is Car specificCar)
        {
            Console.WriteLine($"Top speed: {specificCar.Speed}");
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Хрупкий базовый класс (Fragile Base Class)
> Изменение реализации в базовом классе может непреднамеренно сломать логику во всех его многочисленных потомках. Если вы планируете наследование, делайте базовый класс максимально стабильным и минималистичным.

> [!WARNING] Ошибка: Наследование вместо композиции
> Не используйте наследование только ради доступа к методам другого класса. Используйте наследование только если объект действительно **является** частным случаем другого. В остальных ситуациях используйте **композицию** (включение объекта другого класса как поля).

**Best Practices:**
1.  **Liskov Substitution Principle ([[Liskov Substitution Principle|LSP]]):** Программа должна продолжать работать корректно, если вы замените объект базового класса объектом любого из его наследников.
2.  **[[Абстрактные классы и члены классов|Abstract Classes]]:** Если базовый класс не предназначен для создания экземпляров и лишь описывает общую идею, помечайте его как `abstract`.
3.  **[[Конструкторы]]:** Всегда проверяйте параметры в базовом конструкторе. Это гарантирует, что производные классы не смогут создать объект в невалидном состоянии.
4.  **Скрытие членов:** Избегайте использования ключевого слова `new` для скрытия членов базового класса. Это вносит путаницу в поведение объекта при приведении типов.

> [!TIP] Нюанс: Доступ protected
> Члены с модификатором `protected` видны в производных классах, но остаются закрытыми для внешнего мира. Это основной инструмент для предоставления наследникам доступа к внутренней "кухне" базового класса без нарушения инкапсуляции.
