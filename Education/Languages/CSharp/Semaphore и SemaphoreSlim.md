### Суть (The "What")
**Semaphore** — это примитив синхронизации, который ограничивает количество потоков, имеющих одновременный доступ к ресурсу или пулу ресурсов. В отличие от [[lock]] или [[Mutex]], которые являются взаимоисключающими (пропускают только один поток), семафор имеет **емкость (capacity)**.

Семафор решает задачу **дросселирования (throttling)**. Он используется для ограничения нагрузки на дорогостоящие ресурсы, такие как база данных (не более 10 соединений одновременно) или внешний API, предотвращая перегрузку системы при резких всплесках активности.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Механика счетчика
В основе семафора лежит целочисленный счетчик:
*   **Вход (`Wait` / `WaitOne`):** Поток запрашивает доступ. Если счетчик больше нуля, он уменьшается на 1, и поток проходит. Если счетчик равен 0, поток блокируется.
*   **Выход (`Release`):** Поток освобождает слот. Счетчик увеличивается на 1. Если есть ждущие потоки, один из них разблокируется.

#### 2. Отсутствие привязки к потоку (No Thread Affinity)
Это фундаментальное отличие от `Monitor` (lock) и `Mutex`:
*   Семафор не "знает", какой поток его захватил.
*   Один поток может вызвать `Wait`, а совершенно другой — `Release`. Это позволяет использовать семафоры в асинхронных сценариях, где продолжение задачи может выполняться в другом потоке пула.

#### 3. Виды семафоров
*   **Semaphore (Kernel Object):** Обертка над объектом ядра Windows. Позволяет синхронизировать потоки в **разных процессах**. Медленный из-за переходов в режим ядра.
*   **SemaphoreSlim (Managed):** Легковесная версия, оптимизированная для работы внутри одного процесса. 
    *   Использует "вращение" (spinning) перед блокировкой.
    *   Поддерживает асинхронное ожидание через `WaitAsync()`.
    *   Не использует объекты ядра, если нет большой конкуренции (contention).

> [!INFO] Нюанс: Инициализация
> При создании семафора указываются два параметра: `initialCount` (сколько слотов доступно сразу) и `maxCount` (максимальная емкость). Если `initialCount` равен 0, все первые потоки будут заблокированы, пока кто-то не вызовет `Release`.

### Пример кода (C# 12)

Демонстрация ограничения параллельных HTTP-запросов с использованием `SemaphoreSlim`.

```csharp
using System;
using System.Collections.Generic;
using System.Threading;
using System.Threading.Tasks;

public class ThrottlingService
{
    // Ограничиваем доступ 3 потоками одновременно
    private static readonly SemaphoreSlim _semaphore = new(3, 3);

    public async Task AccessResourceAsync(int id)
    {
        Console.WriteLine($"Запрос {id} ожидает доступа...");
        
        // Асинхронное ожидание свободного слота
        await _semaphore.WaitAsync();

        try
        {
            Console.WriteLine($"Запрос {id} ПОЛУЧИЛ доступ. Работаю...");
            await Task.Delay(TimeSpan.FromSeconds(2)); // Имитация нагрузки
        }
        finally
        {
            Console.WriteLine($"Запрос {id} освобождает слот.");
            _semaphore.Release(); // Увеличение счетчика
        }
    }
}

public class Program
{
    public static async Task Main()
    {
        var service = new ThrottlingService();
        var tasks = new List<Task>();

        for (int i = 1; i <= 10; i++)
        {
            tasks.Add(service.AccessResourceAsync(i));
        }

        await Task.WhenAll(tasks);
        Console.WriteLine("Все запросы обработаны.");
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Release вне finally
> Если внутри критической секции произойдет исключение, и `Release()` не будет вызван, семафор навсегда потеряет один слот ("утечка слотов"). При достижении лимита все последующие потоки зависнут в бесконечном ожидании. Всегда используйте `try-finally`.

> [!WARNING] Ошибка: SemaphoreFullException
> Вызов `Release()` сверх установленного `maxCount` приведет к выбросу исключения. Это часто случается при нарушении логики, когда `Release` вызывается в цикле или ошибочно из нескольких мест для одного захвата.

**Best Practices:**
1.  **SemaphoreSlim по умолчанию:** Всегда используйте `SemaphoreSlim` для внутрипроцессных задач. Он быстрее и поддерживает `async/await`.
2.  **WaitAsync в асинхронном коде:** Никогда не используйте синхронный `Wait()` внутри асинхронных методов — это блокирует поток из пула и может привести к Thread Pool Starvation.
3.  **Использование IDisposable:** `SemaphoreSlim` владеет `WaitHandle` (в случае сильной конкуренции), поэтому его стоит удалять через `Dispose()`, когда он больше не нужен.
4.  **Двоичный семафор:** Семафор с емкостью 1 (`new SemaphoreSlim(1, 1)`) может использоваться как замена `lock`, если требуется поддержка асинхронности.

> [!TIP] Нюанс: Межпроцессный семафор
> Если вам нужно ограничить количество запущенных копий тяжелого процесса в рамках всей операционной системы, используйте именованный `System.Threading.Semaphore`. Его синтаксис имен аналогичен `Mutex`: `Global\MyUniqueName`.
