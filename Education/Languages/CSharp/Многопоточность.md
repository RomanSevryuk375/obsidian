### Суть (The "What")
**Многопоточность** — это форма параллелизма, при которой внутри одного процесса сосуществуют и выполняются несколько независимых потоков управления (threads). Все потоки одного процесса разделяют общую кучу (Heap), но имеют собственные стеки (Stack) для хранения локальных переменных.

Многопоточность решает две основные задачи:
1. **Отзывчивость (Responsiveness):** Выполнение тяжелых вычислений в фоновом потоке, чтобы не блокировать основной (UI) поток.
2. **Производительность (Throughput):** Эффективное использование многоядерных процессоров путем распределения нагрузки.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Потоки vs Процессы
*   **Процесс:** Изолированная единица выполнения с выделенной памятью.
*   **Поток (Thread):** "Легковесный процесс" внутри основного. Переключение между потоками одного процесса быстрее, чем между процессами, но несет накладные расходы на **контекстное переключение (Context Switch)**.

#### 2. Планировщик CLR и ОС
Потоки в .NET являются логическими обертками над потоками операционной системы (Windows Threads).
*   **Квантование времени:** ОС выделяет каждому потоку "квант" времени (обычно ~10-20 мс). Когда время истекает, ОС сохраняет состояние регистров текущего потока в его стек и переключает процессор на другой поток.
*   **Приоритеты:** Вы можете задавать `ThreadPriority`, но планировщик ОС имеет право игнорировать их для предотвращения "голодания" (starvation) других потоков.

#### 3. Память и общие данные
Главная сложность — разделяемая память (Shared State).
*   **Stack:** Каждый поток получает ~1 МБ памяти под стек. Это безопасно, данные здесь изолированы.
*   **Heap:** Все потоки имеют доступ к одним и тем же объектам в куче. Без синхронизации это приводит к **состоянию гонки (Race Condition)**.

#### 4. Пул потоков (Thread Pool)
Создание нового потока — дорогая операция (аллокация стека, системные вызовы). .NET использует `ThreadPool` для переиспользования уже созданных потоков.
*   Потоки пула всегда являются **фоновыми (background)**: они не удерживают процесс от завершения, если основной поток (foreground) закрылся.

> [!IMPORTANT] Нюанс: Порядок инструкций
> Согласно Албахари, компилятор и процессор могут переупорядочивать инструкции (Optimization) или кэшировать переменные в регистрах процессора. В многопоточной среде это может привести к тому, что один поток не увидит изменений, сделанных другим, даже если они обращаются к одной ячейке памяти. Для решения используются барьеры памяти (Memory Barriers) и ключевое слово `lock`.

### Пример кода ([[MOC|C#]] 12)

```csharp
using System;
using System.Threading;

public class ThreadWorker(string id) // Primary constructor
{
    private readonly string _id = id;
    private static readonly object _locker = new(); // Объект синхронизации
    private static int _sharedCounter = 0;

    public void DoWork()
    {
        for (int i = 0; i < 5; i++)
        {
            // Блокировка для обеспечения потокобезопасности
            lock (_locker)
            {
                _sharedCounter++;
                Console.WriteLine($"Thread {_id}: counter = {_sharedCounter}");
            }
            Thread.Sleep(100); // Имитация работы и передача кванта времени
        }
    }
}

public class Program
{
    public static void Main()
    {
        ThreadWorker worker1 = new("A");
        ThreadWorker worker2 = new("B");

        // Создание потоков напрямую (не через пул)
        Thread t1 = new(worker1.DoWork);
        Thread t2 = new(worker2.DoWork);

        // Сделаем потоки фоновыми
        t1.IsBackground = true;
        t2.IsBackground = true;

        t1.Start();
        t2.Start();

        // Ожидание завершения (Join)
        t1.Join();
        t2.Join();

        Console.WriteLine("All work finished.");
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Thread.Abort()
> Никогда не используйте `Thread.Abort()`. Это принудительно прерывает поток в произвольном месте, что может оставить объекты (например, `FileStream`) в поврежденном состоянии или не освободить статические блокировки. Используйте `CancellationToken`.

> [!WARNING] Ошибка: Deadlock (Взаимная блокировка)
> Ситуация, когда Поток 1 ждет освобождения Ресурса А (который у Потока 2), а Поток 2 ждет Ресурс Б (который у Потока 1). Чтобы избежать этого, всегда захватывайте объекты синхронизации в строго определенном порядке.

**Best Practices:**
1. **Используйте TPL (Tasks):** Для большинства современных задач используйте `Task` вместо прямого создания `Thread`. Задачи эффективнее используют пул потоков.
2. **Минимизируйте Lock:** Блокировка — дорогая операция. Если нужно просто инкрементировать число, используйте атомарные операции `Interlocked.Increment(ref counter)`.
3. **Thread Safety:** При использовании коллекций в нескольких потоках выбирайте классы из `System.Collections.Concurrent`.
4. **Immutable State:** Самый простой способ сделать код потокобезопасным — сделать данные неизменяемыми (`readonly`, `init`). Если данные нельзя изменить, их не нужно блокировать.

> [!TIP] Нюанс: Foreground vs Background
> Процесс .NET завершается только тогда, когда завершаются все **foreground** (передние) потоки. Если вы создали поток через `new Thread()`, по умолчанию он foreground. Если забыть его закрыть, приложение будет "висеть" в процессах даже после закрытия окна.
