### Суть (The "What")
**Коллекции** — это специализированные [[Классы и объекты|классы]], предназначенные для группировки объектов и управления ими. В отличие от [[Kлacc Array|массивов]], которые имеют фиксированную длину, большинство коллекций в .NET могут динамически изменять свой размер во время выполнения.

Коллекции решают задачу эффективного хранения, поиска и манипуляции данными. В зависимости от выбранного типа (список, словарь, очередь, множество) изменяется алгоритмическая сложность операций и потребление памяти.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Иерархия интерфейсов
Все коллекции строятся вокруг базового интерфейса [[IEnumerable|IEnumerable]]`<T>`, который позволяет перебирать элементы.
*   **[[Интерфейс ICollection|ICollection]]`<T>`**: Добавляет методы для изменения (Add, Remove), подсчета (Count) и копирования.
*   **[[Интерфейс IList|IList]]`<T>`**: Позволяет доступ по индексу.
*   **[[Dictionary|IDictionary]]`<TKey, TValue>`**: Позволяет доступ по ключу (пары ключ-значение).

#### 2. Основные типы и их устройство
*   **`List<T>`**: Динамический массив([[List]]). При заполнении внутреннего массива создается новый (обычно в 2 раза больше), и элементы копируются. Операция добавления в конец — амортизированная $O(1)$.
*   **`Dictionary<TKey, TValue>`**: Хеш-таблица. Использует хеш-код ключа для определения индекса в массиве (бакете). Обеспечивает поиск за $O(1)$.
*   **[[HashSet]]`<T>`**: Множество уникальных элементов. По сути, это словарь, где есть только ключи без значений.
*   **[[Queue]]`<T>` / [[Stack]]`<T>`**: Реализуют логику FIFO (первым пришел — первым ушел) и LIFO (последним пришел — первым ушел).

#### 3. Управление памятью и Capacity
Коллекции имеют свойства `Count` (сколько элементов сейчас) и `Capacity` (сколько места зарезервировано в памяти). 

> [!INFO] Нюанс: Memory Fragmentation
> Частые расширения `List<T>` (когда Capacity заканчивается) приводят к лишним аллокациям и фрагментации кучи. Если итоговое количество элементов примерно известно, всегда указывайте начальную емкость в конструкторе.

#### 4. [[Generics]] vs Non-generic
Современный .NET использует пространство имен `System.Collections.Generic`. Старые коллекции из `System.Collections` (например, `ArrayList`) работают с типом `object`.

> [!DANGER] Boxing в коллекциях
> Использование `ArrayList` для хранения значимых типов (int, struct) приводит к упаковке (boxing) каждого элемента в кучу. Это катастрофически снижает производительность и увеличивает нагрузку на [[GC]]. Используйте только `List<T>`.

### Пример кода ([[MOC|C#]] 12)

```csharp
using System;
using System.Collections.Generic;

// Использование Collection Expressions (C# 12)
List<int> numbers = [1, 2, 3, 4, 5];

// Словарь с начальной емкостью для оптимизации
var userScores = new Dictionary<string, int>(capacity: 100)
{
    ["Alice"] = 50,
    ["Bob"] = 75
};

// HashSet для мгновенной проверки уникальности
HashSet<string> tags = ["dotnet", "csharp", "coding"];

// Поиск в словаре (Safe Access)
if (userScores.TryGetValue("Alice", out int score))
{
    Console.WriteLine($"Alice's score: {score}");
}

// Работа с Queue (Очередь)
Queue<string> messages = new();
messages.Enqueue("Message 1");
string msg = messages.Dequeue();
```

### Ошибки и Best Practices

> [!WARNING] Ошибка: Модификация во время итерации
> Нельзя изменять состав коллекции (Add/Remove) внутри цикла `foreach`. Это приведет к `InvalidOperationException`, так как итератор проверяет "версию" коллекции при каждом переходе к следующему элементу.

> [!TIP] ReadOnly коллекции
> Если вы передаете коллекцию наружу из класса, не возвращайте `List<T>`. Используйте `IReadOnlyList<T>` или метод `.AsReadOnly()`. Это защитит ваши данные от несанкционированного изменения извне.

**Best Practices:**
1.  **Выбор типа:** Нужен быстрый доступ по индексу? — `List<T>`. Нужен быстрый поиск по ключу? — `Dictionary<K,V>`. Нужна только уникальность? — `HashSet<T>`.
2.  **Initial Capacity:** Всегда задавайте `capacity`, если заранее знаете примерный объем данных. Это предотвращает лишние циклы копирования массива.
3.  **[[LINQ]] vs Cycles:** LINQ удобен для чтения и фильтрации, но в "горячих" участках кода (High Performance) обычный цикл `for` по массиву или `List<T>` всегда будет быстрее и создаст меньше мусора в куче.
4.  **Enumerables:** Если методу нужно только перебрать элементы, принимайте `IEnumerable<T>` вместо конкретного `List<T>`. Это делает код гибче.

> [!IMPORTANT] Нюанс: EqualityComparer
> Для корректной работы `Dictionary` и `HashSet` с пользовательскими типами (class/struct) необходимо правильно переопределить `GetHashCode` и `Equals` или передать `IEqualityComparer<T>` в конструктор коллекции. Без этого поиск будет работать некорректно.
