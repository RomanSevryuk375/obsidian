### Суть (The "What")
**Абстракция** — это процесс выделения значимых характеристик объекта и игнорирования несущественных деталей. В программировании это означает создание упрощенного интерфейса или модели сложной системы, где пользователю доступны только необходимые действия.

Абстракция решает задачу **управления сложностью**: она позволяет программисту работать с высокоуровневыми концепциями (например, "Платеж"), не задумываясь о конкретной реализации (через карту, криптовалюту или наличные) на этапе проектирования основной логики.

---

### Как это работает (The "How" / Nutshell style)

В [[MOC#/MOC|C#]] абстракция реализуется через два основных механизма: **[[Интерфейсы|интерфейсы]]** и **[[Абстрактные классы и члены классов|абстрактные классы]]**.

#### 1. Уровни абстрагирования
*   **Слабая абстракция (Абстрактные классы):** Описывают идентичность ("is-a"). Позволяют частично реализовать логику и хранить состояние (поля). 
*   **Чистая абстракция (Интерфейсы):** Описывают способность ("can-do"). Определяют только контракт. До появления C# 8.0 не могли содержать никакой реализации.

#### 2. Низкоуровневая перспектива ([[CLR]])
На уровне исполняемой среды (CLR) абстракция обеспечивает **полиморфную заменяемость**:
*   **Метаданные:** Типы помечаются как `abstract` в таблице типов. CLR запрещает инстанцирование таких типов.
*   **Слоты в [[VMT]]:** Для абстрактных методов в таблице виртуальных методов (VMT) резервируются слоты, но они не содержат реальных адресов кода до тех пор, пока не будут переопределены в конкретном (concrete) наследнике.
*   **Индирекция:** Программа работает с адресом в VMT, что позволяет подставлять любую реализацию, соответствующую контракту, без перекомпиляции вызывающего кода.

#### 3. Связь с инкапсуляцией
Абстракция и [[Инкапсуляция|инкапсуляция]] взаимосвязаны: абстракция говорит, **что** объект делает (внешний вид), а инкапсуляция скрывает то, **как** он это делает (внутреннее устройство).

> [!INFO] Нюанс: Статическая абстракция (C# 11+)
> Современный C# позволяет использовать `static abstract` члены в интерфейсах. Это позволяет абстрагировать даже статические методы (например, операторы или фабрики), что активно используется в Generic Math (.NET 7/8+).

### Пример кода (C# 12)

```csharp
using System;

// Высший уровень абстракции: контракт "Способность сохранять данные"
public interface IDataStore
{
    void Save(string data);
}

// Уровень частичной абстракции: базовый класс для облачных хранилищ
public abstract class CloudStorage(string endpoint) : IDataStore
{
    protected string Endpoint { get; } = endpoint;

    // Абстрактный метод: конкретные облака реализуют его по-разному
    public abstract void Save(string data);

    // Общая логика для всех облачных хранилищ
    protected void LogConnection() => Console.WriteLine($"Connecting to {Endpoint}...");
}

// Конкретная реализация
public class AzureBlobStorage(string endpoint) : CloudStorage(endpoint)
{
    public override void Save(string data)
    {
        LogConnection();
        Console.WriteLine($"Data '{data}' uploaded to Azure Blob.");
    }
}

public class Program
{
    public static void Main()
    {
        // Работаем с абстракцией IDataStore, не зная, что это Azure
        IDataStore store = new AzureBlobStorage("https://azure.com/my-blob");
        store.Save("Sensitive user data");
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Преждевременная абстракция (Over-engineering)
> Создание интерфейсов для каждого класса в приложении "на всякий случай". Это усложняет навигацию по коду и поддержку. Применяйте абстракцию только там, где реально ожидается более одной реализации или требуется модульное тестирование.

> [!WARNING] Ошибка: Протекающая абстракция (Leaky Abstraction)
> Ситуация, когда детали реализации "просачиваются" в абстрактный интерфейс. Например, если метод интерфейса `Save()` требует специфический для SQL параметр, этот интерфейс перестает быть абстрактным, так как привязан к конкретной технологии БД.

**Best Practices:**
1.  **Принцип инверсии зависимостей ([[DIP]]):** Зависимости должны строиться на абстракциях, а не на конкретных типах. Это делает код модульным.
2.  **Интерфейс vs Абстрактный класс:** Если нужно множественное наследование или описание поведения — используйте интерфейс. Если нужна общая база кода и состояние для иерархии — используйте абстрактный класс.
3.  **Именование:** Интерфейсы всегда должны начинаться с префикса `I` (например, `IRepository`). Абстрактные классы обычно имеют существительное в названии, описывающее общую сущность (например, `BaseService`).
4.  **Минимализм:** Абстрактный контракт должен содержать минимально необходимое количество членов. Чем меньше контракт, тем проще его реализовывать и поддерживать.

> [!TIP] Нюанс: Record в абстракции
> Начиная с C# 9, можно создавать `abstract record`. Это полезно, когда вам нужно абстрагировать модели данных с поддержкой сравнения по значениям (Value Equality).
