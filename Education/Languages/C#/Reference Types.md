### Суть (The "What")
**Ссылочные типы** — это типы данных, переменные которых хранят не само значение, а адрес (ссылку) на область памяти, где расположен объект. Сами данные объекта всегда размещаются в **управляемой куче ([[Managed Heap]])**.

Ссылочные типы решают задачу эффективного управления сложными структурами данных и реализации объектно-ориентированных парадигм ([[инкапсуляция]], [[наследование]], [[полиморфизм]]). Они позволяют нескольким переменным обращаться к одному и тому же экземпляру объекта, обеспечивая совместное использование состояния.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Анатомия объекта в куче
Когда вы создаете экземпляр ссылочного типа (например, `new MyClass()`), в куче выделяется память, которая включает:
*   **Поля данных:** Собственно значения, определенные в классе.
*   **Sync Block Index:** 4-8 байт (в зависимости от разрядности), используемые [[CLR]] для синхронизации потоков (lock) и хранения некоторых хеш-кодов.
*   **Type Object Pointer:** Указатель на объект-тип (Method Table), который описывает метаданные класса (методы, интерфейсы).

#### 2. Управление памятью и [[GC]]
В отличие от значимых типов, которые уничтожаются при выходе из области видимости (в стеке), ссылочные типы зависят от **Garbage Collector (GC)**.
*   Переменная-ссылка в стеке исчезает сразу, но объект в куче остается там до тех пор, пока на него есть хотя бы одна достижимая ссылка.
*   **[[LOH]] (Large Object Heap):** Объекты размером более 85 000 байт (например, большие массивы) выделяются в специальной области кучи, которая оптимизирована для редкого перемещения, так как дефрагментация больших блоков памяти обходится дорого.

#### 3. Копирование и сравнение
*   При присваивании `ref1 = ref2` копируется только 4 или 8 байт адреса. Оба указателя начинают смотреть на один и тот же объект.
*   По умолчанию оператор `==` для ссылочных типов проверяет **ссылочную идентичность** (указывают ли переменные на один и тот же адрес), а не равенство данных. Исключение — тип `string` и типы `record`, где `==` переопределен для сравнения значений.

> [!INFO] Нюанс: Nullability
> Ссылочный тип может иметь значение `null`, что означает "ссылка никуда не указывает". Попытка обращения к члену такой переменной вызывает `NullReferenceException`. В современном C# (8.0+) введены **Nullable Reference Types**, позволяющие на этапе компиляции отслеживать потенциальные обращения к null.

### Пример кода (C# 12)

```csharp
using System;

// Использование record (C# 9+) для ссылочного типа с семантикой значений
public record Person(string Name, int Age);

public class Client(string name) // Primary constructor (C# 12)
{
    public string Name { get; set; } = name;
}

public class Program
{
    public static void Main()
    {
        // 1. Совместное использование состояния
        Client client1 = new("Alice");
        Client client2 = client1; // Копируется ссылка
        client2.Name = "Bob";
        
        Console.WriteLine(client1.Name); // Выведет "Bob", так как объект один
        
        // 2. Сравнение ссылочных типов
        Client c1 = new("Charlie");
        Client c2 = new("Charlie");
        Console.WriteLine(c1 == c2); // False (разные адреса в куче)

        // 3. Records: ссылочный тип, сравнивающий значения
        Person p1 = new("Alice", 30);
        Person p2 = new("Alice", 30);
        Console.WriteLine(p1 == p2); // True (компилятор переопределил == для record)

        // 4. Nullable Reference Types
        string? optionalName = null;
        // Console.WriteLine(optionalName.Length); // Ошибка компиляции (предупреждение)
        Console.WriteLine(optionalName?.Length ?? 0); // Безопасный доступ
    }
}
```

### Ошибки и Best Practices

> [!WARNING] Ошибка: Изменение общего состояния (Side Effects)
> Будьте осторожны при передаче ссылочных типов в методы. Метод может изменить внутреннее состояние объекта, что станет неожиданностью для вызывающего кода. Если это нежелательно, передавайте глубокую копию (Deep Copy) или используйте неизменяемые (immutable) типы.

> [!DANGER] Anti-pattern: String Concatenation в циклах
> Хотя `string` — ссылочный тип, он неизменяем. Каждая конкатенация `s += "next"` создает новый объект в куче. Для сборки строк в циклах всегда используйте `StringBuilder`.

**Best Practices:**
1.  **Класс или Структура?** Используйте `class` для объектов с идентичностью (Identity) и сложным поведением. Используйте `struct` для мелких сущностей, представляющих одно значение.
2.  **Использование [[Record]]s:** Для [[DTO]] (Data Transfer Objects) и простых моделей данных используйте `record`. Это дает встроенную неизменяемость и корректное сравнение "из коробки".
3.  **Null-Safety:** Всегда включайте `<Nullable>enable</Nullable>` в настройках проекта. Проектируйте API так, чтобы минимизировать использование `null`.
4.  **WeakReference:** Для кеширования тяжелых объектов рассмотрите `WeakReference<T>`. Это позволяет GC удалить объект, если в системе не осталось других (сильных) ссылок на него, предотвращая утечки памяти.```yaml
