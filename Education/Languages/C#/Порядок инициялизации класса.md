### Суть (The "What")
Порядок инициализации — это строго детерминированная последовательность действий, которую выполняет [[CLR]] (Common Language Runtime) при создании экземпляра класса или первом обращении к его статическим членам. 

Знание этого порядка решает проблему **целостности состояния объекта**: оно гарантирует, что к моменту выполнения кода в теле конструктора все поля уже имеют базовые значения, а зависимости базовых классов разрешены.

---

### Как это работает (The "How" / Nutshell style)

Процесс разделен на два этапа: **Статическая инициализация** (один раз на тип) и **Инициализация экземпляра** (при каждом `new`).

#### 1. Статическая инициализация (Type Init)
Выполняется при первом обращении к [[Классы и объекты|классу]] (создание объекта, вызов статического метода или обращение к полю).
1.  **Инициализаторы статических полей производного класса.**
2.  **Инициализаторы статических полей базового класса.**
3.  **Статический конструктор базового класса.**
4.  **Статический конструктор производного класса.**

> [!INFO] Нюанс: beforefieldinit
> Если в классе нет явного статического конструктора, компилятор помечает тип флагом `beforefieldinit`. Это дает CLR право инициализировать статические поля в любой момент до обращения к ним, что может ускорить работу за счет оптимизаций [[JIT]]-компилятора.

#### 2. Инициализация экземпляра (Instance Init)
Когда вызывается `new Derived()`:
1.  **Выделение памяти в куче:** Память под весь объект (включая поля всех базовых классов) зануляется (все поля получают `default`).
2.  **Инициализаторы полей производного класса:** Выполняются в порядке их объявления.
3.  **Инициализаторы полей базового класса:** Выполняются аналогично.
4.  **Конструктор базового класса:** Выполняется тело конструктора (начиная с `System.Object`).
5.  **Конструктор производного класса:** Выполняется тело конструктора производного класса.

> [!IMPORTANT] Механика компилятора
> На уровне IL-кода инициализаторы полей не существуют отдельно. Компилятор "вклеивает" их код в начало каждого конструктора перед вызовом `base()`.

### Пример кода ([[MOC#/MOC|C#]] 12)

```csharp
using System;

public class Base
{
    public static int BaseStaticField = InitField("1. Статическое поле Base");
    public int BaseInstanceField = InitField("5. Поле экземпляра Base");

    static Base() => Console.WriteLine("2. Статический конструктор Base");
    public Base() => Console.WriteLine("6. Конструктор экземпляра Base");

    protected static int InitField(string message)
    {
        Console.WriteLine(message);
        return 0;
    }
}

public class Derived : Base
{
    public static int DerivedStaticField = InitField("0. Статическое поле Derived");
    public int DerivedInstanceField = InitField("4. Поле экземпляра Derived");

    static Derived() => Console.WriteLine("3. Статический конструктор Derived");
    
    // Использование первичного конструктора C# 12
    public Derived() : base() 
    {
        Console.WriteLine("7. Конструктор экземпляра Derived");
    }
}

// Вызов: new Derived();
// Вывод подтвердит иерархию: Поля -> Base ctor -> Derived ctor.
```

- **Статика Производного** (Поля)
- **Статика Базового** (Поля)
- **Статика Базового** (Конструктор)
- **Статика Производного** (Конструктор)
- **Поля Экземпляра Производного**
- **Поля Экземпляра Базового**
- **Тело Конструктора Базового**
- **Тело Конструктора Производного**

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Вызов виртуальных методов в конструкторе
> Это критическая ошибка. Если конструктор `Base` вызывает виртуальный метод, переопределенный в `Derived`, то этот метод выполнится **до** того, как отработает тело конструктора `Derived`. Это приведет к работе с неинициализированным состоянием объекта.

> [!WARNING] Ошибка: Зависимость между инициализаторами полей
> Инициализаторы полей выполняются в порядке их текстового объявления в коде. Попытка использовать в инициализаторе поле, объявленное ниже, приведет к использованию значения по умолчанию (например, 0 для int), а не того значения, которое вы ожидаете прописать ниже.

**Best Practices:**
1.  **Простота конструкторов:** Старайтесь делать конструкторы максимально легкими. Сложную логику инициализации (например, асинхронные вызовы или тяжелые расчеты) лучше выносить в фабричные методы.
2.  **Readonly поля:** Всегда инициализируйте `readonly` поля либо сразу в строке объявления, либо в конструкторе. Это гарантирует неизменяемость состояния после завершения процесса инициализации.
3.  **Primary Constructors:** В C# 12 используйте первичные конструкторы для лаконичной инициализации полей данными из аргументов. Параметры первичного конструктора доступны во всех инициализаторах полей класса.
4.  **Static Constructors:** Используйте статические конструкторы только для инициализации сложных статических данных, которые нельзя инициализировать в одной строке. Помните, что исключение в статическом конструкторе делает тип недоступным для использования на все время жизни приложения.

> [!TIP] Нюанс: Структуры (struct)
> Для структур порядок инициализации отличается, так как они не поддерживают наследование и всегда имеют неявный конструктор без параметров (до C# 10), который просто зануляет память. С C# 11 поля структур должны быть полностью инициализированы перед выходом из конструктора.
