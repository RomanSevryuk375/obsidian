### Суть (The "What")
**Свойства** — это члены класса, которые предоставляют гибкий механизм для чтения, записи или вычисления значений закрытых полей. С точки зрения пользователя класса свойство выглядит как публичное поле, но фактически оно является набором специальных методов, называемых **аксессорами (accessors)**.

Свойства решают задачу **инкапсуляции**: они позволяют контролировать доступ к данным, выполнять валидацию, логирование или ленивую инициализацию, не раскрывая внутреннюю структуру объекта.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Механика компиляции
На уровне [[IL]]-кода свойств не существует. Когда вы объявляете свойство `Name`, компилятор [[MOC#/MOC|C#]] генерирует два метода:
*   `get_Name()` — для чтения значения.
*   `set_Name(value)` — для записи значения.

Если используется **автоматически реализуемое свойство (Auto-property)**, компилятор также создает скрытое (backing) поле, к которому нельзя обратиться напрямую из кода C#.

#### 2. Разница между свойством и полем
Несмотря на синтаксическое сходство, между ними есть фундаментальные различия:
*   **Адресация:** Свойство не является переменной и не занимает определенное место в памяти как таковое (это методы). Следовательно, свойство нельзя передать в метод как аргумент с модификатором `ref` или `out`.
*   **Производительность:** Вызов свойства — это вызов метода. Однако [[JIT]]-компилятор часто выполняет **инлайнинг (inlining)** простых свойств, делая их столь же быстрыми, как прямое обращение к полям.

#### 3. Модификаторы и инициализация
*   **`init` (C# 9+):** Позволяет устанавливать значение только в момент создания объекта (через конструктор или инициализатор). После этого свойство становится `readonly`.
*   **`required` (C# 11+):** Обязывает вызывающий код проинициализировать свойство при создании объекта.

> [!INFO] Нюанс: Виртуальные свойства
> Как и методы, свойства могут быть `virtual`, `abstract` или `override`. Это позволяет переопределять логику доступа к данным в производных классах, что невозможно сделать с полями.

### Пример кода (C# 12)

```csharp
using System;

public class Product(string initialName) // Primary Constructor
{
    // 1. Автоматическое свойство только для чтения (с инициализацией)
    public Guid Id { get; } = Guid.NewGuid();

    // 2. Свойство с логикой валидации (Full property)
    private decimal _price;
    public decimal Price
    {
        get => _price;
        set
        {
            if (value < 0) throw new ArgumentOutOfRangeException(nameof(value));
            _price = value;
        }
    }

    // 3. Init-only и Required свойство (C# 11)
    public required string Category { get; init; }

    // 4. Вычисляемое свойство (Expression-bodied member)
    public string DisplayInfo => $"{initialName} (Category: {Category}) - {Price:C}";
}

public class Program
{
    public static void Main()
    {
        // Использование инициализатора для required и init свойств
        var item = new Product("Laptop")
        {
            Category = "Electronics",
            Price = 1200m
        };

        Console.WriteLine(item.DisplayInfo);
        
        // item.Category = "Books"; // Ошибка: init-only
        // item.Id = Guid.Empty;    // Ошибка: нет сеттера
    }
}
```

Специальное поле **field**, которое генерируется компилятором, позволяет обращаться внутри свойства к полю, которое хранит значение для данного свойства. Это избавляет от необходимости явного создания поля для свойства:
```csharp
class Person
{
	public required string Name
    {
        get
        {
            return field;    // возвращаем значение свойства
        }
        set
        {
            field = value;   // устанавливаем новое значение свойства
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Тяжелые вычисления в get
> Аксессор `get` должен работать быстро (сложность O(1)). Пользователь свойства ожидает, что обращение к нему не является ресурсозатратным. Если требуется запрос к БД или сложный расчет, используйте метод (например, `CalculateTotal()`).

> [!WARNING] Ошибка: Побочные эффекты
> Чтение свойства не должно изменять состояние объекта. Избегайте логики в `get`, которая инкрементирует счетчики или меняет другие поля (исключение — ленивая инициализация или кэширование).

**Best Practices:**
1.  **Предпочитайте автосвойства:** Если вам не нужна логика валидации, используйте краткую запись `public string Name { get; set; }`. Это сохранит возможность добавить логику позже без изменения публичного API.
2.  **Инкапсуляция сеттера:** Если свойство должно быть доступно для чтения всем, но изменяться только внутри класса, используйте `public int Score { get; private set; }`.
3.  **Использование Expression-bodied:** Для простых геттеров используйте краткий синтаксис `=>`, это делает код чище.
4.  **Backing fields:** Называйте закрытые поля для свойств с префиксом подчеркивания (например, `_name`), чтобы легко отличать их от локальных переменных.

> [!TIP] Нюанс: Сериализация
> Библиотеки сериализации (например, System.Text.Json) по умолчанию работают только с публичными свойствами. Публичные поля также могут быть сериализованы, но использование свойств позволяет более гибко управлять процессом трансформации данных.
