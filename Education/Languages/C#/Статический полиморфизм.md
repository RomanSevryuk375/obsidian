### Суть (The "What")
**Статический полиморфизм** (или полиморфизм на этапе компиляции) — это механизм, при котором выбор конкретной реализации метода или операции происходит во время компиляции, а не во время выполнения. В отличие от динамического полиморфизма ([[VMT]]), здесь нет накладных расходов на поиск адреса метода в рантайме.

Он решает задачу предоставления единообразного интерфейса для работы с разными типами данных, обеспечивая при этом максимальную производительность и строгую типизацию.

---

### Как это работает (The "How" / Nutshell style)

Статический полиморфизм в [[MOC#/MOC|C#]] реализуется тремя основными способами:

#### 1. Перегрузка методов (Method Overloading)
Компилятор анализирует сигнатуру вызова (количество и типы аргументов) и связывает вызов с конкретным телом метода. Это называется **ранним связыванием (early binding)**.
*   **Нюанс:** Тип возвращаемого значения не является частью сигнатуры. Компилятор не может различать методы только по возвращаемому типу.

#### 2. Перегрузка операторов
Позволяет определить поведение стандартных операторов (`+`, `-`, `*`, `==` и др.) для пользовательских типов. С точки зрения [[CLR]], перегруженный оператор — это статический метод со специальным именем (например, `op_Addition`).

#### 3. Статические виртуальные члены в интерфейсах (C# 11+)
Это наиболее продвинутая форма статического полиморфизма, известная как **Generic Math**. 
*   **Механика:** Интерфейс определяет `static abstract` или `static virtual` методы. 
*   **Реализация:** Обобщенный метод может ограничивать тип параметром, реализующим такой интерфейс. Компилятор и [[JIT]] генерируют специализированный код, вызывающий статический метод конкретного типа напрямую.

> [!INFO] Нюанс: Производительность
> Статический полиморфизм позволяет JIT-компилятору выполнять агрессивный **инлайнинг (inlining)** — подстановку кода метода непосредственно в место вызова. Это невозможно для динамического полиморфизма из-за неопределенности типа в рантайме.

> [!IMPORTANT] Generic Specialization
> При использовании обобщений (Generics) со значимыми типами (struct), CLR создает отдельную копию машинного кода для каждой комбинации типов. Это исключает упаковку (boxing) и позволяет выполнять арифметические операции со скоростью нативного кода.

### Пример кода (C# 12)

```csharp
using System;
using System.Numerics;

// Интерфейс с использованием статического полиморфизма (Generic Math стиль)
public interface IMonoid<T> where T : IMonoid<T>
{
    static abstract T Identity { get; }
    static abstract T operator +(T left, T right);
}

// Реализация структуры
public readonly record struct Vector2(double X, double Y) : IMonoid<Vector2>
{
    public static Vector2 Identity => new(0, 0);

    // Перегрузка оператора - форма статического полиморфизма
    public static Vector2 operator +(Vector2 left, Vector2 right) =>
        new(left.X + right.X, left.Y + right.Y);
}

public class MathLibrary
{
    // 1. Перегрузка методов (классический статический полиморфизм)
    public int Add(int a, int b) => a + b;
    public double Add(double a, double b) => a + b;

    // 2. Статический полиморфизм через Generics (C# 11+)
    // Метод работает с любым типом, реализующим сложение статически
    public static T Accumulate<T>(IEnumerable<T> items) where T : IMonoid<T>
    {
        T sum = T.Identity;
        foreach (var item in items)
        {
            sum += item; // Вызывается статический оператор конкретного типа T
        }
        return sum;
    }
}

public class Program
{
    public static void Main()
    {
        Vector2[] path = [new(1, 1), new(2, 3), new(10, 0)];
        
        // Вызов обобщенного статического метода
        var totalOffset = MathLibrary.Accumulate(path);
        
        Console.WriteLine($"Total Offset: {totalOffset}");
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Несимметричная перегрузка операторов
> При перегрузке операторов сравнения (`==`, `!=`) или арифметики, всегда соблюдайте симметрию и интуитивность. Если перегружен `==`, обязан быть перегружен `!=` и методы `Equals/GetHashCode`.

> [!WARNING] Ошибка: Неоднозначность (Ambiguous Calls)
> Избегайте перегрузок, которые могут вызвать путаницу при неявном приведении типов (например, `Method(int x)` и `Method(short x)`). Компилятор может выдать ошибку, если не сможет однозначно определить лучший вариант.

**Best Practices:**
1.  **Приоритет Generics:** Используйте `static abstract` в интерфейсах (C# 11+) вместо перегрузок для каждого типа, если логика алгоритма идентична.
2.  **Explicit Overloads:** Если методы выполняют логически разные действия, лучше дать им разные имена, чем перегружать одно. Перегрузка уместна, когда действие одно и то же, но над разными данными.
3.  **Использование INumber:** В современном .NET для математических вычислений используйте стандартный интерфейс `System.Numerics.INumber<T>`, который уже реализует все необходимые статические члены для числовых типов.
4.  **Inlining:** Помните, что статический полиморфизм — лучший друг высокой производительности. Если метод вызывается в критическом цикле, используйте перегрузку или дженерики вместо интерфейсов и виртуальных методов.

> [!TIP] Нюанс: Иерархия вызовов
> При перегрузке методов компилятор всегда выбирает наиболее "специфичный" тип. Если есть `Method(object)` и `Method(string)`, при передаче строки будет выбран `Method(string)`. Это происходит на этапе компиляции.
