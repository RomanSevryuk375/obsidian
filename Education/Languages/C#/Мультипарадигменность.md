### Суть (The "What")
[[MOC#/MOC|C#]] является **мультипарадигменным** языком программирования. Это означает, что он не навязывает одну жесткую модель разработки (как, например, ранние версии Java навязывали строгий [[ООП]]), а позволяет комбинировать различные подходы: объектно-ориентированный, функциональный, декларативный, асинхронный и обобщенный.

Мультипарадигменность решает задачу **адаптивности**: разработчик может выбрать наиболее эффективный инструмент для конкретной подзадачи (например, [[LINQ]] для обработки данных или [[асинхронность]] для I/O-операций) в рамках одного проекта и одного синтаксиса.

### Как это работает (The "How" / Nutshell style)

C# эволюционировал от "улучшенного C++" до сложной системы, объединяющей следующие парадигмы:

#### 1. Объектно-ориентированная (OOP) — Фундамент
Базируется на классах, инкапсуляции, наследовании и полиморфизме. В C# все, включая примитивы, может рассматриваться как объект (через механизм boxing/unboxing и иерархию `System.Object`).

#### 2. Функциональная (FP) — Декларативная обработка
C# активно внедряет элементы FP, начиная с версии 3.0:
*   **[[Лямбда-выражения и делегаты]]:** Функции как объекты первого класса (First-class citizens).
*   **LINQ:** Декларативный подход к манипуляции данными.
*   **Immutability (Неизменяемость):** Ключевое слово `readonly`, модификатор `init` и типы `record` минимизируют побочные эффекты.
*   **Pattern Matching:** Позволяет ветвить логику на основе формы данных, а не только их типов.

#### 3. Обобщенная ([[Generics]]) — Параметрический полиморфизм
Позволяет писать алгоритмы, не зависящие от конкретных типов данных. В отличие от шаблонов C++, генерики в .NET компилируются в [[IL]] и специализируются [[JIT]]-компилятором во время выполнения, что сохраняет высокую производительность для значимых типов.

#### 4. Асинхронная (Asynchronous)
Реализована через модель TAP (Task-based Asynchronous Pattern). Компилятор превращает методы с `async/await` в сложные конечные автоматы (state machines), позволяя писать неблокирующий код, который читается как последовательный.

#### 5. Динамическая (Dynamic)
Через ключевое слово `dynamic` и DLR (Dynamic Language Runtime) C# позволяет обходить проверку типов на этапе компиляции, обеспечивая интеграцию с динамическими языками (Python, Ruby) или упрощая работу с COM/JSON.

> [!INFO] Нюанс: Компиляция замыканий
> При использовании функционального стиля (лямбд) и захвате внешних переменных, компилятор C# генерирует скрытый класс (DisplayClass), чтобы продлить жизнь переменным из стека, перенося их в кучу. Это пример того, как функциональная парадигма "накладывается" на объектную структуру [[CLR]].

### Пример кода (C# 12)

Этот пример демонстрирует смешение ООП, функционального программирования и асинхронности.

```csharp
using System;
using System.Collections.Generic;
using System.Linq;
using System.Threading.Tasks;

// 1. ООП + Функциональная парадигма (Record - неизменяемая сущность)
public record Product(string Name, decimal Price);

public class StoreManager
{
    // 2. Обобщения (Generics)
    private readonly List<Product> _products = [];

    // 3. Асинхронная парадигма
    public async Task ProcessDataAsync()
    {
        // 4. Декларативный подход и функциональные методы (LINQ)
        var expensiveProducts = _products
            .Where(p => p.Price > 100) // Лямбда-выражение
            .Select(p => p.Name)
            .ToList();

        await Task.Delay(100); // Имитация I/O
        
        // 5. Pattern Matching (Декларативное ветвление)
        foreach (var product in _products)
        {
            string message = product switch
            {
                { Price: > 1000 } => "Premium Item",
                { Price: > 0 }    => "Standard Item",
                _                 => "Freebie"
            };
            Console.WriteLine($"{product.Name}: {message}");
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Перегрузка парадигмами (Paradigm Overkill)
> Использование `dynamic` там, где можно обойтись интерфейсами, или написание сверхсложных LINQ-запросов (более 10 звеньев), которые невозможно отладить. Выбирайте парадигму, исходя из читаемости кода.

> [!WARNING] Ошибка: Захват переменных в циклах
> При использовании функционального стиля внутри циклов (например, создание лямбд, использующих итератор), будьте осторожны. До C# 5.0 это приводило к захвату одной и той же переменной. В современном C# это исправлено для `foreach`, но остается нюансом для `for`.

**Best Practices:**
1.  **LINQ для чтения, циклы для изменения:** Используйте функциональный стиль для фильтрации и преобразования данных. Для сложной логики модификации коллекций старый добрый `foreach` часто понятнее и быстрее.
2.  **Предпочитайте статику динамике:** Используйте `dynamic` только в крайних случаях (Interop, очень сложный JSON). Сильная типизация — главное преимущество C#.
3.  **Используйте Records для DTO:** Это внедряет лучшие практики функционального программирования (Value Equality, Immutability) в вашу объектную модель.
4.  **Async "до самого верха":** Не смешивайте блокирующий код (`.Result`, `.Wait()`) с асинхронным. Асинхронная парадигма должна пронизывать всё приложение.

> [!TIP] Нюанс: Performance
> Функциональный стиль (LINQ) часто создает больше объектов в куче (делегаты, итераторы) по сравнению с императивными циклами. В высоконагруженных участках (Hot Paths) стоит возвращаться к классическому императивному ООП/процедурному стилю.
