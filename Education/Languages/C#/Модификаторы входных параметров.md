### Суть (The "What")
По умолчанию в [[MOC#/MOC|C#]] аргументы передаются **по значению**. Это означает, что метод получает копию данных (для значимых типов) или копию адреса (для ссылочных типов). Модификаторы параметров изменяют этот механизм, позволяя передавать аргументы **по ссылке**.

Они решают задачи оптимизации памяти (избегание копирования больших структур), возврата нескольких значений из метода и управления мутабельностью данных непосредственно в вызывающем коде.

---

### Как это работает (The "How" / Nutshell style)

На низком уровне модификаторы `ref`, `out` и `in` реализуются одинаково: в стек метода передается не само значение, а **адрес (указатель)** на переменную в вызывающем коде. Разница заключается исключительно в "контракте", который проверяет компилятор.

#### 1. ref (Reference)
Передает параметр "вход-выход". 
*   Переменная **должна** быть инициализирована до вызова метода.
*   Метод может как читать, так и изменять значение.
*   Используется, когда методу нужно модифицировать существующий объект.

#### 2. out (Output)
Передает параметр "только выход".
*   Переменная может быть не инициализирована до вызова.
*   Метод **обязан** присвоить значение параметру до выхода.
*   Часто используется в паттерне `TryParse`.

#### 3. in (Input reference) — C# 7.2+
Передает параметр по ссылке "только для чтения".
*   Позволяет избежать копирования тяжелых структур ([[struct]]), не разрешая их изменять.
*   Если метод попытается изменить параметр `in`, возникнет ошибка компиляции.

> [!WARNING] Нюанс: Защитные копии (Defensive Copies)
> Если вы передаете `struct` через `in`, и метод вызывает её внутренний метод/свойство, не помеченное как `readonly`, компилятор создаст временную копию структуры, чтобы гарантировать неизменяемость. Это может нивелировать выигрыш в производительности. Всегда используйте `readonly struct` с модификатором `in`.

#### 4. params
Позволяет методу принимать переменное количество аргументов одного типа.
*   Внутри метода аргументы представляются в виде массива.
*   Может быть только один `params` и только в конце списка параметров.

> [!INFO] Обновление C# 12
> Начиная с C# 12, модификатор `params` может работать не только с массивами, но и с любыми подходящими типами коллекций (`ReadOnlySpan<T>`, `IEnumerable<T>` и т.д.), что снижает аллокации в куче.

### Пример кода (C# 12)

```csharp
using System;

public struct BigData
{
    public int Value;
    // Пометка метода как readonly критична для модификатора in
    public readonly void Print() => Console.WriteLine(Value);
}

public class ParameterDemo
{
    // Демонстрация всех модификаторов
    public static void Process(ref int counter, out int result, in BigData data, params int[] extra)
    {
        counter++; // Модифицируем оригинал
        result = counter * 10; // Обязаны присвоить значение
        // data.Value = 100; // ОШИБКА: in запрещает изменение
        data.Print();
    }

    public static void Main()
    {
        int myCounter = 5;
        BigData heavyStruct = new() { Value = 42 };

        // Вызов с использованием модификаторов
        Process(ref myCounter, out int myResult, in heavyStruct, 1, 2, 3);

        Console.WriteLine($"Counter: {myCounter}, Result: {myResult}");
        
        // C# 12: params с ReadOnlySpan (эффективнее по памяти)
        PrintStats([10, 20, 30]); 
    }

    static void PrintStats(params ReadOnlySpan<int> numbers)
    {
        foreach (var n in numbers) Console.Write($"{n} ");
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Злоупотребление ref/out
> Использование `ref` и `out` усложняет чтение кода и нарушает чистоту функций (побочные эффекты). Для возврата нескольких значений из метода в современном C# лучше использовать **кортежи (Tuples)**: `public (int sum, int count) GetStats()`.

> [!WARNING] Ошибка: Перегрузка по модификаторам
> Нельзя перегружать методы, если они различаются только наличием `ref` и `out` (например, `Method(ref int x)` и `Method(out int x)`), так как для [[CLR]] они создают одинаковую сигнатуру. Однако различие между обычным параметром и `ref` допустимо.

**Best Practices:**
1.  **in для структур:** Используйте `in` для структур размером более 16-24 байт (размер 2-3 указателей), чтобы повысить производительность.
2.  **TryParse Pattern:** Используйте `out` для методов, которые должны возвращать результат операции и флаг успеха одновременно.
3.  **Readonly Structs:** Всегда помечайте структуры как `readonly struct`, если планируете передавать их через `in`, чтобы избежать скрытых аллокаций на создание "защитных копий".
4.  **Discard (_):** Если вам не нужно значение из `out`, используйте символ подчеркивания: `bool success = int.TryParse("123", out _);`.

> [!TIP] Нюанс: ref для ссылочных типов
> Передача `class` через `ref` позволяет методу заменить сам объект (перенаправить ссылку на новый экземпляр в куче). Без `ref` вы можете менять только внутренние поля объекта.
