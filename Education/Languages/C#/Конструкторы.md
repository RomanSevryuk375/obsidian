### Суть (The "What")
**Конструктор** — это специальный метод класса, который вызывается автоматически при создании нового экземпляра (объекта) с помощью оператора `new`. 

Его главная задача — перевод объекта в валидное начальное состояние (инициализация полей, проверка входных аргументов, выделение необходимых ресурсов). Конструктор гарантирует, что объект не будет доступен остальной части программы в "сыром" или недопустимом виде.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Механика вызова и наследование
Процесс создания объекта строго иерархичен. При вызове конструктора производного класса происходят следующие этапы:
1.  **Инициализаторы полей:** Сначала инициализируются поля самого класса в порядке их объявления.
2.  **Базовый конструктор:** Вызывается конструктор базового класса (`base()`). Если он не указан явно, [[CLR]] ищет и вызывает конструктор без параметров у родителя.
3.  **Тело конструктора:** Выполняется код внутри фигурных скобок конструктора текущего класса.

#### 2. Неявный конструктор по умолчанию
Если в классе не объявлено ни одного конструктора, компилятор [[MOC#/MOC|C#]] автоматически генерирует **публичный конструктор без параметров** (parameterless constructor), который просто вызывает `base()`. Как только вы объявляете хотя бы один конструктор (даже с параметрами), автоматическая генерация прекращается.

#### 3. Статические конструкторы (`static ..ctor`)
- Выполняются один раз для каждого типа, а не для каждого объекта.
- Вызываются автоматически либо перед созданием первого экземпляра, либо перед обращением к любому статическому члену класса.
- **Низкоуровневый нюанс:** CLR гарантирует атомарность и потокобезопасность вызова статического конструктора. Если два потока одновременно обращаются к классу впервые, один будет заблокирован, пока другой выполняет инициализацию.

#### 4. Первичные конструкторы (C# 12 Primary Constructors)
В C# 12 параметры конструктора можно объявлять прямо в заголовке класса.
- Эти параметры доступны во всем теле класса (в инициализаторах полей и методах).
- Если параметр первичного конструктора используется в методе, компилятор автоматически генерирует скрытое поле для "захвата" этого значения.

> [!INFO] Нюанс: Ограничения Primary Constructors
> В обычных классах (не record) параметры первичного конструктора не становятся публичными свойствами автоматически. Они ведут себя как параметры, доступные внутри области видимости класса.

### Пример кода (C# 12)

```csharp
using System;

// 1. Использование первичного конструктора
public class DatabaseConnection(string connectionString)
{
    // Инициализатор поля выполняется ДО тела конструктора
    private readonly string _logPrefix = $"[DB-{DateTime.Now.Ticks}]: ";
    
    // Свойство только для чтения
    public string ConnectionString { get; } = connectionString;

    // 2. Дополнительный конструктор (Constructor Chaining)
    public DatabaseConnection() : this("Server=localhost;Database=Default;")
    {
        // Дополнительная логика для конструктора по умолчанию
    }

    public void Connect() => Console.WriteLine($"{_logPrefix} Connecting to {ConnectionString}");
}

public class SecureConnection : DatabaseConnection
{
    private readonly string _token;

    // 3. Вызов базового конструктора через base
    public SecureConnection(string connStr, string token) : base(connStr)
    {
        if (string.IsNullOrEmpty(token)) throw new ArgumentException("Token required");
        _token = token;
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Вызов виртуальных методов
> Никогда не вызывайте виртуальные методы (`virtual`) в конструкторе. Поскольку конструктор базового класса выполняется раньше конструктора производного, виртуальный метод может обратиться к полям потомка, которые еще не инициализированы. Это классический источник трудноотлавливаемых багов.

> [!WARNING] Ошибка: Исключения в статических конструкторах
> Если статический конструктор выбрасывает необработанное исключение, тип становится "сломанным" (`TypeInitializationException`). Его нельзя будет использовать повторно до перезапуска домена приложения.

**Best Practices:**
1.  **Цепочки конструкторов (Chaining):** Используйте `: this(...)`, чтобы избежать дублирования кода инициализации. Пусть один "главный" конструктор выполняет всю работу, а остальные вызывают его.
2.  **Валидация:** Проверяйте входные данные (Guard Clauses) прямо в конструкторе. Объект должен быть либо полностью готов к работе, либо не создан вовсе.
3.  **Приватные конструкторы:** Используйте `private` конструкторы для классов, которые содержат только статические методы (Utility classes), или для реализации паттерна Singleton, чтобы предотвратить создание экземпляров извне.
4.  **Dependency Injection:** В современной разработке используйте конструкторы для внедрения зависимостей. Первичные конструкторы C# 12 делают этот процесс максимально лаконичным.

> [!TIP] Нюанс памяти: Инициализаторы vs Тело
> По возможности предпочитайте инициализаторы полей (`private int _x = 10;`) логике внутри тела конструктора. Это делает код чище и явно указывает на намерения, однако помните, что в инициализаторах нельзя использовать `this` или обращаться к другим экземплярным методам.
