### Суть (The "What")
**Методы расширения** позволяют добавлять новую функциональность в существующие типы без изменения их исходного кода, [[Наследование|наследования]] или перекомпиляции. Они ведут себя как методы экземпляра расширяемого типа, хотя технически являются статическими.

Эта возможность решает задачу расширения "закрытых" типов (например, системных классов `string`, `int`, `IEnumerable`) и позволяет строить цепочки вызовов (**Fluent API**), значительно улучшая читаемость кода (яркий пример — технология [[LINQ]]).

---

### Как это работает (The "How" / Nutshell style)

#### 1. Синтаксический сахар
Методы расширения — это чистая иллюзия компилятора. На уровне **[[IL]] (Intermediate Language)** вызов `myString.Validate()` превращается в обычный вызов статического метода `StringExtensions.Validate(myString)`. 
*   **Низкоуровневая реализация:** В метаданных сборки такие методы помечаются специальным атрибутом `[ExtensionAttribute]`. Компилятор и IntelliSense используют его, чтобы "предложить" метод как часть расширяемого типа.

#### 2. Правила объявления
Чтобы метод стал расширением, должны быть соблюдены три условия:
1.  Метод должен быть определен в **статическом не-обобщенном классе**.
2.  Метод должен быть **статическим**.
3.  Первый параметр метода должен иметь модификатор **`this`**, за которым следует тип, который мы расширяем.

#### 3. Разрешение имен (Resolution Order)
Компилятор всегда ищет методы в следующем порядке:
1.  **Методы экземпляра:** Если в самом классе есть метод с такой же сигнатурой, он всегда имеет приоритет. Метод расширения никогда не сможет "переопределить" существующий метод класса.
2.  **Методы расширения:** Если метод экземпляра не найден, компилятор ищет методы расширения во всех подключенных пространствах имен (`using`).

#### 4. Область видимости
Поскольку это внешние статические методы, они **не имеют доступа** к `private`, `protected` или `internal` членам расширяемого типа. Они могут взаимодействовать только с публичным API объекта.

> [!INFO] Нюанс: Вызов у null
> В отличие от методов экземпляра, методы расширения **можно вызывать у объектов, равных `null`**. Поскольку это статический вызов, исключение `NullReferenceException` не возникнет автоматически при вызове, а произойдет только если внутри метода будет попытка обращения к членам этого объекта.

### Пример кода ([[MOC#/MOC|C#]] 12)

```csharp
using System;
using System.Collections.Generic;
using System.Linq;

namespace ExtensionsDemo;

// Статический класс для методов расширения
public static class StringExtensions
{
    // Расширяем встроенный тип string
    public static string ToTitleCase(this string input)
    {
        if (string.IsNullOrWhiteSpace(input)) return input;
        
        return char.ToUpper(input[0]) + input.Substring(1).ToLower();
    }

    // Расширяем интерфейс (теперь метод доступен для List, Array и т.д.)
    public static bool IsEmpty<T>(this IEnumerable<T> source)
    {
        return source == null || !source.Any();
    }
}

public class Program
{
    public static void Main()
    {
        string name = "artyom";
        
        // Выглядит как метод экземпляра
        string title = name.ToTitleCase();
        Console.WriteLine(title); // Output: Artyom

        List<int> numbers = [];
        // Вызов метода расширения для интерфейса
        if (numbers.IsEmpty())
        {
            Console.WriteLine("List is empty");
        }

        // Демонстрация вызова у null
        string? nullStr = null;
        Console.WriteLine(nullStr.SafeLength()); // Вызовет расширение без ошибки
    }
}

public static class NullSafeExtensions
{
    // Метод расширения, защищающий от null
    public static int SafeLength(this string? str) => str?.Length ?? 0;
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Загрязнение глобальных пространств имен
> Не создавайте методы расширения в пространстве имен `System`, если только вы не расширяете абсолютно базовые типы. Лучше выносить расширения в специфические пространства имен (напр. `MyProject.Extensions`), чтобы они подключались только тогда, когда действительно нужны.

> [!WARNING] Ошибка: Расширение object
> Избегайте написания методов расширения для типа `object`. Это приведет к тому, что ваш метод появится в IntelliSense абсолютно у всех объектов в проекте, создавая информационный шум и риск конфликтов.

**Best Practices:**
1.  **Расширение интерфейсов:** Это самый мощный способ использования данной фичи. Расширяя `IEnumerable<T>` или `IComparable`, вы добавляете функционал сразу множеству классов (как это делает LINQ).
2.  **Группировка:** Размещайте методы расширения в классах, названия которых отражают их суть (например, `JsonExtensions`, `HttpRequestExtensions`).
3.  **Неявный приоритет:** Помните, что если автор расширяемого класса позже добавит метод с такой же сигнатурой, ваш код может изменить поведение (начнет вызываться метод класса), что приведет к трудноотлавливаемым багам.
4.  **Проверка на null:** Всегда делайте проверку входного параметра `this` внутри метода расширения, если не планируете специфическую логику для `null`-объектов.

> [!TIP] Нюанс: Fluent Interface
> Методы расширения идеально подходят для создания плавных интерфейсов. Если метод расширения возвращает тот же тип, который он расширяет (`return this`), его можно вызывать цепочкой: `myObj.DoA().DoB().DoC();`.
