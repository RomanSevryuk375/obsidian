### Суть (The "What")
**Неявная типизация** — это механизм [[MOC#/MOC|C#]], позволяющий переложить обязанность определения типа переменной на компилятор. Вместо явного указания типа (например, `int` или `List<string>`) используется ключевое слово `var`.

Она решает задачу сокращения избыточности кода ([[DRY]] — Don't Repeat Yourself), упрощает работу со сложными обобщенными типами и является единственным способом работы с **анонимными типами**.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Компиляционное соответствие
Важно понимать: `var` — это **строгая типизация**, а не динамическая. 
*   Тип переменной определяется компилятором в момент анализа исходного кода на основе выражения справа от оператора присваивания.
*   В скомпилированном [[IL]]-коде (Intermediate Language) вместо `var` будет стоять конкретный тип (например, `System.Int32`). 
*   Никаких накладных расходов в рантайме (Runtime) не возникает.

#### 2. Ограничения использования
Ключевое слово `var` можно использовать только для:
*   Локальных переменных внутри методов.
*   В циклах `for` и `foreach`.
*   В блоках `using`.

**Где нельзя использовать `var`:**
*   Для полей класса (полям всегда нужен явный тип).
*   Для параметров методов или возвращаемых значений.
*   Без немедленной инициализации (компилятору не из чего вывести тип).
*   Присваивая `null` без приведения типа.

#### 3. var и анонимные типы
Анонимные типы (`var person = new { Name = "Artyom", Age = 30 };`) создаются компилятором "на лету". Поскольку у такого типа нет имени в исходном коде, `var` является единственным законным способом объявить переменную для хранения ссылки на такой объект.

> [!INFO] Нюанс: Target-typed new (C# 9+)
> Начиная с C# 9, появился "антоним" `var` — **target-typed new**. Это когда тип указан слева, а справа используется просто `new()`: `List<int> list = new();`. Выбор между `var` и `new()` — вопрос стиля, но в обоих случаях компилятор устраняет дублирование.

### Пример кода (C# 12)

```csharp
using System;
using System.Collections.Generic;

public class ImplicitTypingDemo
{
    public void Demonstrate()
    {
        // 1. Очевидный тип (хорошая практика)
        var users = new Dictionary<int, List<string>>();
        
        // 2. Использование в циклах
        var numbers = new[] { 1, 2, 3 }; // массив int[]
        foreach (var n in numbers)
        {
            Console.WriteLine(n);
        }

        // 3. Анонимные типы (обязательное использование var)
        var point = new { X = 10, Y = 20 };

        // 4. Использование с Collection Expressions (C# 12)
        // ВАЖНО: var нельзя использовать с [], так как [] не имеет типа по умолчанию
        // var list = [1, 2, 3]; // ОШИБКА компиляции
        List<int> list = [1, 2, 3]; // Правильно
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Потеря читаемости
> Избегайте `var`, если тип не очевиден из правой части выражения. 
> Плохо: `var data = GetResult();` (читателю нужно лезть в метод `GetResult`, чтобы понять тип).
> Хорошо: `var user = new User();` или `var id = 42;`.

> [!WARNING] Ошибка: Инициализация null
> Код `var x = null;` не скомпилируется. Если нужно использовать неявную типизацию с возможным null, используйте приведение: `var x = (string?)null;`.

**Best Practices:**
1.  **Сложные типы:** Используйте `var`, когда тип слишком длинный (например, в [[LINQ]]-запросах или вложенных [[Generics]]). Это фокусирует внимание на имени переменной, а не на её "шумном" типе.
2.  **Анонимные типы:** Применяйте `var` только там, где типы создаются "на лету" (обычно в `Select` внутри LINQ).
3.  **Имена переменных:** При использовании `var` давайте переменным максимально описательные имена. Поскольку тип скрыт, имя переменной становится единственным источником информации о смысле данных.
4.  **Соблюдение единообразия:** В рамках одного проекта лучше придерживаться одного стиля: либо всегда использовать `var` (где допустимо), либо только там, где тип очевиден.

> [!TIP] Нюанс: var vs dynamic
> Никогда не путайте их. `var` проверяется при компиляции и фиксирует тип навсегда. `dynamic` пропускает проверку типов и может менять тип значения в процессе работы программы, что существенно медленнее и опаснее.
