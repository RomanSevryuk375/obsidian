### Суть (The "What")
`StringBuilder` (из пространства имен `System.Text`) — это изменяемый (mutable) контейнер для последовательности символов. В отличие от `string`, который создает новый объект в куче при любой модификации, `StringBuilder` работает с внутренним буфером, позволяя эффективно изменять, удалять и добавлять данные "на месте".

**Какую проблему решает?**
Устраняет проблему "фрагментации кучи" и избыточных аллокаций при интенсивной работе с текстом. В сценариях с циклами использование обычной конкатенации (`s += "..."`) превращает сложность алгоритма из линейной $O(n)$ в квадратичную $O(n^2)$, тогда как `StringBuilder` сохраняет линейную производительность.

### Как это работает (The "How" / Nutshell style)

#### 1. Внутреннее устройство
Внутри `StringBuilder` хранит массив символов (`char[]`). У него есть два ключевых параметра:
*   **Capacity (Емкость):** Текущий размер выделенного массива.
*   **Length (Длина):** Количество фактически занятых символов в этом массиве.

#### 2. Динамическое расширение
Когда `Length` превышает `Capacity`, `StringBuilder` выделяет новый, более вместительный массив (обычно увеличивая размер вдвое) и копирует туда старые данные.
> [!INFO] Нюанс реализации (Linked List)
> Начиная с .NET Core / .NET 5+, реализация стала сложнее: вместо простого копирования в новый массив, `StringBuilder` может хранить цепочку фрагментов (чанки). Это позволяет избегать аллокаций в "куче больших объектов" (LOH) при сборке очень длинных строк.

#### 3. Стоимость преобразования
Вызов метода `ToString()` копирует данные из внутреннего буфера в новый объект `string`. 
> [!WARNING] Нюанс: Пустой вызов
> Если вызвать `ToString()` повторно, не меняя содержимое `StringBuilder`, современные версии .NET могут вернуть закэшированную строку, однако любая модификация буфера после этого приведет к созданию новой копии.

### Пример кода ([[MOC#/MOC|C#]] 12)

```csharp
using System.Text;

public class StringBuilderExample
{
    public void BuildLargeString()
    {
        // Инициализация с заранее заданным объемом (Best Practice)
        var sb = new StringBuilder(initialCapacity: 1024);

        string[] items = ["Log1", "Log2", "Log3", "Log4"];

        // Fluent Interface (цепочка вызовов)
        sb.Append("Start Logging:")
          .AppendLine()
          .AppendJoin(", ", items)
          .Replace("Log1", "InitialLog");

        // Условная вставка (C# 12 Collection expressions могут использоваться для данных)
        if (sb.Length > 0)
        {
            sb.Insert(0, "[Header] ");
        }

        // Финализация
        string finalResult = sb.ToString();
        
        Console.WriteLine(finalResult);
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Мелкие конкатенации
> Не используйте `StringBuilder` для объединения 2-3 строк за один раз. Накладные расходы на создание объекта `StringBuilder` в куче выше, чем простое использование `string.Concat` или интерполяции `$"..."`. Интерполяция для коротких строк в современном C# оптимизирована через структуры-хендлеры и часто быстрее.

> [!WARNING] Ошибка: Неизвестный размер
> Если вы не указываете `initialCapacity`, `StringBuilder` начинает с маленького буфера (обычно 16 символов). Это приведет к многократным переаллокациям памяти при сборке большой строки.

**Best Practices:**
1.  **Pre-allocation:** Всегда указывайте примерный итоговый размер в конструкторе, если он известен. Это сводит количество выделений памяти к единице.
2.  **AppendJoin:** Используйте метод `AppendJoin` вместо циклов с `Append` и ручным добавлением разделителей. Это чище и оптимизировано внутри [[CLR]].
3.  **Переиспользование:** В высокопроизводительных системах (например, геймдев или высоконагруженные API) можно использовать `ObjectPool<StringBuilder>`, чтобы не создавать новые объекты и не провоцировать работу Garbage Collector.
4.  **Clear vs New:** Если нужно очистить буфер для повторного использования, вызывайте `sb.Clear()`, а не создавайте новый экземпляр `new StringBuilder()`.

> [!TIP] Нюанс производительности
> Если вам нужно просто записать данные в поток (например, в файл или HTTP-ответ), не собирайте сначала огромный `StringBuilder`. Пишите данные напрямую в `StreamWriter`, так как это исключает промежуточную аллокацию строки в памяти.
