### Суть (The "What")
**Событие** — это высокоуровневая языковая конструкция, построенная поверх делегатов, которая реализует паттерн «Издатель-Подписчик» (Publisher-Subscriber). Оно позволяет объекту (издателю) уведомлять другие объекты (подписчиков) о том, что произошло определенное действие.

События решают задачу **строгой [[Инкапсуляция|инкапсуляции]] делегатов**. В отличие от обычного публичного поля-делегата, событие ограничивает доступ внешнего кода: подписчики могут только добавлять (`+=`) или удалять (`-=`) свои обработчики, но не могут вызвать событие напрямую или очистить список других подписчиков.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Синтаксический сахар и аксессоры
Когда вы объявляете событие одной строкой (`public event Action MyEvent;`), компилятор [[MOC#/MOC|C#]] автоматически генерирует:
*   Закрытое (private) поле-делегат.
*   Два публичных метода-аксессора: `add_MyEvent` и `remove_MyEvent`.
*   Логику внутри аксессоров, которая использует `Interlocked.CompareExchange` для обеспечения **потокобезопасности** при подписке и отписке без использования тяжелых блокировок (`lock`).

#### 2. Защита вызова
Ключевое отличие от делегата: метод `Invoke()` для события доступен **только внутри того класса**, где это событие объявлено. Внешний код, имеющий доступ к объекту, увидит событие, но не сможет его «выстрелить» или присвоить ему `null`.

#### 3. Стандартный паттерн .NET
Для обеспечения совместимости и предсказуемости в экосистеме .NET принято использовать:
*   Делегат `EventHandler` или `EventHandler<TEventArgs>`.
*   Первый аргумент `object sender` (кто вызвал).
*   Второй аргумент `TEventArgs e` (данные события).
*   Защищенный виртуальный метод `On[EventName]` для вызова события (позволяет наследникам изменять поведение).

> [!INFO] Нюанс: Field-like vs Property-like
> Вы можете определить аксессоры `add` и `remove` вручную (аналогично свойствам `get`/`set`). Это полезно, если нужно реализовать кастомную логику (например, проксирование события другому объекту или ленивую инициализацию).

### Пример кода (C# 12)

```csharp
using System;

// 1. Определение данных события через Primary Constructor (C# 12)
public class PriceChangedEventArgs(decimal oldPrice, decimal newPrice) : EventArgs
{
    public decimal OldPrice { get; } = oldPrice;
    public decimal NewPrice { get; } = newPrice;
}

public class StockTicker(string symbol)
{
    private decimal _price;

    // 2. Объявление события с использованием стандартного делегата
    public event EventHandler<PriceChangedEventArgs>? PriceChanged;

    public void UpdatePrice(decimal newPrice)
    {
        if (_price == newPrice) return;

        decimal oldPrice = _price;
        _price = newPrice;

        // 3. Безопасный вызов события
        OnPriceChanged(new PriceChangedEventArgs(oldPrice, newPrice));
    }

    // Стандартный метод для вызова события
    protected virtual void OnPriceChanged(PriceChangedEventArgs e)
    {
        // Использование Null-conditional operator для проверки на наличие подписчиков
        PriceChanged?.Invoke(this, e);
    }
}

public class Program
{
    public static void Main()
    {
        var ticker = new StockTicker("MSFT");

        // Подписка
        ticker.PriceChanged += (sender, e) => 
        {
            Console.WriteLine($"Цена изменилась с {e.OldPrice} на {e.NewPrice}");
        };

        ticker.UpdatePrice(150.50m);
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: Утечки памяти (Lapsed Listener)
> Это самая частая причина утечек в .NET. Издатель хранит сильную ссылку на подписчика через делегат. Если объект-подписчик больше не нужен, но он не отписался (`-=`) от события долгоживущего издателя (например, статического класса), [[GC]] никогда не удалит подписчика.

> [!WARNING] Ошибка: Потокобезопасность вызова
> Хотя подписка (`+=`) потокобезопасна, вызов события — нет. Если один поток проверяет `if (PriceChanged != null)`, а другой поток отписывается прямо перед `PriceChanged(this, e)`, возникнет `NullReferenceException`. 
> **Решение:** Всегда используйте `PriceChanged?.Invoke(...)` или копируйте ссылку в локальную переменную перед проверкой.

**Best Practices:**
1.  **EventArgs.Empty:** Если данных для передачи нет, используйте `EventHandler` (без дженерика) и передавайте `EventArgs.Empty` вместо `null`.
2.  **Имена:** Называйте события глаголами в прошедшем времени (`Clicked`, `Finished`) или в процессе (`Closing`), чтобы подчеркнуть момент возникновения.
3.  **Weak Events:** Для предотвращения утечек в сложных UI-системах (WPF/MAUI) рассмотрите использование паттерна `WeakEventManager`.
4.  **Async Events:** Избегайте `async void` в обработчиках событий, если это возможно. Если нужно вызвать асинхронный код, используйте паттерн с `Deferral` (откладыванием завершения), если архитектура это поддерживает.

> [!TIP] Нюанс: Наследование событий
> События не являются виртуальными по умолчанию. Чтобы наследник мог "переопределить" логику вызова, всегда предоставляйте защищенный виртуальный метод `On[EventName]`. Потомок сможет вызвать `base.OnEventName()`, чтобы сохранить уведомление базовых подписчиков.
