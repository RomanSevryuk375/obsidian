# Нисходящее приведение типов (Downcasting)

### Суть (The "What")
**Нисходящее приведение типов (Downcasting)** — это операция приведения ссылки базового [[Классы и объекты|класса]] (или [[Интерфейсы|интерфейса]]) к ссылке на один из его производных классов (или конкретную реализацию). Она представляет собой движение вниз по иерархии наследования (от общего к частному).

Downcasting решает задачу **доступа к специфичной функциональности наследника**, которая отсутствует в базовом контракте. Поскольку эта операция потенциально небезопасна (мы не можем гарантировать на этапе компиляции, что за базовой ссылкой скрывается именно этот наследник), она всегда должна выполняться явно.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Проверка типов во время выполнения (Runtime Type Check)
В отличие от восходящего приведения ([[Upcasting]]), которое проверяется и одобряется компилятором, нисходящее приведение всегда требует **динамической проверки типов** на уровне [[CLR]]:
*   **Чтение заголовка:** Среда выполнения обращается к объекту в [[Куча|куче]] и считывает его [[MethodTable]] Pointer (Type Handle).
*   **Анализ иерархии:** CLR проверяет, присутствует ли целевой тип приведения в цепочке наследования (или списке реализуемых интерфейсов) фактического типа объекта.
*   Если проверка пройдена успешно, возвращается ссылка, приведённая к целевому типу. Если нет — поведение зависит от выбранного синтаксиса приведения.

#### 2. Инструкции [[IL]] и способы приведения
В [[C#]] существует три основных способа выполнить Downcasting, компилирующихся в разные IL-инструкции:

*   **Классическое приведение `(Derived)obj`:** Компилируется в IL-инструкцию `castclass`. Если приведение невозможно, CLR немедленно выбрасывает `InvalidCastException`.
*   **Оператор `as` (`obj as Derived`):** Компилируется в IL-инструкцию `isinst`. Если тип не совпадает, метод возвращает `null` вместо выброса исключения (работает только со ссылочными или [[Nullable-types|nullable-типами]]).
*   **Сопоставление шаблонов (`obj is Derived d`):** Современный и наиболее безопасный подход. Компилятор оптимизирует проверку и приведение в одну эффективную операцию проверки типа и объявления локальной переменной.

### Пример кода (C# 12)

```csharp
using System;

public abstract class Vehicle(string model) // Primary constructor (C# 12)
{
    public string Model { get; } = model;
}

public class Airplane(string model, double maxAltitude) : Vehicle(model)
{
    public double MaxAltitude { get; } = maxAltitude;

    public void Fly() => Console.WriteLine($"{Model} is flying at {MaxAltitude} meters.");
}

public class Crane(string model, double liftCapacity) : Vehicle(model)
{
    public double LiftCapacity { get; } = liftCapacity;

    public void Lift() => Console.WriteLine($"{Model} is lifting {LiftCapacity} tons.");
}

public class Program
{
    public static void Main()
    {
        Vehicle myVehicle = new Airplane("Boeing 747", 13000);

        // --- Способ 1: Классическое приведение (Небезопасно) ---
        try
        {
            // Если бы myVehicle был подтипом Crane, здесь упало бы исключение
            Airplane plane1 = (Airplane)myVehicle; 
            plane1.Fly();
        }
        catch (InvalidCastException ex)
        {
            Console.WriteLine(ex.Message);
        }

        // --- Способ 2: Оператор 'as' (Безопасен, но требует проверки на null) ---
        Crane? crane = myVehicle as Crane;
        if (crane != null)
        {
            crane.Lift();
        }
        else
        {
            Console.WriteLine("Преобразование к Crane через 'as' вернуло null.");
        }

        // --- Способ 3: Pattern Matching (C# 7+ / Рекомендуемый) ---
        // Самый лаконичный и безопасный способ Downcasting
        if (myVehicle is Airplane plane2)
        {
            plane2.Fly();
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Игнорирование проверок при классическом приведении
> Никогда не используйте прямое жесткое приведение вида `(Derived)obj` без предварительной проверки типа, если вы не гарантируете архитектурно на 100%, что объект относится к целевому типу. Падение с `InvalidCastException` в рантайме — признак плохо спроектированного кода.

> [!WARNING] Ошибка: Нарушение принципа Лисков ([[Liskov Substitution Principle|LSP]])
> Частое использование Downcasting (особенно длинные цепочки `if (x is TypeA) ... else if (x is TypeB) ...`) является серьезным архитектурным запахом (Code Smell). Это указывает на то, что базовая абстракция спроектирована неверно, и поведение программы зависит от конкретных реализаций, а не от общего контракта.

**Best Practices:**
1.  **Используйте Pattern Matching:** Отдавайте абсолютный приоритет конструкции `if (obj is Derived d)` вместо старого паттерна `var d = obj as Derived; if (d != null)`. Она безопаснее, читаемее и оптимизирована на уровне JIT.
2.  **Избегайте приведения к интерфейсам в циклах:** Проверка типов при нисходящем приведении к интерфейсам во время выполнения стоит дороже, чем приведение к классам. Избегайте таких проверок в нагруженных циклах.
3.  **Используйте [[Полиморфизм|полиморфизм]]:** Вместо того чтобы приводить объект к наследнику ради вызова специфичного метода, подумайте, нельзя ли вынести этот метод (или его абстрактную версию) в базовый класс в виде `virtual` или `abstract` члена.

> [!TIP] Нюанс: Приведение и Generics
> В обобщенных методах нисходящее приведение может потребовать двойного приведения или использования `as`, если параметр типа `T` не имеет ограничений. Например: `var result = (T)(object)myObj;`. Использование ограничений `where T : class` позволяет избежать таких трюков.
