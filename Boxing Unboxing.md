# Упаковка и распаковка (Boxing / Unboxing)

### Суть (The "What")
**Упаковка (Boxing)** — это процесс неявного преобразования значимого типа ([[Value Types|value type]]) в ссылочный тип ([[Reference Types|reference type]]), такой как `object` или реализуемый им интерфейс. **Распаковка (Unboxing)** — это обратный процесс явного приведения полученной ссылки к исходному значимому типу.

Этот механизм решает задачу **единой системы типов (Unified Type System)** в .NET, позволяя передавать любые значимые типы туда, где ожидаются объекты базового класса `System.Object` или интерфейсы.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Механика упаковки (Boxing)
Когда происходит упаковка значимого типа (например, `int`):
1.  **Выделение памяти в куче:** [[CLR]] выделяет память в управляемой куче ([[Куча|Managed Heap]]). Размер выделяемого блока равен размеру самого значимого типа плюс стандартные накладные расходы объекта (16 байт на x64: Sync Block Index + Method Table Pointer).
2.  **Копирование данных:** Поля значимого типа копируются из [[Стэк|стека]] (или из структуры-владельца) в только что выделенную область кучи.
3.  **Возврат ссылки:** Возвращается адрес созданного объекта в куче.

#### 2. Механика распаковки (Unboxing)
Распаковка выполняется в два шага на уровне [[IL]]-инструкций (`unbox` или `unbox.any`):
1.  **Проверка типа:** CLR проверяет, действительно ли объект в куче является упакованным типом того же типа, к которому идет приведение.
2.  **Копирование назад:** Данные копируются из кучи обратно в стек в переменную значимого типа.

> [!DANGER] Строгое соответствие типов при распаковке
> При распаковке тип должен совпадать **абсолютно точно**. Если упакованный `int` (System.Int32) попытаться распаковать в `double` или даже `short` без предварительной распаковки в `int`, CLR выбросит исключение `InvalidCastException`.

#### 3. Упаковка Nullable-типов
У [[Nullable-types|Nullable-типов]] (`Nullable<T>` или `T?`) особая логика упаковки на уровне CLR:
*   Если объект равен `null` (свойство `HasValue` возвращает `false`), то упаковка возвращает **пустую ссылку (`null`)**, а не упакованный объект со значением по умолчанию.
*   Если значение есть, упаковывается чистый значимый тип `T`, а не структура `Nullable<T>`.

### Пример кода ([[C#]] 12)

```csharp
using System;

public struct Point(int x, int y) : IComparable
{
    public int X = x;
    public int Y = y;

    // Реализация интерфейса приводит к неявному боксингу при вызове через интерфейс
    public int CompareTo(object? obj) => 0; 
}

public class BoxingDemo
{
    public static void Main()
    {
        // 1. Классическая упаковка
        int number = 42;
        object boxedNumber = number; // IL-инструкция: box

        // 2. Распаковка (требует явного приведения к точному типу)
        int unboxedNumber = (int)boxedNumber; // IL-инструкция: unbox.any

        // Ошибка распаковки (InvalidCastException)
        try
        {
            double wrongType = (double)boxedNumber; 
        }
        catch (InvalidCastException ex)
        {
            Console.WriteLine($"Ошибка приведения: {ex.Message}");
        }

        // 3. Упаковка Nullable-типа
        int? nullableInt = null;
        object boxedNull = nullableInt!; // Вернет null-ссылку, а не объект
        Console.WriteLine(boxedNull is null); // True

        // 4. Неявная упаковка через интерполяцию
        Point p = new(10, 20);
        // p упакуется, так как метод Console.WriteLine неявно приводит параметры к object
        Console.WriteLine($"Point coordinates: {p}"); 
    }
}
```

### Ошибки и Best Practices

> [!WARNING] Ошибка: Скрытый боксинг в коллекциях и строках
> Будьте осторожны с методами, принимающими `object` или `params object[]`. Например, `string.Concat(1, " ", 2)` упакует числа `1` и `2`. В высоконагруженных циклах это приводит к взрывному росту аллокаций.

> [!TIP] Оптимизация через IEquatable T 
> Если ваша структура должна сравниваться с другими (например, является ключом в словаре или хранится в `List T`), обязательно реализуйте интерфейс [[IEquatable]] . Это позволит методам поиска (`Contains`, `IndexOf`) использовать сравнение напрямую без упаковки структуры в интерфейс.

**Best Practices:**
1.  **[[Generics|Дженерики]] — лучшая защита:** Всегда используйте обобщенные коллекции (`List T `, `Dictionary K, V`). Они были созданы специально для устранения боксинга, существовавшего во временах `ArrayList` и `Hashtable`.
2.  **[[Статические лямбды]]:** В C# 9+ используйте `static` перед лямбдами, чтобы компилятор не захватывал контекст `this` (что часто вызывает неявную упаковку значимых типов в замыкания).
3.  **[[Интерфейсы]] для структур:** Старайтесь не приводить структуры к интерфейсам. Если вам нужно передать структуру в метод, работающий с интерфейсом, используйте обобщенный метод с ограничением: `void Process T (T item) where T : IMyInterface`. Это исключит упаковку в рантайме.
4.  **ToString():** Перед выводом структуры в строку вызывайте `struct.ToString()` явно. Это предотвратит передачу структуры в сигнатуру `object` и исключит скрытую упаковку.
