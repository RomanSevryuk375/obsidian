### Суть (The "What")
`Stack<T>` — это последовательная обобщенная коллекция, реализующая структуру данных **LIFO (Last-In, First-Out — «последним пришел, первым ушел»)**. Элементы добавляются на вершину стека, и извлекаются также исключительно с вершины.

Стек эффективно решает задачи, требующие **обратного порядка обработки данных**: алгоритмы обхода графов (поиск в глубину — DFS), парсинг и вычисление математических выражений, поддержка операций отмены действия (Undo/Redo) или ведение истории переходов в браузере.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Внутреннее устройство и простая адресация
В отличие от [[Queue]]\<T>, требующей кругового буфера и двух индексов (`head` и `tail`), `Stack<T>` устроен значительно проще:
*   **Массив хранения:** Внутри находится одномерный массив `T[] _array`.
*   **Индекс вершины:** Хранится единственный целочисленный указатель размера `_size`, который указывает на индекс следующего свободного слота (или, логически, на вершину стека).
*   **Push (Добавление):** Элемент записывается по индексу `_size`, после чего `_size` увеличивается на 1.
*   **Pop (Извлечение):** Указатель `_size` уменьшается на 1, и возвращается элемент из ячейки `_array[_size]`.

#### 2. Защита от утечек памяти ([[GC]]Protection)
Важный низкоуровневый нюанс работы метода `Pop()`:
> [!IMPORTANT] Очистка ссылок в Pop
> При извлечении элемента метод `Pop()` не просто сдвигает указатель, но и явно зануляет освободившуюся ячейку массива: `_array[_size] = default;`. Если этого не делать, ссылка на извлеченный объект осталась бы лежать во внутреннем массиве стека, что помешало бы сборщику мусора (GC) утилизировать этот объект, приводя к скрытой утечке памяти.

#### 3. Сложность операций
*   **Push:** Амортизированная **$O(1)$**. Если массив полностью заполнен, происходит перевыделение памяти (размер массива удваивается) и копирование элементов через `Array.Copy`.
*   **Pop:** Стабильная **$O(1)$**. Извлечение всегда идет с конца массива, поэтому сдвигать другие элементы в памяти не нужно (в отличие от `List<T>`).
*   **Peek (Просмотр вершины):** Стабильная **$O(1)$**.

### Пример кода ([[C#]] 12)

```csharp
using System;
using System.Collections.Generic;

public class StackDemo
{
    public static void Main()
    {
        // 1. Инициализация стека с заданной емкостью
        Stack<string> undoStack = new(capacity: 32);

        // 2. Push - добавление на вершину стека - O(1)
        undoStack.Push("Action_1: Create Document");
        undoStack.Push("Action_2: Type Text");
        undoStack.Push("Action_3: Insert Image");

        // 3. Peek - просмотр вершины без удаления - O(1)
        string lastAction = undoStack.Peek();
        Console.WriteLine($"Последнее выполненное действие: {lastAction}"); // Insert Image

        // 4. Pop - извлечение с вершины стека - O(1)
        while (undoStack.Count > 0)
        {
            string action = undoStack.Pop();
            Console.WriteLine($"Отмена действия (Undo): {action}");
        }

        // 5. Безопасное извлечение через TryPop (C# 6+)
        if (undoStack.TryPop(out string? revertedAction))
        {
            Console.WriteLine($"Отменено: {revertedAction}");
        }
        else
        {
            Console.WriteLine("Стек истории пуст.");
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Вызов Pop или Peek на пустом стеке
> Если вызвать метод `Pop()` или `Peek()` у пустого стека (`Count == 0`), рантайм выбросит исключение `InvalidOperationException`. Всегда проверяйте свойство `Count` перед вызовом или используйте безопасные методы `TryPop()` и `TryPeek()`.

> [!WARNING] Ошибка: Многопоточный доступ к стандартному стеку
> Класс `Stack<T>` не является потокобезопасным. Если несколько потоков одновременно выполняют операции `Push` или `Pop`, данные во внутреннем массиве будут испорчены, а указатель `_size` станет некорректным. Для многопоточных сред всегда используйте `ConcurrentStack<T>`.

**Best Practices:**
1.  **Pre-allocation:** Если вы знаете максимальную или среднюю глубину стека (например, глубина рекурсии при обходе папок), инициализируйте стек с начальной емкостью: `new Stack<T>(expectedDepth)`. Это исключит лишнее выделение памяти и копирование массивов.
2.  **Замена ручной рекурсии:** Если ваш рекурсивный алгоритм рискует вызвать переполнение системного стека (`StackOverflowException`), перепишите его на итеративный подход, используя `Stack<T>` в [[Куча|куче]]. Память кучи гораздо больше, чем системный стек потока (который по умолчанию ограничен 1 МБ).
3.  **Не используйте [[List]]\<T> как стек:** Хотя `List<T>` позволяет добавлять и удалять элементы с конца, использование специализированного `Stack<T>` делает намерения разработчика более явными, а API коллекции — защищенным от случайного изменения элементов в середине.
4.  **Сжатие памяти:** Метод `TrimExcess()` освобождает неиспользуемую память внутреннего массива, если стек сильно уменьшился в размере после выполнения алгоритма.

> [!TIP] Нюанс: Обход через foreach
> При переборе стека с помощью `foreach` элементы не удаляются. Порядок обхода строго соответствует логике LIFO: от вершины (последний добавленный) к основанию (первый добавленный). Любая модификация стека во время итерации вызовет `InvalidOperationException`.
