### Суть (The "What")
`Span<T>` и `ReadOnlySpan<T>` — это специализированные [[Value Types|значимые типы]] (`ref struct`), представляющие собой унифицированное, непрерывное и безопасное к типам представление произвольного блока памяти. 

Они решают задачу **высокопроизводительной обработки данных с нулевыми аллокациями в [[Куча|куче]] (zero-allocation)**. С их помощью можно работать с частями (срезами) массивов, строк или нативной памяти без необходимости копировать данные или выделять новые объекты, что радикально снижает нагрузку на Garbage Collector ([[GC]]).

---

### Как это работает (The "How" / Nutshell style)

#### 1. Унифицированный доступ к разным типам памяти
`Span<T>` абстрагирует разработчика от того, где физически находятся данные. Он может ссылаться на:
*   **Управляемую кучу ([[Managed Heap]]):** обычный массив `T[]` или строка `string`.
*   **[[Стек]] (Stack):** буфер, выделенный через инструкцию `stackalloc`.
*   **Неуправляемую кучу ([[Unmanaged Heap]]):** нативная память, выделенная через указатели (C++ interop).

#### 2. Внутреннее устройство (ByRef Fields)
Начиная с .NET 7/8, `Span<T>` реализован с использованием физических `ref` полей (ссылочных полей внутри структур):
```csharp
public readonly ref struct Span<T>
{
    internal readonly ref T _pointer; // Прямая ссылка на элемент в памяти
    internal readonly int _length;     // Длина сегмента
}
```
Благодаря этой структуре, `Span<T>` хранит **внутренний указатель (interior pointer)** на первый элемент сегмента. GC полностью осведомлен об этих указателях: если `Span` ссылается на середину управляемого массива, GC не удалит этот массив из памяти.

#### 3. Безопасность и ограничения ref struct
Поскольку `Span<T>` является `ref struct`, он подчиняется жестким правилам времени компиляции, гарантирующим, что он живет **только в стеке**:
*   Его нельзя упаковать ([[Boxing Unboxing|boxing]]) в `object`.
*   Он не может быть полем обычного класса или структуры (только другой `ref struct`).
*   Его нельзя использовать внутри `async` методов поперек оператора `await` (так как состояние асинхронного метода переносится в кучу).

> [!INFO] В чем разница с ReadOnlySpan?
> `ReadOnlySpan<T>` — это неизменяемая версия. Только через неё можно работать со строками (`string`), так как строки в .NET являются иммутабельными. Вызов `myString.AsSpan()` возвращает именно `ReadOnlySpan<char>`.

### Пример кода ([[C#]] 12)

```csharp
using System;

public class SpanDemo
{
    public static void Main()
    {
        // 1. Срез строки без аллокаций (через ReadOnlySpan)
        string text = "ID:12345-ACTIVE";
        ReadOnlySpan<char> textSpan = text.AsSpan();
        
        // Срез возвращает новый ReadOnlySpan за O(1), копирования строки не происходит
        ReadOnlySpan<char> idSpan = textSpan.Slice(3, 5); 
        Console.WriteLine($"ID: {idSpan.ToString()}"); // "12345"

        // 2. Выделение памяти в стеке (stackalloc)
        // Идеально для небольших временных буферов (до 1-2 Кб)
        Span<int> stackMemory = stackalloc int[5] { 1, 2, 3, 4, 5 };
        MultiplyByTwo(stackMemory);

        // 3. Работа с обычным массивом
        int[] heapArray = [10, 20, 30, 40, 50]; // C# 12 collection expression
        Span<int> arraySpan = heapArray; // Неявное приведение
        
        var slice = arraySpan.Slice(1, 3); // [20, 30, 40]
        slice[0] = 99; // Изменение в срезе меняет оригинальный массив!
        Console.WriteLine($"heapArray[1]: {heapArray[1]}"); // 99
    }

    private static void MultiplyByTwo(Span<int> data)
    {
        for (int i = 0; i < data.Length; i++)
        {
            data[i] *= 2;
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Время жизни stackalloc
> Никогда не возвращайте `Span<T>`, указывающий на `stackalloc`, из метода наружу. Когда метод завершает работу, его кадр стека очищается. Ссылка внутри возвращенного `Span` станет указывать на "мусор" или память другого метода, что приведет к повреждению стека. Компилятор C# отслеживает такие попытки и блокирует их (Safe Context Rules).

> [!WARNING] Ошибка: Попытка использования в асинхронном коде
> Нельзя использовать `Span<T>` в асинхронных сигнатурах. Если вам нужно передавать непрерывные блоки памяти через `async` методы, используйте типы **`Memory<T>`** и **`ReadOnlyMemory<T>`**. Они не являются `ref struct` и могут безопасно жить в куче.

**Best Practices:**
1.  **Парсинг строк:** Всегда используйте `ReadOnlySpan<char>` и методы класса `MemoryExtensions` для парсинга (например, `int.TryParse(span, out ...)`), чтобы избежать создания промежуточных подстрок в куче.
2.  **Параметры методов:** Делайте ваши методы, обрабатывающие коллекции, принимающими `ReadOnlySpan<T>` или `Span<T>` вместо `T[]` или `List<T>`. Это сделает их универсальными (смогут обрабатывать и массивы, и стек, и нативную память).
3.  **Пределы stackalloc:** Используйте `stackalloc` только для небольших размеров (до нескольких сотен байт). Выделение слишком большого объема в стеке может привести к неуловимому `StackOverflowException`.
4.  **u8 Литералы (C# 11+):** Используйте UTF-8 строковые литералы `"myText"u8` для быстрого получения `ReadOnlySpan<byte>` без перекодирования в рантайме.

> [!TIP] Срез за O(1)
> Метод `Slice` работает за константное время $O(1)$ независимо от размера данных. Он просто вычисляет новое смещение указателя и вычитает длину. Это делает обработку гигабайтных массивов или лог-файлов мгновенной.
