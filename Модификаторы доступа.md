### Суть (The "What")
**Модификаторы доступа** — это ключевые слова, определяющие область видимости (accessibility) типов и их членов. Они управляют тем, какой код имеет право обращаться к конкретному классу, методу или полю.

Основная задача — реализация **[[инкапсуляции]]**. Ограничивая доступ к внутренним деталям реализации, разработчик создает "черный ящик" с понятным публичным интерфейсом, что минимизирует зависимости и упрощает поддержку кода.

### Как это работает (The "How" / Nutshell style)

#### 1. Уровни доступа в [[MOC#/MOC|C#]]
C# предоставляет шесть уровней доступа, которые можно разделить на базовые и комбинированные:

| Модификатор          | Описание области видимости                                           |
| :------------------- | :------------------------------------------------------------------- |
| `public`             | Доступ не ограничен. Тип или член виден отовсюду.                    |
| `internal`           | Доступ только внутри текущей **сборки** (Assembly/DLL).              |
| `protected`          | Доступ внутри своего класса и во всех **производных** классах.       |
| `private`            | Доступ только внутри своего класса или структуры (по умолчанию).     |
| `protected internal` | Сборка **ИЛИ** Производные классы за пределами сборки (объединение). |
| `private protected`  | Сборка **И** Производные классы (пересечение).                       |

#### 2. Низкоуровневая реализация ([[CLR]])
На уровне метаданных сборки каждый член типа помечается флагами видимости. Среда выполнения (CLR) проверяет эти флаги при каждой попытке обращения к члену.
*   **Сборка vs Пространство имен:** Важно помнить, что модификаторы привязаны к физическим сборкам (`.dll`/`.exe`), а не к логическим пространствам имен (`namespace`). Пространства имен никак не влияют на видимость.
*   **Default Access:** По умолчанию члены класса/структуры являются `private`. Сами типы (классы, интерфейсы), объявленные непосредственно в пространстве имен, по умолчанию являются `internal`.

> [!INFO] Нюанс: Ограничение видимости
> Уровень доступа члена не может быть "выше", чем уровень доступа его типа. Например, нельзя объявить `public` метод в `internal` классе — метод фактически останется доступным только внутри сборки.

#### 3. Дружественные сборки (InternalsVisibleTo)
Существует механизм исключения — атрибут `[InternalsVisibleTo("AssemblyName")]`. Он позволяет сделать члены с модификатором `internal` одной сборки видимыми для другой конкретной сборки. Это стандартная практика для написания Unit-тестов.

### Пример кода (C# 12)

```csharp
// Assembly_A.dll
namespace CoreLogic;

public class BaseService
{
    // Видно всем
    public void PublicAction() { }

    // Только в этой DLL
    internal void InternalLogic() { }

    // В этой DLL И в наследниках в других DLL
    protected internal void SharedLogic() { }

    // В этой DLL И только в наследниках
    private protected void LocalSubclassLogic() { }

    // Только внутри этого класса
    private void Secret() { }
}

public class DerivedService : BaseService
{
    public void Test()
    {
        PublicAction();       // OK
        InternalLogic();      // OK
        SharedLogic();        // OK
        LocalSubclassLogic(); // OK
        // Secret();          // Ошибка: private
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Anti-pattern: "Public by Default"
> Распространенная ошибка новичков — делать всё `public`. Это нарушает инкапсуляцию и делает рефакторинг невозможным, так как вы не знаете, кто за пределами вашей сборки использует эти методы. Всегда начинайте с `private` и расширяйте доступ только по необходимости.

> [!WARNING] Нюанс: protected internal vs private protected
> Разработчики часто путают эти два модификатора. 
> - `protected internal` — это "шире", чем просто `internal` или `protected`. 
> - `private protected` — это "уже", так как накладывает два ограничения одновременно.

**Best Practices:**
1.  **Принцип наименьших привилегий:** Тип или член должен иметь минимально возможный уровень доступа.
2.  **Свойства вместо полей:** Поля всегда должны быть `private`. Для доступа используйте свойства: `public string Name { get; private set; }`. Это позволяет менять логику записи, не меняя контракт.
3.  **Интерфейсы для публичного API:** Делайте классы реализаций `internal`, а наружу отдавайте только `public` интерфейсы. Это позволяет полностью менять реализацию без вреда для потребителей.
4.  **Top-level types:** Если класс используется только внутри одной сборки, явно помечайте его как `internal` (хотя это и поведение по умолчанию), чтобы подчеркнуть намерение.

> [!TIP] Нюанс: Reflection
> Помните, что `private` не является гарантией безопасности данных. С помощью механизма **[[Reflection]]** (Рефлексия) технически можно получить доступ и изменить любое приватное поле, если у процесса достаточно системных привилегий. Модификаторы — это архитектурный контракт, а не механизм защиты от взлома.
