### Суть (The "What")
**Восходящее приведение типов (Upcasting)** — это операция неявного преобразования ссылки на производный [[Классы и объекты|класс]] к ссылке на его базовый класс или реализуемый им [[Интерфейсы|интерфейс]]. Она представляет собой движение вверх по иерархии [[Наследование|наследования]] (от частного к общему).

Upcasting решает задачу реализации **[[Полиморфизм|полиморфизма]]**. Он позволяет писать обобщенный код, работающий с базовыми абстракциями, но принимающий любые их конкретные подтипы в качестве аргументов.

### Как это работает (The "How" / Nutshell style)

#### 1. Физическое устройство в памяти
С технической точки зрения восходящее приведение является абсолютно "бесплатной" операцией для среды выполнения ([[CLR]]):
*   **Тот же адрес:** Физический адрес объекта в управляемой [[Куча|куче]] (Heap) не меняется ни на один байт. Указатель в [[Стэк|стеке]] продолжает смотреть на ту же самую область памяти.
*   **Расположение полей:** Память под поля объекта выделяется непрерывным блоком. Поля базового класса в структуре объекта всегда располагаются в самом начале, перед полями производного класса. Поэтому ссылка на начало объекта производного класса физически идентична ссылке на начало базового класса.

#### 2. Компилятор против Рантайма
*   **На этапе компиляции:** Меняется статическая видимость объекта. Компилятор начинает воспринимать ссылку через призму ограничений базового типа, запрещая доступ к специфичным методам и свойствам наследника.
*   **На этапе выполнения (Runtime):** [[Метаданные]] объекта не меняются. Указатель на таблицу методов ([[MethodTable]] Pointer в заголовке объекта) по-прежнему ссылается на фактический производный тип. Именно поэтому вызов [[Виртуальные методы и свойства|виртуальных методов]] (`virtual` / `override`) всегда выполняет переопределенную логику наследника, даже после приведения к базовому типу.

#### 3. Приведение к интерфейсу
Приведение к интерфейсу также является формой восходящего приведения. Внутри CLR это может потребовать обращения к картам интерфейсов ([[Interface Map]] в `MethodTable`), но для разработчика на [[C#]] это выглядит как неявное безопасное преобразование.

### Пример кода (C# 12)

```csharp
using System;

public interface IPrintable
{
    void Print();
}

public class Document(string title) : IPrintable // Primary constructor (C# 12)
{
    public string Title { get; } = title;

    public virtual void Print() => Console.WriteLine($"Document: {Title}");
}

public class Invoice(string title, decimal amount) : Document(title)
{
    public decimal Amount { get; } = amount;

    // Переопределение виртуального метода
    public override void Print() => Console.WriteLine($"Invoice: {Title}, Amount: {Amount:C}");

    // Уникальный метод наследника
    public void Pay() => Console.WriteLine($"Paying {Amount:C}");
}

public class Program
{
    public static void Main()
    {
        Invoice invoice = new("INV-001", 1500m);

        // 1. Upcasting к базовому классу (неявно)
        Document doc = invoice; 

        // 2. Upcasting к интерфейсу (неявно)
        IPrintable printable = invoice; 

        // Ограничение видимости компилятором:
        // doc.Pay(); // ОШИБКА компиляции: метод Pay() недоступен через ссылку типа Document

        // Полиморфный вызов: вызовется метод класса Invoice, несмотря на тип ссылки
        doc.Print();       // Output: Invoice: INV-001, Amount: $1,500.00
        printable.Print(); // Output: Invoice: INV-001, Amount: $1,500.00
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Опасность скрытия членов (Member Shadowing)
> Если в производном классе вы скрыли метод базового класса с помощью ключевого слова `new` вместо переопределения `override`, то после восходящего приведения (Upcasting) полиморфизм сломается. Вызов метода через базовую ссылку приведет к выполнению логики базового класса, а не наследника. Избегайте использования `new` для скрытия методов.

> [!WARNING] Нюанс: Потеря специфичной логики
> Приводя объект к базовому типу, убедитесь, что вам действительно не понадобятся его специфичные члены дальше по коду. Обратное приведение ([[Downcasting]]) требует явных проверок типов и снижает безопасность кода.

**Best Practices:**
1.  **Программируйте на уровне интерфейсов:** Всегда неявно приводите конкретные реализации к их интерфейсам при передаче в другие модули (например, передавайте `List<T>` как `IEnumerable<T>`). Это основа слабой связанности (Loose Coupling).
2.  **Не используйте избыточное приведение:** Код вида `Document doc = (Document)new Invoice(...);` содержит лишнее явное приведение, которое засоряет код. Доверяйте компилятору выполнять восходящее приведение автоматически.
3.  **Сохранение null:** Помните, что приведение пустой ссылки (`null`) к любому базовому типу или интерфейсу сохраняет значение `null` и никогда не генерирует исключений.

> [!TIP] Приведение и Generics
> Восходящее приведение для обобщенных интерфейсов (например, приведение `IEnumerable<Dog>` к `IEnumerable<Animal>`) работает благодаря механизму **ковариантности** (параметр `out T`). Без поддержки ковариантности на уровне интерфейса такое приведение вызвало бы ошибку компиляции.
