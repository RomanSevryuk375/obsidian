### Суть (The "What")
`Queue<T>` — это последовательная обобщенная коллекция, реализующая структуру данных **FIFO (First-In, First-Out — «первым пришел, первым ушел»)**. Элементы добавляются в конец очереди, а извлекаются из её начала.

Очередь решает задачу **последовательной обработки задач или сообщений** в порядке их поступления (например, очереди печати, обработка входящих запросов в веб-сервере, фоновые фоновые задачи). Она обеспечивает строгое соблюдение порядка поступления данных без накладных расходов на ручное управление индексами.

---

### Как это работает (The "How" / Nutshell style)

#### 1. Круговой буфер (Circular Buffer)
Внутри `Queue<T>` реализован не связный список и не простой расширяемый массив, а **циклический (круговой) массив**:
*   **Поля состояния:** Класс хранит массив `T[] _array`, индекс головы `_head`, индекс хвоста `_tail` и текущий размер `_size`.
*   **Индексная арифметика:** При вызове `Enqueue` элемент помещается по индексу `_tail`, после чего индекс увеличивается. Если `_tail` доходит до конца физического массива, он циклически "заворачивается" на индекс `0` (используя остаток от деления: `_tail = (_tail + 1) % _array.Length`), если там есть свободное место, освобожденное предыдущими вызовами `Dequeue`.
*   **Отсутствие сдвигов:** При вызове `Dequeue` элемент просто извлекается по индексу `_head`, а сам индекс `_head` смещается вправо (также циклически). В отличие от `List<T>`, где удаление первого элемента требует сдвига всего массива в памяти ($O(n)$), в `Queue<T>` это просто изменение значения числовой переменной.

#### 2. Сложность операций
*   **Enqueue (Добавление):** Амортизированная **$O(1)$**. Если массив полностью заполнен, происходит перевыделение памяти (размер массива удваивается) и копирование элементов в их логическом порядке (от `_head` до `_tail`).
*   **Dequeue (Извлечение):** Стабильная **$O(1)$**. Физическая память не сжимается, просто сдвигается логический указатель.
*   **Peek (Просмотр без удаления):** Стабильная **$O(1)$**.

### Пример кода ([[C#]] 12)

```csharp
using System;
using System.Collections.Generic;

public class QueueDemo
{
    public static void Main()
    {
        // 1. Инициализация очереди (рекомендуется задавать емкость)
        Queue<string> printQueue = new(capacity: 32);

        // 2. Enqueue - добавление в конец очереди - O(1)
        printQueue.Enqueue("Document_1.pdf");
        printQueue.Enqueue("Document_2.pdf");
        printQueue.Enqueue("Document_3.pdf");

        // 3. Peek - просмотр первого элемента без извлечения - O(1)
        string nextDocument = printQueue.Peek();
        Console.WriteLine($"Следующий в очереди на печать: {nextDocument}"); // Document_1.pdf

        // 4. Dequeue - извлечение из начала очереди - O(1)
        while (printQueue.Count > 0)
        {
            string document = printQueue.Dequeue();
            Console.WriteLine($"Печатается: {document}");
        }

        // 5. Безопасное извлечение (Try-паттерн)
        // Исключает генерацию InvalidOperationException на пустой очереди
        if (printQueue.TryDequeue(out string? result))
        {
            Console.WriteLine($"Извлечено: {result}");
        }
        else
        {
            Console.WriteLine("Очередь пуста.");
        }
    }
}
```

### Ошибки и Best Practices

> [!DANGER] Ошибка: Вызов Dequeue или Peek на пустой очереди
> Если вызвать `Dequeue()` или `Peek()` у пустой очереди (`Count == 0`), рантайм выбросит исключение `InvalidOperationException`. Всегда используйте безопасные альтернативы: `TryDequeue` и `TryPeek` (доступны начиная с .NET Core / .NET Framework 4.7.2).

> [!WARNING] Ошибка: Многопоточный доступ к стандартной очереди
> Класс `Queue<T>` не является потокобезопасным. Если один поток вызывает `Enqueue`, а другой одновременно делает `Dequeue`, состояние индексов `_head` и `_tail` сломается, что приведет к порче данных в памяти или бесконечным циклам. Для многопоточности используйте `ConcurrentQueue<T>`.

**Best Practices:**
1.  **Задавайте начальную емкость:** Если вы знаете, что в среднем в очереди будет находиться около 100 элементов, инициализируйте её как `new Queue<T>(100)`. Это спасет от лишних аллокаций и копирований массивов при динамическом расширении.
2.  **Не используйте [[List]]`<T>` как очередь:** Использование `List<T>` с постоянным вызовом `RemoveAt(0)` для реализации FIFO — крайне непроизводительный подход, превращающий извлечение в операцию $O(n)$ вместо $O(1)$.
3.  **Очистка памяти (TrimExcess):** Если очередь временно разрасталась до огромных размеров (например, 1 000 000 элементов), а затем была полностью очищена, внутренний массив всё еще занимает много памяти в куче. Вызовите метод `TrimExcess()`, чтобы уменьшить физический размер массива до реального количества элементов.
4.  **Асинхронные сценарии:** Если вам нужно реализовать очередь для передачи данных между фоновыми потоками, используйте `System.Collections.Concurrent.BlockingCollection<T>` с `ConcurrentQueue<T>` внутри — это избавит от необходимости писать ручные блокировки.

> [!TIP] Нюанс: Перебор через foreach
> Вы можете перебирать очередь через `foreach`, но это не влияет на саму очередь (элементы не удаляются). Порядок обхода будет строго соответствовать логическому порядку FIFO (от `_head` к `_tail`). Изменение очереди во время обхода приведет к ошибке `InvalidOperationException`.
